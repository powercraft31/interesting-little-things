# SOLFACIL VPP v5.14 — 公式大修 + 遙測補全計劃

> 2026-03-06 Alan + Ashe 設計討論產出

## 1. 背景

v5.13 公式審查發現多處邏輯缺陷：
- `calculateDailySavings` 充電成本假設全為離峰，不分來源（光伏免費 vs 電網付費）
- `calculateOptimizationAlpha` 理論最大定義錯誤（一天一循環假設、不考慮光伏/負載時段/多循環）
- 缺少自給自足率（Self-Sufficiency）
- XuhengAdapter 只解析 4 個電池欄位，遺漏 SoH/電壓/電流/溫度等關鍵數據

## 2. 設計原則

- **Baseline 定義**：沒有光伏也沒有電池時的電費（全量從電網買）
- **最佳 TOU 定義**：後驗已知數據下，DP 計算的全局最優充放電排程
- **起始 SoC**：用實際遙測值（不可假設）
- **SoC 下限**：物理約束，通常 5%-10%，電池不會讓你放到 0
- **自發自用率 vs 自給自足率**：用電網進出口餘量推算，不追蹤電池內部來源

## 3. 公式變更清單

### 3.1 刪除

| `calculateDailySavings` | 充電成本邏輯錯誤，被 baseline - actual 取代 |
| `calculateOptimizationAlpha` | 理論最大定義錯誤，被兩個百分比取代 |

| `getRateForHour` | 費率查詢，正確 |
| `calculateSelfConsumption` | `(pvGen - gridExport) / pvGen × 100`，正確 |

### 3.3 新增

| 函數 | 公式 | 說明 |
|------|------|------|
| `calculateSelfSufficiency` | `(load - gridImport) / load × 100` | 自給自足率：負載中多少比例不靠電網 |
| `calculateBaselineCost` | `Σ load[h] × rate(h)` | 原始金額：沒光伏沒電池的電費 |
| `calculateActualCost` | `Σ gridImport[h] × rate(h)` | 實際金額：實際從電網買的費用 |
| `calculateBestTouCost` | DP 動態規劃（見 §4） | 最佳 TOU 金額：後驗最優排程下的電費 |

### 3.4 前端展示指標

```
實際節省率 = (baseline - actual) / baseline × 100
最佳節省率 = (baseline - bestTou) / baseline × 100
自發自用率 = (pvGen - gridExport) / pvGen × 100
自給自足率 = (load - gridImport) / load × 100
```

## 4. DP 最佳 TOU 算法

### 4.1 問題定義

```
minimize: Σ(h=0..23) max(0, battery_delta[h] - net[h]) × rate[h]

where: net[h] = pv[h] - load[h]

subject to:
  soc[0] = soc_initial              （實際遙測值，鎖死）
  soc[h+1] = soc[h] + battery_delta[h]
  soc_min ≤ soc[h] ≤ capacity       （SoC 下限約束，如 10%）
  -max_power ≤ battery_delta[h] ≤ max_power
```

### 4.2 算法

- 動態規劃（Dynamic Programming）
- 狀態：(hour, soc)，SoC 離散化步長 0.1 kWh
- 10 kWh 電池 → 100 個 SoC 級別 × 24 小時 = 2,400 個狀態
- 計算量極小，毫秒級完成
- 無外部依賴，純 TypeScript

### 4.3 驗證

已用兩個場景測試（見 `/tmp/best_tou.js`）：
- 場景 2（3 kWp PV, SoC=4, 下限=1）：最佳 R$0.76，節省率 92.3%
- 場景 3（2 kWp PV, SoC=4, 下限=1）：最佳 R$3.00，節省率 69.7%
- DP 發現手算遺漏的優化：凌晨用初始 SoC 放電供負載（免費電替代付費電）

### 4.4 物理推導 vs DP

兩者結論一致：正確的物理推導 = DP 結果。DP 的優勢是機械化，不會漏算。
