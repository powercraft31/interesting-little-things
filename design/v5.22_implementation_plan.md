## Phase 1: Two-Phase set_reply (3 file changes + 1 index)

**Goal:** Handle `accepted` → `success`/`fail` two-phase protocol without breaking existing single-phase gateways.

### Step 1.1: Modify `command-tracker.ts` — handleSetReply()

**File:** `backend/src/iot-hub/handlers/command-tracker.ts` (140 lines)
**Function:** `handleSetReply()` at line 74

**Current code (lines 88-104):**
- Single UPDATE matching `WHERE result = 'dispatched'`
- Always sets `resolved_at = NOW()`

**Changes:**

1. **Update `SetReplyData` interface** (line 19-23):
   ```typescript
   readonly result?: string; // "accepted" | "success" | "fail"
   ```

2. **Branch on `accepted` vs terminal result** (replace lines 88-104):
   - If `result === "accepted"`:
     - UPDATE `result = 'accepted'` WHERE `result = 'dispatched'`
     - Do NOT set `resolved_at`
     - Do NOT set `error_message`
   - If `result === "success"` or `"fail"`:
     - UPDATE `result = $1, resolved_at = NOW()` WHERE `result IN ('dispatched', 'accepted')`
     - Set `error_message` for fail

3. **Add `pg_notify('command_status', ...)` after each UPDATE** — emit JSON with `{ gatewayId, configName, result }` so SSE can push intermediate status.

4. **Update logging** (lines 124-132):
   - Add `accepted` case: `[CommandTracker] set_reply ACCEPTED (phase 1) for ${gatewayId}`
   - Existing `success`/`fail` logs become `(phase 2)` or `(single-phase)` labels

**Backward compatibility:** If a v1.5 gateway sends only `success`/`fail` (no `accepted`), the `WHERE result IN ('dispatched', 'accepted')` still matches `dispatched`. Works unchanged.

### Step 1.2: Modify `command-dispatcher.ts` — runTimeoutCheck()

**File:** `backend/src/dr-dispatcher/services/command-dispatcher.ts` (178 lines)
**Function:** `runTimeoutCheck()` at line 163

**Current code (lines 165-171):**

**Change:** Replace `result = 'dispatched'` with `result IN ('dispatched', 'accepted')`:

## Phase 2: Backfill Infrastructure (new table + heartbeat detection + requester service)

**Goal:** Detect reconnections and queue backfill requests. Does not yet process responses.

### Step 2.1: Create migration — backfill_requests table

**File:** `backend/migrations/migration_v5.22.sql` (append to file from Step 1.4)

```sql
-- Backfill request queue
CREATE TABLE IF NOT EXISTS backfill_requests (
  id                  BIGSERIAL PRIMARY KEY,
  gateway_id          VARCHAR NOT NULL REFERENCES gateways(gateway_id),
  gap_start           TIMESTAMPTZ NOT NULL,
  gap_end             TIMESTAMPTZ NOT NULL,
  current_chunk_start TIMESTAMPTZ,
  last_chunk_sent_at  TIMESTAMPTZ,
  status              VARCHAR(20) NOT NULL DEFAULT 'pending',
  created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  completed_at        TIMESTAMPTZ,
  CONSTRAINT chk_backfill_status
    CHECK (status IN ('pending', 'in_progress', 'completed', 'failed'))
);

CREATE INDEX idx_backfill_active
ON backfill_requests (created_at ASC)
WHERE status IN ('pending', 'in_progress');
```

### Step 2.2: Modify `heartbeat-handler.ts` — reconnect detection

**File:** `backend/src/iot-hub/handlers/heartbeat-handler.ts` (36 lines)
**Function:** `handleHeartbeat()` at line 11

**Current code (lines 26-33):** Blind UPDATE.

**Changes:**

1. **Before the UPDATE**, add a SELECT to read previous state:
   ```typescript
   const prev = await pool.query(
     `SELECT last_seen_at, status FROM gateways WHERE gateway_id = $1`,
     [gatewayId]
   );
   ```

2. **Calculate gap** between new timestamp and old `last_seen_at`:
   ```typescript
   const oldLastSeen = prev.rows[0]?.last_seen_at;
   const oldStatus = prev.rows[0]?.status;
   const newTimestamp = new Date(deviceTimestamp);
   const gapMs = oldLastSeen ? newTimestamp.getTime() - new Date(oldLastSeen).getTime() : 0;
   const RECONNECT_THRESHOLD_MS = 120_000; // 2 minutes
   ```

3. **If gap detected**, insert backfill request:
   ```typescript
   if (gapMs > RECONNECT_THRESHOLD_MS && oldStatus !== 'online') {
     await pool.query(
       `INSERT INTO backfill_requests (gateway_id, gap_start, gap_end)
        VALUES ($1, $2, $3)`,
       [gatewayId, oldLastSeen, newTimestamp]
     );
     console.log(
       `[HeartbeatHandler] Reconnect: ${gatewayId}, gap=${Math.round(gapMs/60000)}min, backfill queued`
     );
   }
   ```

4. **Existing UPDATE** (lines 26-33) remains unchanged.

**Performance note:** One extra SELECT per heartbeat (every ~30s per gateway). With <100 gateways, negligible.

### Step 2.3: Create `backfill-requester.ts`

**File:** `backend/src/iot-hub/services/backfill-requester.ts` (NEW)

Structure mirrors `CommandPublisher` (command-publisher.ts, 147 lines):

```
export class BackfillRequester {
  private timer: NodeJS.Timeout | null = null;

  constructor(
    private readonly pool: Pool,
    private readonly connectionManager: GatewayConnectionManager,
  ) {}

  start(): void — setInterval(poll, 10_000)
  stop(): void — clearInterval

  private async poll(): Promise<void>
    — SELECT ... FROM backfill_requests WHERE status IN ('pending','in_progress')
      ORDER BY created_at ASC LIMIT 5 FOR UPDATE SKIP LOCKED
    — For each: processRequest()

  private async processRequest(client, req): Promise<void>
    — pending + delay not elapsed → skip
    — pending + delay elapsed → publish first chunk, status='in_progress'
    — in_progress + cooldown not elapsed → skip
    — in_progress + cooldown elapsed + more chunks → publish next chunk
    — in_progress + all chunks done → status='completed'

  private buildGetMissedPayload(gatewayId, startMs, endMs): string
    — Build SolfacilMessage JSON
}
```

**Constants:**
- `DELAY_AFTER_RECONNECT_MS = 30_000` (30s)
- `COOLDOWN_BETWEEN_CHUNKS_MS = 20_000` (20s)
- `CHUNK_DURATION_MS = 30 * 60 * 1000` (30 min)

```
import { parseTelemetryPayload } from '../services/fragment-assembler';
import { DeviceAssetCache } from '../services/device-asset-cache';

export async function handleMissedData(
  pool: Pool,
  gatewayId: string,
  clientId: string,
  payload: SolfacilMessage,
): Promise<void> {
  const data = payload.data;

  // Empty data = gateway had no records for this chunk
  if (!data || Object.keys(data).length === 0) {
    console.log(`[MissedDataHandler] Empty backfill response from ${gatewayId}`);
    return;
  }

  // Parse using shared function (no fragment accumulation needed)
  const recordedAt = new Date(parseInt(payload.timeStamp, 10));
  const parsed = parseTelemetryPayload(clientId, recordedAt, data);
  if (!parsed) return;

  // Resolve asset
  const cache = ... // use same DeviceAssetCache pattern
  const assetId = await cache.resolve(parsed.deviceSn);
  if (!assetId) {
    console.warn(`[MissedDataHandler] Unknown device: ${parsed.deviceSn}`);
    return;
  }

  // INSERT with dedup — ON CONFLICT DO NOTHING
  await pool.query(
    `INSERT INTO telemetry_history (asset_id, recorded_at, battery_soc, ...)
     VALUES ($1, $2, $3, ...)
     ON CONFLICT (asset_id, recorded_at) DO NOTHING`,
    [assetId, parsed.recordedAt, ...]
  );

  // NOTE: Do NOT update device_state (historical data, not current)
  // NOTE: Do NOT fire pg_notify (avoid false SSE updates for old data)
}
```

## Risk Assessment

| Risk | Severity | Phase | Mitigation |
|------|----------|-------|------------|
| Topic routing order bug (`/data` catches `/data/missed`) | HIGH | 3 | Check `/data/missed` before `/data` in routeMessage(); add unit test |
| UNIQUE INDEX on large partitioned table blocks writes | MEDIUM | 3 | Use CONCURRENTLY; run during low-traffic; verify partition strategy |
| HeartbeatHandler extra SELECT adds latency | LOW | 2 | One lightweight SELECT per 30s per gateway; <100 gateways = negligible |
| BackfillRequester publishes to offline gateway | LOW | 2 | Check `isGatewayConnected()` before publish; mark request as `failed` if gateway goes offline mid-backfill |
| Backward compat: v1.5 gateway sends only success/fail | LOW | 1 | `WHERE result IN ('dispatched', 'accepted')` matches both flows |
| Fragment accumulation for backfill data | MEDIUM | 3 | Backfill messages are complete (not fragmented per v1.7 §5), so MissedDataHandler does NOT use FragmentAssembler accumulation — direct parse + INSERT |
| Backfill data triggers device_state update | MEDIUM | 3 | MissedDataHandler explicitly skips updateDeviceState() and pg_notify() |
