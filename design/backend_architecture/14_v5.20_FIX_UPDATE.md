## 1. New Endpoint: `GET /api/gateways/{gatewayId}/detail`

### Purpose

Replace device-level Layer 3 (`GET /api/devices/{assetId}`) with a gateway-level aggregated view. One call returns everything Layer 3 needs: merged device state, telemetry extras, config, and schedule.

### Response Structure

```typescript
{
  success: true,
  data: {
    gateway: {
      gatewayId: string,           // e.g. "WKRD24070202100141I"
      name: string,
      status: "online" | "offline",
      lastSeenAt: string | null,   // ISO datetime
      contractedDemandKw: number | null,
      emsHealth: {                 // from gateways.ems_health JSONB
        cpuTemp: string | null,
        cpuUsage: string | null,
        memoryUsage: string | null,
        diskUsage: string | null,
        wifiSignalStrength: string | null,
        systemRuntime: string | null,
        simStatus: string | null,
        emsTemp: string | null
      } | null
    },
    // Aggregated state — merged from all sub-devices using priority logic (§2)
    state: {
      batterySoc: number | null,
      batSoh: number | null,
      batteryVoltage: number | null,
      batteryCurrent: number | null,
      batteryTemperature: number | null,
      batteryPower: number | null,
      pvPower: number | null,
      gridPowerKw: number | null,      // ← Fix #2: Inverter > Smart Meter
      loadPower: number | null,
      flloadPower: number | null,
      inverterTemp: number | null,
      maxChargeCurrent: number | null,
      maxDischargeCurrent: number | null,
      isOnline: boolean,
      updatedAt: string | null
    },
    telemetryExtra: {
      gridVoltageR: number | null,
      gridCurrentR: number | null,
      gridPf: number | null,
      totalBuyKwh: number | null,
      totalSellKwh: number | null
    },
    config: {
      socMin: number,
      socMax: number,
      maxChargeRateKw: number,
      maxDischargeRateKw: number,
      gridImportLimitKw: number | null,
      defaults: { socMin: number, socMax: number, source: string }
    },
    // Sub-devices list (for device list in Layer 3 header)
    devices: Array<{
      assetId: string,
      name: string,
      assetType: string,    // INVERTER_BATTERY, SMART_METER, etc.
      brand: string | null,
      model: string | null,
      serialNumber: string | null,
      isOnline: boolean
    }>,
    schedule: {
      syncStatus: "synced" | "pending" | "unknown",
      lastAckAt: string | null,
      slots: Array<{ startHour: number, endHour: number, mode: string }>

    _tenant: { orgId: string, role: string }

```

### SQL Queries (4 parallel queries)

**Q1: Gateway info + all devices with state**
```sql
SELECT
  g.gateway_id, g.name, g.status, g.last_seen_at,
  g.contracted_demand_kw, g.ems_health,
  a.asset_id, a.name AS device_name, a.asset_type,
  a.brand, a.model, a.serial_number,
  a.capacidade_kw, a.capacity_kwh, a.operation_mode, a.allow_export,
  ds.battery_soc, ds.bat_soh, ds.battery_voltage, ds.battery_current,
  ds.battery_temperature, ds.battery_power, ds.pv_power,
  ds.grid_power_kw, ds.load_power, ds.flload_power,
  ds.inverter_temp, ds.max_charge_current, ds.max_discharge_current,
  ds.is_online, ds.updated_at
FROM gateways g
LEFT JOIN assets a ON a.gateway_id = g.gateway_id AND a.is_active = true
LEFT JOIN device_state ds ON ds.asset_id = a.asset_id
WHERE g.gateway_id = $1
ORDER BY a.asset_type, a.name
```

**Q2: Latest telemetry extras (from inverter, with smart meter fallback)**
```sql
SELECT th.telemetry_json
FROM telemetry_history th
JOIN assets a ON a.asset_id = th.asset_id
WHERE a.gateway_id = $1 AND a.is_active = true
ORDER BY
  CASE a.asset_type WHEN 'INVERTER_BATTERY' THEN 1 ELSE 2 END,
  th.recorded_at DESC
LIMIT 1
```

**Q3: VPP strategy defaults (same as get-device-detail.ts Q4)**
```sql
SELECT min_soc, max_soc
FROM vpp_strategies
WHERE org_id = $1 AND is_active = true
LIMIT 1
```

**Q4: Schedule from device_command_logs**
```sql
-- Latest 'set' command for this gateway's battery_schedule
SELECT id, payload_json, result, resolved_at, created_at
FROM device_command_logs
WHERE gateway_id = $1
  AND command_type = 'set'
  AND config_name = 'battery_schedule'
ORDER BY created_at DESC
LIMIT 1
```

### Aggregation Logic (in handler code)

```typescript
// Group Q1 rows by asset_type
const inverters = rows.filter(r => r.asset_type === 'INVERTER_BATTERY');
const meters = rows.filter(r => r.asset_type === 'SMART_METER');

// Primary inverter (first online inverter, or first inverter)
const primary = inverters.find(r => r.is_online) || inverters[0] || null;

// Build aggregated state from primary inverter
const state = primary ? {
  batterySoc: toNum(primary.battery_soc),
  batteryPower: toNum(primary.battery_power),
  pvPower: toNum(primary.pv_power),
  gridPowerKw: toNum(primary.grid_power_kw),  // ← from inverter first
  loadPower: toNum(primary.load_power),
  // ... all other fields from primary inverter
} : emptyState();

// Fix #2 fallback: if inverter grid_power_kw is null, try smart meter
if (state.gridPowerKw == null && meters.length > 0) {
  const meter = meters.find(r => r.is_online) || meters[0];
  state.gridPowerKw = toNum(meter.grid_power_kw);
}
```

## 2. Fix #2: Grid Data Priority Logic — Inverter > Smart Meter

### Decision

Implement in the **BFF layer** (new `get-gateway-detail.ts`), NOT in `fragment-assembler.ts`.

**Rationale:**
- `fragment-assembler.ts` writes per-device rows to `device_state`. Each device_state row is correct for its own device.
- The merge is a *presentation concern* — when showing a gateway-level view, we pick the best grid data source.
- Keeping it in BFF means device_state remains a clean per-device table.

### Fallback Chain

```
1. Primary inverter's device_state.grid_power_kw  (INVERTER_BATTERY)
2. First online smart meter's device_state.grid_power_kw  (SMART_METER)
3. null  (no grid data available)
```

Same chain applies to:
- `grid_power_kw`
- `telemetry_extra.grid.*` fields (voltage, current, frequency, power factor)
- `telemetry_extra.totalBuyKwh`, `totalSellKwh`

### Implementation Location

In `get-gateway-detail.ts` handler, after Q1 results are grouped:

```typescript
function mergeGridData(
  inverterState: DeviceStateRow | null,
  meterState: DeviceStateRow | null,
): GridFields {
  // Prefer inverter grid data; fallback to meter
  const src = (inverterState?.grid_power_kw != null)
    ? inverterState
    : meterState;

  return {
    gridPowerKw: toNum(src?.grid_power_kw),
    // telemetryExtra grid fields use same source preference
  };
}
```

### Impact on `get-gateway-energy.ts`

The energy endpoint already queries `telemetry_history` for a single asset. With gateway-level Layer 3, it should aggregate across all devices under the gateway. However, this is a **separate concern** and the current per-asset energy query remains valid as long as the frontend passes the correct (inverter) assetId.

**No changes to `get-gateway-energy.ts` in this fix phase.**

## 3. Schedule API Migration: Device-Level → Gateway-Level

### Current State

| Endpoint | Handler | DB Write |
|----------|---------|----------|
| `GET /api/devices/:assetId/schedule` | `get-device-schedule.ts` | Reads `device_command_logs` by gateway_id (already gateway-level internally) |
| `PUT /api/devices/:assetId/schedule` | `put-device-schedule.ts` | Writes `device_command_logs` with gateway_id from asset lookup |

The DB is already gateway-level (`device_command_logs.gateway_id`), but the API path uses `assetId` and does a lookup to find the gateway.

### New Endpoints

| Old | New | Notes |
|-----|-----|-------|
| `GET /api/devices/:assetId/schedule` | `GET /api/gateways/:gatewayId/schedule` | Direct gateway_id, no asset lookup |
| `PUT /api/devices/:assetId/schedule` | `PUT /api/gateways/:gatewayId/schedule` | Direct gateway_id, no asset lookup |

### Implementation: Modify Existing Handlers

**Option chosen:** Rename/modify existing handlers rather than create new files.

#### `get-device-schedule.ts` → `get-gateway-schedule.ts`

Changes:
1. Extract `gatewayId` from `event.pathParameters.gatewayId` (was `assetId`)
2. Remove asset→gateway lookup query
3. Query `device_command_logs` directly with `gateway_id = $1`

```sql
-- Get latest schedule command and its status
SELECT id, payload_json, result, resolved_at, created_at
FROM device_command_logs
WHERE gateway_id = $1
  AND command_type = 'set'
  AND config_name = 'battery_schedule'
ORDER BY created_at DESC
LIMIT 1
```

Response shape unchanged:
```typescript
{
  syncStatus: result === 'pending' ? 'pending'
            : result === 'success' ? 'synced' : 'unknown',
  lastAckAt: resolved_at,
  slots: payload_json.slots || []
}
```

#### `put-device-schedule.ts` → `put-gateway-schedule.ts`

Changes:
1. Extract `gatewayId` from path (was `assetId`)
2. Remove the asset lookup query (L93-103 in current file)
3. Validate gateway exists: `SELECT 1 FROM gateways WHERE gateway_id = $1`
4. INSERT unchanged (already uses `gateway_id`):

```sql
INSERT INTO device_command_logs
  (gateway_id, command_type, config_name, payload_json, result)
VALUES ($1, 'set', 'battery_schedule', $2, 'pending')
RETURNING id
```

Note: `result` column uses `'pending'` (matching existing index `WHERE result = 'pending'`), not `'pending_dispatch'` which doesn't match the CHECK/index.

### Schedule JSON in `payload_json`

No new tables. The schedule data lives in `device_command_logs.payload_json` JSONB:

```json
{
  "slots": [
    { "startHour": 0, "endHour": 8, "mode": "self_consumption" },
    { "startHour": 8, "endHour": 17, "mode": "peak_valley_arbitrage" },
    { "startHour": 17, "endHour": 24, "mode": "peak_shaving" }
  ]
}
```

### M3 Dispatcher Integration

The existing M3 dispatcher polls `device_command_logs WHERE result = 'pending'`. Schedule commands written with `result = 'pending'` are automatically picked up — **no M3 changes needed**.

### Route Registration

**`local-server.ts` additions:**
```typescript
app.get("/api/gateways/:gatewayId/schedule", wrapHandler(getGatewaySchedule.handler, "gatewayId"));
app.put("/api/gateways/:gatewayId/schedule", wrapHandler(putGatewaySchedule.handler, "gatewayId"));
```
