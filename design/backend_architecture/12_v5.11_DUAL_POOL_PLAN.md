## Section 4: Test Environment Repair

### 4.0 Root Cause Analysis

20 項測試失敗的根本原因有兩類：

**Type A: Cron Job Tests（使用 `getPool()` → solfacil_app，直接 `pool.query()` 未設定 org_id）**

受影響的測試文件：

| 測試文件 | 測試的功能 | 失敗原因 |
|---------|-----------|---------|
| `test/optimization-engine/schedule-generator.test.ts` | `runScheduleGenerator(pool)` | pool.query SELECT FROM assets (RLS) → 空結果 |
| `test/dr-dispatcher/command-dispatcher.test.ts` | `runCommandDispatcher(pool)` | pool.query SELECT FROM trade_schedules (RLS) → 空結果 |
| `test/dr-dispatcher/timeout-checker.test.ts` | `runTimeoutChecker(pool)` | pool.query SELECT FROM dispatch_commands (RLS) → 空結果 |
| `test/dr-dispatcher/collect-response.test.ts` | `createAckHandler(pool)` | pool.query SELECT FROM dispatch_commands (RLS) → 空結果 |

**Type B: BFF Tests（使用 `queryWithOrg()` 但 SOLFACIL_ADMIN 的 org_id 設定可能有問題）**

| 測試文件 | 測試的功能 | 可能失敗原因 |
|---------|-----------|------------|
| `test/bff/get-dashboard.test.ts` | `handler(event)` | SOLFACIL_ADMIN orgId="SOLFACIL" → SET org_id='SOLFACIL' → 但 assets.org_id 是 'ORG_ENERGIA_001'/'ORG_SOLARBR_002' → 不匹配 → 空結果 |
| `test/bff/get-assets.test.ts` | `handler(event)` | 同上 |

**分析 BFF 測試的 SOLFACIL_ADMIN 行為：**

查看 `get-dashboard.test.ts` 的斷言：
- `SOLFACIL_ADMIN` 預期看到 **4 台設備**
- `queryWithOrg()` 當 `orgId = "SOLFACIL"` 時設定 `app.current_org_id = 'SOLFACIL'`
- 但 assets 表中沒有 `org_id = 'SOLFACIL'` 的記錄（只有 `ORG_ENERGIA_001` 和 `ORG_SOLARBR_002`）

**可能的 SOLFACIL_ADMIN 特殊處理：**
- BFF handler 中可能有 `if (role === 'SOLFACIL_ADMIN') { orgId = null }` 的邏輯
- `queryWithOrg(sql, params, null)` → 不設定 `app.current_org_id` → RLS 返回空結果（v5.10 fail-closed）

**這正是 v5.10 雙角色設計的問題：SOLFACIL_ADMIN 原本依賴 "org_id IS NULL = bypass"（v5.9 admin bypass），
但 v5.10 移除了 admin bypass，改為 fail-closed → SOLFACIL_ADMIN 也看不到任何數據。**

| 項目 | 內容 |

**分析 BFF handler 的 SOLFACIL_ADMIN 處理邏輯：**

需要讀取 `src/bff/handlers/get-dashboard.ts` 和 `src/bff/handlers/get-assets.ts`，
確認 SOLFACIL_ADMIN 是否設定 `orgId = null` 傳入 `queryWithOrg()`。

**兩種修復方案：**

| 方案 | 描述 | 影響範圍 |
|------|------|---------|
| **A: BFF 層 Service Pool (不推薦)** | SOLFACIL_ADMIN 使用 service pool 查詢 | BFF 層引入 service pool 依賴，違反設計原則 |
| **B: queryWithOrg 對 orgId=null 使用 Service Pool (推薦)** | `queryWithOrg()` 當 orgId 為 null 時使用 service pool，模擬「管理員看全部數據」 | 修改 `src/shared/db.ts` 的 `queryWithOrg()` |

**方案 B 設計：**

```typescript
export async function queryWithOrg<T extends Record<string, unknown>>(
  sql: string,
  params: unknown[],
  orgId: string | null,
): Promise<{ rows: T[] }> {
  // v5.11: SOLFACIL_ADMIN (orgId = null) uses service pool to bypass RLS
  const pool = orgId ? getAppPool() : getServicePool();
  const client: PoolClient = await pool.connect();
  try {
    await client.query("BEGIN");
    if (orgId) {
      await client.query(`SELECT set_config('app.current_org_id', $1, true)`, [orgId]);
    }
    // When orgId is null (admin), service pool's BYPASSRLS handles cross-tenant access
    const result = await client.query(sql, params);
    await client.query("COMMIT");
    return { rows: result.rows as T[] };
  } catch (err) {
    await client.query("ROLLBACK");
    throw err;
  } finally {
    client.release();
  }
}
```

**安全分析：**
- `orgId = null` 只在 `verifyTenantToken()` 返回 `SOLFACIL_ADMIN` 角色時才發生
- BFF handler 先驗證 JWT → 確認是 SOLFACIL_ADMIN → 才傳 `orgId = null`
- Service pool 的使用受限於已認證的管理員請求

### Task 4.4: Test Setup/Teardown Pattern Documentation

**Cron Job Test Pattern (v5.11):**

```typescript
import { getServicePool, closeAllPools } from "../../src/shared/db";
import { Pool } from "pg";

describe("cron-job-name", () => {
  let pool: Pool;

  beforeAll(() => {
    pool = getServicePool();  // BYPASSRLS — cron job tests need cross-tenant access
  });

  beforeEach(async () => {
    // Insert test data using service pool (no RLS restriction)
    await pool.query(`INSERT INTO ...`);
  });

  afterEach(async () => {
    // Cleanup test data
    await pool.query(`DELETE FROM ...`);
  });

  afterAll(async () => {
    await closeAllPools();  // Close BOTH pools (app + service)
  });

  it("should ...", async () => {
    await runCronJobFunction(pool);
    // Assert...
  });
});
```
