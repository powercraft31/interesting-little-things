> **版本**: v5.10-r2 | **日期**: 2026-03-05
> **來源設計文件**: 00/04/05/08/09/10 v5.10 系列設計文件
> **執行原則**: 每個 Phase 結束後執行測試檢查點。Phase 之間有順序依賴，不可跳步。
>
> **v5.10-r2 重大修正**:
> 1. 以雙角色 DB 架構 (`solfacil_app` + `solfacil_service BYPASSRLS`) 取代危險的 Admin Bypass RLS 策略

> 3. BFF 保留 HTTP 適配器 `extractTenantContext` + `apiError` (在 `auth.ts`)
> 4. 新增 `idx_dispatch_commands_status_org` 複合索引（dashboard KPI 效能）

## Phase 0: DB Migration + Dual-Role Setup（Database Schema v5.10）

### Task 0.1 — 修復 feature_flags UNIQUE 約束

| 項目 | 說明 |
|------|------|
| **目標文件** | `db/migrations/005_v5.10_dual_role_rls.sql`（新建） |
| **修改邏輯** | `ALTER TABLE feature_flags DROP CONSTRAINT IF EXISTS feature_flags_flag_name_coalesce_key;` 然後 `CREATE UNIQUE INDEX IF NOT EXISTS uq_feature_flags_name_org ON feature_flags (flag_name, COALESCE(org_id, ''));` |
| **驗證** | `\d feature_flags` 確認 UNIQUE INDEX 存在 |

### Task 0.2 — dispatch_commands 啟用 RLS（純租戶隔離）

| 項目 | 說明 |
|------|------|
| **目標文件** | `db/migrations/005_v5.10_dual_role_rls.sql`（同上，追加） |
| **修改邏輯** | `ALTER TABLE dispatch_commands ENABLE ROW LEVEL SECURITY;` + 建立 `rls_dispatch_commands_tenant` policy（`org_id = current_setting('app.current_org_id')`）。**不建立 admin bypass policy。** |
| **驗證** | 設定 `app.current_org_id = 'ORG_ENERGIA_001'` 後，只能查到該 org 的 dispatch_commands；不設定 org_id 時返回 **0 行**（fail-closed） |

### Task 0.3 — 新增 dispatch_commands 複合索引

| 項目 | 說明 |
|------|------|
| **目標文件** | `db/migrations/005_v5.10_dual_role_rls.sql`（同上，追加） |
| **修改邏輯** | `CREATE INDEX IF NOT EXISTS idx_dispatch_commands_status_org ON dispatch_commands (org_id, status, dispatched_at DESC);` |
| **效能理由** | GET /dashboard 每次載入時聚合 `SELECT COUNT(*) FILTER (WHERE status = 'completed') FROM dispatch_commands WHERE dispatched_at >= CURRENT_DATE`。50,000+ 設備場景下，此索引將全表掃描優化為索引範圍掃描。 |
| **驗證** | `\di idx_dispatch_commands_status_org` 確認索引存在 |

### Task 0.4 — 建立 solfacil_service 角色（BYPASSRLS）+ 移除 Admin Bypass Policies

| 項目 | 說明 |
|------|------|
| **目標文件** | `db/migrations/005_v5.10_dual_role_rls.sql`（同上，追加） |
| **修改邏輯** | (1) 建立 `solfacil_service` 角色，帶 `BYPASSRLS` 屬性。(2) `DROP POLICY IF EXISTS rls_*_admin_bypass` 移除所有 11 張表的舊 admin bypass policies。(3) GRANT 適當權限。 |
| **安全理由** | 舊 admin bypass 設計（`org_id IS NULL OR = ''` → god-mode）是 P0 安全漏洞。任何忘記設定 org_id 的連線都會靜默獲得全量存取。雙角色架構使 fail-closed 成為預設行為。 |
| **驗證** | `SELECT rolname, rolbypassrls FROM pg_roles WHERE rolname = 'solfacil_service';` 確認 BYPASSRLS = true |

### 🧪 Test Checkpoint 0

```bash
# 執行 migration
psql -d solfacil_vpp -f db/migrations/005_v5.10_dual_role_rls.sql

# 驗證 feature_flags UNIQUE INDEX
psql -d solfacil_vpp -c "\di uq_feature_flags_name_org"

# 驗證 dispatch_commands 複合索引
psql -d solfacil_vpp -c "\di idx_dispatch_commands_status_org"

# 驗證雙角色
psql -d solfacil_vpp -c "SELECT rolname, rolbypassrls FROM pg_roles WHERE rolname IN ('solfacil_app', 'solfacil_service');"

# 驗證 RLS fail-closed（solfacil_app 不設 org_id → 0 行）
psql -U solfacil_app -d solfacil_vpp -c "SELECT COUNT(*) FROM dispatch_commands;"
# 預期：0（fail-closed）

# 驗證 BYPASSRLS（solfacil_service → 全量）
psql -U solfacil_service -d solfacil_vpp -c "SELECT COUNT(*) FROM dispatch_commands;"
# 預期：全量數據

# 驗證無殘留 admin bypass policies
psql -d solfacil_vpp -c "SELECT policyname FROM pg_policies WHERE policyname LIKE '%admin_bypass%';"
# 預期：0 行
```

## 風險與緩解

| 風險 | 影響 | 緩解措施 |
|------|------|---------|
| Phase 1-2 之間 TypeScript 編譯斷裂 | CI pipeline 會失敗 | Phase 1 和 Phase 2 必須在同一 commit 內完成 |
| M4/M8 handler 需更新調用方式 | `extractTenantContext(event)` → `verifyTenantToken(token)` | Task 2.4 需更新 handler 的 token 提取邏輯 |
| solfacil_service 角色密碼管理 | 密碼洩漏可繞過 RLS | 使用 AWS Secrets Manager 或環境變數注入，不 hardcode |
| 50,000+ dispatch_commands 效能 | Dashboard 載入變慢 | Task 0.3 的複合索引覆蓋查詢模式 |
| 舊環境殘留 admin bypass policies | 安全漏洞持續存在 | Task 0.4 明確 DROP 所有 admin bypass policies |
| BFF test mock 未更新 | de-hardcoding 後測試仍 FAIL | Task 3.1 同步更新 get-dashboard.test.ts 的 mock 設定 |
