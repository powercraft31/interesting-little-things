# SOLFACIL VPP v5.17 — 新前端接線 + 累積 Breaking Changes 修復

> **起草**: Ashe (2026-03-08)
> **狀態**: 待 Alan + Gemini 審批
> **範圍**: 僅限 `frontend-v2/` 目錄，BFF 不動

---

## 前因後果

### 背景

項目有兩個活躍前端：
1. **中台**（`admin.html` + `js/admin.js`）— M8 Admin Control Plane，走 `/api/admin/*`（M8 handler），與 BFF（P1-P7）完全割裂。目前正常運作，計劃最後統一修正
2. **新前端 v2**（`frontend-v2/`）— Admin Portal v2，6 頁 SPA（P1-P6），**從未成功接過 BFF**

> 註：舊前端（`index.html` + `js/modules/`）已廢棄，不在本文件討論範圍內

### 問題

v5.12（commit 831a43f）為新前端建了完整的 `data-source.js` adapter（15 個 API 路徑），並在 `local-server.ts` 註冊了對應路由。**但新前端的 6 個頁面 JS 從未呼叫 DataSource** — 全部直接讀 `mock-data.js` 的全域常量。

從 v5.12 到 v5.16，後端做了四輪 BFF 數據結構改動：
- **v5.13**: Dashboard 改用 Tarifa Branca client_savings；Scorecard 的 Savings Alpha 改為 DB 計算；gatewayUptime 去硬編碼
- **v5.14**: Scorecard 大改 — 移除 Savings Alpha，新增 Actual Savings / Optimization Efficiency / Self-Sufficiency
- **v5.15**: performance/savings 移除 `alpha` 欄位；SC/TOU 改為真實歸因（不再用 0.55/0.30/0.15 假比例）；PS 暫設 null
- **v5.16**: performance/savings 的 PS 從 null 改為真實數字

**每個版本的設計文件都標記了「Frontend Impact / Action Required」，但 Claude Code 只改了 BFF，沒碰前端。** 唯一的前端改動是 v5.13 把 `USE_LIVE_API` 從 false 翻成 true — 但沒有任何頁面呼叫它，等於無效。

### 根因：雙層斷裂（Ashe + Gemini 聯合診斷）

經 Ashe 現象觀察 + Gemini 深度代碼掃描，確認問題是**兩層同時斷裂**：

#### L1: DataSource Adapter 層 — 配置孤島 + 靜默降級（Gemini 診斷）

`data-source.js` 的 `USE_LIVE_API` 已設為 `true`（v5.13 翻的），DataSource **確實嘗試打 API**。但：

1. **配置孤島（Configuration Isolation）**：舊前端和中台靠 `js/config.js` 統一管理 API 端點（指向 `https://152.42.235.155:8443/api`），但 `frontend-v2/index.html` **沒有引入任何全域設定檔**。導致 `API_BASE` 無法取得 `window.SOLFACIL_API_BASE`，退化為開發用的本機位置：
   ```js
   var API_BASE = window.SOLFACIL_API_BASE || "http://localhost:3000";
   ```

2. **靜默降級（Silent Fallback）**：API 打向 `localhost:3000` 必然失敗，但 `withFallback()` 攔截器捕捉到錯誤後，只發一個 `console.warn`，**自動退回 mock-data.js 的假數據**。表面上系統正常運作，實際上真實 API 請求全部陣亡。

#### L2: 頁面調用層 — 零次呼叫 DataSource（Ashe 診斷）

即使修好 L1 讓 DataSource adapter 能連到 BFF，**6 個頁面 JS 完全沒有呼叫 `DataSource`**：

```bash
$ grep -rn "DataSource" frontend-v2/js/p*.js frontend-v2/js/app.js
# (零結果)
```

所有頁面直接讀 mock-data.js 的全域常量：
- P1 讀 `FLEET` / `DEVICE_TYPES` / `INTEGRADORES` / `OFFLINE_EVENTS`
- P2 讀 `DEVICES` / `HOMES`
- P3 讀 `DemoStore.get("homeData")` / `_beforeAfterData` / `_crossHomeSummary`
- P4 讀 `MODE_DISTRIBUTION` / `TARIFA_RATES` / `LAST_DISPATCH`
- P5 讀 `VPP_CAPACITY` / `DR_EVENTS` / `LATENCY_TIERS`
- P6 讀 `SCORECARD` / `SAVINGS_BY_HOME`

#### 雙層修復缺一不可

| 只修 L1 | 只修 L2 | 兩層都修 |
|---------|---------|----------|
| DataSource 連上 BFF，但沒人呼叫它 | 頁面呼叫 DataSource，但 API_BASE 指向 localhost 照打不到 | ✅ DataSource 連到 BFF + 頁面呼叫 DataSource |

### 結果

新前端 v2 目前的狀態：
- **L1**: DataSource adapter 嘗試打 API 但打向 localhost:3000，靜默降級回 mock
- **L2**: 6 頁 JS 根本不呼叫 DataSource，直讀 mock 全域常量
- Mock 數據結構與 BFF 回傳結構有 **5 處 breaking mismatch**（v5.13-v5.16 累積）
- 306 個測試全是後端邏輯驗證，前端從未被涵蓋

## v5.17 範圍：雙層修復 + 累積 Breaking Changes

### 第零類：L1 修復 — 破除配置孤島 + 消滅靜默降級

**L1-1. 注入全域配置，統一 API_BASE**

在 `frontend-v2/index.html` 中，於 `data-source.js` 載入前注入配置：

方案 A（引入根目錄 config）：
```html
<script src="../js/config.js"></script>
```
然後修改 `data-source.js`：
```js
var API_BASE = (typeof CONFIG !== "undefined" && CONFIG.BFF_API_URL)
  ? CONFIG.BFF_API_URL
  : (window.SOLFACIL_API_BASE || "http://localhost:3000");
```

**D6 裁定：方案 A — 引入根目錄 config.js，Single Source of Truth。**

**L1-2. 消滅危險的靜默降級（D7 裁定：嚴格模式）**

修改 `data-source.js` 的 `withFallback()` 邏輯：
- `USE_LIVE_API = true` 時：API 失敗 → **throw Error**，**禁止 fallback 回 mock**
- `USE_LIVE_API = false` 時：直接讀 mock，不打 API

```js
function withFallback(apiCall, mockData) {
  if (!USE_LIVE_API) return Promise.resolve(mockData);
  return apiCall();
  // v5.17: NO .catch() — error propagates to caller
  // Page init() must handle errors with Error Boundary UI
}
```

頁面 `init()` 需加入 Error Boundary：API 失敗時顯示系統異常 UI，而非白屏。

---

### 第一類：L2 修復 — 全部 6 頁改為呼叫 DataSource

**A1. 改為 async init，呼叫 DataSource**

目前所有頁面的 `init()` 是同步函數，直接讀 mock 全域常量。需要改為：
```
init() → async init()
  1. 呼叫 DataSource.xxx() 取得數據
  2. 用回傳數據渲染，不再讀全域常量
  3. 錯誤處理依 L1-2 決策（throw 或 banner）
```

**A2. app.js 的 initMockData() 仍保留**

mock-data.js 保留兩個用途：
1. `USE_LIVE_API = false` 時的開發/離線模式
2. P3 的 acPower/evCharge 等 BFF 尚未實作的欄位 fallback

##### Scorecard Breaking Changes（v5.13 → v5.14 累積）

Mock 的 `SCORECARD.optimization` 有 4 項：
```
Savings Alpha (74.2%) / Self-Consumption (96.8%) / PV Forecast MAPE (22.1%) / Load Forecast Adapt (92%)
```

BFF v5.14 的 `optimization` 有 **6 項**：
```
Actual Savings / Optimization Efficiency / Self-Consumption / Self-Sufficiency / PV Forecast MAPE / Load Forecast Adapt
```

差異：
- ❌ `Savings Alpha` → 被替換為 `Actual Savings` + `Optimization Efficiency`
- ➕ 新增 `Self-Sufficiency`
- ✅ `Self-Consumption` / `PV Forecast MAPE` / `Load Forecast Adapt` 保留

**前端 p6-performance.js 讀 `SCORECARD.hardware` / `SCORECARD.optimization` / `SCORECARD.operations` 三個 key，BFF 回傳相同結構，但 optimization 內容不同。** 由於渲染是動態的（遍歷 metrics 陣列），只要接上 BFF 就會自動顯示新的 6 項，不需要改渲染邏輯。但 **i18n key** 需要新增：

需新增的 i18n key：
- `perf.metric.Actual Savings`
- `perf.metric.Optimization Efficiency`
- `perf.metric.Self-Sufficiency`

##### Savings by Home Breaking Changes（v5.15 + v5.16）

Mock `SAVINGS_BY_HOME`：
```json
{ "home": "Casa Silva", "total": 145.0, "alpha": 74.2, "sc": 85, "tou": 40, "ps": 20 }
```

BFF v5.16 回傳：
```json
{ "home": "Casa Silva", "total": 150.00, "sc": 85.20, "tou": 42.60, "ps": 22.20 }
```

差異：
- ❌ `alpha` 欄位已移除（v5.15）
- 前端 `p6-performance.js` 的 Savings Chart tooltip 仍渲染 `home.alpha`，需移除
- `ps` 從假數字 → null（v5.15）→ 真數字（v5.16），目前 BFF 最終狀態是真數字，**前端不需要 null handling**

**P6 修改量**: 中等。接 DataSource + alpha 移除 + 3 個 i18n key。
