### PR2: FragmentAssembler — 消息累積與合併服務

**範圍**：新增 service，不修改任何現有 handler

**檔案變更**：
- `backend/src/iot-hub/services/fragment-assembler.ts` (新建)
- `backend/src/shared/types/solfacil-protocol.ts` (擴充 types)
- `backend/tests/iot-hub/services/fragment-assembler.test.ts` (新建)

**核心邏輯**：

```
class FragmentAssembler {
  // Per-gateway accumulator map
  private accumulators: Map<string, GatewayFragments>

  // 接收分類後的 fragment，存入 accumulator
  receive(clientId: string, type: FragmentType, data: unknown, recordedAt: Date): void

  // 內部：3s debounce timer 或 core 到達時立即 flush
  private scheduleFlush(clientId: string): void

  // 合併所有 fragment → 寫 DB
  private async mergeAndPersist(clientId: string): Promise<void>
}
```

**設計要點**：
1. 純 accumulator 邏輯，不依賴 TelemetryHandler
2. `receive()` 是同步的（只存 Map + reset timer）
3. `mergeAndPersist()` 是 async（寫 DB）
4. 3s safety debounce，但收到 `core` (MSG#5) 時立即 flush
5. 單元測試覆蓋：正常 5 條合併、MSG#5 缺失只寫 ems_health、亂序到達、重複消息

**TDD 順序**：
1. RED: 測試 5 條消息合併後產出正確 ParsedTelemetry
2. RED: 測試 MSG#5 缺失時只寫 ems_health
3. RED: 測試 dido DO0/DO1 正確轉 boolean
4. RED: 測試雙 meter 正確分 key (meter_single / meter_three)
5. GREEN: 實作 FragmentAssembler
6. REFACTOR

**預計 LOC**：~200 (service) + ~300 (tests)

## 風險評估

| 風險 | 影響 | 緩解 |
|------|------|------|
| Fragment 亂序（MSG#5 先到） | 合併可能遺漏 dido/meter | 3s safety debounce 確保等齊 |
| MSG#5 丟失（broker issue） | 不完整的遙測週期 | 只寫 ems_health，不寫 telemetry_history；log warning |
| 重複 MSG（MQTT QoS 1） | 同一 fragment 被 receive 兩次 | Last-write-wins per key，冪等 |
| Timer 累積（長時間運行） | 記憶體洩漏 | clearTimeout on each receive + Map.delete on flush |
| 現有 tests 破壞 | 阻塞其他開發 | PR2 純新增、PR3 先 baseline 再改 |
