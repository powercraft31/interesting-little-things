# v5.16 Peak Shaving (PS) — 設計背景與架構決策文件

**給 Gemini 的審查文件 | 作者：Ashe | 日期：2026-03-07**

---

## 一、為什麼做 PS（業務背景）

巴西工商業電力帳單由兩部分組成：

1. **電量費（Energia）**：實際用電 kWh × 分時費率（Tarifa Branca）
   → v5.15 的 SC/TOU attribution 處理的正是這部分

2. **需量費（Demanda）**：月內最高 15min 積分需量（kVA）× 需量費率（R$/kVA）
   → **v5.16 要處理的是這部分**

需量費的特性：
- **月結制**：整個月只看一個最高峰值，超過合約需量（contracted_demand_kw）的部分按倍率計費
- **一峰定全月**：哪怕只有 15 分鐘超過限額，整個月的超額費都算進來
- **可控性高**：只要在那 15 分鐘內把電池放電撐住，就能避免超限

**PS 的商業邏輯：** 電池在負載高峰時放電，把電網看到的功率壓在合約需量以下，避免觸發超額需量費。

## 二、架構定位（平台 vs 閘道器分工）

**這是 v5.16 最重要的邊界決策，已與 Alan 確認：**

| 職責 | 執行者 | 說明 |
|------|--------|------|
| 即時需量控制 | **閘道器（EMS 盒）** | 閘道器保證 15min 窗口內功率絕對不超限，平台不介入即時控制 |
| 下發 PS 指令 | **平台 M3** | 下發 `{ mode: 'peak_shaving', peak_limit_kva: X }` |
| 切換 PS 模式 | **平台 M2** | 決策何時啟用 PS（按時段、帳單週期位置等） |
| 節省歸因計算 | **平台 M4** | 月底計算「避免了多少需量費」 |
| 負載切斷（DO） | **閘道器硬體** | 電池不夠時觸發繼電器切斷非關鍵負載 |

**核心推論：** 因為閘道器保證「實際 kVA 永遠 ≤ contracted_demand_kw」，所以平台無法用「實際 vs 限額」來計算節省——實際值永遠在限額以下。需要反推「如果沒有 PS，峰值會到多高」。

## 三、PS 節省反事實計算法

**核心等式：**

```
反事實峰值（without PS） = 
    actual_grid_kW              ← 實際流入電網功率（telemetry_history）
  + bat_discharge_kW            ← 電池放電撐住的部分（asset_5min_metrics）
  + load_shed_kW_from_DO        ← DO 觸發切斷的負載（do0/do1 state）
```

**邏輯說明：**
- 電池放電了 X kW → 代表如果沒電池，電網就要多出 X kW
- DO 切了 Y kW 負載 → 代表如果沒切，還有 Y kW 需求在那裡

所以原始的電網需求 = 現在看到的 + 電池補的 + 切掉的

**月度節省公式：**
```
monthly_peak_kva = max( 反事實峰值 ) over all 15-min windows in billing month
avoided_kva = MAX(0, monthly_peak_kva - contracted_demand_kw)
PS_savings_reais = avoided_kva × demand_charge_rate_per_kva
```

**每日 provisional（M4 日結算用）：**
```
daily_ps_savings_reais = 當日 avoided_kva × demand_charge_rate_per_kva / days_in_month
```
月底實際帳單出來後可以做 true-up 修正。

## 四、DO（Digital Output）鏈路

### 4.1 實測確認的 MQTT payload 格式

從旭衡 EMS 設備（WKRD24070202100141I）實際抓包確認：

```json
{
  "data": {
    "dido": {
      "di": [
        { "id": "DI0", "type": "DI", "value": "0" },
        { "id": "DI1", "type": "DI", "value": "0" }
      ],
      "do": [
        { "id": "DO0", "gpionum": "/dev/DO1", "type": "DO", "value": "0" },
        { "id": "DO1", "gpionum": "/dev/DO2", "type": "DO", "value": "0" }
      ]
    }
  }
}
```

- **key 名稱**：`data.dido.do[]`（不是 `doList`）
- **value**：`"0"` = 繼電器開路（未觸發），`"1"` = 閉合（切載中）
- **DI**：不使用，全部忽略
- **DO0 / DO1**：兩路獨立繼電器，各自控制一路負載迴路

### 4.2 切載量自動計算

DO 狀態變化時，從 `telemetry_history.load_power_kw` 計算前後差值：

```
load_shed_kW_DO0 = load_kW_at(DO0_trigger_t - 1min) - load_kW_at(DO0_trigger_t + 1min)
```

**為什麼自動計算而不是靜態設定：**
切斷的負載功率不是固定值（隨時段、設備運行狀態而異），必須從遙測實測差值反推。

```typescript

```

### M4 Market Billing（daily-billing-job.ts）

新增 PS 歸因計算（月底確算 + 每日 provisional）：
1. 從 `asset_5min_metrics` 取每個 5min 窗口的 `bat_discharge_kwh`
2. 從 `telemetry_history` 取 `do0_active`, `do1_active` 狀態變化，計算切載量
3. 重建每 15min 的「反事實峰值」
4. 計算月內最高反事實峰值 → `avoided_kva`
5. `PS_savings = avoided_kva × demand_charge_rate_per_kva`
6. 每日寫入 `revenue_daily.ps_savings_reais`（月底做 true-up）

### BFF（get-performance-savings.ts）

新增 `ps` 欄位（v5.15 已留 `ps: null` 佔位）：
- 讀取 `revenue_daily.ps_savings_reais`
- 回傳真實值，替換之前的 `null`

## 七、待 Gemini 確認的問題

### Q1：15min 需量視窗標準

ANEEL 規定是以 15 分鐘積分需量為計費單位。
我們已有 5min aggregator，每 3 個 5min bucket 平均 = 1 個 15min demand window。

**確認：** 這個 15min 滾動窗口的實現方式是否需要在 `asset_5min_metrics` 之上建一個 `asset_15min_demand` 中間表？還是在 M4 計算時直接 SQL GROUP BY 15min 即可？

### Q2：反事實峰值的 kW vs kVA

需量費以 **kVA** 計算（視在功率），但我們的遙測是 **kW**（有效功率）。

需要功率因數轉換：`kVA = kW / power_factor`

**確認：** power_factor 要存在哪裡？`assets` 表加 `power_factor REAL DEFAULT 0.92`？還是從 `tariff_schedules` 讀取（不同配電公司有不同標準值）？

### Q3：月度 true-up 機制

每日 provisional PS savings 累積到月底，實際帳單出來後需要 true-up（修正誤差）。

**確認：** true-up 是手動觸發（Admin Portal）還是 M4 自動在每月 1 日執行月結修正？

### Q4：多 asset 共用一個 DO

如果一棟建築（home）有 2 台電池（asset），但只有 1 組 DO 繼電器（在其中一台 EMS 上），切載歸因要怎麼分配？

**建議：** DO 歸屬於 EMS 設備（asset_id 層級），切載計算只在那台 asset 上做，不跨 asset 分攤。
