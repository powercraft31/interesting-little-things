**給 Gemini 的審查文件 | 作者：Ashe | 版本：v2（含 Gemini R1 決策更新）| 日期：2026-03-07**

## 一、為什麼做 PS（業務背景）

巴西工商業電力帳單由兩部分組成：

1. **電量費（Energia）**：實際用電 kWh × 分時費率（Tarifa Branca）
   → v5.15 的 SC/TOU attribution 處理的正是這部分

2. **需量費（Demanda）**：月內最高 15min 積分需量（kVA）× 需量費率（R$/kVA）
   → **v5.16 要處理的是這部分**

需量費的特性：
- **月結制**：整個月只看一個最高峰值，超過合約需量（contracted_demand_kw）的部分按倍率計費
- **一峰定全月**：哪怕只有 15 分鐘超過限額，整個月的超額費都算進來
- **可控性高**：只要在那 15 分鐘內把電池放電撐住，就能避免超限

**PS 的商業邏輯：** 電池在負載高峰時放電，把電網看到的功率壓在合約需量以下，避免觸發超額需量費。

## 二、架構定位（平台 vs 閘道器分工）

**已與 Alan 確認的鐵律邊界：**

| 職責 | 執行者 | 說明 |
|------|--------|------|
| 即時需量控制 | **閘道器（EMS 盒）** | 閘道器保證 15min 窗口內功率絕對不超限，平台不介入即時控制 |
| 下發 PS 指令 | **平台 M3** | 下發 `{ mode: 'peak_shaving', peak_limit_kva: X }` |
| 切換 PS 模式 | **平台 M2** | 決策何時啟用 PS（按時段、帳單週期位置等） |
| 節省歸因計算 | **平台 M4** | 月底計算「避免了多少需量費」 |
| 負載切斷（DO） | **閘道器硬體** | 電池不夠時觸發繼電器切斷非關鍵負載 |

**核心推論：** 因為閘道器保證「實際 kVA 永遠 ≤ contracted_demand_kw」，所以平台無法用「實際 vs 限額」來計算節省——實際值永遠在限額以下。需要反推「如果沒有 PS，峰值會到多高」。

## 三、PS 節省反事實計算法

**核心等式：**

```
反事實峰值（without PS） = 
    actual_grid_kW              ← 實際流入電網功率（telemetry_history）
  + bat_discharge_kW            ← 電池放電撐住的部分（asset_5min_metrics）
  + load_shed_kW_from_DO        ← DO 觸發切斷的負載（do0/do1 state）
```

**邏輯說明：**
- 電池放電了 X kW → 代表如果沒電池，電網就要多出 X kW
- DO 切了 Y kW 負載 → 代表如果沒切，還有 Y kW 需求在那裡

**kW → kVA 換算：**
```
counterfactual_kva = counterfactual_kW / billing_power_factor
```

**月度節省公式：**
```
monthly_peak_kva  = MAX( counterfactual_kva ) across all 15-min windows in billing month
avoided_kva       = MAX(0, monthly_peak_kva - contracted_demand_kw)
PS_savings_reais  = avoided_kva × demand_charge_rate_per_kva
```

**每日 provisional（M4 日結算用）：**
```
daily_ps_savings_reais = 當日 avoided_kva × demand_charge_rate_per_kva / days_in_month
```
月底 MonthlyTrueUpJob 自動重算修正。

## 四、DO（Digital Output）鏈路

### 4.1 實測確認的 MQTT payload 格式

從旭衡 EMS 設備（WKRD24070202100141I）實際抓包確認：

```json
{
  "data": {
    "dido": {
      "do": [
        { "id": "DO0", "gpionum": "/dev/DO1", "type": "DO", "value": "0" },
        { "id": "DO1", "gpionum": "/dev/DO2", "type": "DO", "value": "0" }
      ]
    }
  }
}
```

- **key 名稱**：`data.dido.do[]`（不是 `doList`）
- **value**：`"0"` = 繼電器開路（未觸發），`"1"` = 閉合（切載中）
- **DI**：不使用，全部忽略
- **DO0 / DO1**：兩路獨立繼電器，各自控制一路負載迴路

### 4.2 切載量自動計算（決策：自動推算，非靜態設定）

DO 狀態 0→1 時，從 `telemetry_history.load_power_kw` 計算觸發前後差值：

```
load_shed_kW_DO0 = load_kW_at(DO0_trigger_t - 1min) - load_kW_at(DO0_trigger_t + 1min)
```

理由：切斷的負載功率不是固定值（隨時段、設備運行狀態而異），必須從遙測實測差值反推。

### 4.3 孤兒遙測 Fallback（決策：保守降級，等補傳重算）

**場景：** DO 觸發後設備斷網，+1min 的遙測數據缺失。

**處理方式：**
- `load_shed_kW = 0`（不高估節省）
- 標記 `do_shed_confidence = 'low'`（可查，便於稽核）

**自動修正路徑：**
1. 閘道器本地有 5min 數據存儲，網路恢復後補傳（v6.0 Offline Backfill 機制）
2. 補傳數據寫入 `telemetry_history`（UPSERT，覆蓋空值）
3. 月底 `MonthlyTrueUpJob` 重掃完整數據，自動修正 PS savings

**注意（v6.0 待辦）：** `telemetry_history` 目前 PK 是 `(id, recorded_at)`，補傳 UPSERT 需要額外的 UNIQUE 約束：
```sql
-- v6.0 補傳機制上線時才加
CREATE UNIQUE INDEX idx_telemetry_asset_recorded ON telemetry_history (asset_id, recorded_at);
```

### 5.2 tariff_schedules — 新增需量費率與功率因數

```sql
ALTER TABLE tariff_schedules
  ADD COLUMN IF NOT EXISTS demand_charge_rate_per_kva NUMERIC(8,4),
  ADD COLUMN IF NOT EXISTS billing_power_factor        NUMERIC(3,2) DEFAULT 0.92;
```

**決策說明（Gemini R1 確認）：**
- 需量費率放 `tariff_schedules`（與分時費率同一合約，避免多餘 JOIN）
- `billing_power_factor` 是商業計費約定值（非物理量測值），屬於計費合約層，不放 `assets` 物理資產表

### 5.3 revenue_daily — 新增 PS 欄位

```sql
ALTER TABLE revenue_daily
  ADD COLUMN IF NOT EXISTS ps_savings_reais       NUMERIC(10,2),
  ADD COLUMN IF NOT EXISTS ps_avoided_peak_kva    NUMERIC(8,3),
  ADD COLUMN IF NOT EXISTS do_shed_confidence     VARCHAR(10),   -- 'high' / 'low'
  ADD COLUMN IF NOT EXISTS true_up_adjustment_reais NUMERIC(10,2); -- 月底 true-up 專用
```

**true-up 設計說明（Gemini R1 決策）：**
月底修正不 UPDATE 過去每日數據，而是在當月 1 日寫入一筆新的 `true_up_adjustment_reais` 紀錄，保留財務軌跡的不可篡改性（Auditability）。

### M4 Market Billing（daily-billing-job.ts）

**新增 `runDailyPsSavings(pool)` 函式：**

```sql
-- Step 1: 取 PS 模式下每 5min 窗口的反事實 kW
WITH ps_windows AS (
  SELECT
    m.asset_id,
    date_bin('15 minutes', m.window_start, TIMESTAMP '2026-01-01 03:00:00Z') AS window_15,
    SUM(m.grid_import_kwh * 12) + SUM(m.bat_discharge_kwh * 12) AS counterfactual_kw,
    -- DO 切載量（若有 do0/do1 觸發則加入，否則 0）
    SUM(COALESCE(load_shed_do0, 0) + COALESCE(load_shed_do1, 0)) AS do_shed_kw
  FROM asset_5min_metrics m
  JOIN dispatch_records dr ON dr.asset_id = m.asset_id
    AND COALESCE(dr.target_mode, 'UNASSIGNED') = 'peak_shaving'
    AND dr.dispatched_at <= m.window_start
  WHERE m.window_start >= $brt_window_start AND m.window_start < $brt_window_end
  GROUP BY m.asset_id, window_15
),
-- Step 2: kW → kVA，取 15min 最高值
peak_kva AS (
  SELECT asset_id,
    MAX((counterfactual_kw + do_shed_kw) / billing_power_factor) AS peak_kva
  FROM ps_windows
  JOIN tariff_schedules ts ON ... -- 取 billing_power_factor
  GROUP BY asset_id
)
-- Step 3: 計算 avoided_kva 與 savings
SELECT
  pk.asset_id,
  GREATEST(0, pk.peak_kva - h.contracted_demand_kw) AS avoided_kva,
  GREATEST(0, pk.peak_kva - h.contracted_demand_kw) * ts.demand_charge_rate_per_kva AS ps_savings
FROM peak_kva pk
JOIN assets a ON a.asset_id = pk.asset_id
JOIN homes h ON h.home_id = a.home_id
JOIN tariff_schedules ts ON ...
```

**新增 `runMonthlyTrueUp(pool)` 函式（每月 1 日執行）：**
- 重掃上月完整 asset_5min_metrics
- 計算真實 monthly_peak_kva
- 寫入 `revenue_daily.true_up_adjustment_reais`（新增行，不修改舊行）

## 七、已定案的架構決策（Gemini R1 全部確認）

| 問題 | 決策 |
|------|------|
| 15min 視窗實作 | 直接用 `date_bin('15 minutes', ...)` 在 M4 SQL，**不建中間表** |
| kW → kVA 換算 | `billing_power_factor` 放 `tariff_schedules`（計費合約值，DEFAULT 0.92）|
| 月度 true-up | M4 `MonthlyTrueUpJob` 每月 1 日自動執行，**寫新行，不改舊行** |
| 多 asset 共用 DO | DO 嚴格綁定 `asset_id`，BFF 層 `SUM GROUP BY home_id` |
| DO 孤兒遙測 | 保守降級：`load_shed = 0` + `confidence = 'low'`，補傳後 true-up 重算 |

## 八、補傳機制（Offline Backfill）— 獨立議題，v6.0 處理

旭衡閘道器本地有 5min 數據存儲，斷網不丟數據，網路恢復後補傳。

**v5.16 不實作補傳，但 v5.16 的設計必須對補傳相容：**
- M4 PS savings 計算要能重跑（冪等）
- `do_shed_confidence = 'low'` 標記讓月底 true-up 知道哪些窗口需要重算
- v6.0 補傳上線後，需要在 `telemetry_history` 加 UNIQUE 約束以支援 UPSERT
