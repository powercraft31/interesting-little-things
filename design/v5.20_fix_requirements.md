## Fix #1: P2 Layer 3 — Daily Schedule Editor UI

### Current State
Layer 3 right column has a Daily Schedule card, but it is **read-only** — only a timeline bar + table showing existing slots. No way to add, edit, or delete time slots. The Apply button exists at the very bottom of the page but can only send empty schedules.

### Problem
Users cannot configure battery schedules and dispatch them to the Gateway.

### Requirements
- Add **"+ Add Slot"** button below the schedule table
- Clicking it inserts an editable row: Start Hour (dropdown 0-23), End Hour (dropdown 1-24), Mode (dropdown: self_consumption / peak_valley_arbitrage / peak_shaving)
- Each row gets a **🗑 Delete** button
- **Apply button** moves inside the Schedule card (or fixed at right-col bottom), not at page bottom
- Adding/removing slots must update the timeline bar preview in real-time
- Apply flow: `PUT /api/devices/{assetId}/schedule` → BFF writes `device_command_logs` → M3 polls every 10s → MQTT `config/set`

### Files
- `frontend-v2/js/p2-devices.js` → `_buildScheduleCard()` (line 846), `_handleApply()` (line 971)
- `backend/src/bff/handlers/put-schedule.ts`
- `backend/src/dr-dispatcher/services/command-dispatcher.ts`

### Dependency
Depends on Fix #3 (Layer 3 becomes Gateway-level; schedule is per-gateway, not per-device)

## Fix #2: Grid Data Priority Logic (Inverter > Smart Meter)

### Current State
`device_state` stores per-device data. Grid-related fields (grid_power_kw, grid_voltage, grid_current, power_factor, total_buy_kwh, total_sell_kwh) overlap between inverter and smart meter.

### Business Rule (confirmed by Alan)
- If an inverter is present → use inverter's grid data (homes with inverters rarely have separate meters)
- If no inverter → fallback to smart meter's grid data

### Requirements
- In `FragmentAssembler.writeTelemetry()`: when assembling 5 MSG fragments, merge grid data with inverter-first priority
- Or in BFF layer: when returning Gateway-level aggregated data, prefer inverter grid values, fallback to meter
- Tightly coupled with Fix #3 (Gateway-level view needs merged grid data)

### Files
- `backend/src/iot-hub/services/fragment-assembler.ts` → `writeTelemetry()`
- `backend/src/bff/handlers/get-gateway-devices.ts`

### Current DB Evidence
```
device_state has 1 row (GoodWe inverter): grid_power_kw=60.000
Chint meters have no device_state rows (FragmentAssembler only processes batList, not meterList)
```

## Fix #3: Layer 3 — Change from Device-Level to Gateway/Home-Level

### Current State
P2 navigation: Gateway Card → expand → device list → click single device → Layer 3 detail page. Clicking a meter into Layer 3 shows nothing useful (no Energy Flow, no Battery) because those components are inverter-specific.

### Root Problem
Layer 3 describes a **Gateway (home)** — not a single device. Energy Flow, Battery Status, Inverter & Grid are all home-level views.

### Requirements

**Navigation change:**
- Gateway Card → click Gateway name/arrow → enter Layer 3 (Gateway-level)
- Device list remains expandable for quick glance but no longer opens Layer 3 per-device

**Layer 3 left column (monitoring):**
- Energy Flow: entire home (PV + Grid + Battery + Load)
- Battery Status: from the inverter under this Gateway
- Inverter & Grid: aggregated using Fix #2 priority logic

**Layer 3 right column (control):**
- Device Config: inverter config under this Gateway
- Daily Schedule: dispatched to Gateway (not to individual device)
- Gateway Health: CPU/memory/WiFi (see Fix #4 data)

**BFF impact:**
- Existing `GET /api/gateways/:gatewayId/devices` already returns all devices + state
- May need a new `GET /api/gateways/:gatewayId/detail` endpoint that returns merged/aggregated view
- Schedule GET/PUT should use `gatewayId` not `assetId`

### Files
- `frontend-v2/js/p2-devices.js` → `_openLayer3()`, `_buildLayer3()`, all Layer 3 render functions
- `frontend-v2/js/data-source.js` → `devices.deviceDetail()`
- `backend/src/bff/handlers/get-device-detail.ts`, `get-gateway-devices.ts`

### This is the largest structural change — #1 and #2 depend on it.

## Fix #4: Gateway Card — Display EMS Health Data

### Current State
`gateways.ems_health` JSONB has rich real-time data from the EMS firmware:

```json
{
  "CPU_temp": "44.76°C",
  "CPU_usage": "62.62%",
  "memory_usage": "33.77%",
  "disk_usage": "21.91%",
  "wifi_signal_strength": "0.00dBm (无信号)",
  "system_runtime": "1天23小时39分钟",
  "SIM_status": "未插入",
  "ems_temp": "44.42°C"
}
```

BFF `GET /api/gateways` already returns `emsHealth` field. Frontend Gateway Card renders **none of it**.

### Requirements
- Gateway Card (or expanded panel) shows key EMS indicators:
  - 📡 WiFi signal strength
  - 🌡 CPU temperature
  - 💻 CPU usage %
  - 💾 Memory usage %
  - 💿 Disk usage %
  - ⏱ System uptime
- Compact layout: small icons + values, not full cards
- No backend changes needed — data already flows to frontend

### Files
- `frontend-v2/js/p2-devices.js` → Gateway Card render section
- CSS: `frontend-v2/css/pages.css`
