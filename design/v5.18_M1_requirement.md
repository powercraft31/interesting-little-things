# v5.18 需求文件：M1 IoT Hub — MQTT 協議接入層

## 1. 背景

### 1.1 當前狀態

SOLFACIL VPP 平台目前的 M1（IoT Hub）是一個簡化版的 MQTT Bridge，功能有限：
- 只訂閱單一 topic `device/ems/{clientId}/data`
- 只解析電池側部分欄位（soc/power/dailyCharge/dailyDischarge）
- 沒有設備發現能力（deviceList）
- 沒有配置下發能力（config get/set）
- 沒有心跳監測
- 沒有防腐層（Anti-Corruption Layer），直接把 MQTT 原始欄位名寫進 DB

### 1.2 嵌入式協議

旭衡嵌入式團隊已交付正式協議文件 `SolfacilProtocol v1.1`，定義了完整的 MQTT 通信規範：
- 設備列表上報（deviceList）
- 實時數據上報（meterList / gridList / pvList / batList / loadList / flloadList）
- 配置讀取與下發（battery_schedule）
- 心跳包（status）

協議文件存檔：https://152.42.235.155:8443/SolfacilProtocol_v1.1.md

### 1.3 硬件拓撲

每個「家庭」有一台 EMS 網關（clientId），網關下掛：
- **一級子設備（major）**：直接與網關通信 — inverter、meter
- **二級子設備（minor）**：掛載在逆變器下 — battery、PV、grid、load、flload

設備關係由 `fatherSn` 和 `nodeType` 定義。

## 2. 目標

v5.18 要把 M1 升級為完整的 IoT 協議接入層：

1. **訂閱 5 個 topic**（per gateway）接收所有上行數據
2. **發布 2 個 topic**（per gateway）下發配置
3. **防腐層**：將嵌入式協議格式翻譯為平台內部領域模型
4. **數據持久化**：解析後的數據存入對應的 DB 表

## 3. MQTT Topic 清單

### 3.1 訂閱（Subscribe）— 5 個 per gateway

| # | Topic | 觸發時機 | 數據內容 |
|---|-------|----------|----------|
| S1 | `device/ems/{clientId}/deviceList` | 設備變化時 + 網關首次連線 | 一級子設備列表（bindStatus, connectStatus, deviceBrand, deviceSn, productType, protocolAddr 等） |
| S2 | `device/ems/{clientId}/data` | 週期性（約 10-30 秒） | 實時遙測：meterList + gridList + pvList + batList + loadList + flloadList |
| S3 | `device/ems/{clientId}/config/get_reply` | 回應 config/get 請求 | 當前 battery_schedule 配置 |
| S4 | `device/ems/{clientId}/config/set_reply` | 回應 config/set 請求 | 設置結果：success/fail + message |
| S5 | `device/ems/{clientId}/status` | 每 30 秒心跳 | 網關在線狀態 |

### 3.2 發布（Publish）— 2 個 per gateway

| # | Topic | 用途 | 觸發者 |
|---|-------|------|--------|
| P1 | `platform/ems/{clientId}/config/get` | 讀取網關當前配置 | BFF（用戶打開排程編輯器時） |
| P2 | `platform/ems/{clientId}/config/set` | 下發配置到網關 | BFF（用戶按 Apply to Gateway） / M2（算法自動排程） |

## 4. 防腐層（Anti-Corruption Layer）

### 4.1 職責

將嵌入式協議的原始格式翻譯為平台內部領域模型。平台的其他模組（BFF、M2、M3、M4）**只跟領域模型打交道**，不碰協議原始欄位。

### 4.2 需要翻譯的內容

#### 4.2.1 設備發現（deviceList → DB）

協議上報 `deviceList[]`，每個元素代表一個一級子設備。防腐層需要：
- 解析 `productType`（meter / inverter）、`deviceBrand`、`deviceSn`、`connectStatus`
- 在 DB 中建立或更新對應的設備記錄
- 處理設備上下線狀態變更

#### 4.2.2 實時遙測（data → DB）

協議的 `data` 消息包含多種 List，每種對應不同的設備類型。所有值都是 **string 類型**，防腐層需要轉為數值。

**電池側（batList）— 13 個欄位：**

| 協議欄位 | 領域含義 | 單位 | 注意事項 |
|----------|----------|------|----------|
| total_bat_soc | 電池 SOC | % | |
| total_bat_soh | 電池 SOH | % | BMS 直報 |
| total_bat_power | 電池功率 | W | 放電為正，充電為負 |
| total_bat_current | 電池電流 | A | 正放電、負充電 |
| total_bat_vlotage | 電池電壓 | V | ⚠️ 拼寫錯誤，是 vlotage 不是 voltage |
| total_bat_temperature | 電池溫度 | ℃ | |
| total_bat_maxChargeVoltage | 最大充電電壓 | V | 來自 BMS |
| total_bat_maxChargeCurrent | 最大充電電流 | A | 來自 BMS |
| total_bat_maxDischargeCurrent | 最大放電電流 | A | 來自 BMS |
| total_bat_dailyChargedEnergy | 日充電量 | kWh | |
| total_bat_dailyDischargedEnergy | 日放電量 | kWh | |
| total_bat_totalChargedEnergy | 累計充電量 | kWh | |
| total_bat_totalDischargedEnergy | 累計放電量 | kWh | |

**電網側（gridList）— 27 個欄位：**
- 三相電壓/電流/有功/無功/視在功率（各 A/B/C + total）
- 功率因數（A/B/C 相）
- 頻率
- 累計/日買賣電量
- 逆變器溫度

**電表側（meterList）：**
- 單相電表：6 個欄位（電壓/電流/有功/無功/功率因數/頻率）
- 三相電表：29 個欄位（三相電壓/線電壓/電流/有功/無功/功率因數/頻率/正反向電能）

**光伏側（pvList）：**
- 匯總：totalPower / totalEnergy / dailyEnergy
- 每個 MPPT（pv1/pv2）：voltage / current / power

**負載側（loadList + flloadList）：**
- load1（Backup 口）：三相電壓/電流/功率/頻率 + totalPower
- flload（家庭總負載）：三相功率 + totalPower + dailyEnergy

#### 4.2.3 配置翻譯（battery_schedule ↔ 領域模型）

**讀取方向（get_reply → 領域模型）：**
- 協議的 `purpose: "tariff"` + `direction: "charge"` → 平台的 `peak_valley_arbitrage`（谷充段）
- 協議的 `purpose: "tariff"` + `direction: "discharge"` → 平台的 `peak_valley_arbitrage`（峰放段）
- 協議的 `purpose: "self_consumption"` → 平台的 `self_consumption`
- 協議的 `purpose: "peak_shaving"` → 平台的 `peak_shaving`
- 協議的 `export_policy` → 平台排程的附加屬性

**下發方向（領域模型 → config/set）：**
- 反向翻譯，平台模式轉回協議格式
- 時間必須是 60 的整數倍（整點制）
- 所有值轉為 string 類型下發

#### 4.2.4 心跳（status → 網關在線狀態）

- 收到心跳 → 更新網關 last_seen 時間戳
- 超過 N 秒未收到心跳 → 標記網關離線

## 5. DB 變更

### 5.1 新表：gateways（網關註冊表）

網關是入網流程的起點。Commissioning Wizard 寫入此表 → M1 據此訂閱 MQTT。

建議欄位：
- gateway_id (PK)
- client_id（MQTT clientId，即序列號）
- org_id（歸屬 integrador）
- home_id（歸屬家庭）
- mqtt_broker_host
- mqtt_broker_port
- mqtt_username
- mqtt_password
- device_name（如 EMS_N2）
- product_key（如 ems）
- status（online / offline）
- last_seen_at
- commissioned_at
- created_at / updated_at

### 5.2 現有表調整

- `assets` 表：子設備由 M1 收到 deviceList 後動態建立，需要關聯到 gateway_id
- `telemetry_history`：確認欄位能容納完整的遙測數據（目前可能只有電池側部分欄位）

前端和 BFF 都需要做的校驗（雙重保險，網關也會校驗）：

| 欄位 | 類型 | 範圍 |
|------|------|------|
| soc_min_limit | int | 0-100, < soc_max_limit |
| soc_max_limit | int | 0-100, > soc_min_limit |
| max_charge_current | int | ≥0, ≤ BMS 報告上限 |
| max_discharge_current | int | ≥0, ≤ BMS 報告上限 |
| grid_import_limit | int | ≥0, 單位 KW |
| slot.start | int | 0-1380, 60 的整數倍 |
| slot.end | int | 60-1440, 60 的整數倍, > start |
| slots 覆蓋 | - | 必須覆蓋全天 0-1440, 不可重疊 |

## 7. 動態訂閱流程

```
啟動 M1
  → 讀取 gateways 表，取得所有 status != 'decommissioned' 的網關
  → 對每個 gateway 的 client_id 訂閱 5 個 topic
  → 進入事件循環

新網關入網（Commissioning）
  → DB 寫入 gateways 記錄
  → 通知 M1 動態新增訂閱（或 M1 輪詢偵測新記錄）

網關下線
  → 心跳超時 → 更新 status = 'offline'
  → 保持訂閱（不取消），等待重新上線
```

## 8. MVP 架構防禦底線（Addendum — Gemini Review）

### 8.1 時間戳絕對信任（TimeStamp Rule）

針對 `data`（4.2.2）：所有寫入 `telemetry_history` 的遙測數據，其時間戳**必須嚴格解析自 MQTT Payload 頂層的 `timeStamp` 欄位**。嚴禁使用服務器接收時間 `NOW()`，以確保斷網補傳（Backfill）時的冪等性與 M4 結算視窗的絕對準確。

### 8.2 拓撲軟刪除鐵律（Soft-Delete Only）

針對 `deviceList`（4.2.1）：若 M1 對比 DB 發現某已知子設備未出現在最新的 `deviceList` 中，**絕對禁止 DELETE**。必須將該設備的 `is_active` 變更為 `false`（或標記 offline），以死守歷史財務軌跡的完整性。

### 8.3 指令下發異步閉環（Command Tracking）

針對 `config/set_reply`（3.1 S4）：M1 收到設備的 config 設置回應後，必須將結果（`result`, `message`, `timeStamp`）持久化到 DB（例如寫入 `device_command_logs` 表，或更新 `gateways` 對應欄位），確保上層的 BFF 有錨點可查，完成前端 UI 的狀態閉環。

## 9. 非功能需求

- M1 作為常駐服務（systemd），crash 自動重啟
- MQTT 斷線自動重連
- 所有解析錯誤需 log 但不能讓服務掛掉（容錯）
- 遙測數據寫入需考慮批量/防抖（現有 2s debounce 機制可沿用）
