## 2. Feature 1: Two-Phase set_reply

### 2.1 Protocol Change (v1.7 §3.9)

Previously, gateway sent a single `set_reply` with `result = "success"` or `"fail"`.
Now, gateway sends TWO replies:

```
Platform                    Gateway                    Device
   |-- config/set ------------>|                          |
   |                           |-- write registers ------>|
   |<-- set_reply (accepted) --|                          |
   |                           |<-- read-back verify -----|
   |<-- set_reply (success) ---|  (or fail)               |
```

### 2.2 Command Status State Machine

```
                     ┌──────────────────────────────────────────┐
                     │           device_command_logs             │
                     └──────────────────────────────────────────┘

  BFF PUT /schedule     M3 runPendingCommand     M1 CommandPublisher
        │                   Dispatcher                  │
        ▼                      │                        ▼
    ┌────────┐                 │                  ┌────────────┐
    │ pending │────────────────┘                  │ dispatched │
    └────────┘  UPDATE result='dispatched'        └─────┬──────┘
                                                        │
                           ┌────────────────────────────┤
                           │ set_reply(accepted)        │ set_reply(success)
                           ▼                            │ or set_reply(fail)
                     ┌──────────┐                       │
                     │ accepted │───────────────────────┤
                     └──────────┘                       │
                           │                            │
              ┌────────────┼────────────────────────────┤
              │ timeout    │ set_reply(fail)             │ set_reply(success)
              │ (90s)      │                             │
              ▼            ▼                             ▼
        ┌─────────┐  ┌────────┐                   ┌─────────┐
        │ timeout │  │ failed │                   │ success │
        └─────────┘  └────────┘                   └─────────┘
              ▲
              │ timeout (90s from created_at)
              │ WHERE result IN ('dispatched','accepted')
```

**State transitions:**

| From | To | Trigger | sets `resolved_at`? |
|------|----|---------|---------------------|
| pending | dispatched | M3 `runPendingCommandDispatcher` | No |
| dispatched | accepted | `set_reply(accepted)` | **No** — intermediate |
| dispatched | success | `set_reply(success)` — skip accepted | Yes |
| dispatched | failed | `set_reply(fail)` — skip accepted | Yes |
| accepted | success | `set_reply(success)` — phase 2 | Yes |
| accepted | failed | `set_reply(fail)` — phase 2 | Yes |
| dispatched | timeout | `runTimeoutCheck` (90s) | Yes (implicit) |
| accepted | timeout | `runTimeoutCheck` (90s) | Yes (implicit) |

When `handleSetReply` processes any result (`accepted`/`success`/`fail`), fire `pg_notify`:

```sql

-- payload: JSON { gatewayId, configName, result, phase: 1|2 }
```

The existing SSE infrastructure (v5.21) can pick this up. Frontend can optionally show:
- Phase 1 (accepted): "Configuração aceita, aplicando..."
- Phase 2 (success): "Sincronizado ✓"
- Phase 2 (fail): "Falha na aplicação"

### 3.3 Reconnect Detection Logic

**File:** `backend/src/iot-hub/handlers/heartbeat-handler.ts` (currently 36 lines)

Current behavior (line 26-33): blind UPDATE of `last_seen_at` and `status = 'online'`.

New behavior:

```
handleHeartbeat(pool, gatewayId, clientId, payload):
  1. Parse deviceTimestamp (existing, line 17)
  2. SELECT last_seen_at, status FROM gateways WHERE gateway_id = $1   ← NEW
  3. Calculate gap = deviceTimestamp - previous_last_seen_at
  4. If gap > 120_000ms (2 min) AND status !== 'online':               ← NEW
     → INSERT INTO backfill_requests (gateway_id, gap_start, gap_end, status)
       VALUES ($1, old_last_seen_at, new_timestamp, 'pending')
     → Log "[HeartbeatHandler] Reconnect detected, gap=Xmin, queued backfill"
  5. UPDATE gateways SET last_seen_at, status='online' (existing)
  6. pg_notify('gateway_health', ...) (existing, line 35)
```

### 3.4 BackfillRequester Service

**New file:** `backend/src/iot-hub/services/backfill-requester.ts`

```
class BackfillRequester:
  private timer: NodeJS.Timeout | null

  constructor(pool: Pool, connectionManager: GatewayConnectionManager)

  start():  setInterval(poll, 10_000)   — same pattern as CommandPublisher
  stop():   clearInterval

  poll():
    1. SELECT * FROM backfill_requests
       WHERE status IN ('pending', 'in_progress')
       ORDER BY created_at ASC
       LIMIT 5
       FOR UPDATE SKIP LOCKED

    2. For each request:
       a. If status='pending' AND created_at + 30s > NOW():
          → skip (delay not elapsed)

       b. If status='pending' AND delay elapsed:
          → current_chunk_start = gap_start
          → chunk_end = MIN(current_chunk_start + 30min, gap_end)
          → publish get_missed(current_chunk_start, chunk_end)
          → UPDATE status='in_progress', current_chunk_start, last_chunk_sent_at=NOW()

       c. If status='in_progress':
          → If last_chunk_sent_at + 20s > NOW(): skip (cooldown)
          → Advance: current_chunk_start += 30min
          → If current_chunk_start >= gap_end:
            → UPDATE status='completed', completed_at=NOW()
          → Else:
            → chunk_end = MIN(current_chunk_start + 30min, gap_end)
            → publish get_missed(current_chunk_start, chunk_end)
            → UPDATE current_chunk_start, last_chunk_sent_at=NOW()

  publishGetMissed(gatewayId, start, end):
    topic = `platform/ems/${gatewayId}/data/get_missed`
    payload = SolfacilMessage { data: { start: String(startMs), end: String(endMs) } }
    connectionManager.publishToGateway(gatewayId, topic, JSON.stringify(payload))
```

### 3.5 MissedDataHandler

**New file:** `backend/src/iot-hub/handlers/missed-data-handler.ts`

```
handleMissedData(pool, gatewayId, clientId, payload):
  1. Check if data is empty object → log "no data for chunk", return
  2. Call parseTelemetryPayload(clientId, payload)  ← shared with FragmentAssembler
  3. Resolve asset_id via DeviceAssetCache
  4. INSERT INTO telemetry_history (...) VALUES (...)
     ON CONFLICT (asset_id, recorded_at) DO NOTHING
  5. Log count of inserted rows
```

Key difference from live path:
- No FragmentAssembler accumulation needed — backfill messages are complete (not fragmented)
- Uses `ON CONFLICT DO NOTHING` to handle duplicates safely
- Does NOT update `device_state` (backfill is historical, not current state)
- Does NOT fire `pg_notify('telemetry_update')` (avoid false SSE updates)

### 3.8 DDL Changes (Feature 2)

#### 3.8.1 New Table: `backfill_requests`

```sql
-- v5.22: Backfill request queue
CREATE TABLE IF NOT EXISTS backfill_requests (
  id              BIGSERIAL PRIMARY KEY,
  gateway_id      VARCHAR NOT NULL REFERENCES gateways(gateway_id),
  gap_start       TIMESTAMPTZ NOT NULL,
  gap_end         TIMESTAMPTZ NOT NULL,
  current_chunk_start TIMESTAMPTZ,
  last_chunk_sent_at  TIMESTAMPTZ,
  status          VARCHAR(20) NOT NULL DEFAULT 'pending',
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  completed_at    TIMESTAMPTZ,

  CONSTRAINT chk_backfill_status
    CHECK (status IN ('pending', 'in_progress', 'completed', 'failed'))
);

-- Index for polling query
CREATE INDEX idx_backfill_active
ON backfill_requests (created_at ASC)
WHERE status IN ('pending', 'in_progress');
```

#### 3.8.2 Unique Index on `telemetry_history`

```sql
-- v5.22: Prevent duplicate telemetry from backfill
-- NOTE: telemetry_history is partitioned by recorded_at (RANGE).
-- Unique index must include the partition key.
CREATE UNIQUE INDEX CONCURRENTLY IF NOT EXISTS idx_telemetry_unique_asset_time
ON telemetry_history (asset_id, recorded_at);
```

**Partition consideration:** If `telemetry_history` uses declarative partitioning, this unique index is created on the parent table and PostgreSQL automatically creates matching indexes on each partition. The partition key (`recorded_at`) is already included.

## 5. Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| Backfill floods normal telemetry pipeline | Delayed live data display | Separate `data/missed` topic + 20s cooldown + 30min chunks |
| Unique index creation on large partitioned table | Lock contention during creation | `CREATE INDEX CONCURRENTLY` — no table lock |
| `accepted` timeout edge case: gateway sends accepted then goes offline | Command stuck in `accepted` forever | `runTimeoutCheck` now covers `IN ('dispatched','accepted')` |
| Backfill data has duplicate timestamps | Duplicate rows in telemetry_history | `ON CONFLICT (asset_id, recorded_at) DO NOTHING` |
| Topic routing: `/data/missed` vs `/data` | Wrong handler receives backfill data | Check `/data/missed` BEFORE `/data` in `routeMessage()` |
| HeartbeatHandler becomes non-trivial (SELECT before UPDATE) | Extra query per heartbeat (every 30s per gateway) | Acceptable — one lightweight SELECT per 30s per gateway |
