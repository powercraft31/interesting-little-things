第一部分

旭衡電子的業務定位

旭衡電子的核心業務是能源管理系統（Energy Management System，EMS）——幫助用電方、儲能設備擁有者，通過智能化的能源調度，降低用電成本、提升能源效率、參與電力市場獲得收益。

戰略目標：把 EMS 能力打包成白牌多租戶平台，賣給更多客戶。每個客戶（租戶）根據自身市場和業務場景，選擇啟用不同的用例組合。接入第 N 個客戶的增量工作量趨近於零，才是真正的平台。
🇦🇺 UEH（已上線）
  • 澳洲家庭儲能聚合
  • AEMO 電力現貨市場
  • 6 個逆變器品牌，雲端 API 接入
  • 啟用用例：市場套利
🇧🇷 SOLFACIL（設計中）
  • 巴西工商業儲能聚合
  • CCEE 電力現貨市場
  • 硬件網關 + MQTT 接入
  • 啟用用例：市場套利

兩個客戶目前都只啟用了「市場套利」這一個用例。但平台的域設計必須從完整的 EMS 業務出發，而非只為套利服務——否則每個新用例都需要重構。

第二部分

能源管理的本質

2.1 第一性原理

能源管理的本質問題只有一個：

在正確的時間，對正確的設備，做出正確的能源決策——讓能源資源的每一度電都在最優的時機被使用、儲存或出售。

這裡的「正確」有多個維度，由不同用例定義：

維度定義「正確」的標準對應用例
市場時機電價低時充、高時放，差價最大市場套利
自用效率先用太陽能，電池補充，最後才買電自發自用優化
需量管理在用電高峰前放電，削減峰值需量費峰值功率管理
電網協作電網需要削峰/補頻時即時響應需求響應 / 輔助服務
韌性保障保留最低備用電量應對突發停電備用電源管理

2.2 核心操作永遠只有三個

無論哪個用例，對儲能設備的操作指令只有：

⬇️
CHARGE
充電
⬆️
DISCHARGE
放電
⏸️
STOP
停止

三條指令適用於所有用例、所有品牌、所有接入模式。「做什麼」的複雜度在決策層，「怎麼執行」的複雜度在設備通信層。

第三部分

完整用例清單

從旭衡電子的能源管理業務出發，可以列舉出以下 8 個完整用例。不同客戶根據自身市場與業務需求，選擇啟用其中的子集。

A. 市場參與類（Market Participation）
①
市場套利（Price Arbitrage）
根據電力現貨市場電價，在低電價時段充電，在高電價時段放電，賺取差價收益。UEH（AEMO）和 SOLFACIL（CCEE）均啟用此用例。
②
需求響應（Demand Response / DR）
電網運營商發出削峰/填谷事件信號，平台聚合設備響應，按響應量獲得 DR 補償費。信號是事件驅動，收益按次計算，非電價差價。
③
輔助服務（Ancillary Services / Frequency Response）
電網頻率偏離 50Hz 時，設備在秒級內自動響應穩定頻率。響應速度遠快於市場套利，需要實時遙測支撐。
B. 建築/家庭優化類（On-Site Optimization）
④
自發自用優化（Self-Consumption Optimization）
最大化利用屋頂光伏發電：白天太陽能充電池，晚間電池放電，減少從電網購電。需要負載預測和太陽能出力預測支撐。
⑤
峰值功率管理（Peak Demand Management）
工商業客戶按最高需量計費。在用電高峰前提前放電，削減峰值，降低需量費。需要實時負載監測。
⑥
電動車智慧充電（Smart EV Charging）
在電價谷值或太陽能充裕時為電動車充電，避免在高峰電價充電，降低充電成本。支持 V2G（電動車電池回注電網）的場景可視作臨時儲能資源參與套利。
E. 充電服務類（Charging Services）
⑨
公共充電站管理（Public EV Charging Station）規劃中
對外開放的 EV 充電樁，車主掃碼付費，按充電量計費。充電樁可以附帶儲能（電價谷值儲電、高峰向車主放電，降低電費）或不帶儲能（純直接充電）。兩種模式對調度引擎透明，區別在設備能力模型。
C. 韌性類（Resilience）
⑦
備用電源管理（Backup Power / Resilience Mode）
在電池 SoC 達到安全底線前停止一切調度，確保停電時有足夠備用電量。韌性策略優先級高於所有套利策略。
D. 社區類（Community）
⑧
社區共享儲能（Community Energy Sharing / P2P）
社區內鄰居之間直接進行電力撮合交易。需要計量、撮合引擎和獨立的 P2P 結算機制。
第四部分 · 第一層

業務域（Domain）

根據用例的需求分析，EMS 平台定義 8 個業務域。每個域的存在由至少一個用例的業務需求決定，沒有用例需要的概念不應獨立成域。

📡
外部信號域（External Signal）
被用例需要：① 市場套利　② 需求響應　③ 輔助服務
📋
資產登記域（Asset Registry）
被用例需要：① ② ③ ④ ⑤ ⑥ ⑦ ⑧（全部用例）
🧠
調度引擎域（Dispatch Engine）
被用例需要：① ② ③ ④ ⑤ ⑥ ⑦ ⑧（全部用例，策略不同）
🔌
設備網關域（Device Gateway）
被用例需要：① ② ③ ④ ⑤ ⑥ ⑦ ⑧（全部用例）
🔮
預測域（Forecasting）
被用例需要：④ 自發自用　⑤ 峰值管理　⑥ EV 充電（市場套利部分用：CCEE 日前電價預測）
📊
遙測域（Telemetry）
被用例需要：③ 輔助服務　④ 自發自用　⑤ 峰值管理　⑦ 備用電源　⑧ 社區共享（套利部分用）
💰
財務結算域（Financial Settlement）
被用例需要：① 市場套利　② 需求響應　④ 自發自用節省　⑧ P2P 交易
🔌
充電交易域（Charging Transaction）規劃中
被用例需要：⑨ 公共充電站管理　（車主認證 → 充電會話 → 用電計費 → 付款結算）
資產登記 / 調度引擎 / 設備網關 是三個全域共用域——所有用例都需要它們。這三個域是平台的基礎骨架，它們的接口設計必須能支撐任意用例的策略切換。

模塊結構（Module）

標準模塊

📡 外部信號域（External Signal）

職責：接收所有來自電力市場和電網運營商的外部信號，標準化後發布到平台內部。

市場電價連接器
Market Price Connector
每個市場一個實現（插件）。AEMO 輪詢 REST，CCEE 監聽 Webhook。拉取原始電價數據。
需求響應事件接收器
DR Event Receiver
接收電網運營商推送的 DR 事件（削峰/填谷信號），與市場電價信號性質不同——是操作指令，非價格數據。
頻率偏差監測器
Grid Frequency Monitor
實時監測電網頻率偏差（50Hz 基準），秒級觸發輔助服務響應。
信號標準化器
Signal Normalizer
各市場不同格式 → 統一的 ExternalSignal 標準結構，包含信號類型、地區、時間戳、數值。
信號發布器
Signal Publisher
向 Event Bus 發出標準化信號事件，調度引擎訂閱並自主決定響應方式。

📋 資產登記域（Asset Registry）

職責：管理所有參與 EMS 平台的設備元數據、業主信息和授權狀態。所有用例的調度前提。

設備能力模型（Device Capability Model）：平台不以「設備類型」判斷，而是以「設備能力」決定可執行哪些操作。
DeviceCapability {
  hasStorage: boolean       // 能儲能（電池逆變器、帶儲能充電樁）
  isControllableLoad: boolean // 是可調度負載（充電樁、HVAC）
  hasGeneration: boolean    // 能發電（PV 逆變器）
  isV2G: boolean          // 支持車對電網放電
  gatewayProtocol: BRAND_API | OCPP | MQTT
}
設備目錄
Device Catalog
設備技術規格：品牌、型號、SN、容量、能力組合（DeviceCapability）、連接協議。
業主管理
Owner Management
設備業主的個人/企業信息、聯繫方式、銀行賬戶（用於分成）。
授權管理器
Authorization Manager
授權生命周期：待審 → 已授權 → 暫停 → 撤銷。授權狀態決定設備是否可被調度。
位置管理
Location Management
地理位置、電網區域（如 AEMO 的 NSW/VIC/SA 區域），決定適用的電價和 DR 信號。
查詢接口
Query Interface
回答「某地區有哪些已授權設備可以調度」。調度引擎的主要查詢入口。

🧠 調度引擎域（Dispatch Engine）

職責：平台的決策核心。根據外部信號 + 可調度設備列表，按照啟用的策略決定對哪些設備做什麼操作。

策略引擎
Strategy Engine
可插拔策略容器。每個用例對應一個策略實現，租戶配置決定啟用哪些策略和優先級順序。
↳ 套利策略
Arbitrage Strategy
電價低於閾值→充電；電價高於閾值→放電。當前 UEH/SOLFACIL 啟用的策略。
↳ 自發自用策略
Self-Consumption Strategy
太陽能充裕 → 充電；夜間/陰天 → 放電抵消購電。依賴預測域輸入。
↳ 需求響應策略
DR Response Strategy
接收 DR 事件信號 → 立即按指令方向調度指定設備群。響應時效嚴格。
↳ 峰值管理策略
Peak Demand Strategy
預測負載高峰 → 提前放電削峰。依賴遙測域實時負載和預測域負載預測。
↳ 韌性策略
Resilience Strategy
SoC 低於安全閾值 → 停止所有調度，保留備用電量。優先級最高，可覆蓋所有其他策略。
條件評估器
Condition Evaluator
接收信號 + 策略規則 → 判斷是否觸發、觸發哪個方向。與策略解耦，獨立組件。
調度計劃器
Dispatch Planner
查詢資產登記，確定對哪些設備下發什麼指令，生成調度計劃。
狀態追蹤器
State Tracker
記錄當前調度狀態，防止重複下發，處理超時，維護調度生命周期。

🔌 設備網關域（Device Gateway）

職責：將調度引擎的抽象指令（CHARGE/DISCHARGE/STOP）翻譯為各品牌設備的具體通信調用。對調度引擎完全透明。

路由器
Gateway Router
根據 Device.gatewayMode（CLOUD/MQTT）決定走哪條通信路徑。
雲端適配器
Cloud API Adapter
每個品牌一個實現（插件）。AlphaESS / FoxESS / SunGrow / SNJ / GrowWatt / WeiHeng。新品牌 = 新增一個 Adapter 類。
MQTT 網關客戶端
MQTT Gateway Client
向本地網關盒發布 MQTT 指令，訂閱 ACK 回報。SOLFACIL 使用此模式。
OCPP 適配器 規劃中
OCPP Adapter
EV 充電樁的標準通信協議（Open Charge Point Protocol）。一個適配器覆蓋大多數品牌充電樁，與逆變器「每品牌一個適配器」不同。支持充電授權、計量、遠程控制。
指令確認器
Command ACK Handler
處理設備執行結果回報（SUCCESS/FAILED/TIMEOUT），通知狀態追蹤器更新。

🔮 預測域（Forecasting）

職責：預測未來的負載、發電量和電價，為調度引擎的策略提供前瞻性輸入。市場套利基本不需要，自發自用/峰值管理必須依賴。

負載預測
Load Forecasting
預測未來 24 小時的建築/家庭用電負荷。自發自用和峰值管理策略的核心輸入。
太陽能出力預測
Solar Generation Forecasting
根據天氣預報預測 PV 板發電量。自發自用策略決定充放電時機的依據。
電價預測
Price Forecasting
日前市場（如 CCEE）需要提前預測次日電價窗口，提前規劃最優充放電排程。
預測模型管理
Model Management
管理預測模型的訓練、版本和部署。支持各租戶使用不同的預測模型配置。

🔌 充電交易域（Charging Transaction）規劃中

職責：管理公共充電站的完整交易生命周期——從車主到站插槍，到完成付款離開。這是一個獨立的商業交易流程，與能源調度解耦。

充電樁授權器
Charger Authorization
車主掃碼/RFID/APP 認證，確認授權後才能開始充電。對接 Identity 平台服務。
充電會話管理器
Charging Session Manager
記錄每次充電的開始時間、結束時間、充電電量。會話是計費的原子單位。
充電計費引擎
Charging Billing Engine
依據充電電量和定價策略（按電量/按時間/分時段）計算本次應收費用。
支付收款器
Payment Collector
對接支付平台（信用卡/掃碼支付），完成款項收取，觸發財務結算域記錄收益。

📊 遙測域（Telemetry）

職責：收集所有設備和電表的實時狀態數據，提供「現在正在發生什麼」的即時答案。

設備狀態收集器
Device State Collector
收集逆變器/電池的 SoC、充放電功率、在線狀態。市場套利也需要（確認指令執行）。
負載監測器
Load Monitor
實時監測建築/家庭用電量。峰值管理策略必須依賴，用於判斷何時需要放電削峰。
太陽能出力監測器
PV Generation Monitor
實時監測 PV 板發電量。自發自用策略的實時輸入，與 PV 預測結合使用。
數據聚合器
Data Aggregator
將高頻原始遙測數據聚合為分鐘/小時粒度，降低存儲壓力，提供結算用的準確能量數據。
歷史遙測查詢
Historical Telemetry Query
提供歷史能量數據查詢接口，供結算域計算收益時使用。

💰 財務結算域（Financial Settlement）

職責：計算每個用例產生的收益或節省，按配置的分成比例分配給各方，生成報表。永遠不觸發調度。

兩種截然不同的財務模型並存：
模式 A（能源收益型）：平台從市場賺差價 → 按比例分給業主。計費對象是「電量×電價」
模式 B（充電服務型）：車主付費 → 按用電量計費。計費對象是「充電電量×服務單價」
兩種模式在同一個財務結算域內，由不同模塊處理。
套利結算
Arbitrage Settlement
充放電能量 × 對應電價差 = 毛收益。當前 UEH/SOLFACIL 使用的結算模塊。
需求響應結算
DR Event Settlement
按 DR 響應事件計費，與電價套利邏輯完全不同——按響應量和響應質量付費。
自發自用節省計算
Self-Consumption Savings Calculator
計算通過光伏+電池自發自用避免向電網購電所節省的電費金額。
P2P 交易結算
P2P Trading Settlement
社區電力撮合交易的結算。賣電方收益 + 買電方成本 + 平台手續費。
分成分配器
Revenue / Savings Splitter
按租戶配置的分成比例，將收益/節省分配給業主、平台及其他參與方。
報表生成器
Report Generator
生成月度/季度財務報表，供業主門戶和管理後台展示。
第六部分 · 水平組件

平台服務層（Platform Services）

以下組件是水平基礎設施——它們不執行業務邏輯，但定義了七個業務域能否各自獨立運作的前提條件。缺少任何一個，域邊界就無法維持。

🖥️
BFF（前端聚合層）
為管理後台和業主門戶提供業務語義 API。從多個域聚合數據，組合成前端所需的視圖格式。不包含核心業務規則。

缺少時：前端直接對接七個域的 API，自行聚合，維護成本極高。
🔐
Identity（身份認證）
統一處理用戶認證、角色權限和租戶隔離。所有入站請求在此校驗。確保租戶 A 無法訪問租戶 B 的數據。

缺少時：每個域各自實現 Auth，標準不一，安全漏洞難以統一修復。
📨
Event Bus（事件總線）
域間異步通信的通道，解耦發布方與訂閱方。ExternalSignal → 調度引擎，DispatchResult → 遙測域 / 結算域。

缺少時：外部信號域必須直接調用調度引擎，形成硬耦合，域邊界被打破。
⚙️
Tenant Config（租戶配置）
存儲各租戶的運行配置：啟用哪些用例、哪些市場連接器、哪種網關模式、分成比例。純配置讀取，無業務邏輯。

缺少時：新增租戶需要改多個域的硬編碼配置。
🧰
Shared Layer（公共工具）
跨域共用的標準數據結構定義（ExternalSignal / DispatchCommand / DispatchResult / RevenueRecord）、時區工具、日誌工具、通用錯誤類型。
🗄️
Data Persistence（數據持久化）
共用的數據存儲基礎設施，為所有業務域提供持久化能力。
Layer 2 原則（本文件範圍）：
  • 所有域共用同一個物理數據庫
  • 每個域擁有自己的表，不得被其他域直接讀寫
  • 跨域數據訪問只能通過域 API，嚴禁跨域 JOIN
缺少此原則的後果：
  • 任意代碼跨域 JOIN，一張表改動破壞多個域
  • 牽一髮動全身，無法獨立部署或測試單個域
📌 Layer 3（Spec）定義具體的表結構、字段和數據庫技術選擇，不在本文件範圍內。
