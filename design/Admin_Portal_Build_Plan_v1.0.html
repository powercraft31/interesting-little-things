T0.1b Mock 數據生成策略（mock-data.js 核心規範）

時序數據（P3/P5 的圖表）禁止 hardcode JSON 陣列。必須寫 Data Generator 函數。

  • 太陽能發電：generateSolarCurve() — 以 Math.sin() 生成鐘型曲線（06:00 開始，12:00 峰值 3-5kW，18:00 歸零），加 ±5% 隨機擾動
  • 家庭負載：generateLoadCurve() — 雙峰曲線（07:00 早峰 + 18:00 晚峰），基底負載 0.5-0.8kW，峰值 2-3kW
  • 電池：generateBatteryCurve(tariffSchedule, pvCurve, loadCurve) — 根據費率和 PV/Load 計算充放電（off-peak 充，peak 放）
  • 電網：grid_kw = load_kw - pv_kw - battery_kw（能量守恆，不需獨立生成）
  • Dumb Baseline：baseline_grid = load_kw（沒有 PV 沒有電池 = 100% 電網供電）

這樣圖表曲線自然真實，代碼輕量，且可以通過改參數快速生成不同家庭的數據。

🚨 Mock 數據快取（Memoization）：所有 Data Generator（含隨機噪聲）只能在系統初始化（Page Load）時執行一次， 生成的時序陣列存入 DemoStore。後續頁面切換渲染圖表時，直接讀取緩存的陣列， 禁止重複調用 Math.random() 重新生成。 否則從 P3 切到 P4 再切回 P3 時，圖表曲線會「抽搐/變形」（因為 noise 不同），Demo 時會暴露數據是假的。

🚨 時區防禦：所有 24 小時圖表（P3/P5）的 X 軸，禁止使用 JS Date 物件的時間戳。 必須生成純字串陣列 ['00:00', '00:15', '00:30' ... '23:45'] 作為 ECharts 的 X 軸 category。 原因：Demo 可能在亞洲/澳洲的瀏覽器上打開，JS Date 會套用本地時區，導致 Tarifa Branca 尖峰時段（17:00-20:00）在圖表上錯位。 用純字串可以確保在任何國家打開，時段都釘在正確位置。

🚨 Tarifa Branca 基準時段（hardcode 在 mock-data.js 頂部）：
· Peak（尖峰）：17:00–20:00
· Intermediate（中高）：16:00–17:00 及 20:00–21:00
· Off-peak（離峰）：其餘時間（21:00–16:00）
Data Generator 的充放電邏輯必須嚴格對齊這個時段——off-peak 充電、peak 放電。 Claude 不得自行猜測或修改這些時段。

🚨 節費計算邏輯（R$）必須加權費率：
savings_brl = Σ ((baseline_grid_kw[t] - actual_grid_kw[t]) × tariff_price[t] × 0.25)（15分鐘粒度 = 0.25 小時）
不能簡單用 kW 差額。半夜充電增加了電網進口（kW 差額為負），但費率便宜（R$0.41/kWh）； 白天放電減少了電網進口（kW 差額為正），費率昂貴（R$0.89/kWh）。 整體套利利潤 = 昂貴時段省的錢 - 便宜時段多花的錢 = 正值。 P6 的節費效果圖表必須用這個加權公式。

⚠️ ECharts 響應式防雷（charts.js 必須遵守）： 所有 ECharts 實例必須綁定 ResizeObserver 到其容器元素上。 當容器寬高變化時（Sidebar 收合/展開、視窗調整大小），自動呼叫 chart.resize()。 建立一個統一的 createChart(containerId, option) 工廠函數，內建 ResizeObserver 綁定， 所有頁面的圖表都必須通過這個工廠創建，禁止直接 echarts.init()。

🚨 致命雷區：ECharts + display:none 零像素陷阱： SPA 使用 display: none/block 切換頁面。如果在 display: none 狀態下初始化 ECharts， 容器寬高為 0，圖表會縮成一坨或完全不渲染。
鐵律：createChart() 工廠必須實現延遲初始化——只在頁面切換為 display: block 之後， 通過 requestAnimationFrame 觸發圖表的初始化或 chart.resize()。 Router 的頁面切換回調中必須調用 activatePageCharts(pageId)。

🚨 ECharts 單例模式（Singleton）：createChart() 在執行 echarts.init(dom) 前， 必須先檢查 echarts.getInstanceByDom(container)。 如果已存在實例，直接複用並調用 .setOption() + .resize()， 禁止重複 init。否則控制台會噴 "There is a chart instance already initialized on the dom" 警告， 且每次頁面切換都會創建新實例導致內存洩漏。

// DemoStore 極簡 API window.DemoStore = { get(key) { return JSON.parse(sessionStorage.getItem(key)); }, set(key, val) { sessionStorage.setItem(key, JSON.stringify(val)); }, reset() { sessionStorage.clear(); } }; // 使用場景 // P4: DemoStore.set('targetModeDistribution', { self: 10, arb: 25, peak: 12 }); // P3: const modes = DemoStore.get('targetModeDistribution') || DEFAULT_MODES;

⚠️ 執行注意事項

  • Mock 數據一致性：所有頁面的 mock 數據必須統一在 mock-data.js 中管理。P1 說「47 台設備」，P2 的設備列表就必須有 47 筆記錄。P6 的 self_consumption 96.8% 必須與 P3 的跨家庭表對應。
  • 巴西本地化：貨幣格式 R$ 1.234,56（巴西用逗號做小數點、點做千位分隔符）。日期格式 DD/MM/YYYY。時間格式：24 小時制（HH:mm），禁止 AM/PM。語言：英文 UI + 巴西葡語地名（São Paulo / Rio de Janeiro）。
  • ECharts 版本：使用 ECharts 5.x CDN，https://cdn.jsdelivr.net/npm/echarts@5/dist/echarts.min.js
  • 不需要做的事：後端 API / 數據持久化（DemoStore 用 sessionStorage 即可）/ 用戶登入 / 真正的 QR 掃碼 / i18n 多語言 / Admin 頁面的移動端適配（但 Wizard 必須支持平板）
  • ECharts 鐵律：所有圖表必須通過 createChart() 工廠創建，內建 ResizeObserver → chart.resize()。禁止直接 echarts.init()。
  • 時序數據鐵律：P3/P5 的圖表 mock 數據必須用 Data Generator 函數（Math.sin() + noise）生成，禁止 hardcode 96×5 的 JSON 陣列。
  • Phase 執行策略：Phase 0+1 建議合併為一個 Claude Code session（定調）。Phase 2-6 可順序或按需執行。Phase 7 必須在 P1-P6 全完成後執行。
