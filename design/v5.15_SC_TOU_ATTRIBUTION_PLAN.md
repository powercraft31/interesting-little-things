# SOLFACIL VPP v5.15 -- SC/TOU Attribution & 5-min Telemetry Plan

> 2026-03-07 Design Document -- Plan of Record
> **Status**: DESIGN REVIEW (no code changes yet)

---

## 1. Version Overview & Objectives

v5.15 replaces the fake SC/TOU/PS savings split (hardcoded 55%/30%/15% ratios in
`get-performance-savings.ts`) with **real physical attribution** based on:

1. **5-minute telemetry** (`asset_5min_metrics`) -- new partitioned table for high-resolution energy data
2. **Dispatch mode tracking** (`dispatch_records.target_mode`) -- tracks what mode each ESS was actually operating in
3. **Mode-attributed SC/TOU savings** computed daily by M4 billing job

### Why This Matters

- Current `get-performance-savings.ts` lines 57-59 apply `* 0.55 / * 0.30 / * 0.15` multipliers to total savings -- these are fabricated, not physical
- Real attribution requires knowing: (a) what the battery was doing at each moment, (b) what energy flows occurred
- 5-min granularity captures intra-hour mode switches that hourly data loses

## 2. Business Definitions

### Self-Consumption (SC) Mode
Battery stores excess PV for later self-use, reducing grid imports. Savings come from avoided grid purchases.

**SC contribution** (per 5-min window):
```
sc_contribution = (pv_energy_kwh - grid_export_kwh) * rate(hour)
```
- `pv_energy_kwh`: PV generation in this 5-min window (from `asset_5min_metrics`)
- `grid_export_kwh`: grid export in this 5-min window (from `asset_5min_metrics`)
- `rate(hour)`: Tarifa Branca rate for the hour (from `tariff_schedules`)

### Time-of-Use Arbitrage (TOU) Mode
Battery charges during off-peak (cheap) hours and discharges during peak (expensive) hours.

**TOU contribution** (per 5-min window):
```
tou_contribution = bat_discharge_kwh * rate(hour) - bat_charge_from_grid_kwh * rate(hour)

where:
  bat_charge_from_grid_kwh = max(0, bat_charge_kwh - max(0, pv_energy_kwh - load_kwh))
```
- `bat_discharge_kwh`: battery discharge in this 5-min window (from `asset_5min_metrics`)
- `bat_charge_kwh`: battery charge in this 5-min window (from `asset_5min_metrics`)
- Grid-charged energy = total charge minus what PV surplus could have provided
- Revenue = value of discharged energy minus cost of grid-charged energy

### Peak Shaving (PS) Mode
Battery discharges during demand peaks to keep site demand below contracted threshold. **Not implemented in v5.15** -- requires `contracted_demand_kw` and demand charge rate. Pre-work columns added only.

## 3. Why 5-Minute Granularity

### Problem with Hourly Data
The current pipeline aggregates telemetry to hourly in `asset_hourly_metrics`. Within a single hour, a battery may:
- Operate in SC mode for 30 minutes (PV excess → battery)
- Switch to TOU mode for 30 minutes (battery → grid peak discharge)

Hourly aggregation **loses this mode boundary**. The energy flows get mixed, making accurate attribution impossible.

### 5-Minute Resolution
- MQTT raw data arrives every ~5 seconds
- 5-minute aggregation captures mode switches while keeping storage manageable
- At 50,000 devices x 288 records/day = **14.4M rows/day**
- 30-day retention = ~432M rows -- requires partitioning strategy

### Data Loss Without 5-min
Without sub-hourly granularity, we would need to either:
1. Assume entire hours are single-mode (inaccurate for mixed hours)
2. Apply proportional allocation (back to fake ratios)

Neither is acceptable. 5-min windows + dispatch mode JOIN gives physical truth.

## 4. Architecture Decisions

### Decision 1: SC/TOU Attribution Source -- `dispatch_records` (NOT `trade_schedules`)

**Rationale:** Plan != execution. `trade_schedules` contains what M2 *planned* to do; `dispatch_records` contains what M3 *actually dispatched*. Network disconnection or EMS failure means a schedule may never have been dispatched.

**Change:** Add `target_mode VARCHAR(50)` to `dispatch_records`.
- Valid values: `'self_consumption'` | `'peak_valley_arbitrage'` | `'peak_shaving'`
- M3 DR Dispatcher writes this when dispatching a command
- M4 JOINs `asset_5min_metrics` with `dispatch_records` by time window

### Decision 2: `asset_5min_metrics` Partitioning -- PARTITION BY RANGE (daily)

**Rationale:** At 14.4M rows/day, traditional `DELETE`-based cleanup causes:
- WAL bloat (each deleted row generates WAL)
- VACUUM lock risk (autovacuum competes with writes)
- I/O amplification (dead tuple reclamation)

This is a **P0 performance disaster** for production.

**Solution:** PostgreSQL native `PARTITION BY RANGE (recorded_at)` with daily partitions.
- Cleanup: `DROP PARTITION` per day -- O(1) metadata operation, no lock, no WAL
- Retention: 30 days
- Partition creation: automated via daily cron (create tomorrow's partition)

### Decision 3: `allow_export` Placement -- `assets` Table (NOT `vpp_strategies`)

**Rationale:** Export permission is a physical/regulatory constraint from the grid interconnection agreement, not a software strategy option. It belongs on the physical asset, not a strategy configuration.

**Usage:**
- M2 reads `allow_export` as a hard constraint when generating schedules
- M3 enforces `net_power <= 0` when `allow_export = false` (double protection at dispatch layer)

### Decision 4: PS Pre-work (Additive Only, for v5.16)

Add `contracted_demand_kw REAL` to `homes` table -- demand is a site-level contract value, not an asset property.

Demand charge rate goes into `tariffs` table (NOT `homes` or `assets`) -- single responsibility principle. Tariff data belongs with tariff data.

**PS savings calculation is NOT in v5.15 scope.** Only the schema columns are added.

### Decision 5: Aggregator Bug Fix (P0)

**Bug:** Current `telemetry-aggregator.ts` uses factor `1.0/4` in its SQL (line 36-41), assuming 15-minute reporting intervals. Actual Xuheng reporting interval is **5 minutes** (confirmed from MQTT traffic analysis).

**Impact:** All energy values (kWh) in `asset_hourly_metrics` are **inflated by 3x** (dividing by 4 instead of 12).

**Fix:**
1. New `telemetry-5min-aggregator.ts` runs every 5 min, writes to `asset_5min_metrics`
2. Existing `telemetry-aggregator.ts` changes source from `telemetry_history` to `asset_5min_metrics` and fixes factor (no longer needed -- 5-min metrics are already in kWh)

### Decision 6: Partition Pre-creation Strategy (Midnight Crash Prevention)

**Problem:** PostgreSQL does NOT auto-create future partitions. When the clock crosses midnight, if tomorrow's partition doesn't exist, M1's INSERT throws `no partition of relation "asset_5min_metrics" found for row` -- crashing the entire telemetry pipeline.

**Solution:**
1. **Initial migration** pre-creates partitions for current day + next 30 days
2. **Daily cron at 23:00 UTC** pre-creates the next **3 days** of partitions (72-hour buffer against cron failure)
3. All partition DDL uses `CREATE TABLE IF NOT EXISTS` for idempotency

See [10_DATABASE_SCHEMA_v5.15.md](./backend_architecture/10_DATABASE_SCHEMA_v5.15.md) section 8.2 for full SQL.

### Decision 7: UTC-First Time Window Rule (BRT Billing Alignment)

**Problem:** Partition boundaries are UTC midnight-to-midnight. BRT = UTC-3, so a single BRT billing day spans two UTC partitions (BRT day starts at UTC 03:00). Naive BRT-based queries in M4 cause cross-partition scans and performance regression at scale.

**Rule:** All time window queries in M4 use UTC. BRT offset (+3h) is applied only at the presentation layer (BFF/frontend).

**M4 billing window formula:**
```sql
-- "Yesterday in BRT" as a UTC range:
WHERE recorded_at >= (CURRENT_DATE - 1 + INTERVAL '3 hours')   -- yesterday 03:00 UTC
  AND recorded_at <  (CURRENT_DATE     + INTERVAL '3 hours')   -- today 03:00 UTC
```

This ensures partition pruning works correctly and each BRT-day query touches at most 2 UTC partitions.

### Decision 8: NULL Fallback Handling for Additive Columns (UNASSIGNED Bucket)

**Problem:** Additive-only migration means `dispatch_records.target_mode` and `assets.allow_export` are NULL for all existing rows. These NULLs will propagate to M4 attribution logic and M2/M3 scheduling unless handled.

**Solution:**

**`assets.allow_export`** (NULL for existing rows):
- DDL: `DEFAULT false` (new rows safe)
- M2 schedule generator: `COALESCE(allow_export, false)` -- treat NULL as no-export
- M3 command dispatcher: `COALESCE(allow_export, false)` -- net_power constraint applies

**`dispatch_records.target_mode`** (NULL for all pre-v5.15 rows):
- M4 attribution: `COALESCE(target_mode, 'UNASSIGNED')` as effective mode
- **UNASSIGNED bucket rules:**
  - Savings are computed but NOT written to `sc_savings_reais` or `tou_savings_reais`
  - They contribute to `client_savings_reais` (total) only
  - This ensures legacy data before v5.15 doesn't cause TypeError or zero out savings
  - M4 MUST log a warning when UNASSIGNED windows are encountered (for monitoring)

## 6. Formula Derivations

### SC Savings (per 5-min window, mode = 'self_consumption')

```
Variables:
  pv_energy_kwh    -- asset_5min_metrics.pv_energy_kwh
  grid_export_kwh  -- asset_5min_metrics.grid_export_kwh
  rate             -- getRateForHour(hour, schedule)  [from tariff_schedules]

Formula:
  sc_contribution = (pv_energy_kwh - grid_export_kwh) * rate

Interpretation:
  PV energy that was consumed locally (not exported) displaces grid import.
  The value is the avoided grid cost at the applicable tariff rate.
```

### TOU Savings (per 5-min window, mode = 'peak_valley_arbitrage')

```
Variables:
  bat_discharge_kwh      -- asset_5min_metrics.bat_discharge_kwh
  bat_charge_kwh         -- asset_5min_metrics.bat_charge_kwh
  pv_energy_kwh          -- asset_5min_metrics.pv_energy_kwh
  load_kwh               -- asset_5min_metrics.load_kwh
  rate                   -- getRateForHour(hour, schedule)

Derived:
  pv_surplus_kwh         = max(0, pv_energy_kwh - load_kwh)
  bat_charge_from_grid   = max(0, bat_charge_kwh - pv_surplus_kwh)

Formula:
  tou_contribution = bat_discharge_kwh * rate - bat_charge_from_grid * rate

Interpretation:
  Revenue from discharging at current rate, minus cost of grid-charged energy
  at the rate when charging occurred.
  PV-charged energy is free, so only grid portion counts as cost.
```

### Daily Totals

```
sc_savings_reais  = SUM(sc_contribution)  for all 5-min windows where target_mode = 'self_consumption'
tou_savings_reais = SUM(tou_contribution) for all 5-min windows where target_mode = 'peak_valley_arbitrage'
```

Written to `revenue_daily.sc_savings_reais` and `revenue_daily.tou_savings_reais`.

### Edge Cases

- No dispatch_records for a 5-min window (battery idle) -- skip attribution
- Multiple dispatch_records overlapping a 5-min window -- use the one with highest time overlap
- allow_export=false with PV surplus -- verify net_power <= 0 enforcement
- Empty day (no telemetry) -- sc=0, tou=0, both NULL-safe
- First day after migration -- asset_5min_metrics empty, billing job gracefully returns 0
- Cron failure for 2 consecutive days -- 72h partition buffer prevents crash on day 3

## 10. API Contract Change

### `GET /api/performance/savings` -- Response Change

**Before (v5.14):**
```json
{
  "home": "Casa 001",
  "total": 150.00,
  "alpha": 75.2,
  "sc": 82.50,    // FAKE: total * 0.55
  "tou": 45.00,   // FAKE: total * 0.30
  "ps": 22.50     // FAKE: total * 0.15
}
```

**After (v5.15):**
```json
{
  "home": "Casa 001",
  "total": 150.00,
  "sc": 98.30,     // REAL: SUM(revenue_daily.sc_savings_reais)
  "tou": 51.70,    // REAL: SUM(revenue_daily.tou_savings_reais)
  "ps": null        // NOT YET COMPUTED (v5.16)
}
```

**Breaking changes:**
- `alpha` field **removed** (replaced by real sc/tou in v5.14's actual_savings/optimization_efficiency)
- `sc` and `tou` are now real values from DB (not derived from total)
- `ps` returns `null` (not computed until v5.16)
