# v5.22 Requirements: Two-Phase set_reply + Backfill Mechanism

## Context

Protocol v1.7 introduces two changes vs v1.5:
1. **set_reply two-phase response**: `accepted` → `success`/`fail` (was: single `success`/`fail`)
2. **Backfill mechanism**: Platform requests missed data when gateway reconnects

This is a DESIGN TASK. Output design docs + implementation plan. Do NOT write code yet.

## Requirement 1: Two-Phase set_reply

### Protocol Change (v1.7 §3.9)

Gateway now sends TWO set_reply messages after config/set:
- **First**: `result = "accepted"` — gateway validated config, started writing to device registers
- **Second**: `result = "success"` or `result = "fail"` — device write + read-back verification complete

### Current Code (what needs to change)

**File: `backend/src/iot-hub/handlers/command-tracker.ts` (140 lines)**
- `handleSetReply()` at line 78: finds latest `dispatched` command and sets result directly
- Problem: It matches `WHERE result = 'dispatched'` only
- Fix needed: 
  - `accepted` → UPDATE result='accepted' (do NOT set resolved_at)
  - `success` → UPDATE result='success', resolved_at=NOW() — match WHERE `result IN ('dispatched','accepted')`
  - `fail` → UPDATE result='failed', resolved_at=NOW() — match WHERE `result IN ('dispatched','accepted')`
  - Add logging to distinguish phase 1 vs phase 2

**File: `backend/src/dr-dispatcher/services/command-dispatcher.ts` (178 lines)**
- `runTimeoutCheck()` at line 163: times out commands WHERE `result = 'dispatched'`
- Fix needed: Change to `result IN ('dispatched', 'accepted')` — accepted can also timeout if device write hangs

**File: `backend/src/bff/handlers/get-gateway-schedule.ts` (209 lines)**
- syncStatus mapping near line ~50: maps result → frontend status
- Currently: dispatched/pending → "pending"
- Fix needed: Add `accepted` → "pending" (same visual state)

**File: `backend/src/iot-hub/services/command-publisher.ts` (147 lines)**
- No changes needed. It only publishes and sets `dispatched`. The reply path is handled by command-tracker.

**DB: `device_command_logs`**
- result column: varchar(20), no CHECK constraint on values — "accepted" fits, no DDL needed
- Partial index `idx_dcl_dispatched_set`: only covers `dispatched` — may need new index for `accepted` commands too
- Consider: `CREATE INDEX idx_dcl_accepted_set ON device_command_logs (created_at) WHERE result = 'accepted' AND command_type = 'set'`

**Frontend: `frontend-v2/js/p2-devices.js`**
- No changes needed. BFF maps `accepted` → "pending", frontend already handles "pending" state.

### SSE Notification

When `accepted` is received, should we fire a pg_notify so the frontend sees "Accepted..." status in real-time?
- Recommendation: YES, add `pg_notify('command_status', JSON)` in command-tracker after accepted/success/fail
- Frontend SSE handler can show intermediate status: "Configuração aceita, aplicando..." → "Sincronizado ✓"

## Requirement 2: Backfill Mechanism (data/get_missed)

### Protocol (v1.7 §5)

- Platform → Gateway: `platform/ems/{clientId}/data/get_missed` with `{ start, end }` (Unix ms)
- Gateway → Platform: `device/ems/{clientId}/data/missed` — one message per data point, 100ms apart
- Format identical to normal `device/ems/{clientId}/data`
- Gateway stores max 10,000 records locally
- If no data: returns single message with `data: {}`

### Design Decisions (Alan-approved)

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Trigger | Heartbeat resumes after gap | Passive detection (Method A) |
| Delay | 30 seconds after first heartbeat | Let gateway complete init cycle |
| Chunk size | 30 minutes per request | 5-min upload = 6 msgs/chunk = 0.6s |
| Cool-down | 20 seconds between chunks | Let normal telemetry pass through |
| Topic | Separate `data/missed` handler | Don't mix with live `data` pipeline |

### Architecture: What to build

**1. Reconnect Detection (modify `heartbeat-handler.ts`, 36 lines)**

Current: just updates `last_seen_at` and `status = 'online'`.
Add:
- Read previous `last_seen_at` BEFORE updating
- Calculate gap = NOW() - previous_last_seen_at
- If gap > threshold (e.g., 2 minutes) AND status was NOT 'online':
  - Emit a reconnect event (in-memory or pg_notify)
  - Store `reconnected_at` timestamp for backfill-requester to pick up
- Option A: Write `gateways.backfill_requested_at = NULL, backfill_gap_start = old_last_seen_at, backfill_gap_end = NOW()` columns
- Option B: Insert into a `backfill_requests` queue table
- Recommendation: Option B (cleaner separation, queue semantics)

**2. New table: `backfill_requests`**

```sql
CREATE TABLE backfill_requests (
  id BIGSERIAL PRIMARY KEY,
  gateway_id VARCHAR REFERENCES gateways(gateway_id),
  gap_start TIMESTAMPTZ NOT NULL,     -- old last_seen_at
  gap_end TIMESTAMPTZ NOT NULL,       -- reconnect time
  current_chunk_start TIMESTAMPTZ,    -- progress tracking
  status VARCHAR(20) DEFAULT 'pending', -- pending/in_progress/completed/failed
  created_at TIMESTAMPTZ DEFAULT NOW(),
  completed_at TIMESTAMPTZ
);
```

**3. New service: `backfill-requester.ts`**

- Runs on a timer (every 10 seconds, like CommandPublisher)
- Polls `backfill_requests WHERE status = 'pending' OR status = 'in_progress'`
- For pending: wait until `created_at + 30 seconds` has passed (delay)
- For in_progress: check if 20 seconds have passed since last chunk sent
- Publishes `platform/ems/{cid}/data/get_missed` with `{ start, end }` for 30-min chunk
- Updates `current_chunk_start` to track progress
- When all chunks done: `status = 'completed'`
- Uses `GatewayConnectionManager.publishToGateway()` for MQTT publish

**4. MQTT subscription (modify `gateway-connection-manager.ts`, 333 lines)**

- Add topic: `device/ems/${cid}/data/missed` to subscription list (6th topic)
- Add routing: `topic.endsWith('/data/missed')` → `this.handlers.onMissedData()`

**5. New handler: `missed-data-handler.ts`**

- Receives `data/missed` messages
- If `data` is empty object `{}`: log "no data for this chunk", continue
- If `data` has content: reuse FragmentAssembler's parsing logic
- Key difference from live path: 
  - Use `ON CONFLICT (asset_id, recorded_at) DO NOTHING` to avoid duplicates
  - This requires a UNIQUE INDEX on telemetry_history

**6. DB: UNIQUE INDEX on telemetry_history**

```sql
CREATE UNIQUE INDEX idx_telemetry_unique_asset_time 
  ON telemetry_history (asset_id, recorded_at);
```

Note: telemetry_history is partitioned (RANGE on recorded_at). The unique index must be created on each partition, or use a constraint that includes the partition key.

**7. FragmentAssembler refactoring**

Current: `writeTelemetry()` is a private method that does parsing + INSERT.
Needed: Extract parsing logic into a shared pure function that both:
- FragmentAssembler.writeTelemetry() calls (live path)  
- missed-data-handler calls (backfill path)

The INSERT statement differs:
- Live: plain INSERT
- Backfill: INSERT ... ON CONFLICT DO NOTHING

Design decision: Extract `parseTelemetryPayload(acc: Accumulator): ParsedTelemetry` as a standalone export. Both paths call it, then do their own INSERT.

## Module Boundary Rules (IRON LAW)

- backfill-requester.ts lives in `iot-hub/services/` (it's M1 infrastructure)
- missed-data-handler.ts lives in `iot-hub/handlers/` (same as other MQTT handlers)
- M3 (dr-dispatcher) MUST NOT be involved in backfill — it's pure M1
- DB is the only bridge between M1 and M3
