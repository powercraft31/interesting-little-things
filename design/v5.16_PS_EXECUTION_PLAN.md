## 1. Objective

Replace `ps: null` placeholder in `get-performance-savings` with real Peak Shaving savings attribution.

**Peak Shaving (PS)** reduces demand charges (R$/kVA) by discharging batteries during peak load windows to keep site demand below the contracted threshold (`contracted_demand_kw`). Unlike SC/TOU which reduce energy charges (R$/kWh), PS targets the single highest 15-minute demand window in a billing month.

### Architecture Boundary

| Responsibility | Owner | Notes |
|---------------|-------|-------|
| Real-time demand control | **Gateway (EMS box)** | Gateway guarantees 15-min kVA stays below limit; platform does NOT intervene |
| PS mode dispatch | **Platform M3** | Sends `{ mode: 'peak_shaving', peak_limit_kva }` to gateway |
| PS mode scheduling | **Platform M2** | Decides when to activate PS based on demand risk |
| PS savings attribution | **Platform M4** | Counterfactual: "what would demand have been without PS?" |
| Load shedding (DO) | **Gateway hardware** | Relay cutoff when battery alone is insufficient |

## 2. Phase 1: DB Schema Migration

**Files changed:** `migration_v5.16.sql`, `ddl_base.sql`

### Tables Changed

#### 2.1 `telemetry_history` -- +2 Columns

```sql
ALTER TABLE telemetry_history
  ADD COLUMN IF NOT EXISTS do0_active BOOLEAN,
  ADD COLUMN IF NOT EXISTS do1_active BOOLEAN;
```

DO (Digital Output) relay state from gateway. NULL when device doesn't send dido messages.

#### 2.2 `tariff_schedules` -- +2 Columns

```sql
ALTER TABLE tariff_schedules
  ADD COLUMN IF NOT EXISTS demand_charge_rate_per_kva NUMERIC(8,4),
  ADD COLUMN IF NOT EXISTS billing_power_factor        NUMERIC(3,2) DEFAULT 0.92;
```

**HARD CONSTRAINT:** `billing_power_factor` stays in `tariff_schedules` -- it is a commercial contract value (billing convention), not a physical asset property.

#### 2.3 `revenue_daily` -- +4 Columns

```sql
ALTER TABLE revenue_daily
  ADD COLUMN IF NOT EXISTS ps_savings_reais         NUMERIC(10,2),
  ADD COLUMN IF NOT EXISTS ps_avoided_peak_kva      NUMERIC(8,3),
  ADD COLUMN IF NOT EXISTS do_shed_confidence       VARCHAR(10),
  ADD COLUMN IF NOT EXISTS true_up_adjustment_reais NUMERIC(10,2);
```

### HARD CONSTRAINT: NO `asset_15min_demand` Table

Use `date_bin('15 minutes', ...)` in M4 SQL to bin 5-min windows into 15-min demand windows. The existing `asset_5min_metrics` partitioned table is fast enough -- no intermediate table needed.

## 4. Phase 3: M2 Optimization Engine + M3 DR Dispatcher

### M2: schedule-generator.ts

- Read `homes.contracted_demand_kw` for each active asset
- Evaluate peak risk: if recent demand > `contracted_demand_kw * 0.85` (85% threshold)
- Generate `trade_schedules` rows with `target_mode = 'peak_shaving'`, `action = 'discharge'`
- Skip PS slot generation when `contracted_demand_kw IS NULL` (log warning)

See: [02_OPTIMIZATION_ENGINE_v5.16.md](./backend_architecture/02_OPTIMIZATION_ENGINE_v5.16.md)

### M3: command-dispatcher.ts

- When `target_mode = 'peak_shaving'`, emit MQTT payload:
  ```json
  { "mode": "peak_shaving", "peak_limit_kva": <contracted_demand_kw / billing_power_factor> }
  ```
- Set `dispatch_records.target_mode = 'peak_shaving'`

See: [03_DR_DISPATCHER_v5.16.md](./backend_architecture/03_DR_DISPATCHER_v5.16.md)

## 5. Phase 4: M4 Market Billing -- Two New Functions

### 5.1 `runDailyPsSavings(pool, brtWindowStart, brtWindowEnd)`

1. Identify PS-active 5-min windows (most recent dispatch = `peak_shaving`)
2. Reconstruct counterfactual kW: `actual_grid_kW + bat_discharge_kW + DO_load_shed_kW`
3. Bin into 15-min demand windows using `date_bin('15 minutes', ..., '2026-01-01 03:00:00Z')`
4. Convert kW to kVA: `counterfactual_kw / billing_power_factor`
5. Find daily peak kVA, compute `avoided_kva = MAX(0, peak_kva - contracted_demand_kw)`
6. Daily savings: `avoided_kva * demand_charge_rate_per_kva / days_in_month`
7. UPSERT into `revenue_daily.ps_savings_reais`, `ps_avoided_peak_kva`, `do_shed_confidence`

### 5.2 `runMonthlyTrueUp(pool, billingMonth)`

1. Rescan full month's `asset_5min_metrics` for PS-active windows
2. Reconstruct complete 15-min demand series for the month
3. Find true `monthly_peak_kva` (highest 15-min window)
4. Compute `true_ps_savings = avoided_kva * demand_charge_rate_per_kva`
5. Compute `true_up = true_ps_savings - SUM(daily_provisionals)`
6. **INSERT** new row for 1st of month with `true_up_adjustment_reais` -- **NEVER UPDATE** historical rows

**Cron schedule:**
- `runDailyPsSavings`: runs after SC/TOU attribution in existing daily billing cron
- `runMonthlyTrueUp`: new cron entry, 1st of month at 04:00 UTC

See: [04_MARKET_BILLING_MODULE_v5.16.md](./backend_architecture/04_MARKET_BILLING_MODULE_v5.16.md)
