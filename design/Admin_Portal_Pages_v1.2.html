🎨 視覺設計原則

💹
Fintech 交易終端風格，非 SCADA 工業風
SOLFACIL 是金融科技公司（Financial Services），不是電廠。UI 的數據本質是「錢」，不是「瓦」。 採用大型 KPI 數字板（Big Numbers）+ 迷你走勢圖（Sparklines）取代圓形儀表板。 所有財務數字使用等寬字體（Monospace，JetBrains Mono），小數點嚴格對齊。 風格參考：Bloomberg Terminal / Stripe Dashboard。

例外：P3 Energy Behavior 和 P5 VPP & DR 保留工程圖表（功率曲線、能源流向）， 但使用 Fintech 的色彩和排版包裝，不用傳統電力 SCADA 的灰藍工業風。
🌗
按角色物理環境決定主題
SOLFACIL Admin（辦公室，盯螢幕 8 小時）：深色主題 (Dark Mode)。降低視覺疲勞，讓告警紅色和正向綠色極度凸顯。
Integrador（巴西屋頂/戶外施工）：高對比淺色主題 (Light Mode)。白底黑字粗邊框，陽光下可讀。Pilot 階段先預留，不首發。
Customer App（手機端）：Neobank 風格（參考 Nubank：大面積留白、圓角、品牌主色調）。預留，不首發。
技術實現：CSS 使用 theme variables（:root / [data-theme="light"]），不硬編碼顏色。
🎨
三組色彩語意（貫穿全部 6 頁）
綠/青 — 正面：省錢、發電、在線、達標、自發自用
橘/紅 — 負面：花錢、電網進口、離線、未達標、告警
藍/紫 — 中性操作：電池調度、系統動作、VPP 指令、模式切換
🔢
字體排版規範 v1.2 新增
雙字體策略：
· 標籤、說明、UI 文字 → Inter（無襯線，乾淨俐落）
· 所有數據、金額、百分比 → JetBrains Mono（等寬，精密感）

Tabular Nums（表格數字）：所有數據 class 強制加入 font-variant-numeric: tabular-nums;。 防止即時遙測數據跳動時欄位寬度抖動（例：1.23 kW → 9.45 kW 導致卡片/表格閃動）。

正負號強制：所有貨幣和收益數據必須帶正負符號： +R$ 145.00（賺錢/節省）vs -R$ 45.00（花費/損失）。 不能只依賴顏色區分——色盲用戶需要符號冗餘，Admin 掃視大量設備時大腦處理速度也更快。
💀
全局載入策略：Skeleton Loading（骨架屏） v1.2 新增
頁面載入和數據切換（切換安裝商、切換家庭）時，禁止使用居中 Spinner（轉圈圈）。 改用數字骨架屏（灰底色塊 + 脈衝動畫），營造「數據正在從終端串流過來」的速度感。

CSS 實作：@keyframes skeleton-pulse { 0% { opacity: 0.4; } 50% { opacity: 0.7; } 100% { opacity: 0.4; } }
骨架屏形狀應匹配實際數據欄位佈局（KPI 卡片 → 矩形骨架；圖表 → 面積骨架；表格 → 行骨架）。

精靈 Commissioning Wizard（開通精靈） v1.1 新增

安裝商的核心操作流程。Pilot Brief 要求培訓 <90 分鐘、安裝 <2 小時。 Wizard 消除了複雜配置的認知負擔——安裝商只需按步驟點擊「下一步」即可完成開通。

1 輸入 Home ID
→
2 掃描/輸入 Gateway SN
→
3 自動掃描子設備
→
4 通訊測試
→
5 Pass / Fail 報告
  • Step 1：輸入或掃描 Home ID（客戶家庭識別碼），自動關聯安裝商 org_id
  • Step 2：掃描 Gateway 序號 — 全螢幕相機 UI，掃描框置於畫面偏上半部（符合手持手機掃描逆變器條碼的自然角度）；支持手動輸入 fallback
  • Step 3：Gateway 自動掃描已連接的子設備（逆變器/電表/空調/充電樁），顯示發現列表，安裝商確認
  • Step 4：一鍵執行通訊測試——每台設備發送 ping 並等待回應，驗證雙向通訊正常
  • Step 5：結果報告：全部 Pass → 大面積綠色色塊 + 開通完成（自動記錄 commissioning 時間）；任一 Fail → 大面積紅色色塊 + 失敗設備和建議排查步驟

Wizard 完成後，結果自動寫入 Commissioning 追蹤面板和 P6 Performance 的 commissioning_time 指標。

v1.2 新增 戶外施工防護設計：安裝商在巴西屋頂/戶外操作，強光下螢幕可讀性差，手指可能出汗或戴手套。 Integrador 視圖下所有按鈕和觸控區域最小高度 48px（Google Material Design 標準）。 Wizard 各步驟的「下一步」按鈕使用全寬 (100%) + 高對比色，確保強光下可辨識。

HEMS Control 優化規則控制

需求依據 — Pilot Brief "Admin can set rules for HEMS optimization effectively, including for VPP and Energy Trading, eg: rules are set and they are proven to work as evidenced by system-level & device-level performance"
"Ability to load tariffs & economic optimization rules to feed the simulations (whether you do it or we do it)"
"HEMS optimization engine (self-consumption, Time of Use arbitrage, peak shaving)"
"Tarifa Branca can be loaded into the system in <2 days"

操作 優化模式設定

三個 target_mode，對應 Pilot Brief 明確列出的三種 HEMS 優化場景。可對單設備或批量設定。

  • self_consumption（自發自用優化）：最大化光伏自用比例，最小化電網進口。Pilot 驗收指標：>98%
  • peak_valley_arbitrage（TOU 峰谷套利）：off-peak 充電、peak 放電，利用 Tarifa Branca 時段差價。Pilot 驗收：correct charge/discharge timing
  • peak_shaving（峰值管理）：限制峰時最大功率，避免需量超約罰款。核心場景：商業客戶

操作 批量模式下發

支持按條件批量下發模式切換：

  • 按安裝商（org_id）
  • 按地區 / 配電區域
  • 按設備類型
  • 按當前模式（例：把所有 self_consumption 切換到 peak_shaving）

影響範圍預覽（確認前必顯示）：

  • 「此操作將變更 N 台設備的 target_mode」
  • 按設備類型 / 安裝商 / 地區分組的影響統計
  • 離線設備標記（這些設備將在恢復連線後收到新規則）
  • 確認對話框 → 執行

設計決策：平台不做 SoC 保護攔截。SoC 管理在設備層（閘道器/固件），設備會自動在 SoC 低於安全值時忽略調度指令。 平台只負責「下發什麼模式」，不判斷「設備能不能執行」。

操作 Tarifa Branca 費率配置

Pilot Brief 要求費率可載入、<2 天完成。

  • 費率結構：peak / intermediate / off-peak 三個時段 + 對應電價 (R$/kWh)
  • 按配電公司 (DISCO) 區分（CEMIG / Enel / Neoenergia 費率不同）
  • 生效日期設定（費率定期調整）
  • 載入方式：手動輸入 或 CSV 上傳
disco_name peak_price_brl intermediate_price_brl offpeak_price_brl peak_hours effective_date

  • 規則衝突檢測（例：同一設備被指派了矛盾的模式）

視圖 VPP 聚合容量

把所有 Pilot 家庭的電池聚合成一個虛擬電廠，展示整體可調度能力。

total_capacity_kwh available_capacity_kwh aggregate_soc_% max_discharge_kw max_charge_kw devices_dispatchable

v1.1 預留 Net Dispatchable Capacity（淨可調度容量）： 扣除家庭即時負載後的真實可調度電量。需依賴智慧電表即時數據 + 負載預測模型。 Pilot Brief 要求「load forecast adapts within 2 weeks」——Pilot 前兩週此數字不準確。 UI 預留位置，待 load forecast 收斂後點亮。Pilot 階段先展示 Raw Battery Capacity。

操作 DR 事件觸發

Pilot Brief 說 SOLFACIL 會做 "proxy test by controlling loads & logging events"。Admin 需要能手動觸發模擬 DR 事件。

  • 事件類型：放電（Grid Export）/ 充電（Grid Import）/ 負載削減（Demand Limiting）
  • 目標功率 (kW) 或目標能量 (kWh)
  • 持續時間
  • 參與設備選擇：全部 / 按條件篩選（地區、安裝商、設備類型）
  • 執行：立即 / 排程（day-ahead）

📋 設計決策記錄 (Design Decision Log)

記錄所有被提議並經過評估的設計建議，包含採納和否決的理由。確保團隊理解每個決策背後的邏輯。

❌ 否決：Haptic Feedback（震動回饋）— P2 Commissioning Wizard

提議：Step 4 通訊測試 Pass 時觸發手機短震動，Fail 時長震動，讓安裝商在強光下看不清螢幕時憑手感判斷。
否決理由：Web Vibration API 在 iOS Safari 上完全不支持，Android 上支持但需要用戶交互觸發。 安裝商使用什麼設備（iPhone / Android / 平板）我們無法控制。投入產出比太低。 替代方案：Pass/Fail 使用大面積綠/紅色塊（已在 v1.1 Wizard Step 5 實現），強光下色塊比震動更可靠。

❌ 否決：Hold to Confirm / Swipe to Execute — P4 批量下發 & P5 DR 觸發

提議：高風險操作（批量模式切換、DR 事件觸發）使用「長按 3 秒，按鈕進度條填滿」或「滑動以執行」的確認機制，類似加密貨幣錢包或證券 App。
否決理由（此建議被提出兩次，兩次均被否決）：
1. 規模不匹配：Pilot 階段只有 2-3 個家庭。「控制 5 萬台設備的風險」是未來的問題，不是今天的問題。3 台設備的批量下發，最壞情況是 3 台電池放電到 20%（設備層自動保護），財務損失接近零。
2. 平台不匹配：Admin Portal 是桌面端 Web 應用，Admin 用滑鼠操作。Swipe 和 Hold 是移動端 App 的交互模式（Apple Pay、加密錢包），在桌面端做這些動作是反直覺的。
3. SoC 保護在設備層：即使操作失誤，設備固件會在 SoC 低於安全值時自動忽略調度指令，不存在「一次錯誤批量下發導致嚴重財務損失」的場景。
替代方案：影響範圍預覽 + 確認對話框（已在 v1.1 P4 實現）。簡單、桌面端原生、所有用戶都懂。
未來考量：50,000 台規模時可考慮二次確認機制（如：輸入確認碼），但那是 Pilot 之後的事。

❌ 否決：基於 SoC 的策略攔截 — P4 批量下發

提議：批量下發 peak_valley_arbitrage 時，平台自動攔截 SoC 低於 20% 的設備，提示「150 台設備因 SoC 過低，將忽略此套利指令」。
否決理由：SoC 管理是設備層的職責，不是平台層的。 旭衡閘道器/設備固件會在 SoC 低於安全值時自動忽略充放電指令。 UEH（澳洲生產系統）也遵循同一原則：「下發放電到 20%，設備固件自己在 20% 停止」。 平台層做 SoC 攔截是職責越界——如果平台和設備都做 SoC 判斷，兩者的閾值不一致時反而會製造混亂。 設計原則：平台只負責「下發什麼模式」，不判斷「設備能不能執行」。SoC 保護是設備層的事。

❌ 否決：Net Dispatchable Capacity 首發 — P5 VPP 聚合容量

提議：P5 的可用容量應拆分為 Raw Battery Capacity 和 Net Dispatchable Capacity（扣除家庭即時負載後的真實可調度量）。
否決「首發」，但接受「預留」：
1. Pilot 規模問題：2-3 個家庭的聚合容量太小，淨可調度量的波動會非常大，反而顯得系統不穩定。50,000 台時才有統計意義。
2. 數據依賴問題：Net Dispatchable 需要即時負載預測。Pilot Brief 要求「load forecast adapts within 2 weeks」——Pilot 前兩週這個數字根本不準。展示一個不準的數字比不展示更糟。
執行方式：P5 先展示 Raw Battery Capacity；UI 預留 Net Dispatchable 位置（已在 v1.1 實現），待 load forecast 收斂後點亮。

✅ 採納：Commissioning Wizard (v1.1)

P2 新增安裝商開通精靈，5 步流程消除複雜配置的認知負擔。
採納理由：Pilot Brief 硬指標「commissioning <2 hours」+「training <90 minutes」直接支撐。 無 Wizard 的話，安裝商需要理解設備配置細節，培訓時間會顯著超標。

✅ 採納：Dumb Baseline 對照曲線 (v1.1)

P3 能源流向圖疊加「無優化基準線」虛線，差額面積用漸層綠色填充。
採納理由：Pilot Brief 原文使用「shadow benchmark vs. 'dumb' baseline」一詞。 SOLFACIL 自己定義了這個驗收標準（>70% alpha），前端必須直觀呈現。 面積填充比數字更有視覺衝擊力，特別是向投資人展示時。

✅ 採納：Fintech 交易終端風格 (v1.1) + 字體排版規範 (v1.2)

UI 採用 Bloomberg/Stripe 風格而非 SCADA 工業風；雙字體策略 + tabular nums + 正負號。
採納理由：SOLFACIL 是 Financial Services 公司，行業分類和文化決定了他們對「專業」的定義是金融終端，不是電廠控制室。 Tabular nums 防止即時數據跳動時 UI 抖動，是金融界面的基本功。正負號為色盲用戶提供冗餘信號。

✅ 採納：Skeleton Loading 骨架屏 (v1.2)

全局載入策略使用骨架屏替代 Spinner。
採納理由：現代 Web App 標配。骨架屏的感知載入速度比 Spinner 快（用戶看到佈局結構而非空白）， 實作成本低（CSS animation + 灰色色塊），且不影響頁面結構。

✅ 採納：按角色拆 Dark/Light 主題 (v1.1)

Admin Dark Mode / Integrador Light Mode / Customer Neobank 風格。
採納理由：不是從「好看」出發，而是從物理使用環境出發—— Admin 辦公室盯 8 小時（深色減疲勞）vs 安裝商巴西屋頂（淺色防反光）。 Pilot 階段先做 Admin Dark Mode，其他預留。
