### Step 1.4: Create `auth-login.ts` handler

**File:** `backend/src/bff/handlers/auth-login.ts` (NEW)

**Exports:** `createLoginHandler(servicePool: Pool)` → Express handler `(req, res)`

**Logic:**
1. Validate email + password in body
2. `SELECT u.*, uor.org_id, uor.role FROM users u JOIN user_org_roles uor ...` via service pool
3. Check `is_active`, bcrypt compare
4. `jwt.sign({ userId, orgId, role }, secret, { expiresIn: "24h" })`
5. Return `ok({ token, user: {...} })`

**Verify:**

**Commit:** `feat(v5.23): add auth-login handler (POST /api/auth/login)`

## Risk Assessment

| Risk | Severity | Phase | Mitigation |
|------|----------|-------|------------|
| `extractTenantContext()` relies on Lambda `event` type but auth middleware uses Express `req` | HIGH | 2 | Auth middleware overwrites `req.headers.authorization` to raw JSON BEFORE `wrapHandler()` converts to Lambda event. Handler path is unchanged. |
| Removing demo auth breaks existing tests | HIGH | 2 | Keep raw JSON path in `verifyTenantToken()`. Tests can still pass raw JSON auth headers. |
| `getServicePool()` not exposed from current codebase | MEDIUM | 1 | Check if `getServicePool` exists. If not, create it (same pattern as app pool, just with `solfacil_service` role). |
| bcrypt hash computation slow (~100ms per hash at 12 rounds) | LOW | 1 | Only used at login and user creation. Acceptable latency. |
| JWT secret in code fallback (`solfacil-dev-secret`) | LOW | 1 | Dev only. EC2 uses env var. Phase 2 migrates to Secrets Manager. |
| Frontend redirect loop if login.html itself tries to call API | LOW | 3 | `login.html` only calls `/api/auth/login` which is whitelisted. No auth guard on login page. |
| Concurrent login + token refresh race condition | N/A | - | No refresh token in Phase 1. Single token, single state. |

| Phase 3 | Restore hardcoded `getAuthHeader()` in `data-source.js`. Remove `login.html`. Remove logout button. |
