## Requirement 1: SSE Real-time Push (Backend → Frontend)

### Problem
Frontend has no way to know when backend data changes. Users must manually refresh to see new telemetry, device state, or gateway health updates.

### Solution: PostgreSQL NOTIFY + BFF SSE Endpoint + Frontend EventSource

```
M1 writes device_state/telemetry → pg_notify('telemetry_update', gatewayId)
                                         ↓
BFF SSE endpoint (LISTEN) → push event to connected frontends
                                         ↓
Frontend EventSource → auto re-fetch current view data
```

### Changes Required

#### 1.1 M1 — FragmentAssembler (2 lines)
- **File:** `backend/src/iot-hub/services/fragment-assembler.ts`
- **Where:** After `updateDeviceState()` succeeds in `writeTelemetry()`
- **What:** Execute `SELECT pg_notify('telemetry_update', $1)` with `clientId` (gatewayId)
- Also notify after `writeEmsHealth()` succeeds: `SELECT pg_notify('gateway_health', $1)`

#### 1.2 BFF — New SSE Endpoint (~40 lines)
- **File:** New `backend/src/bff/handlers/sse-events.ts`
- **Route:** `GET /api/events`
- **Behavior:**
  - Set headers: `Content-Type: text/event-stream`, `Cache-Control: no-cache`, `Connection: keep-alive`
  - Use PostgreSQL `LISTEN telemetry_update` and `LISTEN gateway_health`
  - On notification: send SSE event `data: { "type": "telemetry_update", "gatewayId": "WKRD..." }\n\n`
  - Heartbeat: send `:keepalive\n\n` every 30s to prevent proxy timeouts
  - On client disconnect: `UNLISTEN` and release DB connection
- **Add route in:** `backend/scripts/local-server.ts`

#### 1.3 Frontend — EventSource (~20 lines)
- **File:** `frontend-v2/js/p2-devices.js`
- **Where:** In `init()` method
- **Behavior:**
  - `new EventSource('/api/events')` (through BFF proxy)
  - On `message` event: if current view is Layer 1, re-fetch gateway list; if Layer 3 and `gatewayId` matches, re-fetch detail
  - On `error`: auto-reconnect (EventSource does this natively)
  - On page navigation away from P2: `eventSource.close()`

#### 1.4 File Server Proxy
- **File:** `/tmp/ashe_file_server/main.py`
- **What:** Proxy `/api/events` to BFF with SSE streaming support (no buffering)

### Constraints
- SSE connection uses one PostgreSQL connection per connected client — acceptable for admin portal (<10 concurrent)
- Must not block BFF's main connection pool — use a dedicated LISTEN connection
- Frontend must handle SSE reconnection gracefully (EventSource auto-reconnects)

## Requirement 2: Config + Schedule Card Merge

### Problem
Current frontend splits Gateway Configuration (SOC limits, charge/discharge currents, grid import limit) and Daily Schedule (time slots) into two separate cards with two "Apply" buttons. But in the protocol (§3.6), these are a SINGLE `battery_schedule` object sent as ONE `config/set` command.

Two separate Apply buttons = two separate MQTT config/set calls = second call overwrites the first. This is architecturally wrong.

### Solution: Merge into One Card, One Apply

```
┌─────────────────────────────────────────────────┐
│  Battery Schedule Configuration                  │
│                                                   │
│  ┌─ Parameters ─────────────────────────────┐    │
│  │ SOC Min (%)        [10  ]                │    │
│  │ SOC Max (%)        [95  ]                │    │
│  │ Max Charge (A)     [100 ]                │    │
│  │ Max Discharge (A)  [100 ]                │    │
│  │ Grid Import (kW)   [3000]                │    │
│  └──────────────────────────────────────────┘    │
│                                                   │
│  ┌─ Daily Schedule ─────────────────────────┐    │
│  │ [====green====][===blue===][==purple==]   │    │
│  │ 00:00-05:00  Tariff (Charge)             │    │
│  │ 05:00-17:00  Self Consumption            │    │
│  │ 17:00-20:00  Peak Shaving                │    │
│  │ 20:00-24:00  Tariff (Discharge, Forbid)  │    │
│  │                        [+ Add Slot]       │    │
│  └──────────────────────────────────────────┘    │
│                                                   │
│  Sync Status: ● Synced  Last: 2026-03-12 09:30  │
│                              [Apply to Gateway]   │
└─────────────────────────────────────────────────┘
```

## Requirement 3: M3→M1 Command Dispatch Pipeline

### Problem
Config/set commands written by BFF never reach the gateway. The chain is broken:
- BFF writes `device_command_logs` with `result = 'pending'`
- M3 picks up pending → marks `dispatched` → calls `publishMqtt()` which is a **STUB** (console.log only)
- Real MQTT connection lives in M1's GatewayConnectionManager, which M3 cannot access

### Architecture Decision (Alan)
M3 = business logic (decide what/when to send). M1 = infrastructure (MQTT publish + subscribe + reply tracking). DB = bridge between M3 and M1. M3 must NOT have any MQTT dependency.

### Solution: M1 Polls Dispatched Commands + Publishes via Own MQTT Client

```
BFF → DB (pending) → M3 polls → validates → marks dispatched
                                                    ↓
                                                    DB
                                                    ↓
                     M1 polls dispatched → builds protocol message
                         → publishToGateway() via real MQTT client
                         → receives set_reply → updates DB (success/fail/timeout)
```

### Status Flow
```
pending → (M3 Job2) → dispatched → (M1 publisher) → [awaiting reply]
                                                         ↓
                               set_reply arrives → success / fail
                               90s no reply → (M3 timeout check) → timeout
                               gateway offline → failed (reason: gateway_offline)
```

### Changes Required

#### 3.1 M3 — command-dispatcher.ts
- **Remove** `import { publishMqtt }` — M3 no longer touches MQTT
- **Simplify** `runPendingCommandDispatcher()`:
  - Poll `device_command_logs WHERE result = 'pending'` with `FOR UPDATE SKIP LOCKED LIMIT 50`
  - Mark as `dispatched` — that's it, no MQTT publish
  - Remove the message-building logic (commandLogId/commandType/payload/timestamp)
- **Keep** `runTimeoutCheck()` as-is (marks `dispatched` → `timeout` after 90s)
- **Keep** `runCommandDispatcher()` Job1 (trade_schedules) as-is for now

#### 3.2 M1 — GatewayConnectionManager
- **Add method** `publishToGateway(gatewayId: string, topic: string, message: string): boolean`
  - Finds the MQTT client for the given gatewayId
  - If found and connected → publish → return true
  - If not found or disconnected → return false
- **Expose** `isGatewayConnected(gatewayId: string): boolean`

#### 3.3 M1 — New: CommandPublisher (~60 lines)
- **File:** New `backend/src/iot-hub/services/command-publisher.ts`
- **Constructor:** Takes `Pool` + `GatewayConnectionManager` reference
- **Polling loop:** Every 10s, query:
  ```sql
  SELECT id, gateway_id, command_type, config_name, payload_json
  FROM device_command_logs
  WHERE result = 'dispatched'
  ORDER BY created_at ASC
  LIMIT 10
  FOR UPDATE SKIP LOCKED
  ```
- **For each command:**
  1. Check `gatewayConnectionManager.isGatewayConnected(gateway_id)`
     - If offline → `UPDATE result = 'failed', error_message = 'gateway_offline'`
  2. Read `payload_json` (DomainSchedule format)
  3. Build protocol message using `buildConfigSetPayload(gateway_id, schedule, messageId)`
  4. Publish via `gatewayConnectionManager.publishToGateway()`
  5. Update `device_command_logs SET message_id = $1` (for reply matching)
  6. Keep `result = 'dispatched'` (waiting for set_reply)
- **Wire up** in `run-m1-local.ts`: create CommandPublisher after GatewayConnectionManager starts

#### 3.4 M1 — command-tracker.ts
- **Change** `handleSetReply` WHERE clause:
  ```sql
  -- Before:
  WHERE result = 'pending' AND command_type = 'set'
  -- After:
  WHERE result = 'dispatched' AND command_type = 'set'
  ```

#### 3.5 Delete mqtt-client.ts Stub
- **File:** `backend/src/iot-hub/mqtt-client.ts`
- **Action:** Delete entirely — no module should import it anymore
- **Verify:** `grep -r "mqtt-client" backend/src/` returns zero hits after M3 cleanup

### Constraints
- M3 MUST NOT import anything from iot-hub/mqtt — clean module boundary
- M1 CommandPublisher uses the same DB pool (solfacil_service, BYPASSRLS)
- 10s polling interval for CommandPublisher matches M3's current interval
- LIMIT 10 per poll (not 50) — config/set is rare, no need for large batches
- Gateway offline = immediate failure, no retry queue (simple for now)
- message_id in device_command_logs must be updated when M1 publishes, so command-tracker can match the reply
