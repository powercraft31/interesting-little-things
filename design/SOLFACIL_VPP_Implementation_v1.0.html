第一部分

SOLFACIL 是誰：金融科技公司，不是電力運營商

1.1 核心定位

SOLFACIL（Solfácil）是拉丁美洲第一個太陽能金融科技平台，成立於 2018 年，總部在聖保羅。行業分類：Financial Services。

"Somos o principal ecossistema de soluções para geração de energia solar no Brasil e a primeira plataforma de fintech solar na América Latina." — SOLFACIL LinkedIn，2026

關鍵事實：SOLFACIL 不是電力零售商（comercializadora de energia）。他們的終端客戶電費仍然支付給當地的分配商（DISCO：CEMIG、Enel、Neoenergia 等）。SOLFACIL 是融資方，不是供電方。

1.2 業務規模（2024 年數據）

指標數字業務意義
合作安裝商（integradores）20,000+遍佈巴西全 27 州，是 SOLFACIL 最大的護城河
已落地太陽能系統100,000+住宅、商業、工業、農場、公共建築
累計裝機容量1 GW2024 年 7 月里程碑
累計融資規模R$ 30 億+QED / SoftBank / Valor Capital / IFC 投資

1.3 多租戶架構的真實含義

這套 VPP 平台採用多租戶架構（org_id 貫穿所有層），但「租戶」的定義：

  • 租戶 = 安裝商合作夥伴（integrador）：每個安裝商管理自己負責部署的設備艦隊，透過 org_id 做 Row-Level Security 隔離
  • SOLFACIL = Super Admin：設計文件中 SOLFACIL_ADMIN 角色，能繞過 RLS 查看全部租戶數據
  • 終端用戶：電池資產的物理持有者（融資客戶），資產屬他們，但調度授權給 SOLFACIL
這不是一個 SOLFACIL 對外販售的白牌 SaaS。SOLFACIL 自己是這個平台的運營方，安裝商合作夥伴是平台的管理層，終端融資客戶是資產的提供方。

1.4 這套平台的生產狀態

這套系統不是展示性 Demo，而是一個正在生產環境測試的平台，目前有 3 個真實用戶在測試中。硬件部署將由 SOLFACIL 的 20,000+ 安裝商合作夥伴執行——這正是他們現有業務的延伸，不需要 SOLFACIL 自建硬件部署能力。

旭衡電子硬件閘道器（工業級 EMS 設備）+ 旭衡 VPP 雲端平台。目前 3 個真實用戶在生產環境測試

2.2 Ampera 失敗的根本原因：技術不到位，不是不賺錢

Ampera Smart Box 從設計上就是免費提供的，SOLFACIL 從未期望它直接盈利。所以「不賺錢」不能解釋它被殺掉的原因——它本來就不是按直接盈利設計的。

真正的問題是硬件可靠性與大規模部署能力。

Ampera 的價值鏈是：設備聯網 → 收集數據 → 展示節電效果 → 客戶黏性提升 → 融資組合品質更好。這個飛輪能轉起來的前提是「100,000 台設備必須可靠聯網且持續回傳數據」。

但 SOLFACIL 有 20,000 個技術水平參差不齊的安裝商，遍佈巴西 27 個州。維持跨這個網絡的 IoT 設備聯網率、在線率、數據品質——這是一個專業 IoT 硬件公司才能解決的問題，不是金融科技公司擅長的領域。

硬件可靠性崩了 → 沒有穩定數據 → 節電效果無法展示 → 用戶感知價值歸零 → 產品死亡。

2.3 為什麼這次與旭衡合作能解決根本問題

Ampera：金融科技公司做硬件
  • 自建硬件，無量產經驗
  • 只做監控（單向），沒有雙向調度
  • 安裝商培訓困難，設備一致性差
  • 硬件維護費用高，問題無法規模化解決
旭衡：工業級 EMS 硬件廠
  • 旭衡閘道器已量產，工業級可靠性
  • MQTT 雙向通信，真正可調度
  • 旭衡已有 EMS 平台能力（工商儲/HEMS）
  • 安裝商部署閘道器是其現有業務延伸
第三部分

三層價值框架：今天 / 中期 / 2028 後

SOLFACIL 的 VPP 平台不需要在第一天就直接盈利。它的商業邏輯是分三個時間層次展開的：

今天（可落地）
C-Side 節費
  • Peak shaving：幫商業客戶避開峰時電費，規避需量超約罰款
  • 自發自用優化：光伏 + 電池，最大化自用比例
  • 誰省錢：終端客戶；不需要 CCEE 監管許可
中期（生態護城河）
戰略資產積累
  • 客戶黏性：電池進入 VPP 網絡後，換融資方成本極高
  • 融資組合品質：節電效果可量化 = 客戶還款能力更可預測
  • 投資人敘事：「50,000 台聯網電池」是比月收入更有力的 equity story
2028 後（監管解鎖）
B-Side 套利
  • CCEE 電力市場套利（CP 39 + Lei 15.269/2025 落地後）
  • 需求響應 DR 補貼（ANEEL 監管框架就位後）
  • 競爭優勢：別人剛開始建系統，SOLFACIL 已有 50,000 台在線資產等著變現
這個平台的核心邏輯不是「今天賺多少」，而是「2028 年監管套利窗口打開時，SOLFACIL 是第一個能立刻變現的人」。Ampera 失敗的技術問題被旭衡解決，監管的時間問題由分三層的商業邏輯橋接。

"a impossibilidade de obtenção de receitas adicionais, como por meio de arbitragem de preços de energia e prestação de serviços ancilares, também reduz as possibilidades de remuneração para a baixa tensão" （在低壓端，透過電價套利和輔助服務獲得額外收益，目前是不可能的）

第五部分

四個用例的推導與現實定位

5.1 從設計規約直接提取

設計文件 v3.md Executive Summary：
"Tarifa Branca arbitrage — Charge batteries during off-peak hours, discharge during peak hours to maximize the R$ 0.57/kWh spread"
"Demand Response (DR) — Coordinated dispatch of 50,000+ battery assets for grid balancing events"

M3 dispatch_tracker 表 target_mode 欄位：
self_consumption | peak_valley_arbitrage | peak_shaving
①
峰谷套利（Market Arbitrage）
今天可做

Tarifa Branca 費率結構下，在離峰時段充電（R$0.25/kWh），在峰時段放電（R$0.82/kWh），節省 R$0.57/kWh 的電費差。這是 C-side 節費——終端客戶在自己的電費帳單上節省，無需 CCEE 市場參與資格。未來監管開放後，可升級為真正的 CCEE 市場 B-side 套利。

target_mode = peak_valley_arbitrage ｜ 定價基礎：Tarifa Branca（今天）→ CCEE PLD（2028 後）
②
需求響應（Demand Response）
2028 後

設計文件描述「協調 50,000+ 台儲能資產響應電網調度事件，獲取 DR 補貼」。巴西目前沒有適用於分散式儲能的 DR 補貼框架，此用例屬於監管卡位設計。M3「DR Dispatcher」在今天實際執行的是 peak shaving 的觸發調度，「DR」命名反映的是未來意圖，不是今日現實。

target_mode 透過 DRCommandIssued event 觸發 ｜ 今日實際功能 = 調度執行層（非 DR 補貼）
④
自發自用優化（Self-Consumption）
今天可做

配合光伏發電，最大化終端客戶的自用電比例，減少從電網購電支出。M2 的 forecast-engine.ts 負責 Load & PV 出力預測，是這個用例的核心預測引擎。這是 SOLFACIL 銷售太陽能 + 電池組合的核心財務說辭。

target_mode = self_consumption ｜ M2 forecast-engine (Load & PV)
⑤
峰值功率管理（Peak Shaving）
今天可做

監控商業客戶用電需量，在接近需量合約上限前主動放電，避免巴西商業電費中的需量超約罰款（demanda）。這個用例不需要 CCEE 市場參與，屬於 behind-the-meter 操作，今天已合法且財務上已有明確 ROI（Colibri One 確認：某些州可完全以 peak shaving 支撐電池投資回報）。

target_mode = peak_shaving ｜ 遙測域實時功率監控觸發

5.2 四個用例的今日現實 vs 未來潛力

用例今日現實2028 後升級
① 峰谷套利C-side 電費節省（Tarifa Branca 差價）B-side CCEE 市場套利（PLD 批發電價）
② 需求響應調度執行架構已就緒，無監管支撐CCEE/ONS DR 補貼直接接入
④ 自發自用C-side 節費（光伏 + 電池自用率最大化）增加電費節省計算精度（SageMaker）
⑤ 峰值管理C-side 避免需量罰款（今日最強 ROI 用例）加入輔助服務頻率響應

用例啟用的業務域

EMS 平台架構定義 8 個業務域。SOLFACIL 的四個用例共啟用 7 個域（充電交易域不在範疇）。

1
外部信號域
全部 4 個用例
  • 今日：Tarifa Branca 費率表（ANEEL 靜態規定）→ M4 tariff_schedules → TariffUpdated 事件 → M2 重算排程
  • 今日：調度觸發信號（前端 BFF 的 DRCommandIssued）→ M3 立即執行
  • 2028：CCEE PLD Horário（pld_horario 表，每小時批發電價）→ 接管套利排程決策
2
資產登記域
全部 4 個用例
  • Assets 表：asset_id / org_id（安裝商）/ region / operation_mode / capacidade_kw
  • Organizations 表：org_id / CNPJ（安裝商合作夥伴法人）/ plan_tier
  • IoT 設備憑證：每個 org_id 對應獨立 IoT Policy，MQTT topic 命名空間隔離
  • RLS：PostgreSQL Row-Level Security，所有查詢強制帶 org_id
3
預測域
SOLFACIL 特有（UEH 無此域）
  • forecast-engine.ts：Load & PV 出力預測，追蹤 MAPE
  • evaluate-forecast.ts：接收 TelemetryReceived → 更新預測模型精度
  • tariff-optimizer.ts：根據 Tarifa Branca 費率計算最優充放電排程窗口
  • 未來：接入 SageMaker 升級預測引擎（設計文件明確規劃）
SOLFACIL 需要預測域而 UEH 不需要：AEMO 是實時電價（響應式）；Tarifa Branca 是固定費率時段（排程式）。自發自用用例還需要 PV 出力與負荷預測來最大化自用率。
4
調度引擎域
全部 4 個用例
  • M2 產生 ScheduleGenerated（24h 排程計劃），覆蓋三種 target_mode
  • M3 執行調度指令（即時派發，不依賴排程的緊急調度）
  • target_mode：策略切換開關，決定本次調度使用哪個優化目標
  • 狀態追蹤：dispatched → acknowledged → completed / failed / timeout（SQS 15 分鐘）
5
設備網關域
全部 4 個用例
  • AWS IoT Core：每台設備一個 Thing + Certificate + Policy
  • MQTT Topic：solfacil/{org_id}/{region}/{asset_id}/command/mode-change
  • Device Shadow：M2 把 24h 排程寫入 Desired State，設備離線重連後自動同步
  • Response Topic：solfacil/{org_id}/{region}/{asset_id}/response/mode-change
6
遙測域
閉環回饋
  • M1 MQTT ingest → Amazon Timestream（device_telemetry，90 天保留）
  • Measures：SoC / power_kw / voltage / temperature / operation_mode
  • 發布 TelemetryReceived → M2 evaluate-forecast 更新預測模型
  • 峰值管理用例需要遙測域提供實時功率監控作為調度觸發
7
財務結算域
全部 4 個用例
  • M4 Market & Billing：Tarifa Branca 費率引擎，計算每次調度的節費金額
  • Optimization Alpha = 實際節費 / 理論最大節費 × 100（效率 KPI）
  • M7 Open API：Webhook 推送客戶節費報告到安裝商後台 / 客戶 App
  • 今日展示：C-side 節費金額（「今天幫你省了 R$X」）
  • 2028 後：B-side 套利利潤（「SOLFACIL 從 CCEE 市場賺了 R$X」）

7.3 M3 命名問題

M3「DR Dispatcher」實際執行的是全部三種 target_mode 的指令派發（peak_valley_arbitrage / self_consumption / peak_shaving），不只是需求響應。命名應改為「Command Dispatcher」——它是策略無關的指令執行層，只負責可靠送達，不決定策略。

Spec 診斷

P0  M1：設備網關域與遙測域無邊界

AWS IoT Core MQTT 接入（設備網關域）和 Timestream 時序存儲（遙測域）在 M1 混合無內部邊界。未來接入新硬件協議時，需要在含遙測邏輯的代碼中改動。

P0  M2：預測域與調度引擎域混合

forecast-engine.ts（預測）和 tariff-optimizer.ts（調度策略）在同一 Lambda stack。未來升級預測引擎（接入 SageMaker），需要觸及套利排程邏輯，違反單一職責原則。

P1  target_mode 策略決策不在調度引擎域

哪個 target_mode 啟用由 M5 BFF 的 DRCommandIssued 事件攜帶決定。等於前端層決定策略模式。策略模式應由調度引擎域根據資產 operation_mode 配置和信號類型自動選擇。

P1  compute-alpha.ts 放置位置語義不對

套利效率計算（實際節費 / 理論最大）是結算指標，語義屬財務結算域，但放在 M2 Algorithm Engine。

P2  用例 ④⑤ 的核心算法未實作

self_consumption：forecast-engine.ts 的 Load & PV 預測算法尚未實作（設計文件標注「future: SageMaker」）
peak_shaving：功率監控觸發邏輯無對應代碼

P2  Demo 層靜態數據未接入真實來源

前端電價數據：market.js 時間計算替代 M4 tariff_schedules 真實查詢
Algorithm KPI（Alpha、MAPE）：data.js 隨機生成替代 M2 Timestream 數據

第九部分

重構路徑

原則：不改業務邏輯，只對齊域邊界。
v5.9 核心算法（套利排程、DR 派發、遙測聚合）已通過 194 個測試。重構目標是讓目錄結構和跨模塊接口符合 EMS 框架，不重寫任何業務邏輯。
P0
域邊界重整（只搬家 + 重命名）
  • M1 拆分：設備網關子目錄（IoT Core / MQTT / Shadow）+ 遙測子目錄（Timestream）
  • M2 拆分：預測子目錄（forecast-engine / evaluate-forecast）+ 調度策略子目錄（tariff-optimizer / run-schedule）
  • M3 重命名：DR Dispatcher → Command Dispatcher
  • compute-alpha.ts 移至 M4 財務結算域
P1
接口契約補全
  • 調度引擎域：target_mode 決策從 BFF 移回調度引擎域；BFF 只觸發調度，不攜帶策略
  • 外部信號域：統一費率信號入口（M4 tariff）與調度觸發信號（M7）到同一個 Signal Publisher
  • 外部信號域：預留 pld_horario 接口——今天用 Tarifa Branca，2028 後無縫切換到 CCEE PLD
P2
用例完整化（④⑤）+ 數據真實化
  • ④ 自發自用：實作 Load & PV 預測算法（或接入 SageMaker）
  • ⑤ 峰值管理：實作遙測域功率監控觸發邏輯
  • 前端電價展示：移除靜態 mock，接入真實 M4 tariff_schedules
  • Algorithm KPI：移除隨機生成，接入真實 M2 Timestream 數據

驗收標準

  • 原有 194 個測試全部通過
  • 每個 EMS 域對應一個獨立的代碼子目錄
  • target_mode 策略決策不在 BFF 層
  • 新增外部信號類型（如 CCEE PLD 接入）只需修改外部信號域，不動其他域
  • 前端 KPI 數字全部來自真實 DB / Timestream，無 mock 數據
