# Solfacil v6.9 Implementation Kickoff Constraints

Date: 2026-04-15 UTC
Status: implementation preflight constraints
Scope: final execution constraints before v6.9 security correction implementation starts

## 1. Purpose

This document is not a new REQ, DESIGN, or PLAN.
It is a short execution constraint layer to remove the last implementation-time ambiguity before coding starts.

The goal is simple:
- do not let implementation teams improvise on unresolved points;
- do not let testing evidence become vague;
- do not let migration bridges become sticky defaults.

## 2. Fixed execution constraints

### 2.1 FE-6 CSP / style allowance rule

v6.9 implementation must follow this rule:
- inline login script and inline `onclick` must be removed or externalized before rollout;
- `script-src 'unsafe-inline'` is not allowed as a general release convenience;
- if an inline script remnant survives temporarily, only nonce/hash-based exception is allowed and must be explicitly tracked;
- `style-src 'unsafe-inline'` is temporarily allowed only for the login page inline style if that style is not removed in the v6.9 window;
- this style allowance must be recorded as a temporary residual risk and cleanup target.

So the implementation default is:
- allow temporary login-page inline style only;
- do not allow broad script-inline relaxation.

### 2.2 Abuse-control parameter set must be fixed before code starts

Implementation must not start with vague “we will throttle somehow.”
Before coding BE abuse-control logic, the team must pin:
- IP burst threshold
- identifier failure threshold
- backoff progression
- success reset rule
- whether every throttle/backoff response emits `Retry-After`

These values may be written into a short engineering note, config constant block, or implementation comment, but they must exist before tests are written.

## 3. Machine bearer scope rule

The design already allows a separate machine/non-browser bearer contract.
Implementation must not treat that as a broad excuse to keep legacy bearer behavior everywhere.

Before coding auth middleware cutover, create a small explicit route-scope note stating:
- which endpoints remain machine-bearer-eligible
- which endpoints are browser-cookie-only
- which endpoints are public

Minimum rule:
- the Solfacil SPA must not use bearer auth
- browser `/api/*` steady state is cookie-first
- any machine bearer path must be explicit and enumerable

## 4. Test evidence format is mandatory

“Tested” is not evidence.
For v6.9, the minimum evidence format is:

### 4.1 Backend slice evidence
- test command output
- relevant curl probe output where HTTP contract changed
- note whether compatibility bridge was ON or OFF

### 4.2 Frontend slice evidence
- browser screenshot or equivalent visible proof for user-facing flow
- devtools network screenshot/export when auth headers/cookies/session probe behavior matters
- storage inspection proof when localStorage/sessionStorage behavior matters
- console/CSP proof when CSP or asset loading behavior matters

### 4.3 Deployment/public-boundary evidence
- `curl -I` output for login and SPA entry
- login response header / cookie proof
- CSP/header proof from actual public boundary
- operator checklist item completion for ingress/store/secret wiring

### 4.4 Final release evidence
The final acceptance matrix in `v6.9-security-plan.md` must be accompanied by recorded evidence per case, not just verbal confirmation.

## 5. Bridge discipline

Migration bridges are emergency migration tools, not comfort blankets.

Required implementation rule:
- compatibility mode defaults OFF unless rollout window explicitly requires it
- dual-cookie bridge defaults OFF unless rollout window explicitly requires it
- any bridge activation must be logged in operator notes
- every bridge must have:
  - enable condition
  - disable condition
  - removal owner
  - maximum allowed window

## 6. Operator-side prerequisite rule

Because part of deployment boundary work lives outside this repo, implementation is not rollout-ready until operator-owned prerequisites exist in concrete form.

At minimum, before rollout approval there must be an operator-side checklist or config diff covering:
- ingress header/CSP changes
- trusted proxy settings
- secure cookie correctness
- shared abuse-store provisioning
- bridge toggle handling

## 7. Practical outcome

If these constraints are followed, implementation can proceed without reopening architecture questions.

If these constraints are ignored, the most likely failures are:
- CSP exceptions creeping wider during implementation
- throttle logic becoming untestable because thresholds were never fixed
- machine bearer scope silently re-expanding
- frontend verification being hand-waved
- bridges lingering as undocumented semi-permanent behavior

## 8. Next step

With this constraint layer in place, the next valid step is implementation preparation and then slice-by-slice execution against the finalized PLAN set.
