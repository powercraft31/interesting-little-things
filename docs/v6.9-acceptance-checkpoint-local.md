# v6.9 Acceptance Checkpoint — Local Surface

Date: 2026-04-16 UTC
Scope: bounded acceptance on local Docker-aligned surface `http://127.0.0.1:3100`
Status: partial acceptance pass — major v6.9 contracts now observable on the running local surface.

## What was verified

### 1. Browser login contract
Verified via `curl` against local running BFF after recreating `solfacil-bff` with v6.9 env wiring.

Observed:
- `POST /api/auth/login` returned `200`
- response body contained `user`
- response body did **not** contain `token`
- `Set-Cookie` was:
  - `__Host-solfacil_session=...`
  - `HttpOnly`
  - `Secure`
  - `SameSite=Strict`
  - `Path=/`
  - `Max-Age=86400`

Verdict:
- browser login contract matches v6.9 target model

### 2. Session endpoint
Verified via `curl` with cookie jar.

Observed:
- `GET /api/auth/session` without cookie → `401`
- `GET /api/auth/session` with browser cookie → `200`
- returned payload included:
  - `userId`
  - `orgId`
  - `role`
  - `name`
  - `email`

Verdict:
- session bootstrap endpoint is working on the local surface

### 3. Machine login contract
Verified via `curl` with header `X-Auth-Contract: machine`.

Observed:
- machine login returned `200`
- response body included `token`
- browser login body did not include `token`

Verdict:
- browser vs machine auth contract separation is observable on the running local surface

### 4. Abuse control
Verified via repeated wrong-password attempts.

Observed:
- attempts 1–5 → `401`
- attempt 6 → `429`
- `Retry-After=900`

Verdict:
- login abuse mitigation is active on the local running surface

### 5. Security headers
Verified on local running surface.

Observed on responses:
- `Content-Security-Policy`
- `X-Frame-Options: DENY`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy`

CSP observed:
- `default-src 'none'`
- `script-src 'self'`
- `style-src 'self'`
- `font-src 'self'`
- `connect-src 'self'`
- `frame-ancestors 'none'`
- no broad `unsafe-inline`

Verdict:
- app-layer browser hardening headers are present on the local running surface

### 6. Browser-side auth storage
Verified in browser after successful login.

Observed:
- page transitioned to `index.html#fleet`
- `localStorage` keys were empty in the browser session used for verification
- `solfacil_jwt` not present in localStorage

Verdict:
- browser no longer depends on localStorage auth token on the verified local surface

### 7. Self-hosted assets / no external dependency
Verified in browser via resource inspection.

Observed:
- no external resource URLs outside current origin
- ECharts loaded from local path:
  - `/js/vendor/echarts.min.js`
- fonts loaded from local paths:
  - `/fonts/inter-latin.woff2`
  - `/fonts/jetbrainsmono-latin.woff2`

Verdict:
- current verified local app surface is using self-hosted assets as intended

## Important runtime issue found during acceptance

### Stale container deployment initially masked reality
When acceptance started, `http://127.0.0.1:3100` was still serving stale runtime behavior:
- old login response still returned token in body
- old cookie name/attributes were present

Root cause:
- running `solfacil-bff` container had not been recreated with the updated code

Action taken:
- recreated `solfacil-bff` and added `solfacil-redis` via Docker Compose using v6.9 env wiring

### Deployment miswire introduced during WS-D edit
A deployment regression was found in `docker-compose.yml` during this acceptance run:
- BFF database URLs had been accidentally left with placeholder `***` instead of real passwords
- container logs showed `password authentication failed for user "solfacil_service"`

Action taken:
- corrected `APP_DATABASE_URL`
- corrected `SERVICE_DATABASE_URL`
- recreated `solfacil-bff`

Verdict:
- issue fixed during acceptance
- this was a deployment wiring bug, not a design gap
