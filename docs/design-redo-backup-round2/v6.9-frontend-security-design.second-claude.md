# Solfacil v6.9 Frontend Security Design

**Date:** 2026-04-15 UTC
**Status:** DESIGN draft
**Parent:** `docs/v6.9-security-design.md` (master)

## 1. Scope

This document covers v6.9 changes to the frontend: login page, authenticated request model, logout/session-expiry behavior, auth guard, localStorage elimination, and browser-side implications of the cookie-based auth model.

### 1.1 Affected frontend files

| File | Change |
|------|--------|
| `login.html` | Remove localStorage write on login success. Remove localStorage-based auth redirect. |
| `js/app.js` | Replace localStorage auth guard with session-validity mechanism. Remove localStorage clear on logout. Clean up residual localStorage on first load. |
| `js/data-source.js` | Remove `Authorization: Bearer` header from all `apiGet()` / `apiPost()` / `apiPut()` calls. |
| `js/config.js` | No change expected. |

### 1.2 Unaffected frontend files

All page modules (`p1-fleet.js` through `p6-alerts.js`), `components.js`, `charts.js`, `i18n.js`, `mock-data.js`, and all CSS files are unaffected. They do not interact with auth state directly — they call `DataSource.*` methods which handle transport internally.

## 2. Login Page (`login.html`)

### 2.1 Current behavior (v6.8)

```
Form submit → POST /api/auth/login { email, password }
  → Success: response.data.token → localStorage("solfacil_jwt") = token
  → Browser also receives Set-Cookie: solfacil_jwt=<token>; HttpOnly; ...
  → Redirect to index.html
```

### 2.2 Target behavior (v6.9)

```
Form submit → POST /api/auth/login { email, password }
  → Success: response.data contains { user } (no token field)
  → Browser has received Set-Cookie: solfacil_jwt=<token>; HttpOnly; SameSite=Lax; Path=/; Max-Age=86400
  → No localStorage write
  → Redirect to index.html
```

Changes:
1. **Remove** `localStorage.setItem("solfacil_jwt", data.token)` from login success handler.
2. **Remove** the pre-login check `if (localStorage.getItem("solfacil_jwt")) { redirect to index.html }`. Replace with: attempt a lightweight authenticated request (e.g., `GET /api/auth/me` or simply let `index.html` handle the redirect — if the cookie is valid, `index.html` will load normally; if not, the auth guard will redirect back).
3. The login form submission must use `credentials: 'same-origin'` (or `'include'` if cross-origin, but current architecture is same-origin) so the browser accepts the `Set-Cookie` response.

### 2.3 Login page auth redirect

The v6.8 login page checks localStorage on load to skip the login form if a token exists. In v6.9, this check cannot use localStorage (the JWT is HttpOnly, invisible to JS).

**Design decision:** The login page does not attempt to detect an existing session. If a user navigates to `login.html` while already authenticated, they see the login form. This is acceptable behavior — the user can simply navigate to `index.html`, or re-login (which refreshes the session).

**Alternative considered and deferred:** A `GET /api/auth/session` endpoint that returns 200 if cookie is valid, 401 if not. The login page could call this on load and redirect if 200. This adds a network round-trip on every login page load. Acceptable to implement if product requires it, but not a security requirement.

### 2.4 Error display

No change to error display behavior. Login failures continue to show translated error messages. New behavior: 429 responses (rate limit exceeded) must be handled — display a user-friendly message such as "Too many attempts. Please wait and try again." with the `Retry-After` value if available.

## 3. Authenticated Request Model (`data-source.js`)

### 3.1 Current behavior (v6.8)

```javascript
// apiGet / apiPost / apiPut pattern:
headers: {
  'Authorization': 'Bearer ' + localStorage.getItem('solfacil_jwt'),
  'Content-Type': 'application/json'
}
```

### 3.2 Target behavior (v6.9)

```javascript
// apiGet / apiPost / apiPut pattern:
headers: {
  'Content-Type': 'application/json'
}
// credentials: 'same-origin' is the fetch default for same-origin requests
// The browser automatically includes the solfacil_jwt cookie
```

Changes:
1. **Remove** all `Authorization` header construction from `apiGet()`, `apiPost()`, `apiPut()`, and any other fetch wrappers.
2. **Remove** all `localStorage.getItem("solfacil_jwt")` references in data-source.js.
3. Ensure `credentials` is not set to `'omit'` anywhere (which would suppress cookie sending). The default `'same-origin'` behavior is correct.

### 3.3 401 handler

The existing 401 handler in data-source.js currently does:
```javascript
// On 401: localStorage.removeItem("solfacil_jwt"); redirect to login.html
```

v6.9 change:
```javascript
// On 401: redirect to login.html (no localStorage cleanup needed)
```

The redirect behavior is unchanged. The localStorage cleanup line is removed since there is nothing to clean.

### 3.4 Mock mode

When `CONFIG.USE_MOCK = true`, no API calls are made. Mock mode is unaffected by auth changes.

## 4. Auth Guard (`app.js`)

### 4.1 Current behavior (v6.8)

At the top of `app.js`:
```javascript
if (!localStorage.getItem("solfacil_jwt")) {
  window.location.href = "login.html";
}
```

### 4.2 Target behavior (v6.9)

The auth guard cannot check for the HttpOnly cookie directly (JS cannot read HttpOnly cookies). Two options:

**Option A — Optimistic load with 401 fallback (recommended):**
Remove the synchronous auth guard entirely. Let the SPA load and make its first API call. If the cookie is missing or expired, the server returns 401, and the existing 401 handler redirects to `login.html`.

Tradeoff: The SPA shell (sidebar, layout) renders briefly before the redirect on an unauthenticated visit. This is a cosmetic concern, not a security concern — no data is exposed because the API calls fail.

**Option B — Lightweight session check on load:**
Add a synchronous or early-async check: `fetch("/api/auth/session")` which returns 200 with user metadata if cookie is valid, 401 if not. If 401, redirect immediately.

Tradeoff: Adds a network round-trip before SPA initialization. Slightly better UX (no flash of empty shell).

**Design decision:** Option A is sufficient. The 401-redirect path already exists and is tested. No new endpoint is needed. The brief SPA shell flash is acceptable for an admin tool. If product requires Option B, it can be added as a minor enhancement without changing the security model.

### 4.3 User metadata availability

The v6.8 frontend extracts user role and orgId from the JWT payload stored in localStorage (it parses the JWT client-side). In v6.9, the JWT is not available to JS.

**Design decision:** The login response body includes user metadata (`userId`, `email`, `name`, `orgId`, `role`) without the raw token. The frontend stores this metadata in `sessionStorage` (not the token — just the non-secret user profile). This supports:
- Role-based UI visibility (admin vs. integrador page filtering)
- User display name in the sidebar
- Role badge rendering

`sessionStorage` is appropriate because:
- It does not persist across tabs or browser restarts (unlike localStorage).
- It contains non-secret user profile data, not auth credentials.
- It matches the existing `DemoStore` pattern (`sessionStorage` with `ds_` prefix).

If the user opens a new tab, the `sessionStorage` is empty, the SPA makes an API call, gets a 401 (if no cookie) or succeeds (if cookie present). On success, the response data can repopulate user metadata — this requires the backend to include user context in a standard place (e.g., the `_tenant` field already present in dev mode, or a dedicated `/api/auth/session` endpoint).

**Open point:** Decide whether to expose a `GET /api/auth/session` endpoint that returns the current user's profile based on cookie auth. This is not strictly required for the security model but improves UX for new-tab scenarios.

### 4.4 Logout

Current behavior:
```javascript
// Sidebar logout button click:
localStorage.removeItem("solfacil_jwt");
window.location.href = "login.html";
```

v6.9 behavior:
```javascript
// Sidebar logout button click:
fetch("/api/auth/logout", { method: "POST" });
// Server clears HttpOnly cookie
sessionStorage.clear(); // Clear user metadata
window.location.href = "login.html";
```

The `POST /api/auth/logout` endpoint already exists in v6.8. The v6.9 change is that the frontend no longer also clears localStorage (nothing to clear).

### 4.5 Session expiry

When the 24h JWT expires, the next API call returns 401. The 401 handler redirects to `login.html`. This behavior is unchanged from v6.8 — the only difference is that the expiry is detected by cookie-based server validation rather than (potentially) client-side token inspection.

## 5. SSE Authentication

### 5.1 No change from v6.8

SSE via `EventSource` at `/api/events` already authenticates via the HttpOnly `solfacil_jwt` cookie. This was the original reason the cookie was introduced in v6.8.

v6.9 does not change SSE authentication. The cookie is now also used for all other API requests, making the auth model consistent.

### 5.2 SSE reconnection after session expiry

If the session expires while an SSE connection is open, the SSE connection itself continues (server does not actively terminate long-lived SSE connections on token expiry). However, if the connection drops and `EventSource` auto-reconnects, the reconnection request will carry the expired cookie, and the server should return 401, closing the SSE stream.

The frontend SSE error handler in `p2-devices.js` logs a warning on SSE errors. It does not redirect to login. This is acceptable — the user will encounter the 401 redirect on their next explicit API call or page navigation.

## 6. localStorage Cleanup

### 6.1 One-time migration

On first load after v6.9 deployment, the frontend should remove the old localStorage key if it exists:

```javascript
// At top of app.js, before any other logic:
localStorage.removeItem("solfacil_jwt");
```

This is defensive cleanup. It ensures no stale JWT lingers in localStorage where an XSS exploit could find it. This line can remain permanently — it is a no-op if the key does not exist.

### 6.2 Other localStorage keys

The frontend uses `localStorage` for:
- `"lang"` — i18n language preference (via `I18n.setLang`)

This is unaffected by v6.9. Only the `"solfacil_jwt"` key is removed.
