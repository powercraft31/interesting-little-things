## 1. Scope

This document covers v6.9 changes to the backend auth boundary: M5 BFF middleware, M6 Identity login/logout handlers, shared-layer token verification, SSE auth path, cookie semantics, login rate limiting, and JWT secret fail-fast behavior.

## 2. Auth Middleware Changes (`bff/middleware/auth.ts`)

### 2.1 Current token extraction (v6.8)

```
1. Check Authorization header exists → 401 if missing
2. Strip "Bearer " prefix
3. verifyTenantToken(token) → { userId, orgId, role }
4. Overwrite req.headers.authorization = JSON.stringify({ userId, orgId, role })
5. next()
```

The SSE handler (`sse-events.ts`) has a separate cookie-reading path — it reads `req.cookies.solfacil_jwt` when no Authorization header is present.

### 2.2 Target token extraction (v6.9)

```
1. Read token from cookie: req.cookies.solfacil_jwt
2. If no cookie: read token from Authorization header (strip "Bearer " prefix)
3. If neither: return 401
4. verifyTenantToken(token) → { userId, orgId, role }
5. Overwrite req.headers.authorization = JSON.stringify({ userId, orgId, role })
6. next()
```

Key changes:
- **Cookie is checked first.** This makes cookie the primary auth mechanism for browser clients.
- **Authorization header is the fallback.** This supports non-browser clients (development tools, future API consumers).
- **If both are present, cookie wins.** This prevents a situation where a stale Authorization header overrides a valid (or invalid) cookie.
- **Cookie parsing:** Requires `cookie-parser` middleware (or equivalent) to be registered before `authMiddleware` in the Express pipeline. Express does not parse cookies by default.

### 2.3 Downstream handler compatibility

The header-rewriting step (step 5) is unchanged. All 47 existing BFF handler files continue to call `extractTenantContext(event)` → `verifyTenantToken(rawJSON)` → Path 1 (raw JSON parse). No handler code changes.

### 2.4 SSE handler convergence

The SSE handler (`sse-events.ts`) currently has its own cookie-reading logic separate from the auth middleware. In v6.9, the auth middleware handles cookie reading for all `/api/*` routes including `/api/events`.

**Design decision:** Remove the SSE-specific cookie fallback from `sse-events.ts`. The auth middleware now handles this uniformly. The SSE handler receives the request after middleware has validated the cookie and rewritten the Authorization header, just like all other handlers.

**Verification:** SSE connections must continue to work after this change. The `EventSource` browser API sends cookies automatically on same-origin connections. The middleware reads the cookie. The SSE handler sees the rewritten header. No change in SSE behavior.

### 2.5 Public routes

Public routes remain: `["/api/auth/login", "/api/auth/logout"]`. These skip auth middleware entirely. No change.

### 2.6 Cookie-parser dependency

Express does not parse cookies natively. The v6.9 middleware requires `cookie-parser` (or a minimal manual parser using `req.headers.cookie`).

**Design decision:** Add `cookie-parser` as an Express middleware, registered before `authMiddleware`:

```
app.use(cookieParser());
app.use(authMiddleware);
```

This is a standard Express pattern. `cookie-parser` has no configuration for HttpOnly cookies — it parses all cookies from the `Cookie` header. HttpOnly only affects browser JS visibility, not server-side parsing.

**Open point:** Verify whether `cookie-parser` is already a dependency. If not, it must be added.

## 3. Login Handler Changes (`bff/handlers/auth-login.ts`)

### 3.1 Current login response (v6.8)

```json
{
  "success": true,
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIs...",
    "user": { "userId": "...", "email": "...", "name": "...", "orgId": "...", "role": "..." }
  }
}
```

The handler also sets `Set-Cookie: solfacil_jwt=<token>; HttpOnly; SameSite=Lax; Path=/; Max-Age=86400`.

### 3.2 Target login response (v6.9)

```json
{
  "success": true,
  "data": {
    "user": { "userId": "...", "email": "...", "name": "...", "orgId": "...", "role": "..." }
  }
}
```

Changes:
1. **Remove `token` field from response body.** The JWT is delivered exclusively via the `Set-Cookie` header.
2. **Cookie attributes unchanged:** `HttpOnly; SameSite=Lax; Path=/; Max-Age=86400`.
3. **Add `Secure` flag** when the application detects HTTPS termination (see section 3.3).

### 3.3 Secure cookie flag

The `Secure` flag ensures the cookie is only sent over HTTPS. In the current deployment model (reverse proxy terminates TLS, forwards to Express on port 3000 over HTTP), Express sees HTTP requests.

**Design decision:** Set the `Secure` flag conditionally:
- If `NODE_ENV === 'production'` or `req.secure` or `req.headers['x-forwarded-proto'] === 'https'`: set `Secure`.
- Otherwise (local development over HTTP): omit `Secure`.

Express's `trust proxy` setting must be enabled for `req.secure` to correctly reflect the proxy's protocol. This is a configuration concern documented in `v6.9-deployment-security-boundary-design.md`.

### 3.4 Logout handler

The logout handler (`POST /api/auth/logout`) currently clears the `solfacil_jwt` cookie via `res.clearCookie(...)`. No change needed. The handler already works correctly.

## 4. Login Rate Limiting

### 4.1 Design

A new middleware function applied exclusively to `POST /api/auth/login`.

```
app.post("/api/auth/login", loginRateLimiter, createLoginHandler(servicePool));
```

### 4.2 Implementation model

In-memory sliding window per client IP.

Data structure:
```
Map<string, { timestamps: number[] }>
```

On each login request:
1. Extract client IP from `req.ip` (respects `trust proxy` for X-Forwarded-For).
2. Remove timestamps older than `WINDOW_MS` from the array.
3. If remaining count >= `MAX_ATTEMPTS`: return 429 with `Retry-After` header.
4. Otherwise: add current timestamp and call `next()`.

### 4.3 Configuration

| Parameter | Env var | Default |
|-----------|---------|---------|
| Window | `LOGIN_RATE_LIMIT_WINDOW_MS` | 900000 (15 min) |
| Max attempts | `LOGIN_RATE_LIMIT_MAX` | 10 |

### 4.4 429 response

```json
{
  "success": false,
  "data": null,
  "error": "Too many login attempts. Try again later."
}
```

Headers: `Retry-After: <seconds remaining in window>`.

### 4.5 Memory management

Stale entries (IPs with no recent requests) are periodically pruned. A simple approach: on every Nth request (e.g., every 100th), iterate the map and remove entries where all timestamps are older than `WINDOW_MS`. This prevents unbounded memory growth.

### 4.6 Placement in middleware chain

The rate limiter runs **before** the login handler but **after** Express JSON body parsing (so the request body is available if needed for logging, though the rate limiter itself only uses the IP).

```
express.json() → loginRateLimiter → createLoginHandler
```

The rate limiter does NOT run through `authMiddleware` (login is a public route).

### 4.7 Affected status codes

The BFF response status code table (M5 doc section 6) must add:

| Code | Meaning | Usage |
|------|---------|-------|
| **429** | Too Many Requests | Login rate limit exceeded |

## 5. JWT Secret Fail-Fast (`shared/middleware/tenant-context.ts` + `local-server.ts`)

### 5.1 Current behavior (v6.8)

```typescript
const jwtSecret = process.env.JWT_SECRET || "solfacil-dev-secret";
```

No startup validation. The dev fallback is always available.

### 5.2 Target behavior (v6.9)

**Startup validation** in `local-server.ts` (or equivalent entry point):

```
On application start:
1. Read JWT_SECRET from env
2. If JWT_SECRET is unset/empty AND NODE_ENV === "production":
   → Log fatal error: "JWT_SECRET must be set in production"
   → process.exit(1)
3. If JWT_SECRET === "solfacil-dev-secret" AND NODE_ENV === "production":
   → Log fatal error: "JWT_SECRET must not use the development default in production"
   → process.exit(1)
4. Otherwise: continue startup
```

**`tenant-context.ts` unchanged.** The fallback `"solfacil-dev-secret"` remains in the code for local development. The startup check ensures it is never reachable in production.

### 5.3 Rationale

The fail-fast is at the application entry point, not in `tenant-context.ts`, because:
- `tenant-context.ts` is a pure, framework-agnostic module. Adding process-level startup concerns violates its design boundary.
- The entry point (`local-server.ts`) is the correct place for environment validation.
- The validation runs once at startup, not on every token verification.

### 5.4 Other modules importing verifyTenantToken

M4 (Market Billing) and M8 (Admin Control) import `verifyTenantToken()` directly. They share the same `JWT_SECRET` env var. The startup check in `local-server.ts` covers all modules because they all run in the same process.

## 6. Cookie Semantics

### 6.1 Cookie attributes

| Attribute | Value | Rationale |
|-----------|-------|-----------|
| Name | `solfacil_jwt` | Unchanged from v6.8 |
| Value | Signed JWT | Unchanged |
| HttpOnly | Yes | Not accessible to JavaScript — prevents XSS token theft |
| Secure | Conditional (see 3.3) | HTTPS-only in production |
| SameSite | Lax | Sent on same-site navigations and top-level GET navigations. Not sent on cross-origin POST — provides CSRF baseline. |
| Path | `/` | Cookie valid for all paths |
| Max-Age | 86400 (24h) | Matches JWT expiry |
| Domain | Not set (defaults to exact origin) | Cookie scoped to the serving domain only |

### 6.2 SameSite=Lax and CSRF

`SameSite=Lax` prevents the cookie from being sent on cross-origin subrequests (images, iframes, POST forms from other sites). This provides effective CSRF protection for state-changing operations (`POST`, `PUT`, `PATCH`, `DELETE`).

**Why not SameSite=Strict?** Strict would prevent the cookie from being sent when a user clicks a link to Solfacil from an external site (e.g., email link). The user would see the login page even if they have a valid session. Lax allows top-level navigations (GET only) to include the cookie, which is the expected UX.

**Why no additional CSRF token?** All state-changing Solfacil API calls use `Content-Type: application/json`, which is not a "simple" content type. Cross-origin requests with non-simple content types trigger a CORS preflight, which is blocked by the existing CORS whitelist. Combined with SameSite=Lax, this provides defense-in-depth against CSRF without requiring a separate token.

### 6.3 Cookie size

A JWT with `{ userId, orgId, role, iat, exp }` signed with HS256 is approximately 200-300 bytes. Well within the 4KB cookie size limit.

## 7. Token Verification Path (`shared/middleware/tenant-context.ts`)

### 7.1 No change to verifyTenantToken()

The `verifyTenantToken(token)` function is unchanged. It continues to support:
- **Path 1 (Raw JSON):** Used internally after middleware rewrites `req.headers.authorization`.
- **Path 2 (JWT):** Used when the middleware passes the raw JWT from cookie or Authorization header.

The function does not know or care where the token came from (cookie vs. header). It receives a string and validates it.

### 7.2 No change to requireRole()

RBAC enforcement is unchanged. `SOLFACIL_ADMIN` bypass, role array check, 403 on failure.

### 7.3 JWT_SECRET env var

The fallback behavior is unchanged in `tenant-context.ts` itself. The fail-fast is at the application startup level (see section 5).

## 8. BFF Handler Impact

### 8.1 Zero handler code changes

All 47 BFF handler files (including the 2 new alert handlers from v6.8) call `extractTenantContext(event)` which reads the rewritten `req.headers.authorization` header. The auth middleware continues to rewrite this header after validating the JWT (now from cookie instead of from the original Authorization header). The handler interface is unchanged.

### 8.2 extractTenantContext compatibility

`extractTenantContext(event)` in `bff/middleware/auth.ts` constructs a Lambda-compatible event from the Express request. It reads `event.headers.authorization` which contains the JSON-stringified `{ userId, orgId, role }`. This is set by the auth middleware regardless of whether the original token came from a cookie or a header.

### 8.3 Background services

Background services (M2 schedule generator, M3 command/timeout, M4 billing, M1 aggregators) use `getServicePool()` directly. They do not go through HTTP or auth middleware. Unaffected.

### 8.4 Webhook routes

Webhook routes (`/webhooks/*`, mock telemetry, dispatch ACK) bypass auth middleware entirely. Unaffected.

## 9. SSE Auth Convergence

### 9.1 Current SSE auth (v6.8)

`sse-events.ts` has its own cookie-reading logic because the main auth middleware only checked the Authorization header:

```
1. Check Authorization header → if present, verify JWT
2. If no header, check req.cookies.solfacil_jwt → verify JWT
3. If neither → 401
```

### 9.2 Target SSE auth (v6.9)

The main auth middleware now reads cookies first (see section 2.2). SSE requests flow through the same middleware as all other `/api/*` requests.

`sse-events.ts` no longer needs its own auth logic. It receives the request after the middleware has validated the token and rewritten the header.

**Change:** Remove the duplicated cookie-reading and JWT-verification code from `sse-events.ts`. The handler should rely on the middleware-injected context, consistent with all other handlers.
