## 3. Security Header Ownership

### 3.1 Boundary principle

Security response headers that apply to all responses (HTML, API, static assets) are most reliably set at the **reverse proxy** layer, because:
- The proxy sees all traffic, including static files that Express may serve without middleware.
- A misconfigured or bypassed Express middleware cannot silently drop headers.
- Header configuration is declarative in Nginx, reducing the chance of code-level bugs.

The Express application sets the **same headers** as defense-in-depth, so that direct-to-Express access (development, or if the proxy is misconfigured) still receives browser hardening.

### 3.2 Header matrix

| Header | Reverse proxy (Nginx) | Express app | Notes |
|--------|----------------------|-------------|-------|
| `Content-Security-Policy` | Yes (primary) | Yes (defense-in-depth) | CSP must match between layers. See section 3.3. |
| `X-Frame-Options` | `DENY` | `DENY` | Redundant with CSP `frame-ancestors 'none'` but included for older browsers. |
| `X-Content-Type-Options` | `nosniff` | `nosniff` | |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | `strict-origin-when-cross-origin` | |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=(), payment=()` | `camera=(), microphone=(), geolocation=(), payment=()` | |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | Not set by app | HSTS must only be set when TLS is guaranteed. The app layer cannot know this. Proxy-only. |

### 3.3 CSP value

```
default-src 'self';
script-src 'self' cdn.jsdelivr.net;
style-src 'self' 'unsafe-inline' fonts.googleapis.com;
font-src fonts.gstatic.com;
img-src 'self' data:;
connect-src 'self';
frame-ancestors 'none'
```

**Open point (from master design):** Verify whether `login.html` or `index.html` contain inline `<script>` blocks. If so, either refactor to external scripts or add `'unsafe-inline'` to `script-src` (weakens CSP) or implement nonce-based CSP.

### 3.4 Nginx configuration pattern

```nginx
server {
    listen 443 ssl;
    server_name solfacil.example.com;

    # TLS config (certs, protocols, ciphers) — not in v6.9 scope

    # Security headers
    add_header Content-Security-Policy "default-src 'self'; script-src 'self' cdn.jsdelivr.net; style-src 'self' 'unsafe-inline' fonts.googleapis.com; font-src fonts.gstatic.com; img-src 'self' data:; connect-src 'self'; frame-ancestors 'none'" always;
    add_header X-Frame-Options "DENY" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Permissions-Policy "camera=(), microphone=(), geolocation=(), payment=()" always;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    location / {
        proxy_pass http://bff:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE support
        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 86400s;
    }
}
```

The `always` directive ensures headers are added on all response codes (including 4xx and 5xx).

### 3.5 Express middleware pattern

```typescript
app.use((req, res, next) => {
  res.setHeader('Content-Security-Policy', "default-src 'self'; ...");
  res.setHeader('X-Frame-Options', 'DENY');
  res.setHeader('X-Content-Type-Options', 'nosniff');
  res.setHeader('Referrer-Policy', 'strict-origin-when-cross-origin');
  res.setHeader('Permissions-Policy', 'camera=(), microphone=(), geolocation=(), payment=()');
  next();
});
```

Registered early in the middleware chain, before route handlers. Does not set HSTS (proxy responsibility).

## 4. Runtime Environment Requirements

### 4.1 Required environment variables for production

| Variable | Required in production | Startup behavior if missing | v6.9 change |
|----------|----------------------|---------------------------|-------------|
| `JWT_SECRET` | **Yes** | Application crashes with fatal error | **New: fail-fast** (was: silent fallback to dev secret) |
| `NODE_ENV` | Recommended (`production`) | No crash, but JWT_SECRET validation is not enforced | Existing behavior; v6.9 uses this to gate secret validation |
| `APP_DATABASE_URL` | Recommended | Falls back to `DATABASE_URL`, then localhost default | No change |
| `SERVICE_DATABASE_URL` | Recommended | Falls back to `DATABASE_URL`, then localhost default | No change |
| `DATABASE_URL` | Fallback for above | Hardcoded localhost default if all three unset | No change |

### 4.2 New optional environment variables

| Variable | Default | Purpose |
|----------|---------|---------|
| `LOGIN_RATE_LIMIT_WINDOW_MS` | 900000 (15 min) | Login rate limit window |
| `LOGIN_RATE_LIMIT_MAX` | 10 | Max login attempts per window per IP |

### 4.3 Express trust proxy

For the `Secure` cookie flag and correct `req.ip` extraction (rate limiting by IP), Express must be configured to trust the reverse proxy:

```typescript
app.set('trust proxy', 1); // Trust first proxy
```

Without this:
- `req.secure` will be `false` even when Nginx terminates TLS.
- `req.ip` will be the proxy's IP, not the client's IP, making rate limiting ineffective.

**Design decision:** Set `trust proxy` to `1` (trust one level of proxy). This is appropriate for the single-proxy deployment topology.

**Security note:** `trust proxy` must only be set when the application is behind a trusted proxy. In direct-to-internet deployment (no proxy), `trust proxy` must be `false` or unset. This is a deployment-time configuration choice.

## 5. Cookie and Proxy Interaction

### 5.1 Secure flag and TLS termination

The `Secure` cookie flag requires HTTPS. In the current topology, TLS is terminated at Nginx. Express sees HTTP.

The login handler sets `Secure` conditionally based on:
- `req.secure` (which is `true` when `trust proxy` is set and `X-Forwarded-Proto: https` is present), OR
- `NODE_ENV === 'production'` (as a safety net: always set Secure in production, assuming production always uses HTTPS)

### 5.2 SameSite and proxy

`SameSite=Lax` works correctly with reverse proxies. The browser evaluates SameSite based on the user-visible origin (the Nginx HTTPS origin), not the internal HTTP hop.

### 5.3 Cookie domain

The cookie `Domain` attribute is not set, so it defaults to the exact origin. This is correct — the cookie should only be sent to the Solfacil domain, not to subdomains.

## 6. Live Surface Verification

### 6.1 Verification checklist

After v6.9 deployment, the following must be verified against the live externally reachable surface:

| # | Check | Method | Expected result |
|---|-------|--------|-----------------|
| 1 | CSP header present | `curl -sI https://<host>/login.html \| grep -i content-security-policy` | Header present with expected value |
| 2 | X-Frame-Options | Same curl | `DENY` |
| 3 | X-Content-Type-Options | Same curl | `nosniff` |
| 4 | Referrer-Policy | Same curl | `strict-origin-when-cross-origin` |
| 5 | Permissions-Policy | Same curl | `camera=(), microphone=(), geolocation=(), payment=()` |
| 6 | HSTS | Same curl (HTTPS only) | `max-age=31536000; includeSubDomains` |
| 7 | Login sets HttpOnly cookie | Browser DevTools → Network → POST /api/auth/login → Response headers → `Set-Cookie` | `solfacil_jwt=...; HttpOnly; SameSite=Lax; Path=/; Max-Age=86400; Secure` |
| 8 | No JWT in login response body | Same request → Response body | `data.user` present, no `data.token` |
| 9 | No localStorage JWT | Browser DevTools → Application → Local Storage | No `solfacil_jwt` key |
| 10 | API works with cookie only | Browser DevTools → Network → any API call | No `Authorization` header in request. 200 response. |
| 11 | Rate limiting active | 11 rapid POST /api/auth/login with wrong password from same IP | 11th returns 429 with `Retry-After` header |
| 12 | Secret validation | Start app with `NODE_ENV=production` and no `JWT_SECRET` | Application refuses to start |
| 13 | CSP not violated | Browser DevTools → Console | No CSP violation errors on normal page load |
| 14 | SSE works with cookie | Browser DevTools → Network → /api/events | SSE connection established, events received |

### 6.2 Automated verification

Checks 1-6 and 8 can be automated as a post-deployment smoke test (curl + grep script). Check 11 can be automated as an integration test. Check 12 is a unit/integration test. Checks 7, 9, 10, 13, 14 are browser-based and best verified via E2E test or manual inspection.

### 6.3 Ongoing monitoring

Security headers should be checked periodically (e.g., weekly automated scan) to detect configuration drift. If Nginx config is modified for other reasons, headers may be accidentally removed.

## 7. Deployment Sequence

### 7.1 Recommended deployment order

Since the BFF serves both the API and the frontend static files, and they are deployed atomically from the same container/process, there is no complex sequencing required:

1. **Pre-deployment:** Ensure `JWT_SECRET` is set in the production environment. Ensure `NODE_ENV=production` is set.
2. **Deploy:** Update the BFF container/process with v6.9 code.
3. **Nginx update:** Apply security header configuration to Nginx. This can be done before or after the BFF deployment — the Express defense-in-depth headers provide coverage during the gap.
4. **Verify:** Run verification checklist (section 6.1).

### 7.2 Rollback

- **BFF rollback:** Redeploy v6.8 container. Users must re-login (see master design section 8).
- **Nginx rollback:** Revert Nginx config to remove security headers. This is independent of BFF rollback.

| # | Open point | Owner | Impact if unresolved |
|---|-----------|-------|---------------------|
| 1 | Verify whether `login.html`/`index.html` have inline `<script>` blocks | Frontend | CSP `script-src` may need `'unsafe-inline'` or nonce |
