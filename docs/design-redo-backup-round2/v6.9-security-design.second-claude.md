## 1. Purpose

This document is the master security design for Solfacil v6.9. It defines the target auth/session architecture, browser hardening model, login abuse controls, secret safety posture, migration path, rollback model, and verification strategy.

It does not prescribe implementation step lists. It states what the system must look like after v6.9, why, and where architectural boundaries fall.

Companion documents:
- `docs/v6.9-frontend-security-design.md` — browser/session model, login page, authenticated request flow, logout/expiry
- `docs/v6.9-backend-auth-boundary-design.md` — M5 + M6 + shared layer auth contract, middleware, cookie semantics, SSE, secret handling
- `docs/v6.9-deployment-security-boundary-design.md` — reverse proxy headers, runtime env/secret requirements, live-surface verification

## 2. Target Auth/Session Architecture

### 2.1 Current state (v6.8)

The v6.8 system is in a split auth model:

| Transport | Mechanism | Purpose | Risk |
|-----------|-----------|---------|------|
| `Authorization: Bearer <jwt>` | JS reads JWT from `localStorage("solfacil_jwt")` | All `fetch()`/XHR API calls | XSS-extractable |
| `solfacil_jwt` HttpOnly cookie | Set by server on login, sent automatically by browser | `EventSource` SSE auth (browser cannot set custom headers on SSE) | Not JS-extractable, but coexists with extractable bearer path |

The server login handler (`auth-login.ts`) returns the JWT in both the response body (for JS) and as an HttpOnly cookie. The frontend stores the body token in `localStorage` and uses it for all API calls. The cookie path is used only for SSE.

This is a half-migrated state. The HttpOnly cookie transport is already present and working, but the primary auth path still relies on JS-readable token storage.

### 2.2 Target state (v6.9)

**Primary auth transport: HttpOnly cookie only.**

After v6.9, the production auth model operates as follows:

| Concern | v6.9 Design |
|---------|-------------|
| **Session credential storage** | HttpOnly, Secure, SameSite=Lax cookie `solfacil_jwt` set by server. No JWT in `localStorage`. |
| **API request authentication** | Browser sends cookie automatically on same-origin requests. No `Authorization` header required for browser-originated requests. |
| **SSE authentication** | Unchanged from v6.8 — cookie already used. |
| **Token lifetime** | Unchanged: 24h expiry, no refresh token (Phase 1). |
| **Logout** | `POST /api/auth/logout` clears cookie server-side. Frontend clears any residual `localStorage` state and redirects. |
| **Session expiry** | Server returns 401 when cookie JWT is expired. Frontend 401 handler redirects to `login.html`. |
| **CSRF protection** | SameSite=Lax provides baseline CSRF resistance for state-changing POST/PUT/PATCH requests (browsers do not send SameSite=Lax cookies on cross-origin POST). Additional CSRF token not required at this stage because all state-changing operations use non-simple content types (`application/json`) which trigger CORS preflight on cross-origin requests. |

### 2.3 Auth contract summary

The v6.9 auth contract between frontend and backend is:

1. **Login:** Frontend POSTs credentials to `/api/auth/login`. Server validates, signs JWT, sets HttpOnly cookie. Server response body returns user metadata (userId, email, name, orgId, role) but **does not return the raw JWT token** in the response body.
2. **Authenticated requests:** Frontend makes same-origin `fetch()` calls with `credentials: 'same-origin'` (default for same-origin). Cookie is sent automatically. No `Authorization` header is set by frontend code.
3. **Backend extraction:** Auth middleware reads JWT from cookie (`req.cookies.solfacil_jwt`). If absent, checks `Authorization` header as fallback (for machine/API clients — see section 2.5).
4. **Logout:** Frontend calls `POST /api/auth/logout`, server clears cookie, frontend redirects.
5. **401 recovery:** On any 401 response, frontend redirects to `login.html`.

### 2.4 What this eliminates

- `localStorage("solfacil_jwt")` is no longer written or read for auth purposes.
- The `Authorization: Bearer <token>` header is no longer set by browser frontend code.
- The JWT is no longer present in the JSON login response body.
- Any XSS exploit can no longer directly extract the session credential.

### 2.5 Machine vs. browser separation

The v6.8 system has no documented machine-to-machine API consumers. All current API consumers are browser-based (the SPA frontend).

However, the auth middleware should still accept `Authorization: Bearer <token>` as a secondary path, for two reasons:
1. Future M7 Open API consumers or third-party integrations may need bearer token auth.
2. Existing development/testing tooling (curl, Postman) uses bearer tokens.

**Design rule:** Cookie takes precedence. If both cookie and Authorization header are present, the cookie is authoritative. The bearer path exists for non-browser clients only. The frontend must not use it.

### 2.6 Token format

No change to JWT payload or signing algorithm. HS256 with `JWT_SECRET`, 24h expiry, payload `{ userId, orgId, role, iat, exp }`.

## 3. Browser Hardening Model

### 3.1 Target header baseline

The following headers must be present on all responses served to browsers from the externally reachable surface:

| Header | Value | Enforcement point |
|--------|-------|-------------------|
| `Content-Security-Policy` | `default-src 'self'; script-src 'self' cdn.jsdelivr.net; style-src 'self' 'unsafe-inline' fonts.googleapis.com; font-src fonts.gstatic.com; img-src 'self' data:; connect-src 'self'; frame-ancestors 'none'` | Reverse proxy (primary) + app layer (defense-in-depth) |
| `X-Frame-Options` | `DENY` | Reverse proxy |
| `X-Content-Type-Options` | `nosniff` | Reverse proxy |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Reverse proxy |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=(), payment=()` | Reverse proxy |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` (if HTTPS terminated at proxy) | Reverse proxy |

### 3.2 CSP rationale

- `script-src 'self' cdn.jsdelivr.net` — the frontend loads ECharts from jsdelivr CDN. No inline scripts should be used. If inline event handlers exist in HTML, they must be refactored to external JS.
- `style-src 'self' 'unsafe-inline' fonts.googleapis.com` — CSS custom properties and inline styles are used extensively. `unsafe-inline` is necessary for the current CSS architecture. Google Fonts stylesheets are loaded from fonts.googleapis.com.
- `frame-ancestors 'none'` — equivalent to X-Frame-Options: DENY, prevents clickjacking.
- `connect-src 'self'` — all API calls and SSE connections are same-origin.

### 3.3 CSP open point

**Open point:** Audit whether any inline `<script>` blocks exist in `index.html` or `login.html`. If so, either move them to external files or add `'unsafe-inline'` to `script-src` (less preferred) or use nonce-based CSP (more complex). Current architecture doc says "all JS files are loaded via `<script>` tags" which suggests external scripts only, but this must be verified before finalizing CSP.

### 3.4 Header ownership

See `docs/v6.9-deployment-security-boundary-design.md` for the full deployment boundary ownership model. Summary: reverse proxy is the primary enforcement point for security headers; the Express app sets them as defense-in-depth via middleware.

## 4. Login Abuse Control

### 4.1 Design choice

Application-layer rate limiting on the login endpoint, implemented as in-memory sliding window per IP address.

### 4.2 Topology

| Component | Responsibility |
|-----------|----------------|
| **Express middleware** (app layer) | Per-IP sliding window rate limit on `POST /api/auth/login`. Configurable window size and max attempts. Returns 429 Too Many Requests when threshold exceeded. |
| **Reverse proxy** (infra layer) | Connection-level rate limiting (if available) as defense-in-depth. Not relied upon as the sole control. |

### 4.3 Parameters

| Parameter | Default | Configurable via |
|-----------|---------|-----------------|
| Window size | 15 minutes | `LOGIN_RATE_LIMIT_WINDOW_MS` env var |
| Max attempts per window | 10 | `LOGIN_RATE_LIMIT_MAX` env var |
| Lockout behavior | Reject with 429 until window expires | — |

### 4.4 Design notes

- **In-memory store:** Acceptable for Phase 1 single-instance deployment. If Solfacil scales to multiple BFF instances, this must migrate to a shared store (Redis, database). This is a documented residual limitation, not a design defect.
- **Per-IP, not per-account:** Prevents enumeration attacks. An attacker cannot lock out a legitimate user by brute-forcing their email from a different IP.
- **No account lockout:** Deliberate. Account lockout creates a denial-of-service vector against legitimate users. IP-based throttling is the correct control for this threat model.
- **Response shape:** 429 response uses the standard `fail()` envelope: `{ success: false, error: "Too many login attempts. Try again later.", data: null }`. The `Retry-After` header is set to the number of seconds remaining in the window.

### 4.5 What this does NOT cover

- CAPTCHA or proof-of-work challenges — not in scope for v6.9.
- Geographic or reputation-based blocking — infrastructure concern, not application design.
- Distributed brute-force from many IPs — acknowledged residual risk (see section 9).

## 5. Secret Safety

### 5.1 Current state

`JWT_SECRET` in `shared/middleware/tenant-context.ts` falls back to `"solfacil-dev-secret"` when the environment variable is unset.

### 5.2 Target state

**Fail-fast on missing secret.** The application must refuse to start if `JWT_SECRET` is not set or is set to a known-unsafe value.

Design:
1. At application startup (in `local-server.ts` initialization), validate `JWT_SECRET`:
   - If `JWT_SECRET` is unset/empty: crash with a clear error message.
   - If `JWT_SECRET` equals `"solfacil-dev-secret"` and `NODE_ENV` is `"production"`: crash with a clear error message.
   - If `NODE_ENV` is not `"production"` (or unset), the dev fallback is permitted for local development convenience.
2. `verifyTenantToken()` continues to read `JWT_SECRET` from env. The fallback string `"solfacil-dev-secret"` remains in code but is gated by the startup check — it can only be reached in non-production environments.

### 5.3 Rationale for gating on NODE_ENV

Removing the fallback entirely would break local development ergonomics (developers would need to set `JWT_SECRET` in every dev environment). The correct fix is to make the unsafe path unreachable in production, not to remove it from development.

### 5.4 Other secrets

| Secret | Current handling | v6.9 change |
|--------|-----------------|-------------|
| `JWT_SECRET` | Env var with unsafe fallback | Fail-fast in production (see above) |
| `APP_DATABASE_URL` | Env var with hardcoded localhost fallback | No change — hardcoded DSN is localhost-only, not a production risk |
| `SERVICE_DATABASE_URL` | Same pattern | No change — same reasoning |
| bcrypt cost factor | Hardcoded 12 | No change — appropriate for current use |

## 7. Migration Model

### 7.1 Migration scope

The auth model migration affects:
1. **Frontend login code** (`login.html`) — stops storing JWT in localStorage, stops reading it for auth guard.
2. **Frontend API calls** (`data-source.js`) — stops setting `Authorization` header.
3. **Frontend auth guard** (`app.js`) — changes from localStorage check to a lightweight session-validity check.
4. **Backend login handler** (`auth-login.ts`) — stops returning JWT in response body.
5. **Backend auth middleware** (`bff/middleware/auth.ts`) — reads cookie first, bearer header second.
6. **Backend startup** (`local-server.ts`) — adds JWT_SECRET validation.
7. **Reverse proxy config** — adds security headers.
8. **Backend rate limit middleware** — new, applied to login route.

### 7.2 Mixed-version behavior

During deployment, there is a brief window where the new backend may be running while a browser still has a cached old frontend, or vice versa.

**New backend + old frontend (cached):**
- Old frontend sends `Authorization: Bearer <token>` header.
- New backend middleware checks cookie first, falls back to Authorization header.
- **Works.** The bearer fallback path handles this case.
- On next login, old frontend stores JWT in localStorage but also receives the cookie. The old frontend continues to work until the user clears cache or the JWT expires.

**Old backend + new frontend (preloaded):**
- This scenario is unlikely in a single-instance deployment (backend and frontend are served from the same process). If it occurred, the new frontend would not send an Authorization header, but the old backend middleware would not find a cookie-reading path.
- **Mitigated by:** deploying backend and frontend atomically (they are in the same repository and served by the same Express instance).

### 7.3 Transition bridge

No explicit bridge mode is designed. The bearer-header fallback in the new middleware is the implicit bridge. It handles stale cached frontends.

The bearer fallback is not a permanent bridge — it exists because non-browser clients (development tools, future M7 consumers) legitimately need it. The fact that it also handles stale frontends is a side effect, not a design goal.

### 7.4 localStorage cleanup

On first load after v6.9 deployment, the new frontend should remove `localStorage("solfacil_jwt")` if present. This is a one-time cleanup, not a persistent behavior.

## 8. Rollback Model

### 8.1 Full rollback to v6.8

If v6.9 must be fully rolled back:
1. Redeploy v6.8 code (backend + frontend).
2. Users whose sessions were established under v6.9 still have valid `solfacil_jwt` cookies. The v6.8 backend auth middleware does not read cookies for non-SSE routes, so these users would need to re-login (their localStorage is empty under v6.9 frontend, and v6.8 frontend expects localStorage).
3. **Impact:** All active sessions are invalidated on rollback. Users must re-login. This is acceptable for a security correction release.

### 8.2 Partial rollback

There is no meaningful partial rollback path. The auth model is either cookie-primary or localStorage-primary. Mixing the two is the problem v6.9 is correcting.

### 8.3 Reverse proxy rollback

Security headers added at the reverse proxy can be removed independently of application code rollback. This is a configuration change, not a code deployment.

## 9. Residual Risks After v6.9

| Risk | Severity | Reason it remains | Mitigation path |
|------|----------|-------------------|-----------------|
| No refresh token — 24h session window | Low | Phase 1 design decision. Re-login required daily. | Phase 2: implement refresh token rotation. |
| No MFA | Medium | Phase 1 scope. Single-factor password auth. | Phase 2: Cognito or TOTP integration. |
| In-memory rate limit not shared across instances | Low | Single-instance deployment. If horizontally scaled, rate limit state is per-instance. | Move to Redis or database-backed rate limiting when scaling. |
| Distributed brute-force (many IPs) | Low | IP-based rate limiting does not stop slow attacks from many sources. | Future: reputation-based blocking, CAPTCHA on threshold. |
| `unsafe-inline` in CSP `style-src` | Low | Required by current CSS architecture (inline styles, CSS custom properties). | Refactor inline styles to classes; move to nonce-based CSP. |
| Raw JSON token path in `verifyTenantToken()` | Low | Exists for backward compatibility with middleware header rewriting. Not exposed externally — only used internally after middleware validation. | Remove when all consumers are migrated to direct context passing. |
| Password reset not implemented | Medium | Phase 1 limitation. Admin must manually reset. | Phase 2: email-based password reset flow. |
| No audit log for login attempts | Low | Login failures are not persisted. Rate limiting is in-memory. | Future: persist login attempt log for forensics. |

## 10. Verification Strategy

### 10.1 Automated tests

| Test category | What it verifies |
|---------------|------------------|
| **Unit: login handler** | Returns user metadata without raw JWT in body. Sets HttpOnly cookie. Clears cookie on logout. |
| **Unit: auth middleware** | Extracts JWT from cookie. Falls back to Authorization header. Rejects requests with neither. |
| **Unit: rate limiter** | Returns 429 after threshold. Resets after window. Per-IP isolation. |
| **Unit: secret validation** | Startup fails when JWT_SECRET missing in production. Startup succeeds in dev without JWT_SECRET. |
| **Integration: login flow** | Full login → authenticated request → logout → rejected request cycle using cookie transport. |
| **Integration: stale frontend** | Bearer-header request succeeds (backward compatibility for non-browser clients). |
| **E2E: browser login** | Login page → cookie set → SPA loads → API calls succeed → logout → redirect to login. |

### 10.2 Deployment verification

| Check | Method |
|-------|--------|
| Security headers present | `curl -I` against live surface; verify all required headers. |
| Cookie attributes correct | Login via browser dev tools; inspect `Set-Cookie` for HttpOnly, Secure (if HTTPS), SameSite=Lax, Path=/. |
| localStorage not used | Browser dev tools → Application → Local Storage → verify no `solfacil_jwt` key after login. |
| Rate limiting active | 11 rapid failed login attempts from same IP → 11th returns 429. |
| JWT_SECRET enforced | Attempt to start application in production mode without JWT_SECRET → verify crash. |
| CSP active | Browser console → verify no CSP violation warnings on normal page load. |

### 10.3 Regression guard

Tests from 10.1 become part of the CI suite. Any future change that reintroduces localStorage token storage, removes cookie auth, or weakens the secret validation will fail these tests.

## 11. Design Questions Answered

(Per REQ section 12)

| # | Question | Answer |
|---|----------|--------|
| 1 | What exact target auth model replaces the current split? | HttpOnly cookie as sole browser session transport. Bearer header retained for non-browser clients only. See section 2. |
| 2 | How will frontend authenticated requests operate? | Same-origin fetch with `credentials: 'same-origin'` (default). Cookie sent automatically. No Authorization header. See `v6.9-frontend-security-design.md`. |
| 3 | What is the migration path? | Backend accepts both cookie and bearer (cookie preferred). Frontend stops using localStorage/bearer. Stale cached frontends handled by bearer fallback. See section 7. |
| 4 | Which browser security headers are set where? | Reverse proxy primary, Express middleware defense-in-depth. See section 3 and `v6.9-deployment-security-boundary-design.md`. |
| 5 | What login throttling design is chosen? | In-memory per-IP sliding window, 10 attempts / 15 min, 429 response. See section 4. |
| 6 | How is JWT secret validation enforced? | Startup crash if JWT_SECRET missing/unsafe in production (`NODE_ENV=production`). See section 5. |
| 7 | What test suite proves closure? | Unit + integration + E2E tests covering all five REQ outcomes. Deployment verification checklist. See section 10. |
| 8 | What residual risks remain? | No MFA, no refresh token, in-memory rate limit, distributed brute-force, unsafe-inline CSP, no audit log. All documented with mitigation paths. See section 9. |
