# v6.9 Redis isolation fix note

Date: 2026-04-17
Scope: Solfacil production Redis isolation correction after temporary shared-host fallback

## Problem

During production rollout, Solfacil BFF was temporarily pointed at the shared host Redis via:

- `.env`: `RATE_LIMIT_REDIS_URL=redis://:***@host.docker.internal:6379`
- production override: BFF reused host Redis instead of a dedicated Solfacil Redis

That was operationally expedient but architecturally wrong:
- shared Redis already served unrelated systems
- Solfacil was not isolated at instance/DB level
- coarse operations on shared Redis could affect Solfacil login abuse-control state

## Root cause

The issue was not that Redis must use port 6379.
The real obstacle was the base `docker-compose.yml` shape:

- `solfacil-redis` published `127.0.0.1:6379:6379`

On the shared production host, host port `6379` was already occupied by the shared Redis.

## Important merge findings

Two concrete Docker Compose findings were verified live:

1. `ports: []` in the production override did **not** clear the base host-port mapping.
2. The effective way to clear inherited ports was:

```yaml
ports: !reset []
```

Also, using:

```yaml
RATE_LIMIT_REDIS_URL: ${RATE_LIMIT_REDIS_URL:-redis://solfacil-redis:6379}
```

was unsafe in production because `.env` still carried the old shared-host Redis URL and silently won the merge.

## Corrected production override

Final steady-state override:

```yaml
version: '3.8'

services:
  solfacil-redis:
    ports: !reset []

  solfacil-bff:
    environment:
      RATE_LIMIT_REDIS_URL: redis://solfacil-redis:6379
```

This gives Solfacil:
- dedicated Redis container
- internal-only Redis exposure
- no host 6379 bind
- BFF connected only through Docker network

### Cleanup completed
The temporary shared-host Redis path was fully cleaned up:
- shared host Redis had no remaining Solfacil `login_*` keys
- shared host Redis had no Solfacil client
- `/opt/solfacil/.env` no longer contains `RATE_LIMIT_REDIS_URL`
- backup created: `.env.bak-20260417-093758`

## Functional regression check

After isolation correction:
- browser login still works
- cookie session contract still works
- `localStorage.solfacil_jwt === null`
- fleet landing page still loads normally

## Final engineering ruling

Shared-host Redis reuse was a bounded rollout compromise, not a valid steady state.
Production is now back to the correct architecture:
- independent Solfacil Redis
- internal-only network path
- no dependency on shared host Redis for abuse-control state
