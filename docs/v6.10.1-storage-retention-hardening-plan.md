## 1. Goal / architecture baseline / tech stack

### 1.1 Goal

Implement the accepted v6.10.1 storage-retention hardening design by:
- hard-capping Docker `json-file` container logs for the local/runtime stack
- extending the existing shared v6.10 runtime-retention executor into the single hourly storage-retention executor
- adding additive archive/schema support before any archive movement runs
- enforcing the committed table-specific lifecycle rules without reopening design choices

### 1.2 Binding architectural baseline

Non-negotiable implementation rules carried into this plan:
- `revenue_daily` stays preserve-in-hot-storage with no TTL and no archive move.
- Docker stays on `json-file` with `max-size: 10m` and `max-file: 5` for `solfacil-bff`, `solfacil-db`, `solfacil-m1`, `solfacil-redis`.
- `device_command_logs`: 90-day hot window, archive-first to `device_command_logs_archive`, then delete hot row; pending rows cannot be swept by age alone.
- `gateway_alarm_events`: 180-day hot window, archive-first to `gateway_alarm_events_archive`, then delete hot row.
- `backfill_requests`: retain active rows; delete terminal rows (`completed`, `failed`) after 14 days; no archive.
- `runtime_issues`: delete closed rows 30 days after `closed_at`; no archive.
- enforcement path is one hourly shared storage-retention executor by extending `backend/src/shared/runtime/retention-job.ts`
- rollback stays behavioral/additive; no destructive schema rollback

### 1.3 Implementation surfaces / tech stack

- PostgreSQL 15 schema + SQL migrations
- TypeScript backend under `backend/`
- shared runtime/storage orchestration under `backend/src/shared/runtime/`
- local runtime bootstrap under `backend/scripts/`
- Docker Compose runtime definition in `docker-compose.yml`
- Jest + TypeScript build/test lanes under `backend/test/`

## 4.2 WS2 — shared executor extension under M9/shared

### Primary target files
- `backend/src/shared/runtime/retention-job.ts`
- `backend/src/shared/runtime/storage-policy.ts` *(new; canonical checked-in M9 storage-governance registry constant)*
- `backend/src/shared/runtime/emit.ts`
- `backend/src/shared/runtime/contract.ts`
- `backend/src/shared/runtime/persistence.ts` *(only if helper query plumbing belongs here)*
- `backend/test/shared/runtime-retention.test.ts`
- `backend/test/shared/storage-policy.test.ts` *(new; if registry lands as a typed constant with invariants worth pinning)*
- `backend/test/shared/db-runtime.test.ts` *(if shared DB/runtime integration helpers are touched)*

### Implementation slice
Extend the current v6.10 retention executor; do not create a separate competing job.
Also land the checked-in storage-governance registry constant in `backend/src/shared/runtime/storage-policy.ts` so the governed surfaces, classes, cutoffs, executor phase names, and rollback posture are declared once in M9/shared rather than being re-encoded ad hoc inside retention helpers.

#### Phase 1 — preserve existing runtime-retention behavior
Keep current behavior intact for:
- recovered issue auto-close
- stale issue auto-close
- `runtime_events` delete > 90 days
- `runtime_health_snapshots` delete > 30 days

#### Phase 2 — add closed `runtime_issues` cleanup
- delete only rows with `state = 'closed'`
- cutoff is `closed_at < now - 30 days`
- no archive table
- active/recovered/suppressed rows are never swept by this phase

#### Phase 3 — add `device_command_logs` archive-first movement
- hot cutoff: rows older than 90 days
- archive into `device_command_logs_archive` first
- only then delete from `device_command_logs`
- eligibility must exclude age-only sweep of non-terminal rows
- terminal eligibility is **exactly**: `result IN ('success','fail','timeout') OR resolved_at IS NOT NULL`
- rows in `pending`, `dispatched`, or `accepted` state are never swept by age alone
- preserve rerun idempotence if archive insert succeeded but delete did not

#### Phase 4 — add `gateway_alarm_events` archive-first movement
- hot cutoff: rows older than 180 days
- archive into `gateway_alarm_events_archive` first
- delete hot rows only after successful archive insert
- keep recent hot-table readers targeting the hot table unchanged

#### Phase 5 — add `backfill_requests` terminal cleanup
- terminal statuses only: `completed`, `failed`
- cutoff: 14 days after `completed_at`, otherwise 14 days after `created_at`
- direct delete only; no archive table
- active/non-terminal rows remain untouched

### Shared execution rules to preserve
- single hourly executor
- per-phase counters in result payload
- per-phase independent failure capture
- archive-first phases tolerate duplicate-safe reruns
- executor failure must emit the concrete runtime-governance housekeeping event code `storage.retention.executor.failed`, registered in `backend/src/shared/runtime/contract.ts`

### Gate evidence
- retention test lane covers every phase independently via `backend/test/shared/runtime-retention.test.ts`
- `backend/test/shared/runtime-retention.test.ts` contains named failure-injection cases proving later phases still execute when an earlier phase fails
- `backend/test/shared/runtime-retention.test.ts` (or a new adjacent shared integration test file if it becomes clearer) contains named duplicate-safe rerun cases for both archive tables
- `backend/test/shared/storage-policy.test.ts` proves the registry constant matches the committed v6.10.1 policy surfaces and cutoffs

## 6. Shared executor implementation details

## 6.1 Expected result/reporting shape

The extended executor result should report at least:
- existing runtime-retention counters
- `runtimeIssuesDeleted`
- `deviceCommandArchived`
- `deviceCommandDeleted`
- `gatewayAlarmArchived`
- `gatewayAlarmDeleted`
- `backfillDeleted`
- per-phase error records
- overall status retaining the current disabled/completed/degraded posture

## 6.2 Transaction and rerun rules

Per design, do not wrap the whole run in one all-or-nothing transaction.
- each phase should commit independently
- archive-first phases may use bounded batch transactions internally
- if archive insert succeeds and delete fails, rerun must remain safe
- duplicate-safe archive identity is mandatory before delete is allowed

## 6.3 Batch discipline

Implementation should keep bounded batch behavior practical for hourly runs.
Committed default for v6.10.1:
- archive/delete helpers use bounded chunking with a default ceiling of **5,000 rows per phase per run**
- the chunk size is an implementation constant under M9/shared runtime, not an operator-facing setting
- if a future environment needs a different ceiling, that is a later operational tuning change, not a v6.10.1 design reopen

## 6.4 Failure surfacing

- shared executor failures must surface through the runtime-governance spine
- fallback logging must remain in place
- one failed phase must not suppress execution of the remaining phases in the same run

## 9. Rollout and rollback

## 9.1 Rollout order

1. merge additive migration and DB contract updates
2. apply migration in local/release environment while existing behavior still runs
3. verify archive tables and helper indexes exist
4. merge shared executor extension
5. verify ad-hoc executor output with `RUNTIME_GOVERNANCE_ENABLED=true npm run runtime:retention`
6. merge/update scheduler wiring in `backend/scripts/local-server.ts`
7. merge Docker compose logging stanza update
8. redeploy / recreate services
9. verify compose logging config on running containers
10. run final acceptance matrix

## 9.2 Rollback posture

Rollback is behavioral/additive only.

Allowed rollback actions:
- disable or stop invoking the extended shared retention executor
- redeploy prior compose/runtime code if executor behavior is suspect
- leave archive tables in place
- leave already archived rows preserved

Explicitly not part of rollback:
- dropping archive tables
- deleting archived data to “undo” rollout
- destructive schema reversal of additive migration

## 9.3 Rollback checklist

1. stop/disable the shared retention executor path
2. redeploy prior application code if needed
3. confirm business endpoints still operate normally
4. confirm archive tables remain readable
5. confirm no further archive/delete movement occurs after executor disablement
