**Cookie correctness chain verification:**

```
1. Browser sends HTTPS request to solfacil.alwayscontrol.net
2. Nginx terminates TLS, sets X-Forwarded-Proto: https
3. Express (trust proxy=1) sees req.secure=true
4. Login handler sets cookie with secure: true
5. Browser receives Set-Cookie with Secure flag
6. Browser accepts cookie (connection is HTTPS)
```

Each link must be verified. If any breaks, the `__Host-` cookie fails.

## 3. Live Curl/Header/Cookie Verification Tasks

After deployment, these verification tasks must be executed against the live surface. They are **not** automated tests — they are operator-run acceptance checks.

### 3.1 Header Verification

```bash
# V-H1: All security headers present on HTML response
curl -sI https://solfacil.alwayscontrol.net/ | grep -iE \
  "content-security-policy|x-frame-options|x-content-type-options|referrer-policy|permissions-policy|strict-transport-security"

# Expected output (6 headers):
# Content-Security-Policy: default-src 'none'; script-src 'self'; ...
# X-Frame-Options: DENY
# X-Content-Type-Options: nosniff
# Referrer-Policy: strict-origin-when-cross-origin
# Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=()
# Strict-Transport-Security: max-age=31536000; includeSubDomains

# V-H2: Security headers on API JSON response
curl -sI https://solfacil.alwayscontrol.net/api/auth/session | grep -iE \
  "content-security-policy|x-frame-options|x-content-type-options"
# Expected: same CSP, X-Frame-Options, nosniff present

# V-H3: No duplicate headers
curl -sI https://solfacil.alwayscontrol.net/ | grep -ci "content-security-policy"
# Expected: 1 (not 2+)
```

### 3.2 Cookie Verification

```bash
# V-C1: Login sets correct cookie
curl -v -X POST https://solfacil.alwayscontrol.net/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"test@test.com","password":"..."}' \
  -c cookies.txt 2>&1 | grep "Set-Cookie"

# Expected: Set-Cookie: __Host-solfacil_session=...; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=86400

# V-C2: Cookie attributes complete
# Check cookies.txt for:
#   - Name: __Host-solfacil_session
#   - HttpOnly flag: TRUE
#   - Secure flag: TRUE
#   - Path: /
#   - No Domain attribute

# V-C3: Session endpoint works with cookie
curl -s https://solfacil.alwayscontrol.net/api/auth/session -b cookies.txt
# Expected: {"success":true,"data":{"userId":"...","orgId":"...","role":"...","name":"...","email":"..."}}

# V-C4: Session endpoint rejects without cookie
curl -s https://solfacil.alwayscontrol.net/api/auth/session
# Expected: 401

# V-C5: Logout clears cookie
curl -v -X POST https://solfacil.alwayscontrol.net/api/auth/logout -b cookies.txt 2>&1 | grep "Set-Cookie"
# Expected: Set-Cookie: __Host-solfacil_session=; ... Max-Age=0
```

### 3.3 Abuse-Control Verification

```bash
# V-A1: Rate limiting triggers on 6th attempt
for i in $(seq 1 6); do
  CODE=$(curl -s -o /dev/null -w "%{http_code}" -X POST \
    https://solfacil.alwayscontrol.net/api/auth/login \
    -H 'Content-Type: application/json' \
    -d '{"email":"ratelimit-test@test.com","password":"wrongpassword"}')
  echo "Attempt $i: $CODE"
done
# Expected: attempts 1-5 → 401, attempt 6 → 429

# V-A2: Retry-After header present on 429
curl -sI -X POST https://solfacil.alwayscontrol.net/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"ratelimit-test@test.com","password":"wrongpassword"}' | grep "Retry-After"
# Expected: Retry-After: <seconds>
```

### 3.4 Machine Contract Verification

```bash
# V-M1: Machine login returns token in body
curl -s -X POST https://solfacil.alwayscontrol.net/api/auth/login \
  -H 'Content-Type: application/json' \
  -H 'X-Auth-Contract: machine' \
  -d '{"email":"test@test.com","password":"..."}' | jq '.data.token'
# Expected: JWT string (not null)

# V-M2: Machine bearer on data route
TOKEN=$(curl -s -X POST https://solfacil.alwayscontrol.net/api/auth/login \
  -H 'Content-Type: application/json' \
  -H 'X-Auth-Contract: machine' \
  -d '{"email":"test@test.com","password":"..."}' | jq -r '.data.token')

curl -s https://solfacil.alwayscontrol.net/api/fleet/overview \
  -H "Authorization: Bearer $TOKEN"
# Expected: 200 with data

# V-M3: Machine bearer rejected on session endpoint
curl -s -o /dev/null -w "%{http_code}" \

```

## 4. Trusted-Proxy Correctness Verification

The `__Host-` cookie correctness depends on a chain of trust from browser through Nginx to Express. If any link breaks, cookies fail.

| Link | What Must Be True | How to Verify |
|------|-------------------|---------------|
| Browser → Nginx | Connection is HTTPS | URL bar shows `https://` |
| Nginx → Express | `X-Forwarded-Proto: https` header set | App log or `req.headers['x-forwarded-proto']` check |
| Express trust proxy | `app.set('trust proxy', 1)` configured | Code review of `local-server.ts` |
| Express `req.secure` | Returns `true` behind HTTPS proxy | Debug log in login handler or test |
| Cookie `Secure` flag | Set to `true` in `res.cookie` call | `Set-Cookie` header inspection |
| Browser acceptance | Browser stores `__Host-solfacil_session` cookie | Application → Cookies in DevTools |

**Failure mode**: If `X-Forwarded-Proto` is missing from Nginx config, Express sees `req.secure = false`. Depending on Express version behavior, the cookie may be set without the `Secure` flag, or may not be accepted by the browser with the `__Host-` prefix. This would silently break authentication.

**Mitigation**: Task D1.2 is critical path. Verify before v6.9 deployment.

## 7. Rollback Procedures — Deployment Layer

| Scenario | Action | Impact |
|----------|--------|--------|
| v6.9 code bug → rollback to v6.8 | Redeploy v6.8 container. Nginx HSTS and proxy headers remain (compatible). Redis service can remain running (unused by v6.8) | Users re-login with v6.8 auth model |
| Cookie `Secure` flag issue behind proxy | Verify `X-Forwarded-Proto` in Nginx. If urgent: set `LEGACY_BROWSER_BEARER=true` temporarily (max 48h) | Bridge allows bearer auth; fix proxy config ASAP |
| CSP breaks a page | Change app-layer CSP to `Content-Security-Policy-Report-Only` temporarily. Identify violation. Fix and re-enforce | Temporary CSP non-enforcement |
| Redis outage | Restart Redis. If unrecoverable: login is blocked (fail-closed by design). Existing sessions continue working (cookie-based, no Redis dependency for verification) | New logins blocked; existing sessions unaffected |
| HSTS too aggressive | Reduce `max-age` or remove from Nginx. Note: browsers cache HSTS for `max-age` duration, so effect is not immediate for users who already visited | Gradual rollback |
