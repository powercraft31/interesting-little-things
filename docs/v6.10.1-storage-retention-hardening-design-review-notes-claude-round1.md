# 1. Overall judgment

The v6.10.1 design set is directionally correct but not ready for PLAN. The matrix packaging is clean, the category boundaries are mostly right, and the docs correctly preserve v6.10 runtime-governance authority while refusing to treat `revenue_daily` as disposable residue. The problem is that too many required answers are still expressed as posture words instead of decisions. Several REQ acceptance questions are only half-answered because the docs say things like "time-window-bounded", "bounded diagnostic window", "defined secondary window", and "deterministic rotation" without the committed values or per-surface cut decisions that PLAN would need.

There are blockers. The biggest ones are unresolved delete-vs-archive cuts, missing concrete retention/log-cap values, and an internal contradiction over whether closed `runtime_issues` cleanup is mandatory or merely recommended.

# 2. What is correct

- **Matrix packaging is correct.** `00_MASTER_ARCHITECTURE_v6.10.1.md` names the right canonical delta docs: M1, M4, M5, M9, DB, and FE. M2/M3/M6/M7/M8 are explicitly inherited, which is the right narrow packaging for this release.
- **Scope discipline is intact.** The docs do not reopen v6.10 runtime-governance semantics, do not invent a new storage dashboard, and do not broaden into warehouse redesign or generalized observability.
- **Category discipline is mostly strong.** M9 defines the right storage classes: volatile runtime logs, runtime-governance history, runtime-governance projection, operational history, queue residue, business history, and audit archive.
- **`runtime_issues` honesty is at least recognized.** Master and M9 both explicitly state that auto-close is not deletion and that `runtime_issues` cannot be called bounded if rows never leave storage.
- **`revenue_daily` is correctly protected.** M4 is clear that it is business/system-of-record history, not runtime residue, and must not receive a blind TTL in v6.10.1.
- **Docker log posture is correctly framed.** M5 keeps `json-file` and adds `max-size`/`max-file` rotation instead of inventing a bigger observability project.
- **Rollback bias is correct.** M9 and DB keep this additive and behavioral: disable executors/config, do not require destructive schema rollback, and keep archive support additive if introduced.
- **Ownership boundaries are mostly consistent.** M1/M4/M5 define local data meaning while M9 owns cross-cutting storage-governance policy and orchestration.

# 3. Blocking issues

- **B1 — Closed `runtime_issues` lifecycle is internally inconsistent across docs.**
  - Master says closed rows *receive* a real cleanup/archive lifecycle.
  - M9 presents cleanup as a *recommended default*.
  - DB says *if* closed-row cleanup is adopted, schema must support it.
  This is not a wording nit. REQ explicitly calls out `runtime_issues` lifecycle honesty as a hard requirement. The design must decide whether v6.10.1 mandates closed-row cleanup or not.

- **B2 — Delete vs archive is not resolved for the governed surfaces.**
  The docs leave the core lifecycle choice as "archive or delete" on the main tables that required hardening:
  - `device_command_logs`
  - `gateway_alarm_events`
  - `backfill_requests`
  - closed `runtime_issues`
  REQ acceptance requires explicit policy, not a menu of options. PLAN cannot implement or verify an "or".

- **B3 — No concrete retention windows or Docker rotation values are committed.**
  REQ asks what Docker settings are adopted and what each table's retention/archive policy is. The design does not commit actual values for:
  - Docker `max-size`
  - Docker `max-file`
  - hot-window duration for `device_command_logs`
  - hot-window duration for `gateway_alarm_events`
  - terminal-row retention duration for `backfill_requests`
  - secondary cleanup window for closed `runtime_issues`
  Without values, the design does not actually answer REQ §12 and does not support capacity budgeting.

- **B4 — Capacity budgeting is promised but not delivered.**
  M9 says every governed surface needs current size, growth unit, retention window/archive cutoff, and steady-state estimate. No such table actually exists in the design set. REQ treats this as a design deliverable, not a future intention.

- **B5 — Enforcement path is still too abstract.**
  M9 says there will be a database lifecycle lane additive to the existing runtime-retention executor, but it does not decide enough to call this real enforcement. Missing cuts include:
  - whether this extends the existing runtime retention job or introduces a second job
  - how archive-then-delete is sequenced safely
  - what happens on partial failure
  - whether executor failure itself is surfaced into runtime governance
  REQ explicitly rejects policy theater. This is still too hand-wavy.

# 4. Non-blocking weaknesses / ambiguities

- **`device_command_logs` safety predicate is too vague.** M5 says pending/timeout behavior must remain compatible before archival/deletion applies, but does not convert that into an explicit eligibility predicate.
- **`backfill_requests` cleanup predicate is incomplete.** M1 says active rows stay queryable and older terminal rows are deleted or archived, but does not fully lock the terminal-state condition into the design language.
- **Archive-table shape is underspecified.** DB says archive tables may be additive and preserve source identity for audit/tracing, but does not name the convention, schema placement, or fidelity requirement.
- **Container-log scope has an exception escape hatch but no actual exception list.** M5 says the same policy applies across Solfacil containers unless a specific exception is justified. The design never says whether `solfacil-db` and `solfacil-redis` follow the same settings.
- **The storage-governance registry is only described conceptually.** M9 does not say whether the registry is a TypeScript constant, configuration artifact, DB table, or just documentation.
- **The synthesis doc is too thin to help reviewers.** It correctly says the matrix docs are canonical, but it is only a pointer stub. A small REQ-to-doc answer map would improve reviewability without changing the matrix truth model.
- **Frontend doc may need one explicit sentence about `#runtime` behavior after closed-row cleanup.** No new page is correct, but the design is silent on whether any copy/expectation changes are needed once old closed issues no longer remain forever visible.

# 5. Decisions that still need Alan

1. Is closed-row cleanup for `runtime_issues` mandatory in v6.10.1, or optional?
2. For closed `runtime_issues`, is the policy archive-first or direct delete after the secondary window?
3. What hot-window duration applies to `device_command_logs`?
4. For `device_command_logs`, is the post-window path archive or delete?
5. What hot-window duration applies to `gateway_alarm_events`?
6. For `gateway_alarm_events`, is the post-window path archive or delete?
7. What diagnostic retention window applies to terminal `backfill_requests` rows?
8. For terminal `backfill_requests`, is archive needed at all, or is direct delete the accepted posture?
9. What exact Docker `max-size` and `max-file` values are adopted?
10. Does one Docker log policy apply to all four observed containers (`solfacil-bff`, `solfacil-db`, `solfacil-m1`, `solfacil-redis`), or are there justified per-container differences?
11. Is `revenue_daily` purely classification-only in v6.10.1, or does this release also introduce future archive hooks for it?
12. Does the DB lifecycle executor extend the existing runtime retention job or ship as a separate job?

# 6. Recommended revisions before PLAN

1. **Resolve the `runtime_issues` contradiction** across master, M9, and DB. One posture only.
2. **Convert every major "archive or delete" branch into a committed per-table decision** in the owning matrix doc.
3. **Add explicit values** for Docker log rotation and every governed retention window.
4. **Add one canonical policy matrix** in M9 or DB with at least: surface, storage class, hot-window, post-window path, executor, rollback knob, owning module.
5. **Actually include the capacity-budgeting table** required by REQ: current size, growth driver, expected rows/day or MB/day, retention/archive cutoff, steady-state estimate.
6. **Name the executor shape concretely**: existing runtime retention job extension vs separate shared job, cadence, failure handling, and archive/delete transaction semantics.
7. **Tighten eligibility rules** for `device_command_logs` and `backfill_requests` so active/pending rows cannot be swept by a loose cutoff.
8. **Either define archive-table conventions now or explicitly defer archive-table design** and commit delete-only for the surfaces where that is acceptable.
9. **Strengthen the synthesis doc** with a brief REQ acceptance-question map pointing to the canonical matrix locations.
10. **If `#runtime` needs no copy change despite closed-row cleanup, say that explicitly in FE** so the no-frontend-work posture is documented, not assumed.
