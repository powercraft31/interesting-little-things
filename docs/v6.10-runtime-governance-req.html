Solfacil v6.10 Runtime Governance Skeleton

這不是再加一個 ops 頁面。v6.10 的核心是把平台的 runtime facts 升格成一級公民：建立中央 doctor / observability spine，讓各模塊輸出標準化運行事實，並由中央治理 severity、lifecycle、summary 與 diagnosis。

Target: v6.10 Theme: Runtime Governance Status: REQ aligned / ready for DESIGN Source: docs/v6.10-runtime-governance-req.md

你先看這裡

一句話： v6.10 要插入一條共同脊椎，不是再長一顆平行腦袋。
  • v6.9 解的是 security closure；v6.10 解的是 runtime governance closure。
  • 當前系統有 domain alarms、asset health、strategy facts，但沒有平台級統一治理語義。
  • 現在可以人工看懂，但系統本身還不能清楚表達自己的運行狀態。
  • 正確方向是中央 spine + 舊模塊標準化發送 facts，而不是每個模塊各自發明 severity/summary/rules。

為什麼現在要做

  • domain telemetry ≠ platform observability
  • runtime truth 目前碎片化在 logs / pages / tables / handlers
  • 沒有 central lifecycle，operator 無法直接知道 issue 是否仍在發生
  • 再不立 spine，後續每個模塊都會各自擴散語義

v6.10 必須交付的能力

  • canonical runtime event schema
  • severity taxonomy + lifecycle semantics
  • platform health summary model
  • priority modules emitting standardized facts
  • operator diagnosis surface
  • retention / dedup / correlation discipline

REQ 主張

v6.10 的設計方向必須是：

  • 中央 spine 定義 event code、severity、lifecycle、summary、retention
  • 舊模塊保留業務邏輯，但改成輸出標準化 runtime facts
  • phase-1 優先接入 BFF / DB / Redis / scheduler / ingest / integrations 等高價值模塊
  • 先讓系統能清楚講自己的病，再談更花俏的觀測面

非目標

  • 不是重寫整個系統
  • 不是把 P6 Alerts 直接包裝成平台 observability
  • 不是 day-1 就變完整外部 observability stack
  • 不是用 UI 掩蓋底層語義缺失

已固定的產品決策

  1. 主要受眾切線：A — phase-1 先服務 internal engineering / ops，不以 customer-facing 簡化為主。
  2. 首版偏向：A — 先做少數關鍵模塊的深治理，不做廣而淺的鋪陳。
  3. surface priority：B — backend contract first，UI 只做最小可用，不讓 dashboard polish 反過來主導 release。

合併起來的 cut line 很清楚：v6.10 phase-1 是 internal-ops-first、narrow-but-deep、backend-governance-first。這樣才能最大化 truth density，避免做出外觀像 ops 系統、語義卻還是鬆散的假骨架。
