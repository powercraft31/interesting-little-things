# Solfacil v6.10 Runtime Governance Skeleton — REQ

Date: 2026-04-18 07:05:31 UTC
Status: REQ aligned / ready for DESIGN
Version baseline: Solfacil v6.9 closure state
Target release: v6.10

## 1. Background

v6.9 is a security correction release. Its job is to close known security defects correctly and stop. That line matters.

Once v6.9 closes, the next problem is no longer primarily browser security. The next structural weakness is that Solfacil still lacks a coherent runtime governance layer for the platform itself.

Today the system can expose domain data such as fleet alarms, gateway states, strategy intents, and device health. But those surfaces are not the same thing as platform observability. They describe business/domain conditions; they do not yet give operators a unified, trustworthy answer to questions like:

- Is the platform healthy right now?
- Which subsystem is degraded?
- Is a failure ongoing, recovered, or merely historical?
- Which signals are actionable versus noise?
- What should operators inspect first?

So v6.10 is not "add another ops page". It is a structural release to make runtime facts first-class and govern them coherently.

## 2. Release decision already settled

The following framing is treated as fixed input for this REQ draft:

1. v6.10 theme is **runtime governance / operations skeleton**, not another isolated feature page.
2. The correct center is a **central doctor / observability spine**.
3. Existing modules should be adjusted to **emit standardized runtime facts** into that spine, rather than each module owning ad hoc operational rules.
4. The goal is to insert a **common spine into the current system**, not to grow a parallel brain or replace existing business modules.

- **metrics surface**: no verified evidence

Bluntly:

## 4. Problem statement

The core problem is not lack of information. The core problem is lack of governed runtime meaning.

Solfacil already emits many facts:

- cron and scheduler activity
- dispatch outcomes
- telemetry aggregation activity
- gateway alarms
- device health information
- strategy reasoning snapshots

But these facts are split across modules, tables, pages, and stdout logs. There is no single authoritative layer that defines:

- what counts as an operational event
- how severity is graded
- when an issue is open vs recovered
- how duplicate symptoms are correlated
- what operators should see first
- what the system itself believes about its own health

The result is five structural failures:

1. **Runtime truth is fragmented**  
   Different modules can expose signals, but there is no common runtime contract.

2. **Operational semantics are inconsistent**  
   A timeout, degraded dependency, backlog, and stale data condition are not normalized into one severity/lifecycle model.

3. **Domain telemetry can masquerade as platform observability**  
   Fleet alarms and asset health exist, but they do not prove the BFF, scheduler, Redis, DB path, or ingest loop are healthy.

4. **Operator triage is log-first and human-memory-dependent**  
   Operators must infer platform state from scattered logs, routes, and pages instead of consuming one diagnosis surface.

5. **Future modules will drift further unless a spine is introduced**  
   If each new module invents its own warning/error/recovery semantics, the platform becomes harder to reason about with every release.

## 5. Root classification

### 5.1 This is not mainly a frontend page problem
A dashboard may be one output, but the real defect is deeper: runtime facts are not first-class system objects with common governance semantics.

### 5.2 This is not merely a logging problem
Adding more logs without common event contracts only produces louder ambiguity.

### 5.3 This is not the same as business alarm center enhancement
P6 Alerts and gateway alarms are domain-facing alarm surfaces. v6.10 is about platform/runtime governance across BFF, scheduler, ingest, DB, Redis, integrations, and cross-module execution paths.

## 6. v6.10 release objective

Solfacil v6.10 is a **runtime governance release** whose objective is:

- to introduce a central runtime governance skeleton (doctor / observability spine)
- to define one authoritative contract for runtime events, severity, lifecycle, and health summaries
- to make existing high-value modules emit standardized runtime facts into that contract
- to give operators one trustworthy diagnosis surface for platform state
- to reduce future observability drift by centralizing governance semantics instead of scattering them across modules

In plain terms:

**v6.10 should make the system able to speak clearly about its own runtime condition.**

## 7. Core product requirements

### 7.1 Central runtime contract must exist
v6.10 must define one platform-level runtime contract covering at least:

- event identity / event code
- source component / subsystem
- severity taxonomy
- state or lifecycle stage
- timestamps
- correlation / dedup keys
- operator-facing summary fields
- machine-readable detail payload

REQ consequence:
- runtime semantics must not remain implicit inside free-form logs.

### 7.2 Central health summary model must exist
v6.10 must define a platform health summary model that can answer at minimum:

- overall runtime posture
- degraded components
- active critical issues
- recovered issues still visible for triage
- last successful self-check / snapshot time

REQ consequence:
- platform health cannot be inferred only from container up/down state.

### 7.3 Event lifecycle semantics must be explicit
v6.10 must define when an issue is:

- detected/open
- ongoing
- acknowledged or suppressed if such states are adopted
- recovered
- closed/retained

REQ consequence:
- operator-facing issue state must not depend on reading historic logs and guessing whether the problem is still happening.

### 7.4 Existing modules must emit facts, not invent governance
At least the high-value runtime-critical modules must be adapted to output standardized runtime facts instead of ad hoc operational judgments.

Initial priority candidates:

- BFF/auth/runtime entry path
- DB connectivity / critical query path
- Redis-backed abuse-control / shared dependency path
- scheduler / cron / background service execution path
- IoT ingest / telemetry pipeline path
- external integration / webhook delivery path

REQ consequence:
- module-local signal generation is allowed
- module-local governance semantics as the long-term authority is not allowed

### 7.5 Operator diagnosis surface must exist
v6.10 must provide an operator-facing surface that summarizes runtime condition without requiring raw log inspection as the primary workflow.

This surface must allow operators to see:

- current overall state
- active high-severity issues
- affected components
- concise diagnosis summary
- drill-down path to underlying facts/events

REQ consequence:
- the primary output is not just storage; it must produce a usable operator view.

### 7.6 Runtime self-check capability must exist
v6.10 must establish platform self-check semantics for critical runtime dependencies and execution loops.

Examples of checks that DESIGN should evaluate:

- DB reachability and minimal critical query success
- Redis reachability when required by runtime path
- scheduler heartbeat / lag
- ingest freshness / stalled pipeline detection
- queue or dispatch timeout accumulation
- config/pathological startup failures

REQ consequence:
- self-check is about platform runtime health, not asset business metrics.

### 7.7 Retention and noise discipline must exist
v6.10 must define retention / visibility rules so the operator surface does not degrade into a permanent junk drawer.

REQ consequence:
- resolved events, dedup policy, and summary retention must be intentionally designed.

## 8. Scope

This REQ covers:

- platform runtime event contract
- severity and lifecycle semantics
- self-check and health summary model
- operator diagnosis surface for runtime governance
- integration of selected existing modules into the new spine
- retention/correlation/dedup rules needed to keep the system usable

This REQ does not yet prescribe exact implementation shape. That belongs to DESIGN.

## 9. Non-goals

v6.10 is **not** intended to:

1. replace existing business modules with a new monolith
2. redesign all legacy pages at once
3. become a full external observability stack replacement on day one
4. solve every historical domain alarm inconsistency
5. turn every debug log into a formal event immediately
6. block release on deep cleanup of unrelated frontend/UI debt
7. confuse asset/fleet business health with platform runtime health

## 10. Architecture context that DESIGN must preserve

### 10.1 Current matrix must be acknowledged
DESIGN must explicitly anchor itself to the current module/system documents instead of pretending v6.10 is a greenfield feature slice.

At minimum DESIGN must state impact or non-impact against:

- `00_MASTER_ARCHITECTURE_v6.9.md`
- `05_BFF_MODULE_v6.9.md`
- `09_SHARED_LAYER_v6.9.md`
- `10_DATABASE_SCHEMA_v6.9.md`
- relevant frontend architecture docs if operator UI is introduced or revised

### 10.2 Preferred insertion model
The preferred architecture direction is:

- introduce one runtime-governance spine
- connect existing modules to it via standardized fact emission
- keep business logic ownership in existing modules
- centralize governance semantics in the spine/shared contract layer

### 10.3 Anti-patterns DESIGN must avoid
DESIGN must not normalize any of the following without explicit approval:

- a parallel ops subsystem that duplicates existing module knowledge without becoming authoritative
- free-form log scraping as the primary semantic source of truth
- module-by-module severity semantics with no central taxonomy
- a UI-first dashboard that lacks governed runtime state underneath it
- a release scope that mutates every subsystem indiscriminately instead of onboarding priority modules first

## 11. Success definition

v6.10 is successful when all of the following are true:

1. Operators can determine platform runtime condition from one central surface.
2. The system has a defined platform health summary, not only domain pages plus logs.
3. High-value modules emit standardized runtime facts into one common contract.
4. Severity, lifecycle, and retention semantics are governed centrally.
5. A runtime issue can be tracked from detection to recovery without relying on operator memory.
6. The release reduces future drift rather than introducing another isolated module.

## 12. Acceptance requirements for DESIGN

DESIGN must answer these questions explicitly:

1. What is the canonical runtime event schema?
2. Where does the runtime governance spine live in the current architecture?
3. Which existing modules are onboarded in v6.10 phase-1 and why?
4. What is the platform health summary model?
5. What lifecycle states exist and what transitions are allowed?
6. What storage/query model is used for active issues vs retained history?
7. What self-checks are mandatory in phase-1?
8. What is the operator-facing surface and how does it map to the governed data model?
9. How are domain alarms kept distinct from platform runtime issues?
10. What rollout/rollback boundary prevents v6.10 from destabilizing core business flows?

## 13. Product decisions now fixed for DESIGN

Alan has resolved the remaining convergence points as **A / A / B**. These are now fixed REQ inputs, not open design questions.

### 13.1 Primary audience cut line — A
Phase-1 is primarily for **internal engineering / ops**.

REQ consequence:
- phase-1 usability should optimize for operators who need trustworthy runtime diagnosis, not for customer-facing simplification;
- customer-side readability may benefit later, but it is not the primary release driver.

### 13.2 Phase-1 visibility bias — A
Phase-1 should be **narrow but deep**.

REQ consequence:
- v6.10 must prefer a smaller number of critical components with governed, trustworthy runtime truth over broad but shallow coverage;
- DESIGN should explicitly select the first onboarding set and justify why those components deliver the highest runtime-governance leverage.

Default release logic:
- skeleton first
- critical-path onboarding second
- broader module coverage later

### 13.3 Surface priority — B
Phase-1 should be **backend contract first with minimal operator UI**, leaving richer UI maturation to a later release.

REQ consequence:
- the primary deliverable is the governed runtime model and operator-consumable backend/API surface;
- any phase-1 UI should remain minimal and functional, only enough to prove the spine is operationally usable;
- DESIGN must not let page design, dashboard polish, or broad frontend refactoring become the release center.

### 13.4 Combined release posture

Putting the three fixed decisions together, v6.10 phase-1 should be interpreted as:

- an internal-ops-first release
- centered on a narrow set of critical platform components
- backend-governance-first, with only minimal UI necessary for operational proof

This is the correct cut line because it maximizes truth density per unit of scope and minimizes the risk of producing a pretty but semantically weak ops layer.

## 14. One-sentence release definition

**Solfacil v6.10 establishes a central runtime governance skeleton so the system can express its own operational condition through standardized facts, governed lifecycle semantics, and a trustworthy operator diagnosis surface.**
