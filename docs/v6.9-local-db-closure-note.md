# v6.9 Local DB Closure Note

Status: closed enough for v6.9 mainline unless a new DB blocker appears.

## Why this note exists

The local DB track started as an incidental discovery during v6.9 work, not as the primary release goal.

The right response was:
- do not ignore real DB defects once discovered
- do not let DB archaeology consume the entire v6.9 release

This note marks the point where the DB track should stop expanding by default.

## Final judgment

The local DB problem was not “one missing migration”.
It was “no single source of rebuild truth”.

That problem is now sufficiently controlled for v6.9 purposes.

## What is now true

1. Canonical rebuild truth exists.
- `backend/scripts/bootstrap.sh` is now the canonical local rebuild entrypoint.
- Default mode is schema+migrations only.
- Optional demo data is an explicit separate layer via `--with-demo-seed`.

2. Migration truth is frozen.
- `backend/scripts/local-migration-manifest.txt` defines the ordered post-v5.10 migration chain.
- `docs/local-db-migration-inventory.md` records what is included, excluded, and why.

3. Runtime-critical compatibility repairs are in the rebuild path.
- 5-minute partition maintenance requirements are included.
- Hourly aggregation schema compatibility is included.
- `solfacil_service` ownership / CREATE privilege requirements are included.

4. Machine-verifiable DB contract exists.
- `backend/scripts/verify-local-db-contract.ts` checks required tables, columns, indexes, privileges, owner, ESS asset-type constraint, and a real partition-creation probe.

5. Minimal runtime smoke path exists.
- `npm run local-db:rebuild-check` performs:
  - rebuild
  - contract verification
  - deterministic smoke fixture insertion
  - 5-minute aggregation smoke
  - hourly aggregation smoke
  - final summary query

6. Schema truth and demo convenience are no longer mixed.
- Canonical rebuild no longer silently depends on demo seed data.
- Demo seed is now opt-in instead of implicit.

## Evidence already obtained

Verified in this repair cycle:
- schema-only rebuild from empty DB succeeded
- seeded rebuild from empty DB succeeded
- contract verifier passed on rebuilt DB
- one-command rebuild-check passed
- 5-minute aggregation smoke passed
- hourly aggregation smoke passed

## What is intentionally deferred

The following are not required to keep the DB track open right now:

1. CI integration for rebuild-check
- Valuable, but not a current v6.9 blocker.

2. Further seed stratification
- e.g. baseline seed vs demo seed vs smoke/test fixture as three separate layers.
- Useful cleanup, not required for current release control.

3. Broader runtime contract expansion
- e.g. more RLS/path-specific smoke checks, auth-seed behavior, wider backfill scenarios.
- Good future hardening, not current release gating.

4. Historical migration archaeology beyond current manifest boundary
- High risk of turning into cleanup for cleanup’s sake.
- Not justified unless it blocks mainline work.

## Reopen conditions

Do not continue DB deep-dive by default.
Reopen this track only if one of the following becomes true:

1. v6.9 mainline work hits a real DB blocker
2. contract verifier fails on the target local rebuild path
3. release acceptance reveals a DB capability gap not covered by current verifier/smoke checks
4. Alan explicitly requests CI-grade DB guardrails now

## Operational recommendation

Treat the DB track as controlled and parked.

Default stance from this point forward:
- do not actively deepen DB work
- do not pretend remaining deferred items are solved
- return focus to the v6.9 mainline

## Practical commands

Schema-only canonical rebuild:
`cd backend && PGPASSWORD=postgres_admin_2026 APP_DB_PASSWORD=solfacil_vpp_2026 SERVICE_DB_PASSWORD=solfacil_service_2026 ./scripts/bootstrap.sh --drop-existing`

Canonical rebuild with demo seed:
`cd backend && PGPASSWORD=postgres_admin_2026 APP_DB_PASSWORD=solfacil_vpp_2026 SERVICE_DB_PASSWORD=solfacil_service_2026 ./scripts/bootstrap.sh --drop-existing --with-demo-seed`

Contract verifier only:
`SERVICE_DATABASE_URL=postgresql://solfacil_service:<password>@127.0.0.1:5433/<db_name> npx ts-node backend/scripts/verify-local-db-contract.ts`

One-command rebuild + verifier + aggregation smoke:
`cd backend && npm run local-db:rebuild-check`

## Decision for v6.9

For v6.9 release steering, local DB repair is no longer the main workstream.
It is an addressed side-track with explicit reopen conditions.

The correct next move is to return to the v6.9 mainline.
