## 1. Overall verdict

The current DESIGN set is directionally correct and aligned with the REQ on the core correction:
- cookie-first browser auth
- no production dependence on localStorage bearer JWT
- unified browser auth path for API + SSE
- fail-fast secret handling
- matrix-aware document ownership

However, the design is not yet ready to proceed to PLAN.

Reason:
The architecture direction is mostly right, but several critical design points are still under-specified or written as if already decided when they are actually still design choices.

This revision round is therefore not a redesign. It is a convergence pass.

## 2. What must remain unchanged

Keep these design positions unless there is a very strong reason to change them:

1. browser admin surface is same-origin and cookie-first
2. production browser auth must not depend on localStorage JWT persistence
3. API and SSE should converge on the same browser auth channel
4. unsafe JWT secret fallback must be eliminated via fail-fast startup/runtime behavior
5. matrix document responsibility must remain explicit
6. any legacy bearer compatibility, if retained at all, must be explicit, temporary, and disabled by default

## 3. Mandatory revision items

### 3.1 Resolve browser contract vs non-browser contract

Current problem:
The current draft assumes a browser-first model but leaves non-browser / machine-client support ambiguous.

Required revision:
The revised DESIGN must explicitly choose one of these and state it clearly:

A. v6.9 only defines the browser admin auth contract; non-browser bearer clients are not supported in this release
or
B. v6.9 defines two contracts:
- browser contract = cookie-first
- machine/non-browser contract = separately documented bearer contract

Do not leave this as a vague discovery item.

If you choose B, clearly separate it from the browser contract and make sure legacy browser bearer mode is not smuggled back in under “machine client”.

### 3.2 Add a real migration / mixed-version / rollback matrix

Current problem:
Migration language is still too high-level.

Required revision:
The revised DESIGN must explicitly address:
- old frontend -> new backend
- new frontend -> old backend
- existing session behavior at cutover
- stale localStorage token behavior after rollout
- cookie name migration behavior if `__Host-solfacil_session` is adopted
- rollback behavior if cookie-first release must be reverted

This belongs in DESIGN because PLAN cannot be valid without a defined transition model.

### 3.3 Tighten abuse-control topology

Current problem:
The design states IP throttling + identifier backoff, but leaves the storage and topology assumptions too open.

Required revision:
The revised DESIGN must state:
- expected deployment assumption for auth-abuse state (single instance / multi-instance / shared store)
- where counters logically live in the chosen design
- what trust model is used for client IP derivation behind proxy
- whether edge rate limiting is required reinforcement or recommended reinforcement
- that application-layer protection is the release-critical layer

Do not leave “memory / Redis / DB” as an unclosed menu unless the doc explicitly declares one as default and the rest as alternatives.

### 3.4 Make CSP first-release stance concrete

Current problem:
The current docs correctly say CSP is mandatory, but still leave the first release target too abstract.

Known current frontend realities that DESIGN must acknowledge explicitly:
- `frontend-v2/login.html` contains inline script
- `frontend-v2/index.html` contains inline `onclick`
- frontend uses Google Fonts
- frontend uses jsdelivr CDN for ECharts

Required revision:
The revised DESIGN must define a first-release CSP strategy, at least at the level of:
- what current constructs block a strict CSP
- whether v6.9 first cut expects frontend cleanup before rollout or accepts documented transitional allowances
- whether the intended direction is nonce/hash/externalization for inline script/event-handler removal
- which third-party origins must exist, if any, in the first-release policy

Do not leave CSP as “mandatory later”. It must be mandatory in v6.9 with a concrete first-release posture.

### 3.5 Clarify deployment boundary ownership beyond “Nginx owns it”

Current problem:
The deployment doc is slightly too Nginx-centric for the actual Solfacil matrix.

Known current architecture/code surfaces that matter:
- backend/scripts/local-server.ts
- backend/scripts/combined-server.ts
- current Express CORS behavior
- documented master architecture / deployment boundary

Required revision:
The revised DESIGN must distinguish:
- production public boundary ownership
- local/direct-BFF/dev boundary ownership
- app-header vs proxy-header responsibility
- precedence/conflict rule when both layers can emit security-related headers

The goal is not to broaden scope into infra redesign. The goal is to avoid pretending the system only has one boundary shape.

### 3.6 Separate “recommended default” from “already decided”

Current problem:
Some values are written too strongly, as if already approved final policy.

Examples:
- `SameSite=Strict`
- `__Host-solfacil_session`
- specific throttle/backoff semantics if present
- exact header defaults

Required revision:
The revised DESIGN must clearly label:
- fixed decisions
- recommended defaults pending review
- explicit open review points

Do not silently upgrade proposed defaults into settled product decisions.
