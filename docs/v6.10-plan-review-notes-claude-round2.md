## 1. Overall judgment

**Judgment: revise plan first before coding.**

This is much closer to an execution plan than a design doc, and the file-targeted slicing is materially better than the usual hand-wavy planning artifacts. But on a strict implementation-readiness review, there are still a few plan-level defects that matter:

1. the gates are not tied tightly enough to passing evidence,
2. the rollout sequence assumes finer-grained flags than the plan actually defines,
3. the integration layer is not segmented enough to match the incremental rollout strategy,
4. retention / auto-close is still under-specified at the execution-and-test level,
5. one or two slices still leave the implementer to choose the testing mechanism rather than naming it.

Those are not design objections. They are execution objections.

## 2. What is execution-ready

These parts are solid enough for implementation handoff once the blocking plan defects below are corrected:

### 2.1 Overall workstream backbone
The top-level order is broadly correct:
- WS0 bootstrap/flags first
- WS1 schema before persistence
- WS2 spine before module emitters
- WS3 operator API before frontend
- module onboarding after the spine is stable
- WS10 hardening last

That respects the canonical authority split from M9/M5/DB/frontend and does not obviously invert ownership.

### 2.2 File-targeted slicing is mostly concrete
The plan usually names real repo targets rather than vague subsystems. That is good enough for implementation delegation in:
- WS1 DB migration/verification
- WS2 shared runtime package
- WS3 BFF runtime routes
- WS4 substrate emitters
- WS5–WS8 module-owned emitter slices

This is the strongest part of the plan.

### 2.3 Slice-level testing is materially better than average
Section 5.1 is usefully concrete. It does not just say “write tests”; it names the failure/recovery behaviors that must be proven, especially in:
- WS2 projection semantics
- WS4 “runtime failure must not fail business request” regressions
- WS5 duplicate/noise discipline
- WS7 recovery after dispatch resumes

That is the right shape for preventing implementers from claiming success off happy-path smoke only.

### 2.4 Separation from design is mostly preserved
The plan mostly stays implementation-oriented and does not substantially reopen architecture. It keeps:
- M9 as the authority,
- M5 as edge only,
- additive DB posture,
- minimal frontend scope,
- separation from P5/P6/domain alarms.

The residual-risk section also mostly stays bounded rather than drifting back into redesign.

## 3. What is still weak or risky in sequencing / testing / gates

### 3.1 **Blocking**: gates are defined as artifact/state checkpoints, not as verified pass checkpoints
The plan says testing is split into ordered layers, but the actual gates in section 3 do not require explicit proof that the corresponding prior-layer tests passed before the next gate opens.

Examples:
- Gate B requires the migration to exist and bootstrap, but does not explicitly require the WS1 contract script/tests to pass before WS2 starts.
- Gate C requires the spine to compile and implement projection/health logic, but does not explicitly require WS2 slice tests to pass before WS3–WS8 begin.
- Gate D requires routes to exist, but not that auth/envelope/wiring tests passed before WS9 starts.
- Gate E requires emitters merged, but not that each emitter slice has passed its own slice suite and integration proof.

Why this matters:
Without binding gates to evidence, the team can “advance” based on code presence instead of validated behavior. That directly weakens segmented testing rigor.

Required revision:
- each gate should name the exact required evidence set,
- each gate should explicitly require the prior workstream’s slice suite to pass,
- integration sets should be attached to the gate that consumes them, not only listed later.

### 3.2 **Blocking**: rollout/rollback assumes per-slice or per-emitter flags that the plan does not actually define
WS0 defines a **single runtime-governance feature gate**. But rollout section 6.1 says module onboarding will be enabled incrementally in sequence, and rollback section 6.2 says to disable a noisy emitter slice via an emitter-specific sub-flag or revert that slice.

That is a real execution mismatch.

As written, the plan supports:
- all runtime governance off, or
- runtime governance on.

It does **not** clearly support:
- enable WS4 only,
- then enable M1,
- then enable M3,
- then enable M2,
- then enable M4,
- or selectively disable one noisy emitter slice while leaving others active.

Why this matters:
The rollout procedure is only operationally credible if the toggles needed for that procedure are planned as deliverables.

Required revision:
Either:
1. explicitly add per-slice/per-module flags to WS0 and to the targeted files/config, plus related tests, **or**
2. simplify rollout/rollback language so it honestly reflects a single global flag.

Do not leave both stories in the same plan.

### 3.3 **Blocking**: integration-layer segmentation is weaker than the rollout strategy requires
The plan is strong at slice tests, but the integration layer does not fully mirror the planned enablement sequence.

Most notably, I4 combines M2 + M3 + M4 into one integration set. That is weaker than the rollout section, which says those emitter slices are enabled incrementally and verified after each step.

Why this matters:
If rollout is incremental, the integration plan should produce incremental evidence. Otherwise the team can defer subsystem interaction problems until a late combined test and still claim the earlier slice was “done.”

Required revision:
Split combined integration into per-slice integration sets aligned to enablement order, e.g.:
- I4a M3 dispatch path
- I4b M2 scheduler path
- I4c M4 billing path
or equivalent naming.

The important part is one isolated integration proof per onboarded module before moving to the next module.

### 3.4 **Blocking**: retention / auto-close remains execution-ambiguous despite being in scope
The plan includes concrete retention and auto-close defaults and even puts them into WS10 acceptance territory. But the plan also says the executor shape is not mandatory and can remain a bounded follow-up if documented.

That leaves an implementation-critical gap:
- what artifact actually enforces 24h recovered auto-close,
- what artifact enforces 72h stale auto-close,
- how that is triggered in local/test environments,
- what test proves it.

Why this matters:
If retention/closure is in acceptance scope, it cannot remain only a policy statement. Otherwise final acceptance can be hand-waved without proving the behavior exists.

Required revision:
Pin one phase-1 enforcement path and one proof path. Example categories are enough; full implementation detail is not needed. But the plan must specify:
- where the executor lives,
- how it runs in local/test environments,
- what slice/integration test proves TTL behavior.

### 3.5 Non-blocking but real: frontend verification path is still too optional/branchy
WS9 says to add `backend/test/frontend/runtime-page-contract.test.ts` **if** frontend regressions are covered there, otherwise add some equivalent UI smoke where the repo already keeps frontend checks.

That is not sharp enough for a strict handoff plan. It leaves the implementer choosing the verification lane.

Why this matters:
The whole plan elsewhere is file-targeted. This one slice regresses to “pick a test home.”

Recommended revision:
Name the exact test location/mechanism expected for WS9, even if it is only one smoke contract file.

### 3.6 Non-blocking but real: final acceptance evidence is partly too informal
The final matrix is directionally good, but some rows still permit soft evidence such as:
- “code review against WS2 files”
- “browser/manual verification”
- “local evidence”

Manual verification is appropriate for the operator page, but some of the backend rows should be tied to named test commands or artifacts rather than reviewer judgment.

Recommended revision:
For each acceptance row, specify whether the required proof is:
- automated test suite,
- scripted local scenario,
- manual browser verification,
- or release rehearsal evidence.

That would make signoff harder to game.

### 3.7 Non-blocking: sequencing is mostly right, but WS10 acceptance logic should not be the first place retention is proven
If retention/auto-close remains a WS10-only concern, late discovery risk is too high. The plan should ensure at least one earlier proof point exists before final signoff.

This is secondary to finding 3.4, but it reinforces the same concern.

## 4. Blocking vs non-blocking findings

### Blocking findings
1. **Phase gates are not tied to mandatory pass evidence.**
2. **Rollout/rollback assumes per-slice flags that WS0 does not define.**
3. **Integration verification is not segmented enough to match incremental onboarding/enablement.**
4. **Retention/auto-close remains under-specified at execution/test level despite being part of acceptance scope.**

### Non-blocking findings
1. WS9 frontend test location/mechanism is still too optional.
2. Some final acceptance evidence categories are too informal.
3. Retention proof appears too late in the plan even aside from the executor ambiguity.

## 5. Verdict: ready for implementation / revise plan first

**Verdict: revise plan first.**

Reason:
The plan is close, and the workstream/file slicing is largely implementation-oriented. But the remaining defects are not cosmetic. They affect whether the team can actually execute the release in controlled slices with credible gates and rollback.

What I would require before coding starts:
1. tie each gate to named passing evidence,
2. reconcile single-flag vs incremental-enable rollout story,
3. split combined integration verification to match rollout slices,
4. pin the phase-1 retention/auto-close execution and proof path,
5. tighten the frontend verification lane.

If those corrections are made, the plan should be ready for implementation without reopening design.
