## 2. Fixed execution constraints

### 2.1 Release boundary
v6.10.1 is storage-retention hardening on top of accepted v6.10 runtime governance.

Do **not** broaden into:
- frontend/storage dashboards
- runtime-governance redesign
- warehouse/archive-productization
- unrelated cleanup of legacy business tables
- `revenue_daily` TTL or archive movement

### 2.2 Hard-fixed storage rules
Implementation must preserve these exact rules:
- Docker logs stay `json-file` with `max-size: 10m`, `max-file: 5` across `solfacil-bff`, `solfacil-db`, `solfacil-m1`, `solfacil-redis`
- `device_command_logs`: 90-day hot window, archive-first to `device_command_logs_archive`, then hot delete
- `device_command_logs` eligibility = `result IN ('success','fail','timeout') OR resolved_at IS NOT NULL`
- `device_command_logs` rows in `pending`, `dispatched`, `accepted` must not be age-swept
- `gateway_alarm_events`: 180-day hot window, archive-first to `gateway_alarm_events_archive`, then hot delete
- `backfill_requests`: active rows retained; terminal rows (`completed`, `failed`) delete after 14 days; no archive
- `runtime_issues`: closed rows delete 30 days after `closed_at`; no archive
- `revenue_daily`: preserve in hot storage; no TTL and no archive move in v6.10.1

### 2.3 Executor architecture
There is exactly one shared executor path:
- extend `backend/src/shared/runtime/retention-job.ts`
- do not create a competing retention scheduler or second orchestration brain
- default cadence remains hourly via `RUNTIME_RETENTION_INTERVAL_MS = 60 * 60 * 1000`
- executor failure must emit `storage.retention.executor.failed`

### 2.4 Registry requirement
A checked-in policy registry is required in:
- `backend/src/shared/runtime/storage-policy.ts`

Do not scatter per-surface cutoffs as hidden constants across helper files without the registry.

### 2.5 Archive idempotence requirement
Archive movement must use:
- preserved-source `UNIQUE`
- `INSERT ... ON CONFLICT DO NOTHING`

No hand-wavy "safe enough" dedupe logic.

## 8. Rollout / rollback execution discipline

### 8.1 Rollout discipline
- archive tables must exist before archive movement code runs
- runtime scheduler wiring and compose changes happen only after executor path is ready
- final acceptance matrix runs after all slices land

### 8.2 Rollback discipline
Allowed rollback means:
- stop/disable extended executor
- redeploy prior code if needed
- leave additive schema and archived data in place

Not allowed:
- dropping archive tables to simulate rollback
- deleting archived rows to hide rollout traces
