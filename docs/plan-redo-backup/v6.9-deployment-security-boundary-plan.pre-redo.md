# Solfacil v6.9 Deployment Security Boundary Plan

Date: 2026-04-15 UTC
Status: PLAN draft for implementation
Design basis:
- `docs/v6.9-security-design.md`
- `docs/v6.9-deployment-security-boundary-design.md`

## 1. Goal

Implement the deployment-facing portion of v6.9 so the shipped public surface actually reflects the designed security model.

Deployment completion means:
- production serves the admin UI as a same-origin HTTPS surface;
- public ingress emits the authoritative browser-hardening header baseline and CSP;
- application and ingress header ownership do not conflict;
- secure cookie transport and trusted-proxy behavior are wired correctly;
- production login-abuse counters use shared ephemeral state;
- local/direct-BFF paths remain useful for verification without redefining the production contract;
- rollout and rollback procedures keep temporary exceptions bounded.

## 2. Current deployment/runtime surfaces in repo

Known files:
- `docker-compose.yml`
- `.env.example`
- `backend/scripts/local-server.ts`
- `backend/scripts/combined-server.ts`
- `backend/package.json`

Current repo realities relevant to deployment planning:
- `docker-compose.yml` runs `backend/scripts/local-server.ts` in a production-like container path
- `.env.example` does not currently describe auth-secret or security runtime inputs
- `backend/scripts/local-server.ts` acts as a direct Express boundary with CORS allowlist behavior
- `backend/scripts/combined-server.ts` uses permissive `cors({ origin: '*' })`
- no ingress config artifact is present in repo, so production ingress tasks must be captured as operator-owned work items even if the actual config lives outside this repo

## 3. Deployment test model used by this plan

Deployment verification is intentionally classified so release owners can see which checks are automated versus environment/operator driven:
- Automated repo test/build: build and any repo automation that validates deployment-facing changes
- Command-line probe: `curl` or equivalent checks against local, staging, or public-boundary surfaces
- Browser/manual verification: real-browser confirmation of effective CSP, page load behavior, and cookie-backed flows
- Operator/infrastructure verification: ingress config, secret input, trusted proxy, store provisioning, and runtime topology checks

Interpretation:
- deployment slice completion requires command-line and operator verification even if repo builds pass
- browser/manual checks are required for effective CSP and real boundary behavior
- deployment slice success still does not replace final full release acceptance

## 4. Implementation slices

## Slice DEP-1 — Runtime input contract and operator checklist

Objective:
- define the runtime/security inputs required before rollout

Tasks:
1. Expand `.env.example` or equivalent operator docs to describe required v6.9 security inputs:
   - `JWT_SECRET`
   - trusted-proxy setting
   - secure-cookie mode expectations
   - shared abuse-store connection/settings
   - compatibility-mode switch values
2. Document that startup must fail when `JWT_SECRET` is unsafe or missing.
3. Document HTTPS and same-origin requirements for the browser contract.
4. Record bridge enable/disable procedures for rollout and rollback.

Files to modify/create:
- `.env.example`
- release/operator runbook docs if maintained in repo
- possibly `docker-compose.yml` comments/examples

Segmented verification:
- Automated repo test/build:
  - ensure changed docs/examples remain buildable or lint-clean if such checks exist
- Command-line probe:
  - not primary for this slice
- Browser/manual verification:
  - not primary for this slice
- Operator/infrastructure verification:
  - operator checklist is sufficient to configure staging/prod without hidden assumptions
  - required inputs are present before smoke testing

## Slice DEP-2 — Public ingress header baseline

Objective:
- serve the authoritative browser-hardening policy from the public boundary

Tasks:
1. Prepare ingress configuration changes outside or inside repo for:
   - `Content-Security-Policy`
   - `Strict-Transport-Security`
   - anti-framing policy (`X-Frame-Options` and/or `frame-ancestors` effective policy)
   - `X-Content-Type-Options: nosniff`
   - `Referrer-Policy`
   - `Permissions-Policy`
2. Ensure login and SPA entry responses both receive the intended baseline.
3. Ensure ingress does not override app headers in a weaker direction.
4. Ensure authenticated/login/session pages are not cacheable in ways that violate the design.

Files to modify/create:
- public ingress config artifact if repo-managed
- otherwise operator task entry in deployment runbook
- optional sample config doc under `docs/` if needed for repo traceability

Segmented verification:
- Automated repo test/build:
  - validate any repo-managed config syntax if tooling exists
- Command-line probe:
  - `curl -I https://<host>/login`
  - `curl -I https://<host>/`
- Browser/manual verification:
  - browser/network inspection of the effective header set on login and SPA entry
- Operator/infrastructure verification:
  - confirm ingress policy is the authoritative source and does not weaken app headers

## Slice DEP-3 — First-release CSP deployment

Objective:
- deliver the first-release CSP that matches actual frontend structure

Tasks:
1. Coordinate with frontend slice FE-6 so ingress CSP matches shipped HTML/JS.
2. Enforce target policy shape:
   - `default-src 'self'`
   - `script-src 'self'` plus nonce/hash only if required
   - `style-src 'self' 'unsafe-inline'` only if login inline style still exists
   - `font-src 'self'`
   - `img-src 'self' data:`
   - `connect-src 'self'`
   - `frame-ancestors 'none'`
   - `object-src 'none'`
   - `base-uri 'self'`
3. Remove third-party font/chart origins from policy because assets are self-hosted before rollout.
4. If nonce/hash contingency is needed, document the exact temporary exception and removal owner.

Segmented verification:
- Automated repo test/build:
  - use config/build validation where available to catch malformed policy changes
- Command-line probe:
  - inspect CSP response header on login and SPA entry
- Browser/manual verification:
  - no unexpected CSP violations in login and SPA flow
  - no dependency on Google Fonts or jsdelivr remains at rollout
- Operator/infrastructure verification:
  - confirm any temporary nonce/hash exception is explicitly documented and owned

## Slice DEP-4 — Cookie transport and proxy correctness

Objective:
- ensure `__Host-solfacil_session` works correctly on the deployed boundary

Tasks:
1. Ensure public admin surface is HTTPS end-to-end from browser perspective.
2. Ensure proxy forwards scheme/proto info so app can safely emit `Secure` cookies.
3. Ensure same-origin serving of SPA, login, and `/api/*` remains intact.
4. Verify no `Domain` rewriting or path mutation breaks `__Host-` cookie requirements.

Targets:
- ingress/proxy config outside repo or deployment manifests if available
- `docker-compose.yml` / local docs for test environment caveats
- local-server/proxy-related code if it needs explicit trust-proxy config

Segmented verification:
- Automated repo test/build:
  - validate repo-managed proxy/config examples where tooling exists
- Command-line probe:
  - login response shows `Set-Cookie: __Host-solfacil_session=...; Secure; HttpOnly; Path=/; SameSite=Strict`
  - session route behaves correctly when the cookie is replayed
- Browser/manual verification:
  - browser returns cookie on same-origin API and SSE requests
- Operator/infrastructure verification:
  - confirm no ingress/proxy layer rewrites cookie domain/path in a way that violates `__Host-` semantics

## Slice DEP-5 — Shared abuse-state topology enablement

Objective:
- make application-layer login-abuse controls production-correct in multi-instance setups

Tasks:
1. Provision shared ephemeral store for production/staging.
2. Wire backend runtime configuration to use the shared store in multi-instance environments.
3. Define local-dev fallback behavior separately from production.
4. Verify edge rate limiting, if used, does not conflict with application response semantics.

Files to modify/create:
- deployment manifests/config outside repo where store is provisioned
- `.env.example` / docs for connection settings
- `docker-compose.yml` only if local reproduction path should include the store

Segmented verification:
- Automated repo test/build:
  - limited to repo-managed config/build checks
- Command-line probe:
  - repeated invalid-login requests demonstrate throttling/backoff behavior at the deployed boundary
- Browser/manual verification:
  - login UX later confirms staging surface presents throttled behavior coherently
- Operator/infrastructure verification:
  - shared store is actually provisioned and selected in staging/prod
  - behavior is consistent across instances in staging

## Slice DEP-6 — Local/direct-BFF boundary alignment

Objective:
- keep local and direct-run entrypoints useful without letting them redefine production security behavior

Tasks:
1. Update `backend/scripts/local-server.ts` to emit fallback hardening headers meaningful for local verification, excluding HSTS.
2. Review `backend/scripts/combined-server.ts` and either:
   - align/document it as non-production demo-only, or
   - tighten it enough that it no longer contradicts the security plan
3. Remove misleading console output and docs that advertise old JWT-header browser semantics.
4. Ensure local server still supports realistic v6.9 auth verification.

Files to modify:
- `backend/scripts/local-server.ts`
- `backend/scripts/combined-server.ts`
- any local run instructions in repo docs

Segmented verification:
- Automated repo test/build:
  - backend build and any local-server tests continue to pass
- Command-line probe:
  - local server exposes fallback headers useful for dev verification
  - local paths still reflect cookie-first auth model
- Browser/manual verification:
  - local browser flow remains usable for realistic v6.9 auth checks
- Operator/infrastructure verification:
  - combined-server is clearly non-authoritative or aligned and cannot be mistaken for production policy

## 5. Testing and verification plan

## 5.1 Segmented deployment verification

From `backend/`:
- Automated repo test/build:
  - `npm run build`
  - `npm test -- --runInBand`
  - `npm run local-server`

If docker-compose is used for local rehearsal:
- Automated repo test/build + operator verification:
  - `docker compose up --build`
  - then run command-line probes against the mapped port

Public-boundary verification commands:
- Command-line probe:
  - `curl -I https://<host>/login`
  - `curl -I https://<host>/`
  - `curl -i -X POST https://<host>/api/auth/login -H 'Content-Type: application/json' --data '{"email":"admin@solfacil.com.br","password": "***"}'`
  - `curl -i https://<host>/api/auth/session --cookie '__Host-solfacil_session=...'`
  - repeated invalid-login requests from same source to confirm `429` / `Retry-After`

Effective-policy checks:
- Browser/manual + operator/infrastructure verification:
  - HTTPS only
  - HSTS present on production HTTPS
  - CSP present and effective
  - anti-framing policy effective
  - nosniff/referrer/permissions headers present
  - auth routes return `Cache-Control: no-store`
  - cookie attributes match design

## 5.2 Cross-workstream integration testing owned partly by DEP

DEP work is not complete when only header curls pass. DEP also participates in staging integration testing.

Required integrated staging checks after DEP changes:
- FE + BE + DEP login/session/protected-API/SSE flow passes through the hardened public boundary
- logout -> cookie cleared -> SSE closed -> next API `401` still behaves correctly after ingress/CSP changes
- stale `localStorage` token does not restore access through the public boundary
- CSP and hardening headers are effective on login and SPA entry while the app remains usable
- throttle/backoff behavior remains observable and not hidden by proxy misconfiguration

Bridge coverage:
- if rollout uses compatibility bridges, run staging integrated checks with bridge ON
- always run staging integrated checks again with bridge OFF
- bridge OFF is the steady-state gate and cannot be skipped because header probes succeeded

## 5.3 Phase gates tied to deployment testing

Phase 3 gate — before production rollout:
- DEP-1 through DEP-6 segmented verification passes
- operator/runtime prerequisites are confirmed
- FE + BE + DEP integrated staging verification passes
- public-boundary header/CSP/cookie checks pass
- compatibility bridge ON rehearsal passes if planned
- compatibility bridge OFF steady-state rehearsal passes

Production release gate contribution:
- passing `npm run build`, `npm test`, or a few `curl -I` commands is not enough
- release approval still requires the full integrated matrix from the master plan, including browser-visible and bridge-OFF scenarios

## 6. Rollout sequence

1. Prepare runtime secrets, trusted-proxy config, and abuse-store infrastructure.
2. Deploy backend changes to staging.
3. Deploy frontend CSP-compatible asset changes to staging.
4. Deploy ingress header/CSP changes to staging.
5. Validate with compatibility bridges ON if used.
6. Validate again with compatibility mode OFF.
7. Promote backend, frontend, and ingress config to production in a coordinated change.
8. After migration window, disable bridge settings and re-verify headers/cookies/session behavior.

## 7. Rollback sequence

1. If production issue is header-policy related, prefer reverting ingress config to the last known secure working variant rather than dropping the entire baseline.
2. If frontend/backend skew requires temporary relief, use documented bridge toggles first.
3. If cookie migration causes churn, re-enable short dual-cookie bridge rather than restoring old steady-state contract.
4. Do not rollback by:
   - removing CSP entirely;
   - dropping secure-cookie behavior on production-capable paths;
   - restoring secret fallback;
   - treating permissive local/demo server behavior as production policy.

2. `docker-compose.yml` currently lacks security runtime inputs for v6.9. Ryan should audit whether local/staging rehearsal environment will realistically cover the new contract.

## 10. Done criteria

Deployment-security work is done when:
- the public boundary emits the intended browser-hardening baseline;
- CSP matches the shipped frontend;
- cookie/proxy/runtime wiring supports the `__Host-` session cookie correctly;
- shared abuse-store infrastructure is active for production topology;
- local/direct-BFF paths no longer contradict the v6.9 security model;
- segmented deployment verification has passed;
- integrated staging verification through the hardened boundary has passed.
