# Solfacil v6.9 Frontend Security Plan

Date: 2026-04-15 UTC
Status: PLAN draft for implementation
Design basis:
- `docs/v6.9-security-design.md`
- `docs/v6.9-frontend-security-design.md`

## 1. Goal

Cut the browser admin surface over from `localStorage` bearer auth to the v6.9 cookie-first browser contract without reopening design decisions.

Frontend completion means:
- login and SPA bootstrap use server-authenticated session truth;
- no production auth token is stored in browser storage;
- request helpers stop setting `Authorization: Bearer` from JS;
- API and SSE behave consistently on login, logout, expiry, and `401`;
- frontend HTML/JS structure is compatible with first-release CSP;
- self-hosted fonts and ECharts are used before rollout.

## 3. Frontend test model used by this plan

Frontend verification is intentionally split into four named categories so implementers do not confuse one kind of check with another:
- Automated test: repo-level test or focused automation covering JS logic where feasible
- Browser/manual smoke: real-browser login/bootstrap/logout/expiry validation
- Network/devtools inspection: browser devtools validation of storage, request headers, cookies, SSE, and CSP behavior
- Staging browser verification: integrated FE + BE or FE + BE + DEP behavior on staging/public-boundary surfaces

Notes:
- If a slice has no useful automated suite yet, the plan says so explicitly.
- Browser verification is mandatory for auth/storage/CSP behavior and is not replaced by `npm test`.
- Passing frontend automated checks alone does not satisfy the v6.9 release gate.

## 4. Implementation slices

## Slice FE-1 — Session bootstrap contract

Objective:
- make server session state the browser source of truth

Tasks:
1. Add a frontend auth/session helper module or equivalent logic for:
   - `GET /api/auth/session`
   - normalized handling of `200`, `401`, and transport failures
2. Replace `localStorage` token-presence redirects in:
   - `frontend-v2/login.html`
   - `frontend-v2/js/app.js`
3. Update login page behavior:
   - render login shell immediately
   - call session probe
   - redirect to SPA only on confirmed authenticated session
4. Update SPA bootstrap behavior:
   - session probe on entry
   - redirect to `login.html` on `401`
   - tolerate transient infrastructure failure with bounded retry or clear failure UX

Files to modify/create:
- `frontend-v2/login.html`
- `frontend-v2/js/app.js`
- create `frontend-v2/js/auth-session.js` or equivalent if implementation prefers separation

Segmented verification:
- Automated test:
  - add or update focused auth-session helper tests if a frontend JS test harness exists
  - if no practical automated harness exists for the page bootstrap flow, record that gap explicitly rather than implying coverage
- Browser/manual smoke:
  - load `login.html` with no session cookie -> remains on login page
  - load `login.html` with valid cookie -> redirects to `index.html`
  - load `index.html` with no valid session -> redirects to login
- Network/devtools inspection:
  - confirm `/api/auth/session` is the bootstrap source of truth
  - confirm no auth decision depends on `localStorage.solfacil_jwt`
- Staging browser verification:
  - repeat unauthenticated and authenticated entry behavior against staging once FE is connected to the new backend session endpoint

Phase/gate relevance:
- this slice is part of the Phase 2 gate and must pass before DEP rehearsal begins

## Slice FE-2 — Login submit cutover

Objective:
- login succeeds via cookie issue, not token return/storage

Tasks:
1. Update login submit handler in `frontend-v2/login.html` to:
   - send credentials to `POST /api/auth/login`
   - rely on browser cookie handling
   - consume only display-safe session metadata if returned
   - stop requiring `result.body.data.token`
2. Preserve language/local UX state handling separately from auth state.
3. Add UX handling for throttled/backoff responses:
   - generic throttled message
   - optional display of retry timing if backend returns safe metadata/header
4. Keep generic auth failure handling and avoid account-enumeration signals.

Files to modify:
- `frontend-v2/login.html`
- possible extracted JS file if inline login script is externalized

Segmented verification:
- Automated test:
  - add focused login submit handler tests only if existing frontend automation can cover them without inventing a new testing architecture during this PLAN pass
  - otherwise document that login page submit verification is browser-led in v6.9
- Browser/manual smoke:
  - successful login navigates into SPA after cookie issuance
  - invalid credentials and throttle responses show distinct but generic UX
- Network/devtools inspection:
  - successful login does not write auth token to `localStorage` or `sessionStorage`
  - login response is consumed without frontend dependence on response-body JWT
- Staging browser verification:
  - perform real staging login and confirm the cookie-backed navigation path works against the integrated backend

Phase/gate relevance:
- this slice is part of the Phase 2 gate and also feeds the final integrated login/throttle release scenarios

## Slice FE-3 — Request helper cutover

Objective:
- remove browser bearer-header injection

Tasks:
1. Replace `getAuthHeader()` and related code in `frontend-v2/js/data-source.js`.
2. Ensure all browser `fetch` calls are same-origin and rely on cookie transport.
3. Centralize `401` behavior:
   - clear in-memory display state only
   - close active SSE connections
   - redirect to login
4. Review any page modules or utility files that directly assume auth header access.

Files to modify:
- `frontend-v2/js/data-source.js`
- `frontend-v2/js/app.js`
- any page/helper module that manually builds authenticated requests

Segmented verification:
- Automated test:
  - update helper-level tests for request construction and `401` handling where automation already exists
  - if no automation exists for direct-fetch callsites, record that browser/devtools inspection is the required proof
- Browser/manual smoke:
  - protected API still succeeds when cookie is present
  - stale `solfacil_jwt` alone does not grant access
- Network/devtools inspection:
  - protected API requests show no frontend-set `Authorization` header
  - removing stale `solfacil_jwt` has no effect on authenticated requests
- Staging browser verification:
  - repeat authenticated API actions in staging and confirm cookie-only transport remains intact

Phase/gate relevance:
- this slice is mandatory for the Phase 2 gate and for the final bridge-OFF release gate

## Slice FE-4 — Logout, expiry, and SSE convergence

Objective:
- make logout/session-loss behavior consistent with cookie-first auth

Tasks:
1. Change logout flow in `frontend-v2/js/app.js` so correctness depends on server cookie clearing, not storage deletion.
2. Keep optional cleanup of old `solfacil_jwt` as hygiene only.
3. Ensure `EventSource('/api/events')` closure is triggered on logout and on centralized `401` handling.
4. Review SSE reconnect logic in `frontend-v2/js/p2-devices.js` to ensure post-logout behavior does not mask auth failure.

Files to modify:
- `frontend-v2/js/app.js`
- `frontend-v2/js/p2-devices.js`
- any other module maintaining long-lived requests or timers

Segmented verification:
- Automated test:
  - add helper-level or event-flow tests where feasible for logout state handling and SSE cleanup hooks
  - if true SSE lifecycle automation is absent, say so and require browser/manual proof
- Browser/manual smoke:
  - logout clears access even if stale `localStorage` token remains
  - expired session causes API `401` and returns browser to login
- Network/devtools inspection:
  - logout closes SSE immediately client-side
  - SSE reconnect fails after logout/expiry as designed
- Staging browser verification:
  - repeat logout and forced-expiry flow on staging with the real backend and SSE path

Phase/gate relevance:
- this slice is part of the Phase 2 gate and the integrated release scenarios for logout, expiry, and SSE

## Slice FE-5 — Legacy storage cleanup and bridge handling

Objective:
- remove browser-side dependence on legacy auth storage while supporting migration hygiene

Tasks:
1. On first cookie-first load, optionally remove `localStorage.solfacil_jwt`.
2. Ensure cleanup failure does not affect auth correctness.
3. Remove any comments/docs in frontend code that describe browser JWT storage as normal behavior.
4. Ensure no fallback path silently reintroduces auth state in JS memory or storage.

Files to modify:
- `frontend-v2/login.html`
- `frontend-v2/js/app.js`
- `frontend-v2/js/data-source.js`

Segmented verification:
- Automated test:
  - add lightweight helper tests only if storage cleanup is implemented through testable utilities
  - otherwise call out that stale-storage verification is browser/manual by design
- Browser/manual smoke:
  - fresh logout/login cycle works with or without storage cleanup having run
- Network/devtools inspection:
  - manual insertion of stale `solfacil_jwt` does not restore auth
  - storage cleanup failure, if simulated, does not change auth correctness
- Staging browser verification:
  - confirm bridge-ON and bridge-OFF staging flows still reject stale storage as an auth source

Phase/gate relevance:
- this slice is a direct Phase 2 gate item and a named final release scenario

## Slice FE-6 — CSP readiness and self-hosted assets

Objective:
- make the frontend compatible with v6.9 CSP rollout

Tasks:
1. Externalize the inline login script from `frontend-v2/login.html`.
2. Remove the inline hamburger `onclick` from `frontend-v2/index.html` and move to JS event binding.
3. Decide whether login inline styles are removed in v6.9 or temporarily tolerated under `style-src 'unsafe-inline'`.
4. Self-host Google Fonts assets and update HTML/CSS references.
5. Self-host ECharts and update `frontend-v2/index.html` script source.
6. Verify no new third-party origins are required by the admin surface.

Files to modify/create:
- `frontend-v2/login.html`
- `frontend-v2/index.html`
- new static JS file for login page if needed
- new self-hosted font assets under frontend static tree
- new local ECharts bundle path under frontend static tree
- any CSS files referencing remote fonts

Segmented verification:
- Automated test:
  - use build/static validation where available to catch broken asset references
  - if no automated CSP-oriented frontend test exists, say so explicitly
- Browser/manual smoke:
  - login and SPA still render and function after inline handler/script cleanup
- Network/devtools inspection:
  - fonts and charts load from self-hosted paths
  - no console CSP violations in the intended steady-state flow
- Staging browser verification:
  - app works with target CSP in staging
  - no `script-src 'unsafe-inline'` is required unless the bounded nonce/hash contingency is explicitly approved

Phase/gate relevance:
- this slice must pass before Phase 3 public-boundary CSP rehearsal can be approved

## 6. Phase gates tied to frontend testing

Phase 2 gate — before leaving frontend cutover:
- FE-1 through FE-5 segmented verification passes
- FE-6 passes to the level needed for DEP CSP rehearsal
- mandatory browser/manual and network/devtools checks are complete
- no implementer may claim Phase 2 completion using only `npm test` or a static build

Phase 3 integration gate — before production rehearsal approval:
- staging browser verification passes with FE + BE integrated
- staging browser verification passes again after DEP header/CSP changes
- compatibility bridge ON flow passes if migration will use it
- compatibility bridge OFF flow passes as the steady-state target

Final release gate contribution:
- frontend is not release-ready until segmented frontend checks, staging browser verification, and final integrated matrix rows involving login/bootstrap/logout/expiry/SSE/CSP all pass

## 7. Regression checklist

Must remain true after each slice:
- language preference may remain in `localStorage`
- auth credentials may not
- mock mode still works where intentionally supported
- role switching and page initialization still work once authenticated
- logout always returns user to login without hidden dependency on storage token deletion

## 8. Rollout sequencing

Frontend rollout order:
1. land session helper and bootstrap changes behind backend readiness
2. cut request helpers to cookie-first mode
3. finish logout/SSE convergence
4. finish CSP-cleanup and self-hosted asset work
5. deploy with backend cookie/session support already available
6. validate browser behavior with backend compatibility mode ON, then OFF

## 9. Rollback sequencing

If frontend regression occurs:
1. prefer rolling frontend back as a single artifact
2. keep backend compatibility bridge ON during rollback if needed
3. do not restore `localStorage` auth as a silent hotfix in the new frontend
4. if rollback requires old frontend, coordinate with backend bridge state and cookie bridge window

## 10. Risks / sequencing notes Ryan should audit

1. `frontend-v2/js/data-source.js` is a central blast-radius file; Ryan should audit whether every authenticated call flows through it or whether there are hidden direct fetches elsewhere.
2. `frontend-v2/js/app.js` currently uses auth guard and logout assumptions from v5.23/v6.8. Ryan should verify there are no secondary route guards elsewhere.
3. CSP readiness depends on both HTML cleanup and asset hosting. Ryan should not approve rollout if only the header is ready but the frontend still relies on blocked origins.
4. SSE auth-loss behavior depends on centralized `401` handling plus client stream closure. Ryan should ensure this is explicitly tested, not assumed.
5. `SameSite=Strict` should be checked in actual operator navigation patterns before sign-off.
6. Frontend automation may remain partial for page-level browser behavior. Ryan should require explicit evidence that mandatory browser/devtools and staging checks were performed rather than assuming frontend test commands cover them.

## 11. Done criteria

Frontend implementation is done when:
- all auth UX works without browser token storage;
- stale `solfacil_jwt` is irrelevant to auth;
- request helpers are cookie-first only;
- CSP-compatible frontend cleanup is complete or documented with bounded exceptions;
- self-hosted fonts and ECharts are in place;
- segmented frontend verification has passed for each slice;
- staging browser verification has passed;
- final integrated release scenarios involving frontend behavior have passed.
