# Solfacil v6.9 Backend Auth Boundary Plan

Date: 2026-04-15 UTC
Status: PLAN draft for implementation
Design basis:
- `docs/v6.9-security-design.md`
- `docs/v6.9-backend-auth-boundary-design.md`

## 1. Goal

Implement the v6.9 backend auth correction across M5 BFF, M6 identity, and M9 shared layer so that:
- browser auth is cookie-first;
- browser API and SSE share the same auth path;
- login no longer returns browser-consumable JWT as the steady-state browser contract;
- application-layer login abuse controls are enforced;
- trusted proxy and CSRF-origin rules are explicit;
- JWT secret validation fails closed;
- temporary compatibility bridges are bounded and removable.

## 2. Current backend surfaces in repo

Primary files already identified:
- `backend/src/bff/handlers/auth-login.ts`
- `backend/src/bff/middleware/auth.ts`
- `backend/src/bff/handlers/sse-events.ts`
- `backend/src/shared/middleware/tenant-context.ts`
- `backend/scripts/local-server.ts`
- `backend/scripts/combined-server.ts`
- `backend/bin/app.ts`

Current security-relevant behaviors observed in repo:
- login handler still returns `token` in JSON body
- login/logout cookie settings are not yet at v6.9 target shape
- auth middleware accepts `Authorization` header or old `solfacil_jwt` cookie
- shared verification still falls back to `solfacil-dev-secret`
- no `GET /api/auth/session` route currently exists
- local server still documents JWT-header auth semantics
- tests heavily assume authorization-header-based protection

## 3. Backend test model used by this plan

Backend verification is split into explicit categories:
- Automated repo test: unit/integration suites run from the repo
- Command-line probe: local or staging HTTP checks using `curl` or equivalent
- Browser/manual verification: only where backend behavior must be observed through frontend/SSE/browser semantics
- Staging integration verification: FE + BE and FE + BE + DEP end-to-end flows in staging

Interpretation:
- backend slice completion requires automated repo coverage and command-line proof where the contract is HTTP-visible
- browser/manual checks do not replace backend tests, but some backend outcomes such as SSE close behavior must also be observed in integrated browser flows
- backend slice success does not equal release success; final release still depends on cross-workstream and public-boundary gates

## 4. Implementation slices

## Slice BE-1 — Secret validation and auth config centralization

Objective:
- remove production-capable secret fallback and define one validated auth config source

Tasks:
1. Create or refactor a shared auth-config utility under `backend/src/shared/` for:
   - validated `JWT_SECRET`
   - cookie name
   - cookie lifetime
   - secure-cookie expectations
   - compatibility-mode flags
2. Move JWT signing and verification to this shared validated config source.
3. Add startup validation in production-capable entrypoints:
   - `backend/bin/app.ts`
   - `backend/scripts/local-server.ts`
   - any other entrypoint that can serve auth traffic
4. Reject missing/placeholder/unsafe secret values before serving requests.

Files to modify/create:
- `backend/src/shared/middleware/tenant-context.ts`
- `backend/src/bff/handlers/auth-login.ts`
- `backend/bin/app.ts`
- `backend/scripts/local-server.ts`
- possible new helper such as `backend/src/shared/auth-config.ts`

Segmented verification:
- Automated repo test:
  - startup/config tests assert fail-fast behavior for missing or placeholder `JWT_SECRET`
  - shared auth config tests assert one validated secret source is used by login and middleware
- Command-line probe:
  - start the app with safe and unsafe auth config values and confirm fail/boot semantics match plan
- Browser/manual verification:
  - not primary for this slice
- Staging integration verification:
  - confirm staging boot fails closed when required runtime input is absent and succeeds when properly configured

## Slice BE-2 — Cookie contract update

Objective:
- converge login/logout on the v6.9 cookie contract

Tasks:
1. Update auth cookie issuance in `backend/src/bff/handlers/auth-login.ts` to:
   - name `__Host-solfacil_session`
   - `HttpOnly`
   - `Secure` on production-capable paths
   - `SameSite=Strict`
   - `Path=/`
   - no `Domain`
   - bounded lifetime aligned to JWT expiry
2. Update logout clearing to match the same cookie attributes.
3. Add `Cache-Control: no-store` to login/logout/session responses.
4. If dual-cookie bridge is enabled, implement explicit old/new cookie handling in one isolated helper rather than scattered route logic.

Files to modify/create:
- `backend/src/bff/handlers/auth-login.ts`
- possible cookie helper module under `backend/src/bff/` or `backend/src/shared/`

Segmented verification:
- Automated repo test:
  - login tests assert cookie issuance, logout clearing, and no steady-state browser token body contract
  - cookie helper tests assert bridge ON/OFF behavior if helper abstraction is introduced
- Command-line probe:
  - `curl` login/logout/session responses and inspect `Set-Cookie` and `Cache-Control` behavior
- Browser/manual verification:
  - optional local browser confirmation, but command-line proof is the minimum slice gate here
- Staging integration verification:
  - verify real deployed login response carries the expected cookie attributes through the staging boundary

## Slice BE-3 — `GET /api/auth/session`

Objective:
- provide dedicated session bootstrap truth source for browser frontend

Tasks:
1. Implement a new handler returning minimal display-safe session metadata only.
2. Register route in server/router entrypoints.
3. Authenticate via same cookie-backed path as other browser routes.
4. Return stable `401` when no valid session exists.
5. Mark response non-cacheable.

Files to modify/create:
- new handler, likely `backend/src/bff/handlers/auth-session.ts`
- `backend/scripts/local-server.ts`
- any routing/bootstrap file that registers auth routes
- related tests under `backend/test/bff/`

Segmented verification:
- Automated repo test:
  - new session endpoint tests for valid cookie, missing cookie, expired cookie, and response shape
- Command-line probe:
  - `curl` the session route with valid and invalid cookies and confirm status/body/cache behavior
- Browser/manual verification:
  - backend slice does not complete on browser proof alone, but the route must later appear in frontend bootstrap verification
- Staging integration verification:
  - confirm integrated frontend bootstrap consumes this endpoint successfully in staging

## Slice BE-4 — Browser cookie auth middleware cutover

Objective:
- make cookie-backed browser auth the default contract for browser traffic while preserving bounded internal compatibility

Tasks:
1. Refactor `backend/src/bff/middleware/auth.ts` to:
   - prioritize cookie-backed browser auth
   - keep internal tenant-context normalization for downstream handler stability
   - isolate any temporary legacy bearer compatibility switch
2. Update auth error messages and status semantics for the new contract.
3. Extend public-route allowlist to include `GET /api/auth/session` as appropriate.
4. Ensure `/api/events` uses the same cookie-backed path.
5. Ensure non-browser bearer contract, if retained, is explicit and not the default browser path.

Files to modify:
- `backend/src/bff/middleware/auth.ts`
- `backend/src/bff/handlers/sse-events.ts`
- route wrappers or helper adapters that still assume header-only extraction

Segmented verification:
- Automated repo test:
  - auth middleware tests assert cookie path, public-route list, SSE auth path, and compatibility-mode behavior
- Command-line probe:
  - protected API requests succeed with cookie auth and fail without valid session
  - legacy browser bearer behavior is accepted only when the compatibility switch is ON
- Browser/manual verification:
  - integrated browser check later confirms SSE/auth behavior is visible in real frontend flows
- Staging integration verification:
  - FE + BE staging flow must prove API and SSE both use the cookie-backed path

## Slice BE-5 — CSRF/same-origin enforcement

Objective:
- enforce the v6.9 same-origin-first posture

Tasks:
1. Add state-changing request checks for `Origin` and, where needed, `Referer` as secondary evidence.
2. Ensure rejection is stable control flow, not accidental `500`.
3. Review CORS behavior in local/public entrypoints to avoid contradicting same-origin intent.
4. Preserve explicit exceptions only where genuinely required for non-browser contract paths.

Files to modify/create:
- `backend/src/bff/middleware/auth.ts` or new dedicated origin-validation middleware
- `backend/scripts/local-server.ts`
- `backend/scripts/combined-server.ts`

Segmented verification:
- Automated repo test:
  - origin-validation tests assert same-origin allow and invalid cross-origin reject behavior
- Command-line probe:
  - send state-changing requests with acceptable and unacceptable origin metadata and inspect status codes
- Browser/manual verification:
  - not primary at slice level
- Staging integration verification:
  - confirm real same-origin browser writes still succeed after DEP boundary changes

## Slice BE-6 — Login abuse controls

Objective:
- add release-critical application-layer abuse resistance

Tasks:
1. Introduce a login-abuse service/module for:
   - source-IP throttle
   - identifier-based progressive backoff
   - reset on successful login
   - generic error semantics
   - audit event emission
2. Support store backends:
   - shared ephemeral store in production
   - in-memory development fallback only for local single-instance dev
3. Derive client IP through trusted-proxy-aware logic.
4. Return `429` with `Retry-After` for throttled/backed-off responses where chosen semantics apply.
5. Add operator-visible logging for success/failure/throttle/backoff.

Files to modify/create:
- `backend/src/bff/handlers/auth-login.ts`
- possible new modules under `backend/src/bff/` or `backend/src/shared/` for abuse tracking and IP derivation
- `backend/scripts/local-server.ts`
- `backend/bin/app.ts` for store/config initialization if needed

Segmented verification:
- Automated repo test:
  - abuse-control tests cover source-IP throttle, identifier backoff, reset on success, and generic error semantics
  - shared-store path is covered as far as repo-level automation reasonably allows
- Command-line probe:
  - repeated bad logins trigger documented throttle/backoff and `Retry-After` behavior where used
- Browser/manual verification:
  - login UX handling is validated in frontend staging checks, not as backend proof by itself
- Staging integration verification:
  - multi-instance staging confirms behavior remains consistent when the shared store is used

## Slice BE-7 — Trusted proxy and source-IP derivation

Objective:
- make IP-based controls correct across ingress and direct-BFF modes

Tasks:
1. Add a trusted-proxy config contract.
2. Ensure forwarded IP headers are honored only through trusted proxy hops.
3. Fail closed to conservative behavior when proxy config is absent/ambiguous.
4. Cover direct local-server behavior using peer address.

Files to modify/create:
- `backend/scripts/local-server.ts`
- shared request/IP helper module
- any express app config handling trust proxy

Segmented verification:
- Automated repo test:
  - trusted-proxy tests demonstrate forwarded IP acceptance only through trusted hops
  - spoofed `X-Forwarded-For` from untrusted direct clients is rejected or ignored
- Command-line probe:
  - local and staging checks validate expected source-IP behavior across direct and proxied paths
- Browser/manual verification:
  - not primary for this slice
- Staging integration verification:
  - staging rehearsal validates the ingress topology actually feeds the expected client IP to abuse controls

## Slice BE-8 — Compatibility bridges and removal-ready isolation

Objective:
- keep temporary bridges isolated and easy to delete

Tasks:
1. Implement legacy browser bearer compatibility behind a dedicated switch.
2. Implement dual-cookie bridge behind a dedicated switch.
3. Ensure both are logged when active.
4. Keep bridge code localized to auth middleware and cookie helper layers.
5. Add explicit removal TODOs in code/comments tied to v6.9 cleanup.

Files to modify/create:
- `backend/src/bff/middleware/auth.ts`
- `backend/src/bff/handlers/auth-login.ts`
- shared auth config/helper module

Segmented verification:
- Automated repo test:
  - bridge ON/OFF behavior is explicitly test-covered
- Command-line probe:
  - toggle bridge settings and verify the HTTP-visible contract changes only as documented
- Browser/manual verification:
  - later staging browser flow confirms bridge-ON and bridge-OFF auth behavior from the user perspective
- Staging integration verification:
  - both bridge-ON and bridge-OFF integrated flows are mandatory; passing ON does not waive OFF

## 5.3 Phase gates tied to backend testing

Phase 1 gate — before frontend cutover begins:
- BE-1 through BE-8 slice-level automated repo tests pass as applicable
- required command-line probes pass for login, session, logout, protected API, SSE auth, origin enforcement, abuse controls, and bridge toggles
- bridge ON/OFF behavior is explicitly verified
- Phase 1 is not complete based on `npm test` alone

Phase 3 integration gate contribution:
- backend must already support the full staging integrated flow before DEP hardening rehearsal starts
- after DEP hardening, backend behavior must still pass the same integrated flows under the public boundary

Final release gate contribution:
- backend slice tests are necessary but insufficient
- final release approval also depends on integrated staging, browser-visible SSE/logout/expiry checks, public-boundary header/CSP/cookie checks, and bridge-OFF steady-state validation

## 6. Dependency ordering

Backend order should be:
1. secret/config centralization
2. cookie helper contract
3. session endpoint
4. auth middleware cutover
5. abuse controls + trusted proxy handling
6. bridge isolation and tests
7. local/direct-BFF entrypoint alignment

Reason:
- frontend cannot switch cleanly until login/session/cookie semantics are stable
- deployment verification depends on stable route and cookie behavior

## 7. Rollout sequence

1. Ship backend with new auth config validation, cookie contract, session endpoint, and auth middleware.
2. Enable compatibility bridges only if rollout plan requires them.
3. Verify frontend works with new backend in staging.
4. After frontend cutover, disable legacy bearer compatibility mode.
5. After migration window, remove old-cookie bridge.

## 8. Rollback sequence

1. If issue is frontend skew, keep backend bridge ON temporarily while rolling frontend back.
2. If issue is cookie-name transition, re-enable dual-cookie bridge instead of reverting steady-state contract.
3. If issue is severe backend auth regression, rollback backend and frontend together as a coherent pair.
4. Never rollback by restoring secret fallback or by turning insecure browser bearer support into default behavior.

## 9. Risks / sequencing notes Ryan should audit

1. Internal downstream handler compatibility relies on normalized tenant-context shims. Ryan should audit that the shim does not accidentally keep legacy browser auth semantics alive longer than intended.
2. The repo has many tests and some handlers built around authorization-header assumptions. Ryan should audit whether a phased adapter layer is enough to avoid broad route churn.
3. `auth-login.ts` currently mixes login DB work, JWT issuance, cookie issue, and response shaping. Ryan should review whether a small auth service extraction is needed to keep abuse controls and bridge logic maintainable.
4. Trusted proxy handling is security-critical because it feeds abuse controls. Ryan should require explicit configuration and failure-closed semantics.
5. `combined-server.ts` currently uses permissive CORS and should not remain an implicit alternative contract.
6. Exact throttle/backoff thresholds remain an implementation review point; Ryan should confirm they are documented before approval.
7. Backend teams may over-weight repo automation and under-weight integrated browser-visible checks. Ryan should ensure the release gate still requires those integrated checks even after backend suites pass.

## 10. Done criteria

Backend auth-boundary work is done when:
- login/logout/session routes implement the v6.9 contract;
- cookie-first browser auth works for API and SSE;
- login abuse controls are live and testable;
- trusted-proxy-aware IP derivation is correct;
- startup fails without safe auth configuration;
- bridges are bounded, switchable, and removal-ready;
- segmented backend verification has passed;
- integrated staging flows depending on backend behavior have passed.
