## 1. Goal

Implement the v6.9 security correction release as a coordinated frontend + backend + deployment cutover that:
- removes browser dependence on `localStorage` bearer JWT;
- converges browser API and SSE onto the cookie-first session contract;
- introduces release-critical application login-abuse controls;
- removes production-capable JWT secret fallback behavior;
- deploys the required browser-hardening headers and CSP;
- includes bounded compatibility bridges with explicit removal steps.

This is an implementation plan, not a redesign. The settled v6.9 decisions are fixed inputs:
- cookie name: `__Host-solfacil_session`
- `SameSite=Strict`
- dedicated `GET /api/auth/session`
- production auth-abuse state in shared ephemeral store
- Google Fonts and ECharts self-hosted before rollout

## 2. Brief architecture summary

v6.9 changes the browser auth truth from split storage/cookie behavior to a same-origin cookie-first model:
- Frontend login and SPA bootstrap trust server-authenticated session state, not browser token presence.
- M5 BFF authenticates browser `/api/*` and `/api/events` through the auth cookie.
- M6 login/logout/session endpoints own cookie issuance, clearing, and session metadata.
- M9 shared verification keeps internal tenant-context normalization but must stop using insecure secret fallback.
- Production ingress owns authoritative CSP and browser hardening headers; application owns cookie correctness, auth semantics, cache semantics, and local/direct-BFF fallback headers.

## 4. Test strategy overview and release gates

The v6.9 PLAN uses three distinct test layers. They are cumulative and none of them substitutes for the others.

### 4.1 Layer 1 — Slice-level / segmented testing

Purpose:
- prove each FE, BE, or DEP slice works before the next slice depends on it

Allowed methods in this layer:
- automated repo test
- command-line probe
- focused browser/manual verification
- operator/infrastructure verification

This layer answers: "did this individual slice land correctly?"

This layer does not answer: "is the full v6.9 release contract complete?"

### 4.2 Layer 2 — Cross-workstream integration testing

Purpose:
- verify FE + BE integration once the browser contract is connected
- verify FE + BE + DEP integration once ingress/CSP/header hardening is active
- exercise the migration bridges in realistic staging flows

Required environments and modes:
- staging with compatibility bridge ON when migration rehearsal requires it
- staging with compatibility bridge OFF for steady-state validation
- public-boundary verification for effective headers, CSP, cookie attributes, and cache behavior

This layer answers: "do the workstreams operate together in staging and at the public boundary?"

### 4.3 Layer 3 — Final release acceptance / full integrated testing

Purpose:
- verify the full v6.9 security contract in one explicit release gate

This layer is represented by one explicit integrated matrix in Section 9, not by scattered bullets alone.

Release approval requires all of the following, not just `npm test`, `npm run build`, or a few curl probes:
- segmented backend tests passing
- segmented frontend verification, including mandatory browser checks, passing
- segmented deployment/operator verification passing
- cross-workstream staging flow verification passing
- public-boundary header/CSP verification passing
- bridge ON migration rehearsal passing if bridge is used during rollout
- bridge OFF steady-state verification passing

Why first:
- frontend session-probe and cookie-first bootstrap depend on server support
- deployment header verification depends on stable route behavior

Slices:
1. Secret fail-fast utilities and startup validation
2. Cookie contract update for login/logout
3. `GET /api/auth/session`
4. browser cookie auth path for API + SSE
5. CSRF/same-origin enforcement on state-changing requests
6. login abuse controls and audit events
7. temporary compatibility bridge for old browser bearer and old cookie name

Phase 1 gate — must pass before Phase 2 starts:
- backend supports new cookie and session endpoint
- backend can run with compatibility mode ON for migration testing
- backend segmented automated tests and targeted command-line probes pass for secret validation, cookie contract, session endpoint, API auth, SSE auth, CSRF/origin checks, abuse controls, and bridge ON/OFF behavior
- this gate is not satisfied by `npm test` alone; required slice probes and auth-flow verification must also pass

## Phase 2 — Frontend browser cutover

Why second:
- frontend must target the stabilized backend session contract

Slices:
1. login page session probe and cookie-first submit path
2. SPA bootstrap from server session truth
3. request-helper removal of `Authorization` header injection
4. unified `401` handling and SSE cleanup
5. legacy storage cleanup and compatibility code removal from frontend
6. inline-script/event-handler cleanup and self-hosted asset path updates required for CSP

Phase 2 gate — must pass before Phase 3 starts:
- frontend operates with compatibility mode OFF
- stale `localStorage` auth no longer affects auth correctness
- frontend slice-level verification passes for each FE slice with explicit automated/browser/manual coverage
- mandatory browser verification confirms no auth token storage, no frontend-set `Authorization` header, correct session bootstrap, correct logout/expiry handling, and SSE cleanup behavior

## Phase 3 — Deployment boundary hardening and rollout rehearsal

Slices:
1. ingress/browser hardening headers and CSP
2. local/direct-BFF fallback header behavior
3. trusted-proxy and secure-cookie deployment wiring
4. shared abuse-state store enablement
5. end-to-end staging verification with compatibility bridge ON then OFF

Phase 3 gate — must pass before production rollout:
- DEP slice-level verification passes for runtime inputs, ingress headers, CSP, cookie transport, shared abuse-store topology, and local/direct-BFF alignment
- cross-workstream staging tests pass once with FE + BE connected, and again with DEP hardening active
- public-boundary verification confirms effective CSP, header baseline, cache semantics, and cookie attributes on login and SPA entry
- compatibility bridge ON rehearsal passes if rollout will use the bridge
- compatibility bridge OFF steady-state rehearsal passes

## Phase 4 — Production rollout and bridge removal

Slices:
1. coordinated FE+BE+DEP cutover with bounded bridge enabled
2. live verification
3. bridge disablement after migration window
4. post-cutover doc closure

Production release gate — must pass before rollout is considered complete:
- the final integrated acceptance matrix in Section 9 passes in the release environment
- live verification confirms bridge OFF steady-state behavior before bridge-removal closure
- no team treats partial test success (`npm test`, `npm run build`, or isolated curl checks) as final v6.9 release approval

## 7. Compatibility bridges and removal plan

## 7.1 Bridge A — Legacy browser bearer compatibility mode

Purpose:
- short-lived migration or emergency rollback support only

Implementation expectation:
- backend-only switch
- disabled by default
- explicit logging when enabled
- test-covered ON/OFF behavior

Removal plan:
1. Enable only during coordinated rollout window if required.
2. Confirm new frontend works with compatibility mode OFF in staging.
3. Disable in production after migration window.
4. Remove code/tests/docs for legacy browser bearer acceptance in the next cleanup change of the v6.9 train, not later as unowned debt.

## 7.2 Bridge B — Dual-cookie migration (`solfacil_jwt` -> `__Host-solfacil_session`)

Purpose:
- preserve some in-flight browser sessions during cutover

Implementation expectation:
- short dual-read/dual-issue or equivalent bounded bridge
- explicit duration and operator removal step
- no reuse of old cookie name as long-term steady state

Removal plan:
1. Roll out with dual-cookie bridge enabled if minimizing forced re-login is desired.
2. Monitor session continuity and auth error rate.
3. Disable old-cookie read/issue after bounded window.
4. Remove old cookie handling from backend, tests, and docs.

## 7.3 Bridge C — CSP nonce/hash exception for inline remnants

Purpose:
- contingency only if inline-script cleanup misses the train

Implementation expectation:
- no broad `script-src 'unsafe-inline'`
- only nonce/hash-based exception if absolutely needed
- removal tracked as release blocker follow-up before approval if used

Removal plan:
- remove once `frontend-v2/login.html` inline script and `frontend-v2/index.html` inline handler are externalized

## 9. Verification strategy and final integrated matrix

## 9.1 Slice-level / segmented verification

Classification used in this section:
- Automated repo test: repository test suite or focused automated test command
- Command-line probe: `curl` or equivalent local/staging HTTP check
- Browser/manual verification: real-browser validation, including devtools/network/storage inspection
- Operator/infrastructure verification: environment, ingress, store, or runtime-config confirmation

Run from `backend/` where applicable:
- Automated repo test:
  - `npm test -- --runInBand`
  - `npm run build`
  - `npm run lint`

Suggested focused backend test commands as slices land:
- Automated repo test:
  - `npm test -- auth-login.test.ts --runInBand`
  - `npm test -- auth-middleware.test.ts --runInBand`
  - `npm test -- tenant-context.test.ts --runInBand`
  - `npm test -- --runInBand bff`

Suggested live/local verification commands after local server changes:
- Command-line probe:
  - `curl -i -X POST http://127.0.0.1:3000/api/auth/login -H 'Content-Type: application/json' --data '{"email":"admin@solfacil.com.br","password": "***"}'`
  - `curl -i http://127.0.0.1:3000/api/auth/session --cookie '__Host-solfacil_session=...'`
  - `curl -i -X POST http://127.0.0.1:3000/api/auth/logout --cookie '__Host-solfacil_session=...'`
  - repeated invalid-login curls to confirm `429` / `Retry-After`

Suggested browser-side verification:
- Browser/manual verification:
  - confirm no auth JWT is written to `localStorage` or `sessionStorage`
  - confirm network requests succeed without frontend-set `Authorization` header
  - confirm `EventSource('/api/events')` authenticates through cookie
  - confirm logout closes SSE and subsequent API calls return `401`

Operator/runtime prerequisites before Phase 3 rehearsal:
- Operator/infrastructure verification:
  - confirm safe `JWT_SECRET` is configured
  - confirm trusted proxy configuration is present
  - confirm shared abuse-state store is reachable
  - confirm self-hosted frontend assets required for CSP are deployed in the target environment

## 9.2 Cross-workstream integration testing

Integrated staging verification is required after FE + BE are connected and again after DEP hardening is active.

Integrated staging flow set:
- Mixed browser/manual + command-line verification:
  - successful login -> session bootstrap -> protected API -> SSE
  - logout -> cookie cleared -> SSE closed -> next protected API returns `401`
  - expired/invalid session -> frontend redirect and backend `401` semantics align
  - stale `localStorage` token does not restore access
  - repeated invalid login triggers documented throttle/backoff behavior

Bridge coverage:
- run the integrated staging flow set once with compatibility bridge ON if rollout uses it
- run the integrated staging flow set again with compatibility bridge OFF
- the OFF run is the steady-state gate and cannot be skipped just because the ON run passed

Public-boundary verification after DEP changes:
- Command-line probe:
  - `curl -I https://<host>/login`
  - `curl -I https://<host>/`
  - `curl -i https://<host>/api/auth/session`

Validate presence/effectiveness of:
- `Content-Security-Policy`
- `X-Frame-Options` and/or `frame-ancestors`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy`
- `Permissions-Policy`
- `Strict-Transport-Security` on production HTTPS

Cookie verification:
- confirm `__Host-solfacil_session; Secure; HttpOnly; Path=/; SameSite=Strict`
- confirm no `Domain` attribute
- confirm login/logout/session responses carry `Cache-Control: no-store`

## 9.4 Final release acceptance requirements

The final release gate is broader than slice tests and broader than a single staging smoke run.

v6.9 release approval requires all of the following:
- segmented backend tests pass
- segmented frontend automated checks and mandatory browser verification pass
- segmented deployment/operator verification passes
- cross-workstream staging verification passes with FE + BE integration and with DEP hardening active
- public-boundary header/CSP/cookie verification passes
- bridge ON migration rehearsal passes if bridge is used during rollout
- bridge OFF steady-state verification passes

Passing `npm test`, `npm run build`, or isolated curl probes does not by itself satisfy this release gate.

## 10. Rollout sequence

1. Provision/verify production `JWT_SECRET`, trusted-proxy config, and shared abuse-state store.
2. Deploy backend with:
   - fail-fast config validation
   - new cookie contract
   - `GET /api/auth/session`
   - abuse controls
   - compatibility mode ON only if migration window requires it
   - dual-cookie bridge ON only if migration window requires it
3. Deploy frontend with cookie-first bootstrap, no bearer injection, self-hosted assets, and CSP-compatible HTML/JS.
4. Deploy ingress/browser header config.
5. Run live smoke checks on login, session probe, protected API, logout, SSE, and repeated invalid login.
6. After stability window:
   - disable legacy bearer compatibility mode
   - disable/remove dual-cookie bridge
7. Close matrix docs and release notes with final steady-state behavior.

## 11. Rollback sequence

Rollback objective is service restoration without silently restoring insecure steady-state defaults.

Preferred rollback order:
1. If new frontend is still deployed and backend issue appears, temporarily enable backend legacy compatibility mode only if necessary for emergency continuity.
2. If cookie-name migration caused unacceptable churn, temporarily re-enable dual-cookie bridge rather than reverting to long-term old-cookie behavior.
3. If frontend regression is primary, rollback frontend first while keeping backend bridge ON during the rollback window.
4. If backend regression is primary and compatibility mode cannot stabilize service, rollback backend and frontend together to the last known coherent pair.
5. Do not rollback by:
   - removing CSP from public surface as a convenience fix;
   - reintroducing secret fallback;
   - making insecure production cookies non-secure.

Rollback prerequisites:
- bridge toggles must exist and be operator-documented before production rollout
- release package must preserve last-known-good frontend and backend artifacts as a pair

## 12. Risks and sequencing notes Ryan should audit

1. Frontend/backend skew remains the largest delivery risk; the plan assumes coordinated FE+BE release.
2. The repo currently has many handler tests and middleware assumptions centered on `Authorization` headers. Ryan should audit the blast radius of test updates and any internal adapter strategy used to avoid route-by-route churn.
3. `backend/scripts/local-server.ts` currently applies auth middleware before CORS and still advertises JWT-header semantics. Ryan should confirm local/dev boundary behavior stays aligned with the new browser contract.
4. `backend/scripts/combined-server.ts` is permissive and must not be left as a misleading security reference surface.
5. `SameSite=Strict` must be validated against real operator login/navigation flows before approval.
6. Application abuse controls depend on trusted client-IP derivation. Ryan should audit the exact trusted-proxy configuration contract and failure-closed behavior.
7. CSP rollout depends on finishing or tightly controlling inline-script cleanup and self-hosted asset delivery. Ryan should require explicit confirmation that those file changes are ready before approving rollout.
8. Because v6.9 keeps stateless JWT sessions, SSE revocation remains limited. Ryan should confirm the documented residual risk is acceptable without server-side active disconnect machinery.
9. The final release gate now depends on explicit browser/manual and public-boundary verification. Ryan should confirm owners and environments are assigned so those checks are not skipped in practice.
