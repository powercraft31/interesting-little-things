## 2. Remaining correctness issues before proceeding further

### 2.1 B5 cookie attributes drifted from accepted design

Current problem in `backend/src/bff/handlers/auth-login.ts`:
- browser login cookie still uses `sameSite: "lax"`
- logout cookie clear also uses `sameSite: "lax"`

Why this is wrong:
- accepted design baseline is `SameSite=Strict`
- implementation must not regress to the weaker historical default

Required correction:
- browser login cookie and logout clear path must use `sameSite: "strict"`
- preserve current browser/machine split and session cookie name behavior
- update tests accordingly

### 2.2 B6 exists but is not yet wired into runtime

Current problem:
- `backend/src/bff/middleware/rate-limit.ts` and tests exist
- but `backend/scripts/local-server.ts` still registers login route directly as:
  `app.post("/api/auth/login", createLoginHandler(servicePool));`
- there is no visible B6 middleware hookup on login route yet

Why this matters:
- B6 is not complete until the abuse-control middleware is actually active in runtime
- tests of the middleware in isolation are not enough

Required correction:
- initialize/select rate-limit store in `local-server.ts`
- attach abuse-control middleware only to `POST /api/auth/login`
- ensure non-dev startup still fails if `RATE_LIMIT_REDIS_URL` is absent/unusable
- add or update targeted tests if hookup logic needs coverage

### 2.3 B6 should not silently depend on uninstalled Redis package at runtime

Current problem:
- `rate-limit.ts` uses runtime `require("ioredis")`
- Claude hit permission denials trying to install `ioredis`
- current repo state may pass tests without ever exercising Redis path, but production path still depends on that package existing

Required correction:
- if `ioredis` is the selected implementation, add it to backend dependencies explicitly
- verify package manifest is updated
- if package install cannot be performed in this environment, state it explicitly rather than pretending B6 production path is complete
