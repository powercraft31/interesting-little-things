## 1. Scope

This document covers the frontend-side changes required by v6.9. It is the frontend companion to the master security design and addresses:

- Login bootstrap via `GET /api/auth/session`
- Elimination of localStorage/sessionStorage as auth truth
- 401/session-loss recovery behavior
- CSP blockers in the current frontend structure
- Self-hosting of third-party assets

### Architecture Surface

| v6.8 Frontend Artifact | v6.9 Impact |
|------------------------|-------------|
| `login.html` | **Rewritten**: inline script extracted; onclick removed; auth check uses session endpoint |
| `index.html` | **Updated**: ECharts `<script>` points to self-hosted path; Google Fonts `<link>` replaced with local `@font-face`; cache-bust version bumped |
| `js/app.js` | **Updated**: auth guard uses `GET /api/auth/session` instead of localStorage check; logout removes no localStorage; role/user state populated from session endpoint response |
| `js/data-source.js` | **Updated**: `apiGet`/`apiPost`/`apiPut` no longer set `Authorization` header; rely on cookie; 401 handler simplified (no localStorage cleanup) |
| `js/config.js` | **Unchanged** |
| `js/p1-fleet.js` … `js/p6-alerts.js` | **Unchanged** — page modules do not touch auth directly |
| `js/p2-devices.js` (SSE) | **Minimal change**: `EventSource` URL unchanged; cookie name change is transparent to JS |
| `css/variables.css`, `css/base.css` | **Updated**: `@font-face` declarations for self-hosted Inter + JetBrains Mono |
| `css/layout.css`, `css/components.css`, `css/pages.css` | **Unchanged** |

## 2. Login Bootstrap — Fixed Design

### 2.1 v6.8 Behavior (Historical — Being Replaced)

> **Historical context only** — the following describes the pre-v6.9 browser auth flow. None of these patterns (localStorage token, `solfacil_jwt`, token-in-body) are part of the v6.9 design.

```
login.html load:
  1. Check localStorage("solfacil_jwt")
  2. If token exists → redirect to index.html (no server validation)
  3. If no token → show login form

Form submit:
  4. POST /api/auth/login → receive { token } in body
  5. Store token in localStorage("solfacil_jwt")
  6. Redirect to index.html
```

**Problems being fixed**: Token in localStorage is XSS-extractable. The "already logged in?" check trusts a client-side string without server validation.

### 2.2 v6.9 Design

```
login.html load:
  1. Call GET /api/auth/session (cookie attached automatically)
  2. If 200 → user already has valid session → redirect to index.html
  3. If 401 → show login form

Form submit:
  4. POST /api/auth/login { email, password }
  5. Server sets __Host-solfacil_session cookie via Set-Cookie header
  6. Response body contains { user } but NO token field
  7. Redirect to index.html

index.html load:
  8. Call GET /api/auth/session
  9. If 200 → store user info (userId, orgId, role, name) in JS runtime variable
  10. If 401 → redirect to login.html
```

### 2.3 Auth Guard Contract

| Location | v6.8 | v6.9 |
|----------|------|------|
| `login.html` page load | `localStorage.getItem("solfacil_jwt")` | `GET /api/auth/session` — server-validated |
| `app.js` init | `if (!localStorage.getItem("solfacil_jwt")) redirect` | `GET /api/auth/session` — if 401, redirect |
| User state source | Decoded from localStorage JWT (or not decoded — role came from login response stored separately) | Response from `GET /api/auth/session` |

## 3. No-localStorage Auth — Fixed Design

### 3.1 What Is Removed

| v6.8 localStorage Usage _(historical — all removed)_ | v6.9 Replacement |
|--------------------------|------------------|
| `localStorage.setItem("solfacil_jwt", token)` in login flow | Removed. Cookie is set by server |
| `localStorage.getItem("solfacil_jwt")` in auth guard | Replaced by `GET /api/auth/session` |
| `localStorage.removeItem("solfacil_jwt")` in logout | Removed. Server clears cookie |
| `Authorization: Bearer ${localStorage.getItem("solfacil_jwt")}` in every API call | Removed. Cookie auto-attached by browser |

### 3.2 What Remains in localStorage

| Key | Purpose | Auth-related? |
|-----|---------|---------------|
| `lang` | i18n language preference | No |

### 3.3 What Remains in sessionStorage

| Key pattern | Purpose | Auth-related? |
|-------------|---------|---------------|
| `ds_*` | DemoStore cross-page ephemeral state | No |

No auth-related data is stored in localStorage or sessionStorage after v6.9.

### 3.4 Runtime User State

After `GET /api/auth/session` returns 200, the frontend stores user info in a JS module-level variable (not persisted to storage):

```
// In app.js — runtime only, lost on page close/refresh (re-fetched on load)
let currentUser = null;  // { userId, orgId, role, name, email }
```

On every full page load (refresh, new tab), `GET /api/auth/session` is called to re-populate this variable. This is the single source of truth for the frontend.

## 4. Authenticated Request Model

### 4.1 `data-source.js` Changes

The `apiGet()`, `apiPost()`, `apiPut()` helper functions are simplified:

| Aspect | v6.8 | v6.9 |
|--------|------|------|
| Headers | `{ 'Authorization': 'Bearer ' + localStorage.getItem('solfacil_jwt'), 'Content-Type': 'application/json' }` | `{ 'Content-Type': 'application/json' }` (cookie auto-attached) |
| `credentials` | Not explicitly set (defaults to `same-origin` for same-origin requests) | Same default behavior — no change needed |
| 401 handling | Remove localStorage token, redirect to `login.html` | Redirect to `login.html` (no localStorage cleanup) |

### 4.2 SSE (`EventSource`)

The SSE connection in `p2-devices.js` requires no functional change. `EventSource` already relies on cookies for auth (it cannot set custom headers). The only difference is the cookie name (`__Host-solfacil_session` instead of the historical pre-v6.9 `solfacil_jwt`), which is transparent to JS — the browser attaches cookies automatically.

```javascript
// Unchanged in v6.9
self._sseSource = new EventSource(base + "/api/events");
```

## 5. 401 / Session-Loss Behavior

### 5.1 API Call Returns 401

```
apiGet/apiPost/apiPut receives 401:
  → Redirect to login.html immediately
  → No retry, no token refresh (no refresh token in v6.9)
```

### 5.2 SSE Connection Error

```
EventSource onerror fires:
  → Call GET /api/auth/session
  → If 401 → redirect to login.html
  → If 200 → SSE error was transient (network); EventSource auto-reconnects per spec
```

### 5.3 Page Load Session Check Failure

```
index.html loads → app.js init:
  → GET /api/auth/session
  → If 401 → redirect to login.html (user never sees SPA content)
```

### 5.4 Logout Flow

```
User clicks logout:
  → POST /api/auth/logout
  → Server clears cookie
  → Frontend redirects to login.html
  → No localStorage/sessionStorage cleanup for auth
```

## 6. CSP Blockers — Current Frontend Structure

The v6.9 CSP target (`script-src 'self'; style-src 'self'; font-src 'self'`) requires resolving these blockers in the current frontend.

### 6.1 Blocker Inventory

| # | Blocker | File | Current Code Pattern | Resolution |
|---|---------|------|---------------------|------------|
| B1 | ECharts CDN script | `index.html` | `<script src="https://cdn.jsdelivr.net/npm/echarts@5/dist/echarts.min.js">` | Download `echarts.min.js` (~1MB), place at `frontend-v2/js/vendor/echarts.min.js`, update `<script src>` to relative path |
| B2 | Google Fonts CSS | `index.html` | `<link href="https://fonts.googleapis.com/css2?family=Inter:wght@...&family=JetBrains+Mono:wght@...">` | Download Inter and JetBrains Mono WOFF2 files. Place in `frontend-v2/fonts/`. Add `@font-face` declarations to `variables.css` or a new `fonts.css`. Remove the Google Fonts `<link>` and `<link rel="preconnect">` |
| B3 | Inline `<script>` in `login.html` | `login.html` | Login form logic in `<script>...</script>` block | Extract to `frontend-v2/js/login.js`. Load via `<script src="js/login.js">` |
| B4 | Inline `onclick` in `login.html` | `login.html` | `<button onclick="handleLogin()">` (or similar) | Replace with `document.getElementById('login-btn').addEventListener('click', handleLogin)` inside `login.js` |
| B5 | Google Fonts `preconnect` | `index.html`, `login.html` | `<link rel="preconnect" href="https://fonts.googleapis.com">` | Remove after self-hosting fonts |

### 6.2 Self-Hosting Requirements

| Asset | Source | Target Path | Size Estimate |
|-------|--------|-------------|---------------|
| `echarts.min.js` | `cdn.jsdelivr.net/npm/echarts@5/dist/echarts.min.js` | `frontend-v2/js/vendor/echarts.min.js` | ~1MB (gzipped ~300KB) |
| Inter (WOFF2, 4 weights: 300,400,500,600) | `fonts.gstatic.com` | `frontend-v2/fonts/inter-*.woff2` | ~100KB per weight |
| JetBrains Mono (WOFF2, 1-2 weights: 400,500) | `fonts.gstatic.com` | `frontend-v2/fonts/jetbrains-mono-*.woff2` | ~50KB per weight |

### 6.3 Nonce/Hash Exception — Last Resort Only

If a specific inline script cannot be extracted (unlikely), the CSP may include a `sha256-<hash>` exception for that exact script content. The exception must be:
- Specific to that one script block (not a broad allowance)
- Documented with reason in the CSP middleware comment
- Tracked for removal in a future release

Broad `'unsafe-inline'` in `script-src` is never acceptable.

## 7. `login.html` — Design Constraints

### 7.1 Structural Changes

| Aspect | v6.8 | v6.9 |
|--------|------|------|
| Script location | Inline `<script>` block | External `<script src="js/login.js">` |
| Event binding | `onclick="..."` attributes | `addEventListener` in `login.js` |
| Auth check on load | `localStorage.getItem("solfacil_jwt")` | `GET /api/auth/session` |
| Login success handling | Store token in localStorage, redirect | Redirect only (cookie set by server) |
| Font loading | Google Fonts `<link>` | Self-hosted `@font-face` (same CSS as `index.html`) |
| Error display | Inline i18n strings | Maintained; no change to UX |

### 7.2 `login.js` Responsibility

The extracted `login.js` file handles:
1. On page load: call `GET /api/auth/session` → if 200, redirect to `index.html`
2. Form submit: call `POST /api/auth/login` → if 200, redirect to `index.html` (cookie already set); if 401/429, show error
3. No localStorage read/write for auth
4. No inline event handlers

## 8. `index.html` — Design Constraints

### 8.1 Script Tag Changes

| v6.8 | v6.9 |
|------|------|
| `<script src="https://cdn.jsdelivr.net/npm/echarts@5/dist/echarts.min.js">` | `<script src="js/vendor/echarts.min.js?v=6.9">` |
| `<link href="https://fonts.googleapis.com/css2?...">` | Removed. `@font-face` in CSS |
| `<link rel="preconnect" href="https://fonts.googleapis.com">` | Removed |
| `<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>` | Removed |

### 8.2 No Inline Scripts

`index.html` in v6.8 does not contain inline `<script>` blocks (all logic is in external JS files loaded via `<script src>`). No CSP blocker from `index.html` script tags.

## 10. Bounded Open Points

| Open Point | Constraint |
|------------|------------|
| Exact `@font-face` weight/style subset for Inter and JetBrains Mono | Must match the weights currently loaded from Google Fonts (Inter: 300,400,500,600; JetBrains Mono: 400,500). Only WOFF2 format needed (all target browsers support it) |
| Whether to create a separate `fonts.css` or add `@font-face` to `variables.css` | Either is acceptable. Single-file (`variables.css`) is simpler given the zero-build-tool constraint |
| ECharts version pin | Must match current CDN version (5.x). Pin to exact minor version in filename for cache clarity |
| `login.js` — whether to use ES modules or IIFE | IIFE, consistent with all other frontend JS files (no build step, no module loader) |
