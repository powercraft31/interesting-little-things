## 1. What improved materially

The third pass is much closer to the intended design discipline than the second pass.

Recovered successfully:
1. `__Host-solfacil_session` is back as the fixed browser cookie baseline.
2. `SameSite=Strict` is back as the intended browser baseline.
3. `GET /api/auth/session` is back as a fixed browser bootstrap/session-truth contract.
4. Browser vs machine contracts are explicitly separated again.
5. Self-hosting of Google Fonts and ECharts is back as the desired first-release baseline.
6. Broad `script-src 'unsafe-inline'` is again rejected.
7. Redis/shared ephemeral abuse-control topology is back as the production primary design.
8. Bridge discipline is tighter than the second pass.

## 2. Remaining problems that must be corrected before PLAN

These are not optional polish items. They are design-level correctness issues.

### 2.1 Do not fail open on production abuse-control backing-store loss

Current problem:
- master design still states a production posture equivalent to: if Redis is unavailable at startup, log warning and allow login without rate limiting.

Why this is wrong:
- the accepted design logic says login abuse control is release-critical and application-owned;
- if production starts without the shared backing store, the intended production protection is absent;
- "fail open with warning" silently turns a release-critical control into best effort.

Required correction:
- production-capable runtime must not start in normal mode without the configured shared abuse-control store;
- if the production/shared store is required by config and unavailable, startup or auth route enablement must fail closed;
- in-memory abuse control may remain only for local/dev single-process mode, not as production degradation behavior.

### 2.2 Remove any lingering browser baseline wording that re-normalizes `solfacil_jwt` or `SameSite=Lax`

Current problem:
- backend design still contains a line describing a login behavior that sets `solfacil_jwt` cookie with `SameSite=Lax`.

Why this is wrong:
- even if intended as a comparison or leftover legacy note, it is too easy to read as an active acceptable contract;
- this reintroduces ambiguity into a design that is supposed to freeze the browser baseline.

Required correction:
- the steady-state browser contract must be described only with:
  - `__Host-solfacil_session`
  - `HttpOnly`
  - `Secure`
  - `SameSite=Strict`
  - `Path=/`
  - no `Domain`
- if legacy pre-v6.9 behavior is referenced, it must be clearly labeled as historical old state only, not a candidate implementation path.

### 2.3 Tighten secret-governance wording so unsafe fallback is not normalized as a valid runtime behavior

Current problem:
- the docs still carry wording that can be read as permitting `solfacil-dev-secret` fallback in some non-production runtime paths.

Why this is still too soft:
- the whole point of v6.9 is to stop normalizing unsafe secret fallback behavior;
- design should prefer explicit local/dev secret provisioning over code-level fallback normalization;
- otherwise PLAN and implementation can drift back toward convenience-first behavior.

Required correction:
- design should state that production-capable code paths do not rely on a built-in unsafe fallback;
- local/dev may use explicit local configuration, but the design should not present `solfacil-dev-secret` fallback as the standard mechanism to keep;
- if a transitional compatibility note is necessary, it must be framed as legacy debt to eliminate, not as a desirable preserved behavior.

### 2.4 Tighten machine login contract wording so it does not accidentally weaken the browser-first closure

Current problem:
- backend design introduces machine login discrimination using a request header and separate body-token response shape.

This is not automatically wrong, but it is still under-specified.

Required correction:
- keep browser and machine contracts separate;
- make clear that machine login/body-token behavior is not the browser baseline and must not reopen browser bearer semantics by convenience;
- mark exact machine-eligible route scope as a bounded open point for PLAN/implementation, not as a vague general fallback.

## 3. Revision request to Claude Code

Revise the four DESIGN docs in place.

Revision priorities:
1. production abuse-control store unavailability must not degrade to fail-open normal operation;
2. remove or relabel every lingering `solfacil_jwt` / `SameSite=Lax` browser-baseline phrasing so no ambiguity remains;
3. tighten secret-governance wording to avoid normalizing `solfacil-dev-secret` fallback as an accepted runtime design;
4. sharpen machine-contract wording so browser bearer cannot creep back in through vague compatibility language.

## 4. Acceptance standard for the revised DESIGN

Do not move to PLAN unless the revised design clearly satisfies all of the following:
- browser steady-state contract is unambiguously `__Host-solfacil_session` + `SameSite=Strict`;
- `GET /api/auth/session` remains fixed;
- production abuse-control does not fail open when the shared backing store is absent;
- secret handling wording no longer normalizes built-in unsafe fallback behavior;
- machine contract remains explicitly separate and bounded from the browser contract.
