## 1. Workstream Map

v6.9 is organized into four parallel-capable workstreams with explicit dependency edges.

| WS | Name | Companion Plan Doc | Owner Scope |
|----|------|--------------------|-------------|
| **WS-F** | Frontend Security | `v6.9-frontend-security-plan.md` | `frontend-v2/` — login, auth guard, CSP blockers, self-hosted assets |
| **WS-B** | Backend Auth Boundary | `v6.9-backend-auth-boundary-plan.md` | `backend/src/bff/`, `backend/src/shared/`, `backend/scripts/` — middleware, session endpoint, abuse control, secret validation |
| **WS-D** | Deployment Security Boundary | `v6.9-deployment-security-boundary-plan.md` | Nginx config, `docker-compose.yml`, Redis provisioning, env vars, live verification |
| **WS-I** | Integration & Acceptance | This document §7–§9 | Cross-workstream integration, final acceptance, rollout |

### Dependency Graph

```
WS-B (backend)  ─────────────────────────────────────────►│
                                                           │  WS-I (integration
WS-F (frontend) ──── depends on WS-B session endpoint ───►│   & acceptance)
                                                           │
WS-D (deployment) ── depends on WS-B env var contracts ──►│
```

- **WS-B** has no upstream dependency. Start first.
- **WS-F** depends on WS-B delivering `GET /api/auth/session` and the cookie-first login response. Frontend work on CSP blockers (self-hosting, inline script extraction) is independent and can start immediately.
- **WS-D** depends on WS-B defining env var contracts (`JWT_SECRET` fail-fast, `RATE_LIMIT_REDIS_URL`). Nginx/HSTS work and Redis provisioning are independent and can start immediately.
- **WS-I** starts only after all three workstreams pass their slice-level gates.

## 4. Fixed Tasks vs. Bounded Open Implementation Decisions

### Fixed — No Design Latitude

| Task | Fixed Decision |
|------|---------------|
| Cookie name | `__Host-solfacil_session` (prod), `solfacil_session` (dev) |
| Cookie attributes | `HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=86400` |
| Session endpoint | `GET /api/auth/session`, cookie-only, returns user state |
| Browser login response | No `token` field in body |
| Abuse-control scope | `POST /api/auth/login` only |
| Abuse-control limits | 5/email, 10/IP per 15-min window |
| Production backing store | Redis, fail-closed if unavailable |
| Secret fail-fast | `process.exit(1)` if `JWT_SECRET` missing/weak in non-dev |
| CSP baseline | `script-src 'self'`, no `unsafe-inline` |
| Self-hosting | ECharts + Google Fonts (Inter, JetBrains Mono) |
| Machine login discrimination | `POST /api/auth/login` + `X-Auth-Contract: machine` header; no separate endpoint in v6.9 |
| Legacy browser bearer | **Not part of default implementation path**. Emergency-only rollback toggle (`LEGACY_BROWSER_BEARER`), disabled by default, max 48h lifetime if ever enabled. Not implemented unless rollback scenario demands it |
| Dual-cookie bridge | Not implemented |

### Bounded Open — Implementation Decision Required During Phase 1

| Decision | Constraint | When to Decide |
|----------|------------|----------------|
| Session endpoint: new file `auth-session.ts` vs. added to `auth-login.ts` | Either acceptable; new file is cleaner | WS-B implementation start |
| `jwtSecret` passing: parameter vs. module-level init | No runtime fallback allowed either way | WS-B implementation start |
| Redis client library | Must support TTL key expiry; `ioredis` expected | WS-B abuse-control implementation |
| ~~Machine login discrimination~~ | **Moved to Fixed** — see Fixed table above | — |
| `@font-face` location: `variables.css` vs. new `fonts.css` | Either acceptable | WS-F font self-hosting |
| ECharts exact version pin | Must match current CDN version (5.x) | WS-F asset self-hosting |
| `login.js` pattern: IIFE (expected, consistent with codebase) | No ES modules (no build step) | WS-F login script extraction |
| CSP report-only phase (24-48h before enforcement) | Deployment-time decision; policy content is fixed | WS-D rollout |
| Production Redis: managed service vs. Docker Compose | Must support TTL key ops | WS-D provisioning |

## 7. Cross-Workstream Integration Test Procedure

After Gates G1, G2, G3 pass:

### 7.1 Environment Setup

1. `docker-compose up` with Redis service, `JWT_SECRET` set, `RATE_LIMIT_REDIS_URL` pointing to Redis
2. Nginx configured with HSTS + proxy headers (or run directly for initial integration)
3. Frontend built with self-hosted assets

### 7.2 Integration Test Script (curl-based)

```
# 1. Browser-contract login
curl -v -X POST https://host/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"test@test.com","password":"..."}' \
  -c cookies.txt
# VERIFY: Set-Cookie has __Host-solfacil_session with correct attributes
# VERIFY: Response body has NO "token" field

# 2. Session endpoint with cookie
curl -v https://host/api/auth/session -b cookies.txt
# VERIFY: 200 with { userId, orgId, role, name, email }

# 3. Session endpoint without cookie
curl -v https://host/api/auth/session
# VERIFY: 401

# 4. API call with cookie
curl -v https://host/api/fleet/overview -b cookies.txt
# VERIFY: 200 with data

# 5. Machine-contract login
curl -v -X POST https://host/api/auth/login \
  -H 'Content-Type: application/json' \
  -H 'X-Auth-Contract: machine' \
  -d '{"email":"test@test.com","password":"..."}'
# VERIFY: Response body HAS "token" field, no Set-Cookie

# 6. Machine bearer API call
curl -v https://host/api/fleet/overview \
  -H 'Authorization: Bearer <token-from-step-5>'
# VERIFY: 200 with data

# 7. Machine bearer on session endpoint (should fail)
curl -v https://host/api/auth/session \
  -H 'Authorization: Bearer <token-from-step-5>'
# VERIFY: 401

# 8. Security headers check
curl -I https://host/
# VERIFY: Content-Security-Policy, X-Frame-Options: DENY, X-Content-Type-Options: nosniff,
#         Referrer-Policy, Permissions-Policy, Strict-Transport-Security (if behind Nginx)

# 9. Abuse control
for i in $(seq 1 6); do
  curl -s -o /dev/null -w "%{http_code}" -X POST https://host/api/auth/login \
    -H 'Content-Type: application/json' \
    -d '{"email":"test@test.com","password":"wrongpassword"}'
  echo " attempt $i"
done
# VERIFY: Attempts 1-5 return 401, attempt 6 returns 429 with Retry-After header

# 10. Logout
curl -v -X POST https://host/api/auth/logout -b cookies.txt -c cookies.txt
# VERIFY: Cookie cleared (Max-Age=0 or empty)

# 11. Post-logout session check
curl -v https://host/api/auth/session -b cookies.txt
# VERIFY: 401
```

## 9. Rollout & Rollback Model

### 9.1 Rollout Sequence

| Step | Action | Verification |
|------|--------|-------------|
| R1 | Deploy Redis to production (or verify managed Redis access) | `redis-cli ping` returns `PONG` |
| R2 | Update Nginx config: HSTS + `X-Forwarded-Proto` + `X-Forwarded-For` | `curl -I` shows HSTS; `X-Forwarded-Proto` confirmed via app log |
| R3 | Provision `JWT_SECRET` (strong random value) and `RATE_LIMIT_REDIS_URL` in production env | Env var check |
| R4 | *Optional*: Deploy with `Content-Security-Policy-Report-Only` for 24-48h | Monitor browser console / report endpoint for violations |
| R5 | Deploy v6.9 application code | Full acceptance matrix (§8) |
| R6 | Switch CSP to enforcement mode (if R4 was used) | `curl -I` shows `Content-Security-Policy` (not `-Report-Only`) |
| R7 | Monitor for 24h | No unexpected 401s, no 429 false positives, no broken pages |

### 9.2 Rollback Procedures

| Scenario | Procedure | Impact |
|----------|-----------|--------|
| **Critical bug in v6.9 code** | Redeploy v6.8 code. Users re-login (v6.8 re-issues `solfacil_jwt` cookie + localStorage token). `__Host-solfacil_session` cookie persists in browsers but is ignored by v6.8. No data loss. | All active sessions lost; users must re-login |
| **v6.9 deployed but cookie issue (e.g., `Secure` flag problem behind proxy)** | **Emergency only**: set `LEGACY_BROWSER_BEARER=true` as last-resort rollback toggle. This is NOT part of the normal implementation or rollout path. Fix the proxy/cookie issue immediately. Disable within 48h. Remove env var after fix confirmed. | Users may need to re-login; bridge logs warnings per request |
| **CSP breaks a page** | Switch to `Content-Security-Policy-Report-Only` temporarily. Identify and fix the violating content. Re-enable enforcement. | No security degradation beyond CSP; other headers remain |
| **Redis outage in production** | Restart Redis. If Redis cannot be restored, production login is unavailable (fail-closed by design). Do NOT remove the Redis requirement. | Login blocked until Redis is restored; existing sessions continue to work |

### 9.3 Rollback Governance

| Bridge | Default | Max Lifetime | Who Disables |
|--------|---------|-------------|-------------|
| `LEGACY_BROWSER_BEARER` | `false` (not provisioned by default — only add to env if emergency demands it) | 48 hours max | Operator — must remove env var after fix confirmed; do not leave dormant |
| CSP `Report-Only` | Not used (enforcement is default) | 48 hours if used | Operator — switch to enforcement after violation review |
