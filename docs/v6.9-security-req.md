# Solfacil v6.9 Security Correction Release — REQ Draft

Date: 2026-04-15 UTC
Status: REQ draft / ready for DESIGN
Version baseline: Solfacil v6.8
Target release: v6.9

## 1. Background

Solfacil v6.8 has reached a point where several security issues are no longer hypothetical hygiene items; they are already observable in the current login/auth surface and surrounding delivery model.

The investigation already completed three necessary things:
1. verified the externally visible behavior from the production login surface;
2. traced the corresponding local repository code paths;
3. separated code-layer problems from deployment/configuration-layer problems.

This release is therefore not a generic "security improvement" release. It is a correction release.

The purpose of v6.9 is to remove known and fixable security design errors from the current system using the most correct engineering approach, not the easiest patch path.

## 2. Product decision already settled

The following decisions are already fixed and should be treated as REQ inputs rather than open questions:

1. v6.9 covers code-layer issues plus necessary configuration-layer closure.
2. Necessary compatibility changes are acceptable.
3. Acceptance standard:
   - after v6.9 release, all discovered and fixable code-layer security issues are closed using the correct engineering approach;
   - any residual risk must be explicitly documented;
   - no known insecure design may remain silently tolerated by default.

## 3. Problem statement

The current security posture has four structural weaknesses:

1. authentication state handling is architecturally inconsistent;
2. browser-facing security hardening is incomplete;
3. login abuse resistance is insufficiently enforced at the application layer;
4. secret/default handling still permits unsafe fallback behavior.

The deeper issue is not just the presence of isolated vulnerabilities. The deeper issue is that the current auth and browser trust model is internally split between older convenience patterns and newer safer patterns.

That split weakens the system in three ways:
- it increases attack surface;
- it makes security reasoning ambiguous;
- it encourages future partial fixes rather than coherent closure.

## 6. Verified findings that drive v6.9

### 6.1 Frontend auth model stores JWT in localStorage
Verified in repository code and matched against the production login page payload.

Current behavior includes:
- login success returns a token to frontend JavaScript;
- frontend stores the token in `localStorage`;
- subsequent API requests read the token from `localStorage` and send it via `Authorization: Bearer ...`.

Implication:
- any XSS-capable script execution can directly extract the bearer token;
- session security is over-coupled to "entire frontend must remain entirely XSS-clean forever".

REQ consequence:
- v6.9 must eliminate localStorage-based primary auth token handling from the production auth model.

### 6.2 Backend already sets auth cookie, but system is not truly cookie-first
Verified in backend login handler.

Current behavior includes both:
- server sets an HttpOnly auth cookie;
- server also returns the token in the JSON response body for JS consumption.

Implication:
- the system is currently in a half-migrated state;
- safer session transport exists, but insecure JS-readable token handling remains active.

REQ consequence:
- v6.9 must converge the auth model into one coherent primary pattern instead of preserving split behavior as a permanent state.

### 6.3 Application-layer login throttling is not evident
Observed from production behavior and repository inspection.

Current findings:
- repeated invalid login attempts returned ordinary auth failures;
- repository inspection did not find application-layer rate limiting / lockout controls for login.

Implication:
- the internet-facing login surface appears to lack sufficient application-native abuse resistance;
- external reverse-proxy or WAF protections, if any, are not an acceptable substitute for clear system-level design unless explicitly modeled and verified.

REQ consequence:
- v6.9 must provide explicit, testable login abuse mitigation.

### 6.4 Browser security headers are incomplete on production surface
Observed from production response headers.

Verified missing or not observed on the login page response:
- Content-Security-Policy
- X-Frame-Options and/or equivalent frame-ancestors policy
- X-Content-Type-Options
- Referrer-Policy
- Permissions-Policy

Implication:
- browser-side damage containment is weaker than required for an externally reachable admin/login surface.

REQ consequence:
- v6.9 must define and deploy a browser hardening baseline, including the necessary application and/or reverse-proxy configuration required to make it real.

### 6.5 JWT secret handling permits dangerous fallback
Verified in backend auth code.

Current behavior allows a default fallback when `JWT_SECRET` is missing.

Implication:
- a production misconfiguration can silently degrade into a predictable secret path;
- this is an unsafe production default.

REQ consequence:
- v6.9 must eliminate insecure auth-secret fallback behavior for production-capable runtime paths.

## 7. Root classification: what is code, what is configuration

### 7.1 Code-layer problems
The following are treated as code/design defects:
- localStorage as primary bearer-token storage;
- frontend auth flow depending on JS-readable token transport;
- half-migrated cookie + bearer split auth model;
- lack of explicit login abuse controls in application behavior;
- unsafe JWT secret fallback in runtime code.

### 7.2 Configuration-layer problems
The following are treated as deployment/configuration closure items:
- missing CSP and related browser hardening response headers;
- any reverse-proxy enforcement required to make browser policy effective;
- any environment/secret provisioning changes required to safely run the corrected auth model.

### 7.3 Coupled problems
Some items must be fixed across both code and configuration to count as solved:
- session model migration;
- secure cookie semantics;
- header deployment;
- secret provisioning and startup validation.

REQ rule:
- no item may be marked complete if only one side of a coupled issue is patched while the other side remains insecure.

## 8. v6.9 release objective

Solfacil v6.9 is a security correction release whose objective is:

- to remove all discovered and fixable code-layer security defects from the current system;
- to add whatever configuration-layer changes are necessary to close those defects in reality;
- to prefer the most correct and maintainable security model over the lowest-effort patch path.

In plain terms:
- v6.9 is not allowed to keep a known-wrong design merely because it is already wired into the current frontend or deployment.

## 9. Required outcomes

### 9.1 Auth model outcome
The system must converge onto a coherent production auth model with these properties:
- primary session/auth state is not exposed to arbitrary frontend JavaScript as a bearer secret;
- frontend authenticated access does not depend on `localStorage` token persistence for primary login state;
- logout, session expiry, and unauthorized recovery behavior remain operational and understandable;
- the resulting model is explicit enough that future engineers can reason about it without ambiguity.

### 9.2 Browser hardening outcome
The externally reachable web surface must expose a defined browser security baseline appropriate for an admin/login system, including:
- content injection containment policy;
- anti-framing policy;
- MIME sniffing prevention;
- referrer policy;
- other intentionally chosen response hardening headers rather than accidental defaults.

### 9.3 Abuse resistance outcome
The login surface must provide explicit, testable protection against repetitive abuse attempts, including:
- request throttling and/or lockout semantics;
- behavior that is observable and verifiable under repeated failure conditions;
- a design that does not rely solely on undocumented infrastructure assumptions.

### 9.4 Secret-safety outcome
Authentication secret handling must fail safely:
- production-capable runtime paths must not silently fall back to predictable development secrets;
- startup or deployment must clearly fail or reject unsafe auth-secret configuration;
- the operating model for secret provisioning must be explicit.

### 9.5 Regression-proofing outcome
The corrected security posture must be protected by tests and verification artifacts so the system does not drift back into the same class of mistakes.

## 10. Non-goals

The following are not v6.9 goals unless required by the chosen security design:
- broad feature redesign unrelated to security closure;
- cosmetic frontend refresh work;
- security theater changes that improve scanner output but do not materially reduce risk;
- arbitrary scope expansion into unrelated backend modules with no direct security value.

## 11. Compatibility stance

Necessary compatibility changes are allowed.

Interpretation:
- if the correct security model requires retiring current localStorage/bearer behavior, v6.9 may do so;
- compatibility should be preserved only where it does not preserve an insecure design;
- any temporary bridge mode must be explicitly justified, bounded, and removable.

REQ rule:
- "existing behavior" is not by itself a reason to preserve insecure auth behavior.

## 12. Design questions the next phase must answer

Before DESIGN is considered complete, it must also explicitly identify the design-document set required for v6.9.

Minimum required DESIGN outputs:

1. one v6.9 master security design document that states target auth/session architecture, migration model, verification model, and rollback model;
2. one frontend security design update derived from `docs/FRONTEND_ARCHITECTURE_v6.8.md` covering login page, authenticated request model, logout/session-expiry behavior, and browser-side implications;
3. one backend auth-boundary design update covering M5 + M6 + shared-layer interaction, including middleware, cookie semantics, token verification, SSE impact, and secret fail-fast behavior;
4. one deployment/security-boundary design update covering reverse proxy headers, runtime env/secret requirements, and live-surface verification expectations.

These may be delivered either as:
- one merged DESIGN with explicit sections mapped to the existing matrix documents; or
- a master DESIGN plus module-specific addenda.

But DESIGN is not allowed to remain vague about document ownership. It must explicitly say which architecture documents are updated for v6.9.

The DESIGN phase must answer at least the following:

1. What exact target auth model replaces the current split localStorage + cookie behavior?
2. How will frontend authenticated requests operate after migration?
3. What is the migration path from current clients/pages to the corrected model?
4. Which browser security headers are set where: app layer, reverse proxy, or both?
5. What exact login throttling/abuse-control design is chosen?
6. How is JWT secret validation enforced so unsafe fallback is impossible in production?
7. What test suite and deployment verification prove the closure is real?
8. What residual risks, if any, remain after v6.9 and why?

## 13. Acceptance criteria

v6.9 is accepted only if all of the following are true:

1. all discovered and fixable code-layer security issues identified in this REQ are closed;
2. necessary configuration-layer work has been applied where required to make the closure real;
3. the system no longer depends on localStorage-held bearer JWT as the primary production auth model;
4. browser security hardening headers are actually present on the live externally reachable surface as designed;
5. login abuse mitigation is implemented and demonstrably testable;
6. unsafe JWT secret fallback behavior is removed or made impossible in production-capable execution paths;
7. compatibility changes, if introduced, are intentional, documented, and verified;
8. any unresolved item is explicitly documented as residual risk with reason, impact, and next-step ownership.

## 14. Release verdict standard

This release is successful only if the following statement is true:

"After v6.9 release, Solfacil's discovered and fixable code-layer security issues have been closed using the correct engineering approach; any remaining risk is explicitly documented, and no known insecure design remains silently tolerated by default."
