# v6.10 WS10 — Runtime Retention Executor Runbook

Date: 2026-04-18
Scope: WS10 release hardening — retention / auto-close executor operator checks only.

This runbook is the focused WS10 artifact referenced in
`docs/v6.10-runtime-governance-plan.md §4.11`. It does NOT redefine M9 semantics
and does NOT extend v6.10 scope into dashboard polish or unrelated infra
cleanup. It only documents the operator-facing surface of the retention
executor and the release-check checklist the on-call operator needs.

---

## 1. What the executor does

The runtime retention executor lives in
`backend/src/shared/runtime/retention-job.ts`. Phase-1 responsibilities
(canonical TTLs from `docs/v6.10-runtime-governance-plan.md §4.11`):

| Action | Threshold | Source column |
|---|---|---|
| Auto-close recovered issues | `recovered_at < now - 24h` | `runtime_issues.recovered_at` |
| Auto-close stale detected/ongoing issues | `last_observed_at < now - 72h` | `runtime_issues.last_observed_at` |
| Delete raw events | `observed_at < now - 90 days` | `runtime_events.observed_at` |
| Delete health snapshots | `captured_at < now - 30 days` | `runtime_health_snapshots.captured_at` |

Auto-close writes go through the shared-layer mutation path
(`applyOperatorClose` → `upsertRuntimeIssue`) with:
- `operator_actor = "system.retention"`
- `operator_note = "auto-closed by runtime retention (...)"`

No ad-hoc UPDATE statements touch `runtime_issues`. Closed/suppressed rows are
never reopened by retention. `runtime_events` and `runtime_health_snapshots`
are deleted using bounded `DELETE ... WHERE <ts> < $1` statements.

## 2. Disabled-mode contract

When `RUNTIME_GOVERNANCE_ENABLED=false` (or unset):
- `runRuntimeRetention({ flags })` returns
  `{ status: "disabled", recoveredAutoClosed: 0, staleAutoClosed: 0,
    eventsDeleted: 0, snapshotsDeleted: 0, errors: [] }`.
- No DB connection is opened, no SQL runs.
- `local-server.ts` does not register the retention interval when the flag is
  off, so flagging the spine off is sufficient behavioral rollback.

## 3. Failure posture

All errors are caught per-phase inside the executor:
- fingerprint-scoped errors (auto-close of a single issue row) do not abort the
  batch; other rows continue being processed.
- Top-level errors produce
  `{ status: "degraded_fallback", errors: [ { phase, error, fingerprint? } ] }`
  and a `[runtime-retention:fallback] {...}` stderr/fallback-logger line.
- Retention NEVER throws into a business request path. The in-process
  scheduler (`scheduleRuntimeRetention()` in `backend/scripts/local-server.ts`)
  wraps the call in an extra `try/catch` as defense in depth.

## 4. Local operator smoke (pre-release)

1. `RUNTIME_GOVERNANCE_ENABLED=false npm run runtime:retention`
   — must print `{ "status": "disabled", ... }` and exit 0 without opening pg
   connections.
2. `RUNTIME_GOVERNANCE_ENABLED=true npm run runtime:retention`
   — must print a JSON envelope with counters. Non-zero exit is expected only
   when at least one phase reports `degraded_fallback`.
3. From a running local-server, confirm the retention tick fires at least once
   after `RUNTIME_RETENTION_INTERVAL_MS` (default 1h; override for smoke with
   a smaller value, e.g. `RUNTIME_RETENTION_INTERVAL_MS=60000`).
4. Verify `GET /api/runtime/issues` no longer lists retention-eligible
   recovered / stale rows after one tick.
5. Verify no business endpoint regresses under the retention interval — run
   the standard v6.9 smoke (auth, fleet, P5, P6) with retention enabled.

## 5. Rollout / rollback

### 5.1 Rollout

1. Apply migration `002_runtime_governance.sql` (WS1 gate). Retention writes
   nothing until the governance flag is on, so rollout is additive.
2. Deploy code with `RUNTIME_GOVERNANCE_ENABLED=false`. The scheduler in
   `local-server.ts` will not register.
3. Flip `RUNTIME_GOVERNANCE_ENABLED=true` per the canonical per-slice order
   (see plan §6.1). Retention will activate on the next process boot.
4. Watch `[runtime-retention:fallback]` stderr lines — they are the in-band
   operator signal if persistence degrades.

### 5.2 Rollback

1. Flip `RUNTIME_GOVERNANCE_ENABLED` back to `false`.
2. No schema changes, no data loss. Already-closed rows stay closed — retention
   does not reopen anything.
3. If a specific run misbehaves, `npm run runtime:retention` is safe to stop
   (SIGINT) — there are no multi-statement transactions; the worst-case partial
   state is a subset of eligible rows processed, which the next tick will
   catch up on.

## 7. Residual gaps intentionally deferred

- Deep partition-detach automation for `runtime_events` is deferred beyond
  v6.10 per plan §7.3. Phase-1 uses a bounded `DELETE ... WHERE observed_at`,
  which is correct for phase-1 volumes.
- Snapshot compaction / aggregation prior to deletion is NOT in WS10 scope.
- Retention metrics (Prometheus, structured logs at INFO) are NOT in WS10
  scope — fallback stderr lines are the only phase-1 operator signal.
