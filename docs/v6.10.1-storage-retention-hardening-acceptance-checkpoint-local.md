## 4. Rollout posture from current local evidence

Current local evidence supports the intended rollout order and posture:

1. **Schema-first foundation is present before archive behavior**
   - `backend/src/shared/migrations/003_storage_retention_hardening.sql` is additive only.
   - It creates `device_command_logs_archive` and `gateway_alarm_events_archive` plus helper retention indexes with `IF NOT EXISTS`.
   - Local contract verification confirms those objects exist in the current DB.

2. **Single shared executor path is present**
   - `backend/src/shared/runtime/retention-job.ts` remains the sole shared executor.
   - Archive phases use `INSERT ... ON CONFLICT (id) DO NOTHING` and delete hot rows only when the archive row exists.
   - Per-phase failure capture is implemented and verified by test.

3. **Scheduled and ad-hoc paths both point at the same executor**
   - `backend/scripts/local-server.ts` schedules `runRuntimeRetention(...)` with default `RUNTIME_RETENTION_INTERVAL_MS = 60 * 60 * 1000`.
   - `backend/scripts/run-runtime-retention.ts` invokes the same `runRuntimeRetention(...)` codepath and emits the operator JSON envelope.
   - `backend/package.json` exposes `runtime:retention` for the ad-hoc lane.

4. **Compose/runtime policy is rollout-ready locally**
   - `docker compose config` and `docker inspect` both showed the committed `json-file`/`10m`/`5` logging policy on all four Solfacil containers.

5. **Current live behavioral note**
   - `npm run runtime:retention` returned a disabled-mode report with zero counters and exit `0`.
   - This is acceptable local evidence for the operator path and JSON envelope, but it is not a proof of live destructive retention execution because governance was not enabled in this run.

Conclusion: local rollout posture is **plausible and correctly ordered** from current code/config/runtime evidence.

## 5. Rollback posture from current local evidence

Current local evidence supports the accepted additive/behavioral rollback posture:

1. **Rollback does not require schema destruction**
   - Kickoff guardrail says rollback is: stop/disable extended executor, redeploy prior code if needed, leave additive schema and archived data in place.
   - `003_storage_retention_hardening.sql` is additive and contains no destructive rollback coupling.

2. **Executor can be behaviorally disabled**
   - `runRuntimeRetention(...)` returns a disabled-mode result when governance is off.
   - `npm run runtime:retention` locally demonstrated that disabled posture cleanly exits `0` with zero side effects in the current environment.
   - `local-server.ts` only schedules retention when governance is enabled.

3. **Archive-first phases are rerun-safe rather than rollback-fragile**
   - Archive tables preserve source identity as PK.
   - Executor uses `ON CONFLICT DO NOTHING` and only deletes hot rows when archive rows exist.
   - `runtime-retention-storage.test.ts` passed duplicate-safe rerun proof cases for both archive surfaces.

4. **Failure posture is observable rather than silent**
   - Executor failure is wired to `storage.retention.executor.failed` and per-phase failure capture is covered by `runtime-retention.test.ts`.
   - This makes “stop/disable and revert code” behaviorally plausible without needing schema rollback.

Conclusion: rollback posture is **plausible, additive, and consistent with accepted constraints**.
