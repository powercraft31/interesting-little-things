## 3. Auth Contracts — Fixed Design

### 3.1 Two Distinct Auth Contracts

v6.9 defines two mutually exclusive auth contracts. The SPA must not use the machine contract.

| Property | Browser Contract | Machine Contract |
|----------|-----------------|------------------|
| **Credential transport** | `__Host-solfacil_session` cookie | `Authorization: Bearer <token>` header |
| **Who uses it** | SPA (`index.html`, `login.html`), browser `EventSource` | CLI tools, service-to-service, scripts, future external API consumers |
| **Cookie attributes** | `HttpOnly; Secure; SameSite=Strict; Path=/` | N/A — no cookie |
| **Token in JSON body** | **NOT returned** to browser login response | Returned in machine-login response |
| **Session bootstrap** | `GET /api/auth/session` | N/A — stateless bearer |
| **401 recovery** | Redirect to `login.html` | Caller re-authenticates |
| **Default state** | **Primary**. All browser traffic uses this | **Available but separate**. Not used by SPA |

### 3.2 Browser / Machine Contract Discrimination

The auth middleware distinguishes browser from machine requests by checking for the presence of the session cookie versus the `Authorization` header:

| Condition | Contract Applied |
|-----------|-----------------|
| `__Host-solfacil_session` cookie present, no `Authorization` header | Browser contract |
| `Authorization: Bearer <token>` header present, no session cookie | Machine contract |
| Both present | Cookie takes precedence; `Authorization` header is ignored |
| Neither present | 401 Unauthorized |

### 3.3 Cookie Baseline — Fixed

| Attribute | Value | Rationale |
|-----------|-------|-----------|
| **Name** | `__Host-solfacil_session` | `__Host-` prefix enforces Secure + no Domain + Path=/ at the browser level |
| **HttpOnly** | `true` | Not readable by JavaScript |
| **Secure** | `true` | Transmitted only over HTTPS (enforced by `__Host-` prefix) |
| **SameSite** | `Strict` | No cross-site request attachment |
| **Path** | `/` | Required by `__Host-` prefix |
| **Domain** | _(none)_ | Required by `__Host-` prefix — bound to exact origin |
| **Max-Age** | `86400` (24h) | Matches current JWT expiry |

**Dev-mode exception**: Local development (`NODE_ENV=development`) over plain HTTP cannot use `__Host-` prefix (requires Secure). Dev mode uses cookie name `solfacil_session` without `Secure` flag. This is acceptable because dev mode is localhost-only.

### 3.4 Session Endpoint — Fixed Contract

| Property | Value |
|----------|-------|
| **Route** | `GET /api/auth/session` |
| **Auth** | Cookie only (bearer not accepted on this endpoint) |
| **Purpose** | Browser session bootstrap and truth check |
| **Success response** | `{ success: true, data: { userId, orgId, role, name, email } }` |
| **Failure response** | `401` — cookie missing, expired, or invalid |
| **When called** | (1) `login.html` page load — determines if already authenticated; (2) `index.html` page load — bootstraps session state; (3) any 401 recovery flow |

This endpoint replaces `localStorage.getItem("solfacil_jwt")` as the session truth source for browser clients.

## 4. Browser Session Model

### 4.1 Login Flow (v6.9)

```
Browser                          BFF
  │                               │
  ├── POST /api/auth/login ──────►│  { email, password }
  │                               │  1. Check abuse-control (§5)
  │                               │  2. Validate credentials (bcrypt)
  │                               │  3. Sign JWT
  │                               │  4. Set-Cookie: __Host-solfacil_session=<jwt>; ...
  │◄── 200 { success, data: {    │
  │      user: { userId, orgId,  │
  │      role, name, email } } } │
  │    (NO token field in body)   │
  │                               │
  ├── Redirect to index.html      │
  │                               │
  ├── GET /api/auth/session ─────►│  (cookie attached automatically)
  │◄── 200 { user state }        │  Frontend stores user info in JS memory
  │                               │
  ├── GET /api/fleet/overview ───►│  (cookie attached automatically)
  │◄── 200 { data }              │
```

**Key change from v6.8**: The login response no longer includes a `token` field in the JSON body for browser clients. The credential is transported exclusively via `Set-Cookie`.

### 4.2 Authenticated Request Model (v6.9)

All browser requests (fetch, XHR, EventSource) rely on the cookie. No `Authorization` header is set by frontend code for browser-originated requests.

| Request type | v6.8 behavior | v6.9 behavior |
|-------------|---------------|---------------|
| `fetch('/api/...')` | `Authorization: Bearer <localStorage token>` | Cookie auto-attached; `credentials: 'same-origin'` (default for same-origin) |
| `new EventSource('/api/events')` | Cookie (`solfacil_jwt`) _(historical)_ | Cookie (`__Host-solfacil_session`) |
| Logout | `POST /api/auth/logout` + clear localStorage | `POST /api/auth/logout` (clears cookie server-side) |

### 4.3 Session Expiry and 401 Handling

| Scenario | Behavior |
|----------|----------|
| Cookie expires (24h) | Next API call returns 401 → frontend redirects to `login.html` |
| Server rejects cookie (invalid signature, tampered) | 401 → redirect to `login.html` |
| `GET /api/auth/session` returns 401 on page load | Redirect to `login.html` immediately |
| SSE connection gets 401 | `EventSource` `onerror` fires → frontend checks `GET /api/auth/session` → redirect if 401 |

### 4.4 Logout

1. Frontend calls `POST /api/auth/logout`
2. BFF clears `__Host-solfacil_session` cookie (`Set-Cookie` with `Max-Age=0`)
3. Frontend redirects to `login.html`
4. No localStorage cleanup needed (localStorage is no longer used for auth)

## 5. Abuse-Control Topology

### 5.1 Fixed Design Decisions

| Decision | Value |
|----------|-------|
| **Scope** | `POST /api/auth/login` only (v6.9 scope) |
| **Primary production backing store** | Shared ephemeral state: Redis (or equivalent key-value store with TTL support) |
| **Keying** | Composite: `login_ip:<ip>` + `login_email:<normalized_email>` (both tracked independently) |
| **Per-IP limit** | 10 failed attempts per 15-minute sliding window |
| **Per-email limit** | 5 failed attempts per 15-minute sliding window |
| **Lockout behavior** | `429 Too Many Requests` with `Retry-After` header. Response body: `{ success: false, error: "Too many login attempts. Try again later." }` |
| **Successful login** | Resets the counter for that email. IP counter is not reset |
| **Release-critical** | Yes — must ship with v6.9 |

### 5.2 Topology Assumptions

| Environment | Backing Store | Notes |
|-------------|---------------|-------|
| **Production** | Redis (or equivalent) | Shared across all BFF process instances. Connection string via `RATE_LIMIT_REDIS_URL` env var |
| **Development / single-process** | In-memory Map with TTL | Acceptable for local dev only (`NODE_ENV=development`). Must NOT be the production code path and must NOT be a production fallback |
| **Production: Redis unavailable at startup** | **Fail closed** | If `RATE_LIMIT_REDIS_URL` is configured (required in production) and the connection fails, the process must refuse to enable the login route (or refuse to start). Login abuse control is release-critical and must not silently degrade to unprotected in production |

### 5.3 Bounded Open Points

| Open Point | Constraint |
|------------|------------|
| Exact Redis client library | Must support TTL-based key expiry. `ioredis` is the expected choice but not fixed |
| CAPTCHA as secondary gate | Out of v6.9 scope |
| Account lockout (disable `is_active`) vs. time-based throttle only | v6.9 uses time-based throttle only. Account lockout is Phase 2 |

## 6. Migration / Mixed-Version / Rollback

### 6.1 Migration Model

The transition from v6.8 (localStorage-bearer + `solfacil_jwt` cookie) to v6.9 (cookie-first `__Host-solfacil_session`) is a **hard cutover** for the browser contract. Users must re-login after v6.9 deployment.

| Aspect | Design |
|--------|--------|
| **Re-login required** | Yes. The old `solfacil_jwt` cookie and localStorage token are not recognized by v6.9 |
| **Legacy browser bearer bridge** | **Available but default OFF**. Env var `LEGACY_BROWSER_BEARER=true` enables temporary acceptance of `Authorization: Bearer` from browser origins. Purpose: emergency rollback aid only |
| **Dual-cookie bridge** | **Not implemented**. v6.9 does not read the old `solfacil_jwt` cookie. Clean break |
| **Machine bearer** | Unaffected. Machine clients using `Authorization: Bearer` continue to work (they never used the cookie) |

### 6.2 Rollback Model

| Scenario | Procedure |
|----------|-----------|
| Rollback to v6.8 code | Deploy v6.8. Users re-login (v6.8 issues `solfacil_jwt` cookie + localStorage token). No data loss. The `__Host-solfacil_session` cookie persists in browsers but is ignored by v6.8 |
| v6.9 deployed but needs bearer bridge temporarily | Set `LEGACY_BROWSER_BEARER=true`. Auth middleware accepts bearer tokens from browser requests as fallback. **Bounded lifetime: must be disabled within 48 hours** |

### 6.3 Legacy Bridge Governance

| Bridge | Default | Purpose | Lifetime Bound |
|--------|---------|---------|----------------|
| `LEGACY_BROWSER_BEARER` | `false` | Emergency: allow browser bearer tokens during rollback window | Max 48h. Must be actively disabled |
| Dual-cookie (`solfacil_jwt` read) | Not implemented | — | — |

## 7. Secret Governance

### 7.1 JWT Secret — Fixed Design

| Aspect | v6.8 Behavior | v6.9 Design |
|--------|---------------|-------------|
| Secret source | `process.env.JWT_SECRET` | `process.env.JWT_SECRET` |
| Dev fallback | `"solfacil-dev-secret"` silently used when env var missing | **Removed from production-capable code path** |
| Missing secret at startup | Silent fallback to dev secret | **Process exits with error** (fail-fast) |
| NODE_ENV gate | None | Non-development: missing or weak `JWT_SECRET` is a fatal startup error |

### 7.2 Startup Validation Contract

At process startup (`local-server.ts` initialization), before any route is registered:

```
IF NODE_ENV !== 'development':
  IF JWT_SECRET is not set
     OR JWT_SECRET is empty
     OR JWT_SECRET === 'solfacil-dev-secret':
    LOG FATAL: "JWT_SECRET must be set to a strong secret in production"
    process.exit(1)
```

In development mode (`NODE_ENV=development`):
- Missing `JWT_SECRET` → **preferred**: developer provisions an explicit local secret via `.env` or shell export
- **Legacy transitional behavior**: if no `JWT_SECRET` is set, a hardcoded dev fallback is used with a loud console warning. This is legacy debt from v6.8 retained only to avoid blocking local onboarding. It must not be treated as the standard local-dev mechanism and should be eliminated in a future release
- This is the **only** code path where any fallback is tolerated, and it must never execute outside `NODE_ENV=development`

### 7.3 Secret Rotation Model

| Aspect | v6.9 Design |
|--------|-------------|
| Rotation mechanism | Restart with new `JWT_SECRET` env var |
| Impact | All existing JWTs (cookies) are invalidated; all users must re-login |
| Dual-secret support | Out of v6.9 scope. Single secret only |

## 8. Browser Hardening Baseline

### 8.1 Required Headers — Fixed

| Header | Value | Owner |
|--------|-------|-------|
| `Content-Security-Policy` | See §8.2 | App-layer (Express middleware) |
| `X-Frame-Options` | `DENY` | App-layer |
| `X-Content-Type-Options` | `nosniff` | App-layer |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | App-layer |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=(), payment=()` | App-layer |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | Ingress/reverse-proxy only |

**Ownership rule**: All headers except HSTS are set by the Express app layer so they are effective in all deployment topologies (local dev, Docker, production behind proxy). HSTS is set at ingress because it requires the ingress to guarantee HTTPS termination.

### 8.2 CSP Baseline — First Release

The first-release CSP assumes self-hosted fonts and charts. Third-party CDN origins are not in the baseline policy.

```
Content-Security-Policy:
  default-src 'none';
  script-src 'self';
  style-src 'self';
  img-src 'self' data:;
  font-src 'self';
  connect-src 'self';
  frame-ancestors 'none';
  base-uri 'self';
  form-action 'self'
```

**Blockers that must be resolved before this CSP is enforceable:**

| Blocker | Current v6.8 State | Required Resolution |
|---------|---------------------|---------------------|
| ECharts loaded from CDN | `<script src="https://cdn.jsdelivr.net/...">` in `index.html` | Self-host: download `echarts.min.js`, serve from `/js/vendor/` |
| Google Fonts (Inter, JetBrains Mono) | `<link>` to `fonts.googleapis.com` + `fonts.gstatic.com` | Self-host: download font files, serve from `/fonts/`, update CSS `@font-face` |
| Inline `onclick` in `login.html` | `<button onclick="...">` | Refactor to `addEventListener` in external JS file |
| Inline `<script>` in `login.html` | Login logic embedded in `<script>` tag | Extract to `login.js`, load via `<script src="login.js">` |

These blockers are detailed in the frontend security design.

### 8.3 CSP Escalation Path

If a specific blocker cannot be resolved before release, a per-script `nonce` or `sha256` hash exception is acceptable **only** for that specific script. Broad `'unsafe-inline'` is not acceptable as a baseline.

## 9. Boundary Ownership Summary

| Responsibility | Owner | Notes |
|----------------|-------|-------|
| Cookie attributes (`__Host-solfacil_session`) | App (Express) | Set in login handler; app controls all cookie semantics |
| Security response headers (CSP, X-Frame-Options, nosniff, etc.) | App (Express middleware) | Effective regardless of deployment topology |
| HSTS | Ingress (Nginx / ALB) | Requires HTTPS termination guarantee |
| TLS termination | Ingress | App runs HTTP behind proxy |
| `X-Forwarded-For` / `X-Real-IP` trust | Ingress config + Express `trust proxy` | Required for IP-based abuse control |
| Login abuse control | App (Express middleware) | Redis-backed; app-owned logic |
| Static asset caching | Ingress or app | Cache-busting via `?v=` already in place |

## 10. Residual Risks After v6.9

| Risk | Severity | Reason Accepted | Mitigation Path |
|------|----------|-----------------|-----------------|
| No refresh token — 24h hard session | Low | Phase 1 scope; admin-only system with small user base | Phase 2: refresh token rotation |
| No MFA | Medium | Phase 1 scope; login abuse control reduces brute-force risk | Phase 2: TOTP or WebAuthn |
| Single JWT secret (no rotation without downtime) | Low | Small deployment; secret rotation = restart + re-login | Phase 2: dual-secret rotation |
| No CSRF token | Low | `SameSite=Strict` + `form-action 'self'` in CSP mitigates CSRF | Re-evaluate if SameSite is ever relaxed |
| Webhook endpoints unauthenticated | Medium | Existing M7 design; out of v6.9 scope | Future: HMAC or API-key validation |
| `__Host-` prefix requires HTTPS | N/A in prod | Production is HTTPS. Local dev uses non-`__Host-` variant | Documented dev-mode exception (§3.3) |
