# Solfacil v6.9 Security Design

Date: 2026-04-15 UTC
Status: DESIGN revision for v6.9 security correction release
Baseline: v6.8
Source REQ: `docs/v6.9-security-req.md`
Related review baseline: `docs/AUTH-BOUNDARY-REVIEW-2026-04-10.md`

## 1. Purpose

This document is the master DESIGN for the v6.9 security correction release.

It defines, at design level:
- the supported auth contracts;
- browser session transport and CSRF posture;
- frontend/backend/deployment boundary allocation;
- login abuse-control topology;
- migration, mixed-version, and rollback behavior;
- fail-fast secret governance;
- first-release browser hardening posture;
- verification and residual-risk expectations.

This is a convergence pass, not a redesign. The core direction remains unchanged:
- cookie-first browser auth;
- no production browser dependence on `localStorage` JWT persistence;
- unified browser auth path for API and SSE;
- fail-fast secret handling;
- explicit matrix ownership.

## 2. Design outcome in one statement

v6.9 converges Solfacil Web onto a same-origin, cookie-first browser session model while explicitly separating any non-browser bearer usage into a distinct machine contract, removing browser `localStorage` bearer dependence, unifying browser API and SSE auth, making application-layer login abuse controls release-critical, requiring fail-fast secret validation, and assigning browser hardening responsibilities across application and deployment boundaries with explicit migration and rollback rules.

## 5. Decision summary

### 5.1 Fixed decisions

1. v6.9 defines two auth contracts, not one vague hybrid:
   - browser admin contract: same-origin, cookie-first;
   - machine/non-browser contract: separate bearer contract, documented separately from the browser contract.
2. The shipped browser admin surface must not depend on `localStorage`, `sessionStorage`, or JS memory for production JWT persistence.
3. Browser API requests and browser SSE use the same cookie-backed auth path.
4. Legacy browser bearer compatibility, if retained at all, is an explicit temporary compatibility mode and disabled by default.
5. Application-layer login abuse controls are release-critical. Edge rate limiting is reinforcement, not the primary acceptance criterion.
6. Production-capable auth entrypoints fail fast when auth secret inputs are missing or unsafe.
7. Production public browser hardening is owned by the public ingress boundary, while cookie correctness, cache semantics, and auth rejection behavior remain application-owned.
8. CSP is mandatory in v6.9, not deferred.

### 5.2 Fixed defaults accepted for v6.9

1. Cookie name adopted for v6.9: `__Host-solfacil_session`, with a short explicit dual-cookie migration bridge allowed during rollout.
2. Browser cookie `SameSite` for v6.9: `Strict`.
3. Browser bootstrap/session truth source for v6.9: dedicated `GET /api/auth/session` endpoint returning minimal display-safe session metadata.
4. Production auth-abuse state store for v6.9: shared ephemeral store such as Redis or equivalent so counters are consistent across multiple BFF instances.
5. First-release CSP target for v6.9:
   - no `unsafe-inline` in `script-src`;
   - remove inline login script and inline `onclick` before production rollout, or use tightly bounded nonce/hash exceptions only if cleanup misses the release train;
   - self-host Google Fonts and ECharts before rollout so first-release policy does not depend on third-party font/chart origins;
   - temporary `style-src 'unsafe-inline'` is allowed only while `frontend-v2/login.html` still contains inline `<style>`.

### 5.3 Remaining review points after default settlement

1. Exact threshold values for throttle/backoff.
2. Whether active SSE disconnect logic is needed beyond frontend stream closure.

## 6. Supported auth contracts

### 6.1 Browser admin contract

Fixed decision:
- this is the production browser contract for the Solfacil admin UI.

Contract rules:
- same-origin browser surface;
- auth credential transported in an HttpOnly Secure cookie;
- cookie automatically accompanies same-origin `fetch`/XHR and `EventSource`;
- login response may return minimal display-safe metadata, but not the JWT string;
- frontend auth state is derived from authenticated server response, not token presence in browser storage;
- browser requests do not rely on `Authorization: Bearer` from frontend JS.

### 6.2 Machine / non-browser contract

Fixed decision:
- v6.9 continues to allow a separate bearer-based machine contract for non-browser callers only.

Contract rules:
- bearer tokens are for non-browser clients that can intentionally set `Authorization` headers;
- this contract is not the browser admin contract and must not be documented as such;
- the Solfacil SPA may not use it;
- browser compatibility mode cannot be smuggled back in under the label “machine client”; browsers remain cookie-first;
- route/operator scoping for machine use remains a review/implementation detail, but the design split is now explicit.

Reason for choosing two contracts instead of browser-only scope:
- Solfacil already has backend and integration surfaces where intentional bearer usage is a plausible non-browser pattern;
- the real design problem was contract ambiguity, not the existence of all bearer usage;
- separating the contracts avoids reintroducing browser bearer semantics while preserving a bounded path for legitimate non-browser callers.

## 7. Cookie and CSRF posture

### 7.1 Cookie contract

Fixed v6.9 cookie contract:

| Property | Design stance |
|---|---|
| Name | Fixed: `__Host-solfacil_session`; a short dual-cookie migration bridge may temporarily coexist during rollout |
| HttpOnly | Fixed: required |
| Secure | Fixed: required on production-capable paths |
| SameSite | Fixed: `Strict` |
| Path | Fixed target: `/` |
| Domain | Fixed target: not set |
| Lifetime | Fixed: bounded and aligned to JWT expiry |
| Cache interaction | Fixed: login/logout/session bootstrap responses are non-cacheable |

### 7.2 CSRF posture

Fixed decision:
- v6.9 uses a same-origin-first CSRF posture and does not require a full token framework in this release.

Required controls:
- same-origin admin surface by default;
- auth cookie with `SameSite=Strict`;
- `Origin` validation on state-changing requests;
- `Referer` may be used as secondary evidence where topology requires it;
- no cross-site browser writes supported by default;
- no third-party embedding contract.

## 8. Frontend and backend session model

### 8.1 Browser login and bootstrap

Target browser flow:
1. Browser submits credentials to same-origin `POST /api/auth/login`.
2. Backend authenticates and issues session cookie.
3. Backend returns display-safe session metadata only.
4. Browser stores no auth token in web storage.
5. Login page or SPA bootstrap confirms session via dedicated `GET /api/auth/session` server-authenticated state.

### 8.2 Authenticated browser requests

Fixed decision:
- browser API and SSE are authenticated through the same cookie-backed session path.

Implications:
- `fetch('/api/...', { credentials: 'same-origin' })` or equivalent same-origin default behavior;
- no browser-side `Authorization` injection from token storage;
- `EventSource('/api/events')` relies on the same cookie;
- `401` becomes the main browser-visible session-loss signal.

## 9. Migration, mixed-version, and rollback matrix

### 9.1 Migration principle

Migration is from split auth truth to single browser auth truth:
- old state: browser bearer in `localStorage` for API plus cookie for SSE;
- new state: cookie-backed browser auth for API and SSE.

Compatibility bridges are allowed only as explicit temporary controls.

### 9.2 Mixed-version and rollback matrix

| Scenario | Expected behavior | Design rule / operator action |
|---|---|---|
| Old frontend -> new backend, compatibility mode OFF | Old frontend login or API calls fail because backend no longer returns browser JWT body / no longer accepts browser bearer path | Allowed but not target. Preferred rollout is coordinated FE+BE cutover. If this scenario must be tolerated briefly, compatibility mode must be explicitly enabled. |
| Old frontend -> new backend, compatibility mode ON | Old frontend can continue briefly using returned/accepted bearer semantics while new cookie is also issued | Temporary rollback/migration bridge only. Disabled by default. Sunset required. |
| New frontend -> old backend | New frontend may still succeed if old backend issues cookie and session probe can validate server auth, but risks remain because old backend still exports browser JWT and may expect bearer usage in some flows | Not a supported steady-state pairing. Deploy frontend and backend together. Avoid prolonged skew. |
| Existing browser session at cutover, cookie name unchanged (`solfacil_jwt`) | Existing cookie may continue until expiry; new frontend stops using `localStorage`; stale storage token is ignored | Lowest-risk operational cutover if immediate cookie-name migration is deferred. |
| Existing browser session at cutover, cookie name changed to `__Host-solfacil_session` without dual-issue bridge | Existing sessions do not carry the new cookie name and are forced to re-authenticate | Acceptable if planned. Must be called out in release notes. |
| Existing browser session at cutover, cookie name changed with temporary dual-issue/dual-read bridge | Existing sessions survive through migration window; both cookie names accepted/read for bounded period | Allowed as migration tool if operators want lower forced-login impact. Must have explicit expiry and removal plan. |
| Stale `localStorage` token remains after rollout | New frontend ignores it; logout does not rely on clearing it for auth correctness; optional cleanup may delete known legacy keys for hygiene | Fixed rule: stale storage token must not restore auth. |
| New frontend logout after rollout | Cookie cleared server-side; frontend clears only UI-local state; SSE closed client-side | Fixed behavior. |
| Rollback from cookie-first frontend/backend to legacy backend with compatibility mode OFF | New frontend cannot fall back to browser bearer; login/session flow breaks | Therefore rollback requires either coordinated frontend rollback or explicit backend compatibility mode. |
| Rollback using backend compatibility mode ON | Backend temporarily re-accepts legacy browser bearer semantics while operators recover | Allowed emergency path only. Disabled by default, time-bounded, and treated as residual risk. |

### 9.3 Release stance derived from the matrix

Fixed decision:
- PLAN must assume coordinated frontend/backend rollout.

Fixed migration default:
- adopt `__Host-solfacil_session` in v6.9;
- use a short explicit dual-cookie bridge during rollout to reduce avoidable session disruption;
- remove the bridge after the bounded migration window.

## 10. Login abuse-control topology

### 10.1 Release-critical layer

Fixed decision:
- the release-critical abuse control layer is the application layer, not the proxy.

Required application behavior:
- source-IP throttling;
- identifier-based progressive backoff independent of account existence;
- generic external errors;
- audit logging for success, failure, throttle, and backoff events.

### 10.2 State topology

Fixed v6.9 default:
- production uses a shared ephemeral store so counters are consistent across multiple BFF instances;
- local/direct-BFF development may use in-process memory;
- DB-backed counters are a fallback contingency only if deployment constraints block the preferred shared ephemeral store.

Logical ownership of counters:
- application owns the counter semantics;
- counters are keyed by normalized login identifier and derived client IP;
- counter storage must be shared across all production auth-serving instances if production is multi-instance.

### 10.3 Source-IP trust model

Fixed decision:
- client IP is trusted only when derived through a configured trusted-proxy chain.

Rules:
- on the public production boundary, the app may derive client IP from forwarded headers only when the immediate proxy/ingress is trusted;
- `X-Forwarded-For` must not be accepted from arbitrary direct clients;
- for direct-BFF/local use without trusted proxy, the app uses the socket/connection peer address;
- if trusted-proxy configuration is absent or ambiguous, the app must fail closed to a conservative single-hop interpretation rather than trust user-supplied headers.

### 10.4 Edge reinforcement stance

Fixed decision:
- edge rate limiting is recommended reinforcement for public production ingress, but application-layer abuse controls are the release gate.

This wording is intentional:
- public ingress should still absorb obvious burst traffic where available;
- however, PLAN and release acceptance may not assume the edge alone solves login abuse.

## 11. Browser hardening and CSP stance

### 11.1 Current frontend realities acknowledged by design

The current frontend structure includes:
- `frontend-v2/login.html` inline `<script>`;
- `frontend-v2/login.html` inline `<style>`;
- `frontend-v2/index.html` inline `onclick` on the hamburger button;
- Google Fonts are currently loaded externally;
- ECharts is currently loaded from `https://cdn.jsdelivr.net`.

### 11.2 First-release CSP stance

Fixed decision:
- v6.9 ships with CSP enabled on the public admin surface.

Fixed first-release implementation stance:
- remove or externalize the inline login script and inline `onclick` before rollout so first-release `script-src` does not require `'unsafe-inline'`;
- if that cleanup is not completed, use nonce/hash-based allowances rather than broad permanent `'unsafe-inline'` in `script-src`;
- temporary `style-src 'unsafe-inline'` is acceptable only as a documented transition while login page styles remain inline.

Concrete first-release target policy shape:
- `default-src 'self'`;
- `script-src 'self'` plus nonce/hash entries only if rollout still contains known inline script remnants;
- `style-src 'self' 'unsafe-inline'` for the first cut if inline login styles remain;
- `font-src 'self'`;
- `img-src 'self' data:`;
- `connect-src 'self'`;
- `frame-ancestors 'none'`;
- `object-src 'none'`;
- `base-uri 'self'`.

Longer-term tightening direction:
- remove inline login styles;
- eliminate all nonce/hash exceptions if full externalization is completed.

## 12. Runtime secret validation and fail-fast rules

### 12.1 Fixed rules

1. `JWT_SECRET` is mandatory for production-capable auth entrypoints.
2. Missing or unsafe secret causes startup failure before serving requests.
3. Known placeholder/development values are rejected.
4. Signing and verification use the same validated secret source.
5. Verification code contains no silent fallback constant.

### 12.2 Development stance

Fixed decision:
- local development must still use explicit secret injection or explicit dev bootstrap outside production-capable runtime behavior.

No production-capable path may silently invent its own auth secret.

## 13. Verification strategy

| Area | Proof required |
|---|---|
| Browser contract split | frontend uses cookie-only; any non-browser bearer path is documented separately and not used by SPA |
| Frontend auth storage | no production auth token written to `localStorage` or `sessionStorage` |
| Mixed-version behavior | cutover and rollback scenarios behave as declared in the matrix |
| Cookie behavior | cookie flags, name choice, and logout clearing semantics match chosen design |
| SSE | same cookie authenticates `/api/events`; reconnect fails after logout/expiry |
| Abuse control | configured throttle/backoff triggers with correct IP/identifier behavior across the expected topology |
| Source-IP derivation | trusted proxy vs direct-BFF behavior is testable and documented |
| Headers/CSP | live surface exposes required header baseline with first-release CSP policy |
| Secret fail-fast | startup refusal occurs on missing/unsafe secret |

## 14. Residual risks and explicit review points

### 14.1 Residual risks accepted in v6.9

1. Stateless JWT remains weaker than server-side sessions for immediate revocation and privilege-change invalidation.
2. If first-release CSP still needs temporary allowances for inline remnants or inline styles, those allowances remain a bounded residual risk until cleanup is complete.
3. SSE remains authenticated at connect time, so already-open streams are not guaranteed immediate revocation without future server-side session or active disconnect machinery.

### 14.2 Remaining explicit review items after default settlement

1. Exact threshold values for throttle/backoff.
2. Whether active SSE disconnect logic is needed beyond frontend stream closure.

## 15. Final design verdict

v6.9 is now defined as a cookie-first browser correction release with an explicit separate machine contract, a concrete migration/rollback model, an application-owned abuse-control topology, a concrete first-release CSP posture, and a non-ambiguous deployment ownership split. Remaining review items are now visible as review points rather than hidden ambiguities.
