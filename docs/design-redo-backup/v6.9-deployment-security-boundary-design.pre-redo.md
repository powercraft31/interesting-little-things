# Solfacil v6.9 Deployment Security Boundary Design

Date: 2026-04-15 UTC
Status: DESIGN revision addendum for v6.9
Primary source: `docs/v6.9-security-req.md`
Conceptually updates: deployment sections of `design/backend_architecture/00_MASTER_ARCHITECTURE_v6.8.md` and the boundary allocations in `docs/AUTH-BOUNDARY-REVIEW-2026-04-10.md`

## 1. Document role in the matrix

This addendum defines the deployment-facing part of the v6.9 security correction.

It covers:
- production public boundary ownership;
- local/direct-BFF boundary ownership;
- browser security header placement and precedence;
- runtime secret and cookie transport requirements;
- first-release CSP delivery grounded in the current frontend structure;
- edge reinforcement vs application-layer auth protection;
- live-surface verification expectations.

## 2. Deployment decision summary

### 2.1 Fixed decisions

1. Solfacil Web is deployed as a same-origin HTTPS admin surface.
2. Production public browser hardening is owned by the public ingress boundary, which may be Nginx or an equivalent reverse proxy / ingress tier.
3. Application still owns cookie issuance, cache semantics, auth rejection behavior, and direct-BFF fallback headers.
4. Header precedence must be explicit when both app and proxy can emit security-related headers.
5. CSP is mandatory in v6.9 and must reflect current frontend realities.
6. Application-layer login abuse controls are release-critical; edge rate limiting is reinforcement.
7. Trusted-proxy configuration determines whether forwarded client-IP headers are usable.

### 2.2 Fixed deployment defaults accepted for v6.9

1. Production ingress serves the authoritative CSP and browser-hardening baseline.
2. Direct-BFF local/dev entrypoints emit fallback non-HSTS headers so local validation is still meaningful.
3. First-release CSP avoids `script-src 'unsafe-inline'` by frontend cleanup or nonce/hash exceptions; temporary `style-src 'unsafe-inline'` is allowed only while inline login styles remain.
4. Google Fonts and ECharts are self-hosted before rollout; first-release CSP does not depend on third-party font/chart origins.
5. Cookie migration to `__Host-solfacil_session` happens in v6.9 with a short explicit bridge.
6. Production auth-abuse topology is multi-instance-capable and uses shared ephemeral state.

### 2.3 Remaining deployment review points after default settlement

1. Exact first-release CSP header contents after implementation-ready frontend cleanup is confirmed.

## 3. Boundary shapes the design must cover

### 3.1 Production public boundary

Production shape:
- browser
- public ingress / reverse proxy / LB
- BFF
- database and internal services

Design consequence:
- production browser-security headers are judged at the public ingress response surface, not only at the Express app surface.

### 3.2 Local/direct-BFF boundary

Current non-production code surfaces that matter:
- `backend/scripts/local-server.ts`
- `backend/scripts/combined-server.ts`
- current Express CORS behavior in those entrypoints

Design treatment:
- these are real boundary shapes for local/dev/demo use;
- they are not allowed to redefine the production browser contract;
- they must still honor the cookie-first session model and fail-fast secret rules if they are treated as production-capable test paths.

### 3.3 Current code reality acknowledged

Current entrypoint facts relevant to deployment design:
- `local-server.ts` is a direct Express boundary with its own CORS allowlist behavior;
- `combined-server.ts` is a demo HTTPS surface with permissive `cors({ origin: '*' })` behavior;
- neither local script is the authoritative definition of the production public browser boundary;
- production deployment documents must not pretend the system has only one Nginx-shaped boundary.

## 4. Ownership split across ingress and application

### 4.1 Ingress-owned responsibilities on the public surface

The public ingress layer owns:
- TLS termination and redirect to HTTPS where applicable;
- `Strict-Transport-Security` on production HTTPS entrypoints;
- public CSP delivery;
- anti-framing policy;
- MIME sniffing prevention;
- referrer policy;
- permissions policy;
- cache behavior that must not serve stale authenticated HTML or login responses incorrectly;
- optional IP-oriented edge rate limiting.

### 4.2 Application-owned responsibilities

The application owns:
- `Set-Cookie` correctness;
- login/session/logout response semantics;
- `Cache-Control: no-store` on login/logout/session-bootstrap responses;
- auth enforcement and origin validation;
- application-layer login abuse controls;
- fallback hardening headers for direct-BFF/local use when no public ingress is present, excluding HSTS.

### 4.3 App-header vs proxy-header precedence rule

Fixed precedence rule:
- for public browser-hardening headers (`Content-Security-Policy`, HSTS, framing, referrer, permissions, nosniff), the public ingress value is authoritative on the public surface;
- the app must not intentionally emit weaker conflicting values;
- for auth-correctness headers and semantics (`Set-Cookie`, cache-control on auth routes, `Vary` related to origin decisions), the app is authoritative;
- in direct-BFF/local mode with no ingress, app-emitted values are authoritative by definition.

Conflict rule:
- if both layers emit a hardening header, the deployment must resolve to the stricter intended effective policy;
- if a conflict would weaken public behavior or make behavior ambiguous, that is a deployment defect and not an acceptable steady state.

## 5. First-release CSP stance grounded in the current frontend

### 5.1 Current frontend facts that block a fully strict CSP today

Confirmed current structure:
- `frontend-v2/login.html` contains inline `<script>`;
- `frontend-v2/login.html` contains inline `<style>`;
- `frontend-v2/index.html` contains inline `onclick`;
- frontend uses Google Fonts (`fonts.googleapis.com`, `fonts.gstatic.com`);
- frontend loads ECharts from jsdelivr (`cdn.jsdelivr.net`).

### 5.2 v6.9 first-release decision

Fixed decision:
- public production rollout must include CSP from day one.

Fixed first-release implementation stance:
- remove or externalize the inline login script and inline `onclick` before rollout so `script-src` does not need broad `'unsafe-inline'`;
- if inline script cannot be removed in time, use nonce/hash-based exceptions rather than open-ended `'unsafe-inline'` in `script-src`;
- accept temporary `style-src 'unsafe-inline'` only while `login.html` still carries inline CSS;
- self-host Google Fonts and ECharts before rollout so the first-release public CSP does not need third-party font/chart origins.

### 5.3 Concrete first-release policy target

Target policy shape:
- `default-src 'self'`;
- `script-src 'self'` plus nonce/hash exceptions only if rollout still contains known inline script remnants;
- `style-src 'self' 'unsafe-inline'` while inline login styles remain;
- `font-src 'self'`;
- `img-src 'self' data:`;
- `connect-src 'self'`;
- `frame-ancestors 'none'`;
- `object-src 'none'`;
- `base-uri 'self'`.

Longer-term tightening direction:
- self-host fonts and ECharts;
- remove inline login styles;
- remove any nonce/hash allowances once inline script/event handlers are gone.

## 6. Runtime env and secret requirements

### 6.1 Mandatory runtime inputs

The deployment boundary must provide, at minimum:
- validated `JWT_SECRET`;
- deployment mode/config inputs that make `Secure` cookies valid in the actual request path;
- trusted-proxy configuration when ingress-derived client IP is used;
- any explicit allowlist/origin settings required by same-origin policy enforcement.

### 6.2 Secret governance rules

Deployment rules:
- `JWT_SECRET` must be injected through managed secret/config handling or equivalent controlled runtime configuration;
- deployment may not rely on code fallback secret values;
- startup validation failure blocks rollout;
- secret rotation consequences for stateless JWT sessions must be acknowledged operationally.

## 7. Cookie and transport boundary requirements

### 7.1 Required transport conditions

Because the design uses an HttpOnly Secure cookie:
- the public admin surface must be HTTPS;
- proxy forwarding must preserve scheme/proto information correctly;
- no production-capable path may silently downgrade the browser into insecure-cookie behavior.

### 7.2 Host/path scoping

Preferred deployment-compatible contract:
- host-only cookie;
- `Path=/`;
- no `Domain` override;
- same-origin serving of SPA, login page, and `/api/*`.

This is what enables the recommended `__Host-` cookie shape if approved.

## 8. Edge rate limiting vs application rate limiting

### 8.1 Fixed split

v6.9 uses both layers for different reasons:
- application-layer protection is the release-critical layer;
- public edge rate limiting is recommended reinforcement for source-IP burst absorption.

### 8.2 Deployment consequence

Release acceptance requires:
- application throttle/backoff working correctly;
- any configured edge reinforcement not conflicting with application semantics;
- no claim that edge protection alone satisfies the v6.9 requirement.

## 9. Trusted proxy and source-IP derivation

Fixed trust rule:
- forwarded client-IP headers are only valid when supplied through configured trusted ingress/proxy hops.

Deployment implications:
- public ingress must preserve and document the forwarded-IP chain used by the app;
- direct-BFF/local paths must not trust arbitrary forwarded headers from clients;
- if no trusted ingress configuration exists, the app falls back to socket peer address.

## 10. Live-surface verification expectations

The deployment boundary is accepted only if live verification demonstrates:
- HTTPS delivery of the public admin surface;
- required hardening headers on login and SPA entry responses at the public boundary;
- CSP behavior matching the declared first-release policy;
- auth cookie observed with `HttpOnly`, `Secure`, reviewed `SameSite`, and correct path semantics;
- unauthenticated and authenticated cache behavior matching design intent;
- trusted-proxy vs direct-BFF IP derivation behaving as designed;
- repeated login abuse triggering application controls and any configured edge reinforcement;
- startup failure when required secret inputs are absent or unsafe.

## 11. Rollback and operational exception handling

### 11.1 Allowed temporary exceptions

Examples of acceptable temporary exception mechanisms:
- enabling documented backend compatibility mode for legacy browser bearer acceptance during emergency rollback;
- temporary dual-cookie bridge during cookie-name migration;
- temporary CSP nonce/hash exceptions for known inline script remnants;
- temporary edge-rate adjustments to reduce false positives.

### 11.2 Not acceptable

Not acceptable:
- removing CSP from the public admin surface;
- reintroducing predictable secret fallback;
- removing `Secure` cookie semantics on production-capable paths;
- treating `combined-server.ts` permissive CORS behavior or `local-server.ts` local allowlists as the production security contract;
- turning rollback into permanent undocumented dual-mode operation.

## 12. Residual risks / review points

Remaining deployment review is now implementation-tightening, not direction-setting:
- exact first-release CSP header contents after frontend cleanup is verified in code;
- verification that `SameSite=Strict` works correctly in real operator navigation flows.

Deployment design verdict: v6.9 now distinguishes the public ingress boundary from local/direct-BFF boundaries, grounds CSP in the actual frontend structure, defines header ownership and precedence, and keeps application-layer auth protection as the release-critical control.
