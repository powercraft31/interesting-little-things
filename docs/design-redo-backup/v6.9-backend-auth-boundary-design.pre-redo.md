# Solfacil v6.9 Backend Auth Boundary Design

Date: 2026-04-15 UTC
Status: DESIGN revision addendum for v6.9
Primary source: `docs/v6.9-security-req.md`
Conceptually updates: `design/backend_architecture/05_BFF_MODULE_v6.8.md`, `design/backend_architecture/06_IDENTITY_MODULE_v6.8.md`, `design/backend_architecture/09_SHARED_LAYER_v6.8.md`

## 1. Document role in the matrix

This addendum defines the backend auth/session boundary for v6.9 across:
- M5 BFF request handling and middleware;
- M6 identity/login/logout semantics;
- M9 shared-layer token verification and secret validation contract.

It supersedes the v6.8 backend auth description where the browser contract is split between:
- bearer-header auth for normal browser API traffic;
- cookie fallback for SSE;
- login returning a browser-consumable token body;
- runtime secret fallback to a predictable development value.

## 2. Backend decision summary

### 2.1 Fixed decisions

1. The backend defines two distinct auth contracts:
   - browser admin contract: cookie-first;
   - machine/non-browser contract: bearer-based and separate from the browser contract.
2. Browser API and browser SSE share the same cookie-backed auth entry path.
3. Login no longer exports the JWT string as part of the steady-state browser contract.
4. Legacy browser bearer compatibility, if retained, is explicit, temporary, and disabled by default.
5. Application-layer login abuse controls are mandatory and release-critical.
6. Client IP is trusted only through configured trusted-proxy rules.
7. Missing or unsafe auth secret configuration blocks startup.

### 2.2 Fixed backend defaults accepted for v6.9

1. Cookie name for v6.9: `__Host-solfacil_session`.
2. Browser cookie `SameSite` for v6.9: `Strict`.
3. Browser bootstrap helper for v6.9: `GET /api/auth/session` returning minimal session metadata.
4. Production abuse-state topology for v6.9: shared ephemeral store across all auth-serving instances.

### 2.3 Remaining backend review points after default settlement

1. Exact threshold values for throttle/backoff.
2. Whether active SSE disconnect logic is needed beyond frontend stream closure.

## 3. Supported backend auth contracts

### 3.1 Browser admin contract

Steady-state browser contract:
- login issues signed JWT inside an HttpOnly Secure same-origin cookie;
- auth middleware validates that cookie for browser `/api/*` traffic;
- SSE and non-SSE browser traffic share the same cookie-based path;
- login response returns display-safe session metadata only;
- browser handlers do not require `Authorization: Bearer` from frontend JS.

### 3.2 Machine / non-browser contract

Separate machine contract:
- intentional bearer-token use is allowed only for non-browser callers;
- this contract must be documented and reasoned about separately from the browser contract;
- it must not be used by the Solfacil SPA;
- any route eligibility for machine bearer use must be explicit, not assumed from legacy browser behavior.

Design consequence:
- browser bearer acceptance is no longer the default external contract for `/api/*`.

## 4. M5 BFF design changes

### 4.1 Middleware contract

The v6.8 M5 statement that auth middleware validates `Authorization: Bearer` on normal browser API routes is replaced, for browser traffic, by cookie-backed validation.

Target middleware responsibilities:
- parse auth cookie from same-origin browser requests;
- reject missing/invalid/expired cookie with stable `401` behavior;
- normalize verified tenant context for downstream handlers;
- enforce auth-related origin policy on modifying requests;
- emit stable rejection semantics for unauthorized, throttled, and malformed auth requests.

### 4.2 Bearer compatibility stance

Legacy browser bearer acceptance is not part of the target browser contract.

If retained during migration or rollback:
- it must sit behind an explicit compatibility switch;
- it must be disabled by default;
- it must be logged/documented as an operational exception;
- it must have a removal/sunset expectation.

### 4.3 Downstream handler compatibility

Existing handlers may continue receiving normalized tenant context in their current internal form to avoid unnecessary route-wide redesign.

Design rule:
- internal compatibility shims may preserve handler stability;
- they may not preserve the old insecure browser auth model.

## 5. M6 identity/login/logout design changes

### 5.1 Login handler semantics

The v6.8 identity statement that login performs token return plus cookie issue is replaced by cookie issue plus optional metadata return for browsers.

Target browser-login semantics:
- validate credentials;
- evaluate application-layer login-abuse controls before and after credential check;
- on success, issue auth cookie with required flags;
- return display-safe user/session metadata only;
- reset failure counters/backoff state;
- log security-relevant auth event;
- use externally generic error messages.

Machine/non-browser login/token issuance, if needed, is separate from the browser contract and must not drive frontend design.

### 5.2 Logout handler semantics

Target logout semantics:
- clear the auth cookie using exactly matching cookie attributes where required by browser behavior;
- return stable success regardless of prior session presence;
- support idempotent logout;
- not depend on frontend-side token deletion for correctness.

### 5.3 Session expiry semantics

Target expiry semantics:
- expired token yields `401` on subsequent API requests;
- expired token prevents new SSE connections/reconnections;
- frontend transitions to logged-out UX on `401`;
- immediate mid-stream revocation remains limited while the system uses stateless JWT.

## 6. Cookie contract

### 6.1 Fixed cookie properties

Fixed target properties:
- HttpOnly;
- Secure on production-capable paths;
- `Path=/`;
- no `Domain` attribute;
- explicit bounded lifetime aligned to JWT expiry.

### 6.2 Fixed cookie shape for v6.9

Fixed adopted shape:
- name: `__Host-solfacil_session`;
- `SameSite=Strict`.

Migration note:
- temporary dual-cookie read/issue is allowed only as an explicit short migration bridge;
- reuse of `solfacil_jwt` beyond the bridge window is not the v6.9 target state.

### 6.3 Migration semantics for cookie-only vs compatibility mode

| Mode | Browser behavior | Backend rule |
|---|---|---|
| Cookie-only steady state | browser authenticates only via cookie | target mode; default after cutover |
| Compatibility mode | backend also accepts legacy browser bearer semantics temporarily | disabled by default; time-bounded rollback/migration exception |
| Dual-cookie bridge | backend issues/reads old and new cookie names for a short window | allowed only to reduce forced re-login during cookie-name migration |

## 7. CSRF and same-origin enforcement

### 7.1 Same-origin decision

The Solfacil admin BFF is treated as a same-origin web backend by default.

Consequences:
- cross-origin browser use is not a default supported product capability;
- CORS allowlisting, where present, is exceptional rather than normative;
- origin rejection must be explicit and stable, not a `500` side effect.

### 7.2 CSRF controls for v6.9

Required v6.9 backend posture:
- reviewed `SameSite` cookie setting;
- `Origin` validation on state-changing requests;
- `Referer` as secondary evidence where needed;
- no state-changing endpoint accepts cross-site browser writes by default.

A dedicated CSRF token framework is not required for this release if the same-origin controls above are implemented correctly.

## 8. Login abuse-control design

### 8.1 Required controls

The backend auth boundary must implement application-owned abuse resistance at login.

Required controls:
- source-IP throttling;
- identifier-based progressive backoff independent of account existence;
- audit logging for auth success/failure/throttle/backoff events;
- generic external error semantics;
- testable threshold behavior.

### 8.2 Response semantics

Recommended response semantics:
- invalid credentials: generic `401`;
- throttle/backoff active: `429` with `Retry-After` and generic body;
- disabled/disallowed account states should not provide richer enumeration signals than necessary.

### 8.3 State topology

Fixed decision:
- application owns the semantics and acceptance criteria.

Fixed default for v6.9:
- production uses shared ephemeral counters across all auth-serving instances;
- local/direct-BFF single-instance use may use in-memory counters;
- DB-backed counters are fallback contingency only if the preferred shared ephemeral store is unavailable.

Logical counter placement:
- IP counters keyed by derived client IP;
- identifier counters keyed by normalized login identifier;
- success clears identifier backoff state;
- production multi-instance deployments require shared state for correctness.

### 8.4 Source-IP trust stance

Fixed decision:
- the app trusts forwarded client-IP headers only when the request arrives through configured trusted proxy hops.

Rules:
- trusted public ingress may provide `X-Forwarded-For` / equivalent for client-IP derivation;
- direct untrusted clients may not override IP via headers;
- local/direct-BFF paths derive IP from the socket peer address;
- ambiguous proxy configuration must fail to a conservative interpretation instead of trusting user-supplied headers.

## 9. JWT signing and verification design

### 9.1 Token content

JWT content remains minimal:
- `userId`;
- `orgId`;
- `role`;
- standard `iat` and `exp`.

### 9.2 Signing rules

Signing rules:
- single validated secret source;
- bounded expiration;
- no fallback secret;
- no environment ambiguity between signing and verification paths.

### 9.3 Verification rules

Verification rules:
- reject missing, malformed, expired, or signature-invalid tokens with `401`;
- treat secret misconfiguration as startup failure, not runtime degraded mode;
- retain downstream tenant-context normalization only as an internal compatibility mechanism.

## 10. Startup fail-fast rules

### 10.1 Mandatory startup validation

Before serving requests, the backend validates:
- `JWT_SECRET` is present;
- it is not empty;
- it is not a known placeholder or legacy default;
- it meets minimum security policy;
- cookie `Secure` assumptions match deployment mode expectations.

### 10.2 Failure mode

On violation of any mandatory auth configuration rule:
- process startup fails;
- health checks do not go green;
- the service does not enter a partially secure runtime.

This is a deliberate fail-closed decision.

## 11. SSE impact on backend design

### 11.1 Auth simplification

SSE authentication is no longer a separate special path. It uses the same cookie-backed auth state as other browser requests.

### 11.2 Remaining limitation

Because the session remains stateless JWT in v6.9:
- SSE authorization is evaluated at connection establishment;
- already-open streams are not guaranteed immediate revocation on privilege change or secret rotation;
- logout should close the browser stream from the client side.

## 12. Backend verification requirements

The backend auth boundary is accepted only if verification demonstrates:
- browser login issues the cookie with the chosen attributes;
- browser login no longer returns the bearer token in steady-state mode;
- API access works through cookie auth without browser bearer injection;
- any machine bearer path is documented separately and is not required by SPA flows;
- modifying requests reject invalid cross-origin attempts through stable control flow;
- repeated failed logins trigger configured throttle/backoff behavior across the expected topology;
- trusted-proxy vs direct-BFF IP derivation behaves as designed;
- service refuses startup when `JWT_SECRET` is missing or unsafe;
- SSE connects with the same cookie-backed auth state and fails appropriately after logout/expiry.

## 13. Residual risks / review points

Points still requiring explicit review:
- exact threshold values for throttle/backoff;
- whether active SSE disconnect logic is needed beyond frontend stream closure.

Backend design verdict: M5, M6, and M9 now converge around a cookie-first browser contract, a distinct non-browser bearer contract, explicit abuse-state topology and IP trust rules, and fail-fast secret governance.
