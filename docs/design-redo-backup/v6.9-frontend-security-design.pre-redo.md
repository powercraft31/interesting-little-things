# Solfacil v6.9 Frontend Security Design

Date: 2026-04-15 UTC
Status: DESIGN revision addendum for v6.9
Primary source: `docs/v6.9-security-req.md`
Conceptually updates: `docs/FRONTEND_ARCHITECTURE_v6.8.md`

## 1. Document role in the matrix

This addendum is the frontend-specific v6.9 security design derived from `docs/FRONTEND_ARCHITECTURE_v6.8.md`.

It supersedes the frontend auth/session portions of v6.8 where the browser contract is described or implied as:
- `JWT (localStorage) + HttpOnly cookie`;
- login bootstrap based on browser token storage;
- authenticated fetch/XHR using `Authorization: Bearer` sourced from `localStorage`;
- logout as dual cleanup of cookie plus `localStorage` token.

All other frontend architecture content remains valid unless it depends directly on those old auth assumptions.

## 2. Frontend decision summary

### 2.1 Fixed decisions

1. The Solfacil browser admin surface is a same-origin cookie-first client.
2. The browser does not own the primary auth credential in JS-readable storage.
3. Browser API and browser SSE share the same cookie-backed auth path.
4. The shipped SPA may not depend on browser bearer-header injection.
5. Stale `localStorage` auth tokens after rollout do not restore or preserve auth.
6. Browser-visible session truth comes from server-authenticated response, not token-presence heuristics.

### 2.2 Fixed frontend defaults accepted for v6.9

1. Use a dedicated `GET /api/auth/session` probe for login bootstrap and SPA bootstrap.
2. On first cookie-first release, proactively remove legacy `solfacil_jwt` from `localStorage` as cleanup hygiene, while not relying on that cleanup for correctness.
3. Remove inline login script and inline `onclick` before production rollout so `script-src` can avoid `'unsafe-inline'`.
4. Self-host Google Fonts and ECharts before production rollout so first-release CSP does not depend on external font/chart origins.

### 2.3 Remaining frontend review points after default settlement

1. Exact UX behavior for forced re-login during the temporary cookie-bridge removal window, if any.

## 3. Updated frontend architecture statements

### 3.1 Technology stack correction

The v6.8 frontend stack line:
- `Auth | JWT (localStorage) + HttpOnly cookie | Bearer token via Authorization header for fetch/XHR; HttpOnly solfacil_jwt cookie for browser-native SSE`

is replaced for v6.9 by:
- `Auth | HttpOnly Secure same-origin session cookie | Browser API and SSE both authenticate through the same cookie-backed session path; no production auth token persisted in localStorage`

### 3.2 Request model correction

The SPA request model changes from bearer injection to cookie transport.

v6.9 browser request contract:
- browser-originated API requests are same-origin requests;
- browser includes session cookie automatically;
- frontend request helpers no longer read auth token from web storage;
- request helpers handle `401` by transitioning to logged-out UX;
- machine/non-browser bearer usage, if any, is not a frontend concern and is not exposed through SPA helpers.

## 4. Login page design

### 4.1 Current reality acknowledged

Current frontend structure that affects the first release:
- `frontend-v2/login.html` contains inline `<style>` and inline `<script>`;
- `frontend-v2/login.html` currently reads `localStorage` for language preference and historically also participated in auth-token flow;
- `frontend-v2/index.html` contains an inline `onclick` handler;
- Google Fonts and jsdelivr ECharts are external dependencies in current HTML.

### 4.2 Entry behavior

`login.html` is no longer allowed to use `localStorage("solfacil_jwt")` or similar token-presence checks as its primary decision point.

Target behavior:
- render the login shell immediately;
- perform a lightweight same-origin session probe;
- if the probe confirms a valid session, redirect to the SPA;
- if the probe returns `401`, remain on login and show the form;
- if the probe errors for infrastructure reasons, keep the user on login and show a retryable generic message.

Fixed decision:
- use dedicated `GET /api/auth/session` returning minimal display-safe user/session metadata.

### 4.3 Login submit behavior

Target browser flow:
- submit credentials to same-origin `POST /api/auth/login`;
- expect `Set-Cookie` to establish the session;
- consume optional display-safe session metadata only;
- do not store token in `localStorage`, `sessionStorage`, or a long-lived JS variable;
- after success, navigate into the SPA and bootstrap from authenticated server state.

### 4.4 Error behavior

Frontend should distinguish only the UX categories it actually needs:
- generic auth failure;
- temporary throttled/backed-off login;
- retryable transport/system error.

Frontend must not attempt to infer account existence from richer response details.

## 5. SPA bootstrap and route gating

### 5.1 App bootstrap source of truth

`index.html` / `app.js` bootstrap from server-authenticated state.

Target behavior:
- initial app bootstrap uses same-origin authenticated session truth;
- if bootstrap returns `401`, redirect to login;
- no route gate based on token presence in browser storage;
- in-memory user metadata may exist for UX, but it is derived from authenticated server response.

### 5.2 `localStorage` after migration

`localStorage` remains acceptable only for low-value UX state such as language or theme.

It is not acceptable for:
- JWT;
- session token;
- refresh token;
- equivalent auth secrets.

Stale-token handling after rollout:
- new frontend ignores existing legacy auth keys;
- optional startup cleanup may delete `solfacil_jwt` from storage for hygiene;
- auth correctness does not depend on whether cleanup ran.

## 6. Authenticated request helper design

### 6.1 Fetch/XHR helper contract

The request helper layer replaces bearer semantics with cookie semantics.

Target contract:
- send same-origin requests;
- rely on browser cookie handling;
- never construct `Authorization: Bearer` from browser storage;
- preserve common error-envelope parsing;
- centralize `401` handling.

### 6.2 `401` handling

On `401`:
- clear in-memory display-safe session state;
- close active `EventSource` instances;
- redirect to login;
- optionally display a session-expired message.

This replaces the v6.8 behavior that cleared a stored JWT value.

## 7. SSE design impact

### 7.1 Auth channel

The frontend SSE client continues to use `EventSource('/api/events')`, but in v6.9 this is no longer a special exception. It uses the same browser session cookie as normal API requests.

### 7.2 Session-loss behavior

Expected behavior:
- reconnect attempts continue only while the browser session remains valid;
- on explicit logout, frontend closes the stream immediately;
- when API and SSE both indicate auth failure, frontend converges to one logged-out path.

## 8. Mixed-version and rollback behavior as seen by the browser

| Scenario | Browser-visible behavior | Frontend stance |
|---|---|---|
| Old frontend -> new backend, compatibility mode OFF | login/API failures due to contract mismatch | Not supported steady state. Coordinated rollout required. |
| Old frontend -> new backend, compatibility mode ON | legacy pages may continue briefly | This is a backend compatibility exception, not a frontend target. |
| New frontend -> old backend | may partially work if old backend still issues cookie, but remains unsupported skew | Avoid prolonged mixed-version deployment. |
| Cookie name unchanged at cutover | existing sessions may survive until expiry | Lowest-disruption rollout path. |
| Cookie name changed without bridge | users with old sessions are forced to log in again | Acceptable if planned and announced. |
| Cookie name changed with temporary bridge | session continuity possible during migration window | Temporary only, with explicit expiry. |
| Backend rollback without compatibility mode | new frontend cannot use old browser bearer path | Rollback must include frontend rollback or backend compatibility mode. |

## 9. Frontend browser-security implications

### 9.1 First-release CSP posture grounded in current structure

Current blockers to a strict no-exception CSP:
- `frontend-v2/login.html` inline `<script>`;
- `frontend-v2/index.html` inline `onclick`;
- `frontend-v2/login.html` inline `<style>`;
- external dependencies on Google Fonts and jsdelivr, both of which are slated for self-hosting before v6.9 rollout.

v6.9 first-release stance:
- CSP is mandatory on day one;
- preferred rollout cleans up the inline login script and inline `onclick` before production so `script-src` can remain free of broad `'unsafe-inline'`;
- if cleanup misses the release train, any temporary inline-script allowance must be nonce/hash-based and explicitly documented;
- temporary `style-src 'unsafe-inline'` is acceptable only while the login page still uses inline CSS;
- first-release allowlisted third-party origins, if still used, are limited to:
  - `https://fonts.googleapis.com`
  - `https://fonts.gstatic.com`
  - `https://cdn.jsdelivr.net`

### 9.2 Framing and embedding

The admin SPA is not designed for third-party embedding.

Frontend therefore assumes:
- no iframe embedding contract;
- no cross-origin app-hosting contract;
- no browser extension-style dependency on token readability.

## 10. Frontend verification requirements

v6.9 frontend closure is verified only if all of the following are true:
- login succeeds without writing auth JWT to `localStorage` or `sessionStorage`;
- app bootstrap succeeds using cookie-backed auth only;
- no normal API request depends on `Authorization` header injection by frontend JS;
- stale legacy storage tokens do not keep the user authenticated;
- logout works without auth-token cleanup as a correctness requirement;
- `401` handling consistently redirects to login and closes SSE;
- the frontend operates correctly under the deployed CSP and header baseline.

## 11. Residual risks / review points

Residual frontend risk now narrowed to implementation/verification, not product-direction choice:
- ensure the dedicated session probe does not become a silent source of bootstrap regressions;
- ensure cookie-bridge removal is coordinated with release timing;
- ensure `SameSite=Strict` is validated against real operator navigation flows during verification.

Final frontend design decision: the browser UI stops owning the auth credential, treats stale legacy storage as irrelevant to auth, and becomes a same-origin client of a cookie-authenticated BFF under a concrete first-release CSP posture.
