Solfacil v6.9 Implementation Kickoff Constraints

這不是新一輪 REQ / DESIGN / PLAN，而是 implementation 開始前的最後收口層。目的很直接：把剩下最容易在執行期漂移的點先釘死，避免 agent 或工程師在實作時自行發明規則。

Status: implementation preflight constraints Date: 2026-04-15 UTC Scope: v6.9 security correction

你先看這裡

  • inline login script / inline onclick 必須清掉；不允許用廣義 script-src 'unsafe-inline' 偷過。
  • login abuse control 不能帶著模糊參數開工，threshold / backoff / reset / Retry-After 必須先定。
  • machine bearer contract 必須先列出 scope；不能靠「反正有 machine contract」把 legacy bearer 全留著。
  • 手動驗證必須有證據格式，不接受口頭「測過了」。
  • bridge 是短期工具，不是舒適圈。

固定執行約束

FE-6 / CSP

  • 移除或外部化 inline login script 與 inline onclick
  • 不接受廣義 script-src 'unsafe-inline'
  • 若仍有 script 殘留，只能走 nonce/hash 例外
  • login page inline style 可暫時保留，但必須列為 temporary residual risk

Abuse-control

  • IP burst threshold
  • identifier failure threshold
  • backoff progression
  • success reset rule
  • Retry-After 規則

Machine contract / Bridge discipline

Machine bearer scope

  • 先列出 machine-bearer-eligible endpoints
  • 列出 browser-cookie-only endpoints
  • SPA 不得使用 bearer

Bridge discipline

  • compatibility mode 預設 OFF
  • dual-cookie bridge 預設 OFF
  • 每個 bridge 都要有 enable / disable 條件、removal owner、最大存在時間

測試證據格式

Backend

  • test command output
  • 必要的 curl probe output
  • bridge ON/OFF 狀態說明

Frontend

  • browser screenshot
  • devtools network screenshot/export
  • storage inspection proof
  • console / CSP proof

Deployment / Public boundary

  • curl -I output
  • login response header / cookie proof
  • CSP/header proof from actual public boundary
  • operator checklist completion

Final release

  • final acceptance matrix 每一格都要有證據
  • 不接受口頭“測過了”

Operator-side prerequisite

因為 deployment boundary 有一部分不在 repo 裡，所以 implementation 完成 ≠ rollout-ready。至少要有 operator-side checklist 或 config diff 覆蓋：

  • ingress header / CSP changes
  • trusted proxy settings
  • secure cookie correctness
  • shared abuse-store provisioning
  • bridge toggle handling
