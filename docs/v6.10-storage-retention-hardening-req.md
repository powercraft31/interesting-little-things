# Solfacil v6.10.x Storage Retention Hardening — REQ

Date: 2026-04-19 UTC
Status: REQ draft / ready for DESIGN
Version baseline: Solfacil v6.10 runtime-governance local acceptance closure
Target release: v6.10.x hardening track

## 1. Background

Solfacil v6.10 established a runtime-governance spine and proved that part of the platform can now express governed runtime facts with lifecycle and retention semantics.

That solved one class of problem: runtime governance tables are no longer pure append-only drift.

It did **not** solve the broader storage-growth problem across the system.

The live system still contains multiple log/history surfaces with no hard cap, no TTL, and no explicit archive policy. That means the system can still accumulate data until disk pressure becomes the de facto stop line.

So this follow-up REQ is not about adding another dashboard or polishing v6.10 UI. It is about making Solfacil's storage behavior intentional.

## 2. Release decision already settled

The following framing is treated as fixed input for this REQ draft:

1. This is a **post-v6.10 hardening track**, not a redefinition of v6.10's runtime-governance objective.
2. The problem is **storage-governance inconsistency**, not lack of visibility.
3. The priority is to **cap unbounded growth surfaces first**, especially those that can consume disk without warning.
4. The solution must distinguish between:
   - platform/runtime log retention,
   - operational history retention,
   - business history retention,
   - audit evidence that may require longer preservation.
5. Blunt deletion of business tables is not acceptable unless the data class is explicitly downgraded from system-of-record to disposable operational residue.

## 3. Verified current-state evidence

This REQ is grounded in current repo + live local runtime evidence.

### 3.1 Live runtime / host evidence

Observed on the canonical local stack:

- running containers: `solfacil-bff`, `solfacil-db`, `solfacil-m1`, `solfacil-redis`
- all four containers currently use Docker log driver `json-file`
- current Docker log config for all four containers is empty: no `max-size`, no `max-file`
- current root filesystem free space: `125G`

REQ consequence:
- container stdout/stderr logs are currently bounded only by remaining disk space.

### 3.2 Runtime-governance retention that already exists

Verified in code under `backend/src/shared/runtime/retention-job.ts`:

- recovered issue auto-close TTL = 24h
- stale issue auto-close TTL = 72h
- `runtime_events` retention = 90 days
- `runtime_health_snapshots` retention = 30 days

Verified in schema / behavior:

- `runtime_self_checks` is latest-state keyed by `check_id`, not append history
- `runtime_issues` is a single mutable row per `fingerprint`
- retention currently auto-closes `runtime_issues` but does **not** delete closed rows

REQ consequence:
- only part of the runtime-governance surface is time-window bounded.

### 3.3 Live table state and growth posture

Observed in the live local DB:

|---|---:|---:|---|
| `runtime_events` | 0 | 0 B | time-window bounded (90d) |
| `runtime_health_snapshots` | 0 | 48 KB | time-window bounded (30d) |

### 3.4 Verified code paths for unbounded writers

Observed in repo:

- `iot-hub/handlers/alarm-handler.ts` writes `gateway_alarm_events` via pure `INSERT`
- `iot-hub/handlers/telemetry-handler.ts` inserts `backfill_requests`

Observed from migrations / schema:

- `gateway_alarm_events`, `backfill_requests`, and `device_command_logs` are append-oriented tables
- no verified cleanup/retention executor currently covers them

REQ consequence:
- the remaining growth problem is structural, not accidental.

## 4. Problem statement

The system currently mixes multiple data classes under one implicit storage policy: "keep writing until something forces cleanup later."

That is acceptable for neither operations nor engineering governance.

The core problem is not that Solfacil lacks logs. The core problem is that it lacks **one explicit retention model across storage classes**.

The result is four failures:

1. **Disk risk is delegated to luck**  
   Container logs currently have no rotation and can grow until the host runs out of space.

2. **Append-heavy operational tables have no stop line**  
   `device_command_logs`, `gateway_alarm_events`, and `backfill_requests` can keep accumulating without a formal TTL or archive rule.

3. **Business history and operational residue are not cleanly separated**  
   Tables like `revenue_daily` may be legitimate historical records, but they are currently governed no differently from disposable logs.

4. **Current retention semantics are fragmented**  
   v6.10 runtime tables have one set of lifecycle rules, while adjacent operational tables remain outside any unified storage-governance contract.

## 5. Root classification

### 5.1 This is not mainly a dashboard problem
A page can show sizes, but the real defect is missing storage-governance policy.

### 5.2 This is not mainly a database-size problem
Current live sizes are still small. The problem is not present size; it is missing bounding semantics.

### 5.3 This is not the same as deleting history aggressively
Some tables are true business history and should not be treated like throwaway logs.

### 5.4 This is an operations hardening problem with schema consequences
The solution must combine Docker runtime settings, DB retention policy, and explicit data-class boundaries.

## 6. Release objective

Solfacil v6.10.x storage-retention hardening should:

- introduce one explicit storage-governance model for high-growth log/history surfaces
- cap container stdout/stderr growth with deterministic Docker rotation
- define per-table retention/archive policy for append-heavy operational records
- preserve business-history tables unless and until a separate archival decision is made
- make disk growth predictable enough to budget, monitor, and review

In plain terms:

**the system should stop relying on infinite growth as the default storage policy.**

## 7. Core product requirements

### 7.1 Container log hard caps must exist
The runtime stack must define Docker log rotation explicitly for all Solfacil containers that currently rely on `json-file` logging.

REQ consequence:
- host disk consumption from stdout/stderr must no longer be implicitly unbounded.

### 7.2 Storage classes must be classified explicitly
DESIGN must classify at least these data classes separately:

- platform/runtime logs
- runtime-governance projections/history
- operational append-only history
- queue/state tables
- business/system-of-record history
- audit evidence requiring longer retention

REQ consequence:
- one blanket TTL for "all logs/tables" is not acceptable.

### 7.3 Append-heavy operational tables must have a bounded policy
At minimum, DESIGN must define formal retention/archive handling for:

- `device_command_logs`
- `gateway_alarm_events`
- `backfill_requests`
- `runtime_issues` closed-row lifecycle

REQ consequence:
- these tables must not remain forever-growing by omission.

### 7.4 Business-history treatment must be explicit
For tables such as `revenue_daily`, DESIGN must state whether they are:

- system-of-record history retained indefinitely,
- archived out of the hot path after a window,
- or downgraded to operational history with TTL.

REQ consequence:
- business-history policy cannot be left implicit.

### 7.5 Runtime-governance retention must remain consistent
The existing v6.10 retention semantics for `runtime_events` and `runtime_health_snapshots` must remain explicit and must not regress.

REQ consequence:
- this hardening track extends storage governance; it does not weaken the v6.10 runtime-governance spine.

### 7.6 Capacity budgeting model must exist
DESIGN must define a way to estimate retained footprint by table/log class using:

- current total size
- growth driver
- expected rows/day or MB/day
- retention window or archive cutoff
- resulting steady-state estimate

REQ consequence:
- retention policy is incomplete if operators still cannot estimate storage ceiling behavior.

### 7.7 Cleanup execution path must be real, not aspirational
For every bounded surface, DESIGN must name the actual enforcement path:

- Docker daemon/container config
- scheduled DB retention executor
- partition drop policy
- archive job
- manual operator runbook only, if and only if that is consciously accepted

REQ consequence:
- policy without enforcement is not considered solved.

## 8. Scope

This REQ covers:

- Docker/container log rotation for Solfacil runtime containers
- storage classification of current Solfacil log/history tables
- retention/archive treatment for append-heavy operational tables
- budgeting / predictability rules for retained data
- explicit preservation rules for business-history tables

This REQ does not yet prescribe exact implementation shape. That belongs to DESIGN.

## 9. Non-goals

This hardening track is **not** intended to:

1. redesign v6.10 runtime-governance APIs or UI
2. erase historical business data blindly for the sake of small tables
3. build a full external observability stack
4. solve every future storage problem across unrelated services on the host
5. turn every table into a partitioned warehouse immediately
6. normalize one retention number for data classes with different legal/operational meaning

## 10. Architecture context that DESIGN must preserve

### 10.1 Existing v6.10 runtime-governance semantics stay authoritative
DESIGN must preserve the already-accepted runtime-governance contract and its retention semantics.

### 10.2 Platform/runtime data and business data remain distinct
DESIGN must not collapse domain/business history into the same disposal lane as runtime noise.

### 10.3 Enforcement should stay narrow-but-deep
The preferred insertion model is:

- first cap the truly dangerous uncapped surfaces
- then govern the highest-growth append tables
- leave low-risk historical/business tables explicit but not prematurely mutilated

### 10.4 Anti-patterns DESIGN must avoid
DESIGN must not normalize any of the following without explicit approval:

- deleting business history just because it grows
- pretending auto-close is equivalent to row deletion
- calling a table "bounded" when only issue state changes but rows never leave
- shipping a policy document with no actual executor or config enforcement
- broad, invasive repartitioning of unrelated modules when narrow retention hooks are sufficient

## 11. Success definition

This hardening track is successful when all of the following are true:

1. Container stdout/stderr growth is deterministically capped.
2. Every major append-heavy operational table has an explicit lifecycle: retain, archive, or delete.
3. `runtime_issues` is honestly classified as either retained projection history or receives a real closed-row cleanup policy.
4. Business-history tables are explicitly preserved or archived by design, not by neglect.
5. Operators can explain the storage ceiling behavior of the stack without hand-waving.

## 12. Acceptance requirements for DESIGN

DESIGN must answer these questions explicitly:

1. What are the canonical storage classes in Solfacil?
2. Which surfaces are hard-capped, time-window-bounded, archive-bounded, or intentionally indefinite?
3. What Docker log rotation settings are adopted, and why?
4. What is the retention/archive policy for `device_command_logs`?
5. What is the retention/archive policy for `gateway_alarm_events`?
6. What is the retention/archive policy for `backfill_requests`?
7. What is the lifecycle policy for closed `runtime_issues` rows?
8. How is `revenue_daily` classified and governed?
9. What executor path enforces each policy?
10. What rollback posture preserves data safety if the hardening rollout must be disabled?

## 13. Proposed defaults carried into DESIGN

These defaults are the current recommended baseline for DESIGN unless Alan explicitly overrides them.

### 13.1 Priority cut line
First fix the surfaces that can consume disk or rows indefinitely without meaningful operator choice:

- Docker container logs
- `device_command_logs`
- `gateway_alarm_events`
- `backfill_requests`
- `runtime_issues` closed-row accumulation

### 13.2 Business-history default
`revenue_daily` should be treated as business history first, not disposable runtime noise.

### 13.3 Rollout bias
Prefer additive enforcement and reversible retention executors over destructive schema redesign.

## 14. One-sentence release definition

**Solfacil v6.10.x storage-retention hardening makes data growth intentional by capping uncapped log surfaces, assigning explicit lifecycle policy to append-heavy operational records, and preserving true business history under a separate rule set.**
