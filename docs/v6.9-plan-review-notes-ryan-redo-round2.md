## 1. Overall judgment

The new PLAN set is materially better than the old plan chain.
It is concrete, file-targeted, phased, and includes layered testing plus final acceptance.
So the direction is correct.

But there are still a few places where PLAN is softer than the revised DESIGN and could drift during implementation.
These should be corrected now, before implementation starts.

## 2. Required corrections

### 2.1 Do not normalize `solfacil-dev-secret` fallback in PLAN

Current problem:
- backend plan still contains a dev-mode step equivalent to: if `JWT_SECRET` is unset in development, warn and use legacy `solfacil-dev-secret` fallback.

Why this is still wrong at PLAN level:
- DESIGN already frames this as legacy debt, not the desired operating model;
- PLAN should move implementation toward explicit local/dev provisioning, not encode the old fallback as the normal plan;
- otherwise implementation will preserve the very convenience path we are trying to retire.

Required correction:
- PLAN should require explicit local/dev secret provisioning for normal development runs;
- any remaining legacy fallback mention must be framed only as temporary debt encountered in current code, not as the target implementation behavior;
- remove plan wording that tells implementers to deliberately keep or add `solfacil-dev-secret` fallback behavior.

### 2.2 Stop reopening machine-login discrimination as an implementation option list

Current problem:
- master/backend plan still contains wording like:
  - `X-Auth-Contract: machine` header preferred; separate endpoint acceptable
  - machine login discrimination left as a plan-start option

Why this is a problem:
- PLAN should operationalize the chosen design, not reopen the contract shape;
- leaving this as an option means implementation can fork the contract during coding.

Required correction:
- choose one concrete implementation path for v6.9 PLAN;
- default for this plan should be:
  - keep `POST /api/auth/login`
  - use explicit `X-Auth-Contract: machine` discrimination for machine callers
  - browser default remains cookie-first with no token in body
- if a separate endpoint is considered later, treat it as out-of-scope/future work, not a current plan branch.

### 2.3 Tighten legacy-browser-compat wording so PLAN does not quietly preserve it

Current problem:
- deployment plan still includes commented optional `LEGACY_BROWSER_BEARER` environment entries.

Why this is risky:
- bridge/default-off semantics are correct, but PLAN should not casually normalize extra compatibility toggles unless they are truly required for this rollout;
- the current design trend is hard-cut browser contract, not convenience-preserved browser bearer.

Required correction:
- unless there is a clearly justified rollout need, remove `LEGACY_BROWSER_BEARER` from the default implementation plan;
- if retained as an emergency-only toggle, label it clearly as disabled-by-default emergency rollback tooling with bounded lifetime and not part of the normal rollout path.

### 2.4 Keep acceptance and verification aligned with the hard-cut browser contract

Current requirement to preserve:
- final acceptance must continue to prove:
  - no browser token body in normal login
  - no `solfacil_jwt` browser contract in v6.9 steady state
  - `GET /api/auth/session` is fixed and working
  - production-mode startup fails without working Redis abuse-control store

This is mostly present already. Preserve it while revising the softer points above.

## 3. Revision request to Claude Code

Revise the four PLAN docs in place.

Required edits:
1. remove/replace dev-mode plan language that preserves `solfacil-dev-secret` fallback as the target implementation path;
2. freeze machine-login implementation path to one concrete v6.9 choice instead of option-listing;
3. tighten or remove `LEGACY_BROWSER_BEARER` rollout language so it is not normalized as part of the default implementation path;
4. keep all current phased structure, testing layers, file targets, and final acceptance matrix intact.

## 4. Acceptance standard before implementation

Do not treat PLAN as ready for coding until:
- `solfacil-dev-secret` fallback is no longer written as a normal implementation step;
- machine login discrimination is one concrete path, not a reopened option;
- browser legacy-compat toggles are either removed from the normal path or explicitly framed as emergency-only disabled-by-default tools;
- final acceptance still proves the hard-cut browser contract and fail-fast abuse-control startup semantics.
