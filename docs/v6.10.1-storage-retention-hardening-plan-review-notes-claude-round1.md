# v6.10.1 Storage-Retention Hardening — Plan Review (Claude, round 1)

Reviewer: independent second opinion on PLAN only
Date: 2026-04-19 UTC
Plan under review: `docs/v6.10.1-storage-retention-hardening-plan.md`

## 1. Overall judgment

The plan is implementable and faithful to the accepted v6.10.1 design set. It derives from the matrix, does not reopen REQ/design decisions, respects the fixed guardrails (revenue_daily preserved, Docker `json-file` 10m×5 across the four containers, device/alarm archive-first, backfill/runtime_issues direct-delete, shared hourly executor extension under M9/shared, additive rollback). Workstream segmentation is sane and ordered correctly (schema → executor → wiring → compose → compatibility → acceptance). File targets for WS1/WS2/WS3/WS4 are precise against the live tree; WS5/WS6 drift into an overly long file list that does not match their own "compatibility verification, not new policy" framing. Gate criteria are mostly evidence-linked; a few hinge on vague phrases that the plan should pin before implementation starts.

No blockers. Several non-blocking tightenings are worth doing before anyone opens an editor on `retention-job.ts`.

## 2. What is correct

- **Scope discipline.** Plan refuses to add storage UI, refuses to TTL `revenue_daily`, refuses to repartition, refuses destructive rollback. Matches REQ §9 and DESIGN §7 explicitly.
- **Executor shape.** `backend/src/shared/runtime/retention-job.ts` is the correct extension anchor. The current `runRuntimeRetention` already has per-phase try/catch, `degraded_fallback` status, fallback logger, and disabled-mode early return — the five-phase extension fits cleanly on top without replacing the job.
- **Phase ordering.** Runtime retention → runtime_issues closed cleanup → device_command archive → gateway_alarm archive → backfill terminal cleanup matches `09_SHARED_LAYER_v6.10.1.md` §3.1 verbatim.
- **Archive-first semantics.** "archive insert before hot delete" + "duplicate-safe archive identity" + "rerun must remain safe after archive-ok/delete-fail" lines up with DB §3.3 and M9 §3.3.
- **Safety predicates.** Pending-row exclusion for `device_command_logs`, terminal-only eligibility for `backfill_requests`, 30d-after-`closed_at` for `runtime_issues`, preserve `revenue_daily` — all carried forward correctly.
- **Migration file targeting.** `003_storage_retention_hardening.sql` is the right next slot given existing `001_p5_strategy_triggers.sql` and `002_runtime_governance.sql`. Manifest/contract-verify touchpoints are the right ones.
- **Docker enforcement.** Exact stanza, exact four service names (`solfacil-bff`, `solfacil-db`, `solfacil-m1`, `solfacil-redis`), `docker compose config` + post-recreate inspect as acceptance lanes. Clean.
- **Rollback posture.** Behavioral/additive only, archive tables remain, no schema surgery on disable. Matches M9 §7.
- **Frontend non-scope.** §10 explicitly carries FRONTEND_ARCHITECTURE_v6.10.1's non-central commitment.

## 3. Blocking issues

None.

Nothing in the plan reopens REQ or design decisions, mangles the executor contract, leaks scope into UI or warehouse redesign, or assumes a file that does not exist. Implementation can begin from this plan. The rest of this review is non-blocking tightening.

## 4. Non-blocking weaknesses / ambiguities

1. **Terminal-eligibility predicate for `device_command_logs` is softened vs DESIGN.** Plan §4.2 Phase 3 says `result != 'pending' and/or resolved_at IS NOT NULL based on actual schema semantics`. DESIGN (05 BFF §3) committed the explicit terminal set `{success, fail, timeout}` or non-null `resolved_at`. The current `ddl_base.sql` plus `dr-dispatcher/services/command-dispatcher.ts` also use intermediate states (`dispatched`, `accepted`) that are *not* terminal. The plan's "and/or … based on actual schema semantics" invites implementer drift toward age-only sweeps on `dispatched`/`accepted` rows. This should be restated as the committed DESIGN predicate, not deferred to "implementation detail".

2. **Executor failure-event code is left optional.** M9 §3.3 names a concrete failure event (e.g. `storage.retention.executor.failed`) that the executor must emit on failure so the governance spine reports its own housekeeping failure. Plan §4.2 lists `contract.ts` as "*if* new executor event code registration is required there". It is required. The plan should commit the event code and its contract-registration touchpoint, not leave it conditional.

3. **Storage-governance registry commitment is silently omitted.** M9 §2.1 requires a checked-in TypeScript policy constant under M9/shared runtime describing each governed surface (class, owner, posture, cutoff, executor, rollback). Plan workstream list does not pin a file for this registry. Either the plan should call out where it lives (e.g. `backend/src/shared/runtime/storage-policy.ts` plus corresponding test) or explicitly defer that commitment — silent omission is the worst option.

4. **WS5 / WS6 file lists are overbroad for a "compatibility verification" workstream.** Plan §4.5 says WS5 is "compatibility verification, not new retention policy design", yet it lists ~10 BFF/IoT production files (including `get-fleet-overview.ts`, `get-fleet-integradores.ts`, `get-alerts-summary.ts`, `publish-config.ts`, `command-publisher.ts`) among its target files. If the slice truly is "verify current readers target the hot table and writers keep terminal evidence", the production files are read-only review items, not edit targets. The list will mislead an implementer into thinking every file is in scope for code change. Either relabel them as "review surfaces" or trim to the small set that may genuinely need a test/comment nudge.

5. **Batch/chunk size is explicitly deferred without a default.** §6.3 says batch size "can be an implementation constant" and leaves the number unpinned. With an hourly cadence and live sizes still small, even an unbounded delete will not hurt — but the plan should either commit a concrete ceiling (e.g. 5_000 rows/phase/run) or explicitly say "unbounded for v6.10.1 given live volumes; revisit if growth forces it". Right now the plan reads as "pick a number, any number".

6. **Cadence is not pinned against the existing scheduler.** Round-2 design review already flagged this. Plan §4.3 says "keep the existing interval model and point it at the extended shared executor" — but the M9 design commits to an *hourly* shared executor. The plan should either (a) confirm `backend/scripts/local-server.ts` already schedules hourly and this is a no-op, or (b) commit a scheduler-config change to guarantee hourly cadence. "Keep the existing interval model" is underspecified.

7. **Idempotence mechanism for archive insert is not named.** Plan §5.1 step 3 calls for "archive constraints/indexes for idempotent preserved-source inserts", but does not say whether this is a `UNIQUE` constraint on the preserved source id plus `INSERT ... ON CONFLICT DO NOTHING`, or a `NOT EXISTS`-guarded select, or something else. Both shapes are defensible; leaving it open in the plan is a future rerun-safety bug waiting to be re-debated at PR-review time.

8. **Integration lane "seed or fixture proof" is not concretized.** §8.2 lanes B/C/D/E all require proof by seed/fixture, but the plan does not pin which fixture harness or which existing test file owns that proof (new `test/shared/runtime-retention-storage.test.ts`? Extensions to `test/shared/runtime-retention.test.ts`? A new integration lane under `test/scripts/`?). Without naming the owning file, the "cross-workstream integration" lane is vaporware relative to the slice lanes, which are concrete.

9. **Gate C failure-path acceptance has no matching named test.** Plan claims "tests prove later phases still execute when an earlier phase fails" and "tests prove duplicate-safe rerun behavior for both archive tables". Neither acceptance bullet is tied to a named Jest case or fixture path. Good engineering; needs a pointer.

10. **Capacity budgeting refinement is mentioned in round-2 caution but absent from plan.** Round-2 review asked PLAN to turn the budgeting matrix into concrete rows/day or MB/day where practical. Plan §7 passes on this entirely. Not blocking because DESIGN's matrix is enough for v6.10.1 acceptance, but the plan should either acknowledge the deferral or add one-line budgeting evidence per surface in release notes.

11. **Minor file-target ambiguities.** `run-local-migrations.sh` and `local-db-rebuild-check.ts` are listed under WS1 with "only if…" qualifiers. Either confirm the touch is needed or drop them. Qualifiers in a target-file list dilute the gate.

## 5. Decisions that still need Alan

None are strictly required before implementation begins; the guardrails are committed. The items below are optional tightenings Alan may want to pin so the implementer does not re-litigate them mid-PR:

1. **Confirm the explicit terminal-state set for `device_command_logs` eligibility.** DESIGN says `{success, fail, timeout}` plus non-null `resolved_at`. Plan softens this. One-line confirmation in the plan avoids drift.
2. **Confirm the storage-policy registry commitment.** Either instruct implementer to land the TypeScript policy constant file under `backend/src/shared/runtime/` in WS2, or explicitly defer to a later release.
3. **Commit cadence.** Either "hourly = already scheduled in `local-server.ts` as confirmed by code read" or "WS3 must set scheduler interval to 3_600_000 ms".
4. **Commit the executor-failure event code** (e.g. `storage.retention.executor.failed`) so `contract.ts` lands the registration deterministically.
5. **Commit idempotence mechanism** (UNIQUE + ON CONFLICT DO NOTHING is the obvious pick) to avoid PR-time debate.

All five are implementation-detail tightenings, not reopened design decisions.

## 6. Recommended revisions before implementation

Minimum set that would make this plan implementation-ready without further reviewer cycles:

1. **Tighten WS2 Phase-3 predicate.** Replace the "and/or … based on actual schema semantics" phrasing with the committed DESIGN predicate: terminal eligibility = `result IN ('success','fail','timeout') OR resolved_at IS NOT NULL`, age cutoff applied after that predicate. Pending/dispatched/accepted rows are never swept by age alone.
2. **Promote the executor failure event from optional to required in WS2.** Commit the event code name and register it in `backend/src/shared/runtime/contract.ts` as part of the workstream exit criteria.
3. **Pin a file for the M9 storage-governance registry** if v6.10.1 is meant to land it; otherwise add one line in §11 or a new non-scope bullet saying the registry ships in a later release.
4. **Trim WS5/WS6 target-file lists** to the files that actually require edits/test tweaks. Move the rest into a "review surfaces" sub-list so the implementer does not treat them as edit targets.
5. **Pin batch/chunk size** or explicitly declare "unbounded per-phase in v6.10.1 at current live volumes" so the implementer does not guess.
6. **Pin scheduler cadence**: add a one-line confirmation or change directive for `backend/scripts/local-server.ts` so hourly is not left implicit.
7. **Name the archive-insert idempotence mechanism** in §5.1 (UNIQUE on preserved source id + `ON CONFLICT DO NOTHING`).
8. **Name the test files that own the integration lanes** (B/C/D/E) — either extensions to `test/shared/runtime-retention.test.ts` or a new file — and list the failure-injection + rerun-duplicate-safety cases as explicit lane items under §8.2.
9. **Resolve WS1 qualifiers** (`run-local-migrations.sh`, `local-db-rebuild-check.ts`) to definite "touch" or "no-touch".
10. **Optional**: append a short per-surface rows/day or MB/day estimate appendix tying to the M9 §5.2 matrix, to close round-2's capacity-budgeting caution.

With those edits the plan is green for implementation without further review cycles.
