# 1. Overall judgment

The tightened v6.10.1 storage-retention hardening design set is now ready for PLAN.

This round resolves the core blockers from the prior review. The design is no longer relying on posture-only wording. It now commits concrete retention windows, concrete Docker rotation values, explicit delete-vs-archive cuts per governed surface, a real shared executor shape under M9/shared, a budgeting matrix, and a consistent closed-row lifecycle for `runtime_issues`.

The matrix packaging remains correct. v6.10 runtime-governance semantics remain authoritative. Frontend remains appropriately non-central. `revenue_daily` remains correctly protected as business/system-of-record history rather than being collapsed into disposable residue.

# 2. What is now resolved

- **REQ acceptance coverage is now explicit.** The synthesis doc now includes a REQ-answer map that points each acceptance question to the canonical matrix location.
- **`runtime_issues` lifecycle contradiction is resolved.** Master, M9, and DB now align on one rule: closed rows are direct-deleted 30 days after `closed_at`, with no archive lane because issue-cycle truth remains in `runtime_events`.
- **Delete vs archive decisions are now committed per surface.**
  - `device_command_logs`: 90-day hot window, archive-first into `device_command_logs_archive`, then delete hot row.
  - `gateway_alarm_events`: 180-day hot window, archive-first into `gateway_alarm_events_archive`, then delete hot row.
  - `backfill_requests`: active rows retained; terminal rows (`completed`, `failed`) direct-delete after 14 days; no archive.
  - `runtime_issues`: closed rows direct-delete after 30 days; no archive.
  - `revenue_daily`: preserve in hot storage in v6.10.1; no TTL and no archive move in this release.
- **Concrete retention / rotation values now exist.** Docker stays on `json-file` with `max-size: 10m` and `max-file: 5`, uniform across `solfacil-bff`, `solfacil-db`, `solfacil-m1`, and `solfacil-redis`.
- **Budgeting matrix now exists.** M9 now includes a canonical policy/budgeting matrix covering storage class, live size, growth driver, hot-window rule, post-window path, and executor.
- **Executor shape is now real enough for PLAN.** M9 commits to one hourly shared storage-retention executor by extending the existing v6.10 runtime-retention job, with named phase ordering, archive-before-delete sequencing, independent per-phase commits, duplicate-safe reruns, and runtime-governance surfacing on executor failure.
- **Safety predicates were tightened.**
  - `device_command_logs` pending rows are not swept by age alone.
  - `backfill_requests` terminal-state eligibility is explicitly defined.
- **Archive-table convention is now documented.** DB names the archive tables, additive schema posture, preserved source identity, mirrored hot-table columns plus `archived_at` and `archive_reason`, and idempotent insert expectations.
- **Frontend ambiguity is resolved.** FE explicitly states that closed `runtime_issues` cleanup does not require a frontend change and that `#runtime` remains a hot-state/operator-triage surface.

# 3. Remaining blocking issues

None.

I do not see any remaining design-level blocker that should stop the work from proceeding to PLAN.

# 4. Non-blocking weaknesses / cautions

- **The budgeting matrix is structurally correct, but some growth inputs remain qualitative.** That does not block PLAN, but PLAN should turn the matrix into more concrete rows/day or MB/day estimates where practical so storage ceilings can be forecast more precisely.
- **Executor cadence should be carried carefully into PLAN.** The design now commits to an hourly shared executor extension, but PLAN should make sure that cadence is reconciled cleanly with the existing v6.10 retention job schedule and rollout behavior.
- **Eligibility-scan index needs should be checked during PLAN.** DB correctly stays additive and avoids premature repartitioning, but PLAN should explicitly verify whether helper indexes are needed to keep retention/archive scans efficient at the 90d / 180d / 14d / 30d cut lines.
- **M4 still mentions optional future archive hooks for `revenue_daily`.** That is not a contradiction because the master rule for v6.10.1 is still preserve-in-hot/no TTL/no archive move, but PLAN should avoid drifting into scope expansion for `revenue_daily` in this release.
- **The synthesis doc remains intentionally thin.** That is acceptable because the matrix docs remain canonical, but reviewers will still rely on the matrix set rather than the synthesis note for full detail.
