# v6.9 header ownership fix note

Date: 2026-04-17
Scope: `solfacil.alwayscontrol.net` production ingress cleanup

## Problem

Public 2xx responses were returning **two sets** of security headers:

- **App (Solfacil BFF / Express)** emitted:
  - `Content-Security-Policy: default-src 'none'; script-src 'self'; style-src 'self'; img-src 'self' data:; font-src 'self'; connect-src 'self'; frame-ancestors 'none'; base-uri 'self'; form-action 'self'`
  - `X-Frame-Options: DENY`
  - `X-Content-Type-Options: nosniff`
  - `Referrer-Policy: strict-origin-when-cross-origin`
  - `Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=()`

- **Shared-host nginx global `http {}`** also emitted:
  - `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`
  - `X-Frame-Options: SAMEORIGIN`
  - `X-Content-Type-Options: nosniff`
  - a broader `Content-Security-Policy` allowing `'unsafe-inline'`, Google/Cloudflare/fonts, aliyuncs, etc.

This created duplicated/conflicting `CSP` and `X-Frame-Options` on public responses.

## What was verified before fixing

### Source of duplication

1. Production app direct response from `127.0.0.1:3100` already contained the strict Solfacil app headers.
2. Production nginx config (`/docker/nginx/conf/nginx.conf`, bind-mounted into `nginx-web`) defined global `add_header` rules in `http {}`.
3. `solfacil.alwayscontrol.net` proxied to `http://localhost:3100` and had **no local `add_header` override**, so it inherited the global nginx header set.

### Browser semantics

Empirically tested with a local lab server:

- Multiple `Content-Security-Policy` headers were **not** treated as “last one wins”.
- Browser behavior followed the **stricter effective policy** (inline script/style still blocked if one policy forbade them).
- Dual `X-Frame-Options: DENY` + `SAMEORIGIN` resulted in blocked iframe embedding in browser tests.

Conclusion before fix:
- Current public behavior was still effectively strict.
- But header **ownership was wrong**: dual authorities, poor auditability, future maintenance trap.

## Correct target truth

- **nginx owns HSTS only** (TLS-terminating ingress concern)
- **Solfacil app owns all other browser-hardening headers**

This matches the app middleware design and test intent in:
- `backend/src/bff/middleware/security-headers.ts`

## Fix applied

Changed only the `server` block for `solfacil.alwayscontrol.net` in:
- `/docker/nginx/conf/nginx.conf`

Added:

```nginx
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload";
```

inside the Solfacil `server { listen 443 ssl; server_name solfacil.alwayscontrol.net; ... }` block.

## Why this exact fix

nginx `add_header` inheritance rule:
- if a child context has **no** `add_header`, it inherits parent headers
- if a child context defines **any** `add_header`, it stops inheriting the parent `add_header` set

Therefore this is the minimal safe fix:
- preserve ingress-owned HSTS for Solfacil
- cut inheritance of shared-host global CSP/XFO/nosniff headers
- avoid changing global nginx behavior for unrelated sites on the same host

### nginx validation

- `nginx -t` passed
- `nginx -s reload` succeeded
- backup created:
  - `/docker/nginx/conf/nginx.conf.bak-20260417-074109`

### Public response validation

Re-checked:
- `GET /login.html`
- `GET /index.html`
- `POST /api/auth/login`

Observed final state:

- **From app only**:
  - CSP (strict Solfacil policy)
  - X-Frame-Options: DENY
  - X-Content-Type-Options: nosniff
  - Referrer-Policy
  - Permissions-Policy
- **From nginx only**:
  - Strict-Transport-Security

Duplicate/conflicting nginx CSP/XFO headers no longer appeared.

### Functional regression check

Browser login and fleet landing page still worked after the fix:
- login success
- cookie session flow intact
- `localStorage.solfacil_jwt === null`
- fleet dashboard loads normally
- no homepage system error introduced by header cleanup

## Final engineering ruling

This fix was worth doing because the previous state was:
- not a release blocker
- but not engineering-correct

New state is correct:
- single authority for app-layer security headers
- ingress retains only HSTS
- no collateral change to unrelated shared-host sites
