現有架構上下文（REQ 必須承認的現實）

這不是單純的 login 頁修補。Solfacil 現有設計是矩陣式架構，v6.9 的安全修復必須對齊既有架構文檔，而不是脫離模組邊界另起一套抽象說法。

  • 前端主文檔：docs/FRONTEND_ARCHITECTURE_v6.8.md
  • 後端主藍圖：design/backend_architecture/00_MASTER_ARCHITECTURE_v6.8.md
  • 矩陣模組：M1 IoT Hub、M2 Optimization、M3 DR Dispatcher、M4 Market Billing、M5 BFF、M6 Identity、M7 Open API、M8 Admin Control、09 Shared Layer、10 Database Schema

對 v6.9 直接相關的邊界至少包括：Browser Frontend、Nginx Public Boundary、M5 BFF、M6 Identity、Shared Layer token verification、以及 DB-backed user/org/role mapping。

問題定義

1. 前端主認證模型仍依賴 localStorage JWT

登入成功後 token 會進入 localStorage，後續 API 由前端 JavaScript 讀取並拼成 Authorization: Bearer ...。這代表任何成功執行的 XSS 都可能直接偷走主認證憑證。

2. 後端已設 HttpOnly cookie，但系統仍停留在 split auth model

後端已經有較安全的 cookie 路徑，但同時仍把 token 回傳給前端 JS 使用。這不是完成遷移，而是半遷移狀態。

3. Login abuse protection 沒有形成明確應用層保護

目前可見登入失敗行為沒有展示出足夠清晰的應用層節流/鎖定設計。即使基礎設施外層有保護，也不能替代應用層設計本身。

4. Browser hardening headers 不完整

production login 面未觀察到 CSP、anti-frame、nosniff、referrer-policy、permissions-policy 等完整基線。這屬於必要配置層 closure，不是可有可無的裝飾。

5. JWT secret handling 存在危險 fallback

當 JWT_SECRET 缺失時，程式仍允許退回預設開發 secret。這讓 production misconfiguration 有可能無聲降級成危險運行模式。

REQ distilled

  • v6.9 必須收斂為單一、清楚、可推理的 production auth model。
  • 主認證狀態不得再以 JS 可讀 bearer token 形式作為 production 主路徑。
  • 對外 web surface 必須具備明確 browser security baseline。
  • 登入面必須具備可驗證的 abuse mitigation。
  • auth secret handling 必須 fail-safe，不能默默 fallback。
  • 修復必須由測試與部署驗證保護，避免回退。

Non-goals

  • 與安全無直接關聯的大面積功能重構
  • 只為了看起來安全而做的 scanner theater
  • 不帶安全價值的 UI cosmetic refresh
  • 用「兼容性」當理由保留已知錯誤安全設計

Design phase 必答題

target auth/session architecture frontend request model after migration cookie semantics and CSRF stance browser headers at app vs proxy layer login rate-limit / lockout model JWT secret validation and startup rules test and deployment verification plan rollback and residual risk treatment

Design 文檔交付要求

因為現有架構是矩陣式，所以 DESIGN 不能只交一份模糊總說明。它至少要明確說出哪些現有架構文檔會被 v6.9 更新。

  • 一份 v6.9 master security design：目標 auth/session、migration、verification、rollback
  • 一份 frontend security design update：從 FRONTEND_ARCHITECTURE_v6.8.md 派生，覆蓋 login、authenticated request model、logout/session expiry、browser-side implications
  • 一份 backend auth-boundary design update：覆蓋 M5 + M6 + Shared Layer 的 middleware、cookie semantics、token verification、SSE impact、secret fail-fast
  • 一份 deployment/security-boundary design update：覆蓋 reverse proxy headers、runtime env/secret requirements、live-surface verification

允許做成一份 merged DESIGN + 清楚的矩陣映射，也允許 master + addenda；但不允許對文檔責任保持含糊。

验收标准

  • 已发現且可修復的代碼層安全問題全部收口。
  • 必要配置層工作已落地，且不是只停留在 repo 理論上。
  • 系統不再依賴 localStorage bearer JWT 作為 production 主認證模型。
  • live surface 上實際可驗證安全頭存在且符合設計。
  • login abuse mitigation 已實作且可測。
  • unsafe JWT secret fallback 已移除或在 production-capable path 上變得不可能。
  • 任何殘留風險都已明確記錄，不再默許已知錯誤設計。
