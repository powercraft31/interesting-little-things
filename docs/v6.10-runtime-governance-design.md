# Solfacil v6.10 Runtime Governance Skeleton — DESIGN

> **Status**: DESIGN aligned with REQ `docs/v6.10-runtime-governance-req.md`
> **Baseline**: Solfacil v6.9 closure state
> **Target release**: v6.10
> **Date**: 2026-04-18
> **Scope posture**: A / A / B — internal-ops-first, narrow-but-deep, backend-contract-first
> **Anchored architecture**: v6.9 normalized matrix lineage (M1–M9)

This document is DESIGN, not PLAN. It defines the shape of the v6.10 runtime-governance spine — where it lives, what it contracts, and how existing modules emit into it — without prescribing task-level implementation work.

---

## 0. Executive Design Judgment

v6.10 inserts one **runtime-governance spine (运行治理脊柱)** into the normalized v6.9 matrix lineage. The spine is deliberately thin: a canonical event contract, a lifecycle/severity taxonomy, an active-issue projection, a health summary, and a small registry of self-checks. It is **not** a second brain, not an ops monolith, and not a UI feature.

The architectural commitment:

1. **The spine lives in the shared layer (M9)** as a new sub-package, not as a module. Shared-layer placement is the only placement consistent with the REQ's "insert a common spine into current matrix architecture" instruction — M9 is already the only layer every module depends on, and it already owns cross-cutting contracts (`types/*`, `db.ts`, `tenant-context.ts`).
2. **Existing modules stop inventing operational judgment and start emitting facts.** M1/M3/M5/M9 existing hot paths are adapted to call `emit(...)` with standardized events. Governance semantics (dedup, lifecycle, retention, active-issue aggregation) move into the spine. Business logic ownership stays in the modules.
3. **Storage is additive and append-first.** Three new operational tables (`runtime_events`, `runtime_issues`, `runtime_self_checks`) plus one periodic snapshot table (`runtime_health_snapshots`). No existing table is restructured. No existing query path is rerouted.
4. **Operator surface is a backend contract first.** `/api/runtime/*` endpoints are the phase-1 product. The single minimal UI page (`#runtime`) exists only to prove the spine is operationally usable to internal engineering/ops. Polish, charts, alert routing, customer-facing surfaces — all deferred.
5. **Platform runtime is strictly separated from domain alarms.** `gateway_alarm_events`, P6 Alerts, and strategy intents (P5) are untouched. A namespace convention on `event_code` (`bff.*`, `m1.*`, `m9.*`, `db.*`, `ingest.*`, `dispatch.*`, `scheduler.*`, `external.*`) prevents cross-contamination with domain-alarm codes.

Bluntly: v6.10 makes the platform able to speak about itself without asking modules to invent their own grammar.

- **M9 already owns the dual-pool DB, RLS-scoped query helpers, and migrations** — the spine's persistence layer rides on `getServicePool()` / `withTransaction()` with no new connection strategy.

## 3. Platform Runtime vs Domain Alarms — strict separation

This separation is not documentary; it is enforced at four layers.

| Layer | Platform runtime (v6.10 spine) | Domain alarms (existing) |
|-------|--------------------------------|---------------------------|
| **Storage** | `runtime_events`, `runtime_issues`, `runtime_self_checks`, `runtime_health_snapshots` | `gateway_alarm_events`, `offline_events`, `gateway_outage_events`, strategy_intents/posture_overrides |
| **Write path** | `shared/runtime/emit.ts` only | M1 alarm handler, M2 strategy engine, M3 timeout checker |
| **Read path** | `/api/runtime/*` | `/api/alerts`, `/api/alerts/summary`, `/api/p5/*`, `/api/fleet/offline-events` |
| **UI surface** | `#runtime` (new) | P5 Strategy, P6 Alerts (unchanged) |
| **Event-code namespace** | `bff.*`, `db.*`, `redis.*`, `ingest.*`, `dispatch.*`, `scheduler.*`, `external.*`, `m9.*` | N/A — domain alarms do not use `event_code` |
| **Severity taxonomy** | 5-tier platform scale (§5.1) | Domain-specific alarm levels on `gateway_alarm_events.alarm_level` |

**Rule**: the spine never reads from or writes to `gateway_alarm_events`, `offline_events`, `strategy_intents`, `posture_overrides`, or P6 endpoints. A fleet alarm is **not** a platform runtime event. If an operator needs to correlate them, the drill-down links out — it does not duplicate the data.

A `gateway.heartbeat_missed` platform event and a `gateway_alarm_events` row may coexist for the same gateway in the same minute. That is intentional: one answers "is the platform ingest loop healthy?", the other answers "does this customer's site have a problem?". Collapsing them would destroy both.

## 4. Canonical Runtime Event Schema

### 4.1 Type definition (lives in `shared/types/runtime.ts`)

```ts
export type RuntimeSeverity =
  | 'info'        // informational; never alone produces an issue
  | 'notice'      // anomaly worth retaining but not degrading
  | 'warning'     // subsystem imperfect, fully functional
  | 'degraded'    // subsystem impaired, partially functional
  | 'critical';   // subsystem failing or failed

export type RuntimeLifecycle =
  | 'detected'    // first observation
  | 'ongoing'     // re-observed while still open
  | 'recovered'   // spine observed recovery signal or self-check pass
  | 'closed'      // operator-closed or retention-TTL closed
  | 'suppressed'; // operator-declared mute (not auto-emitted)

export interface RuntimeEvent {
  readonly event_id: string;          // ULID, client-generated
  readonly event_code: string;        // namespaced, e.g. "db.query.timeout"
  readonly source: string;            // component id, e.g. "bff", "m1.telemetry", "m3.dispatcher"
  readonly severity: RuntimeSeverity;
  readonly lifecycle_hint: 'open' | 'recovery' | 'heartbeat';
  readonly occurred_at: string;       // ISO-8601 UTC, emitter clock
  readonly observed_at: string;       // ISO-8601 UTC, spine clock (set by emit.ts)
  readonly fingerprint: string;       // dedup key; see §4.3
  readonly correlation_id?: string;   // optional trace/request id
  readonly tenant_scope: string | null; // org_id if tenant-relevant, null = platform-wide
  readonly summary: string;           // single-line operator-facing English/PT-BR text
  readonly detail: Record<string, unknown>; // machine-readable structured payload
}
```

### 4.2 Event-code namespace (registry lives in `runtime/contract.ts`)

Phase-1 namespaces are finite and closed. Adding a new code requires adding a row in the registry — this is the single governance gate against event-code sprawl.

| Namespace | Owner | Examples (phase-1) |
|-----------|-------|--------------------|
| `bff.*` | M5 BFF | `bff.boot.ok`, `bff.handler.unhandled_exception`, `bff.auth.jwt_invalid_burst` |
| `db.*` | M9 shared | `db.pool.exhausted`, `db.query.timeout`, `db.connection.failed`, `db.self_check.ok` |
| `redis.*` | M9 shared | Namespace reserved only in phase-1; examples such as `redis.connection.failed` and `redis.self_check.ok` remain dormant until redis is confirmed as an onboarded runtime dependency in a later release |
| `ingest.*` | M1 | `ingest.telemetry.stale`, `ingest.fragment.backlog`, `ingest.parser.failed` |
| `dispatch.*` | M3 | `dispatch.dispatcher.heartbeat`, `dispatch.timeout_checker.heartbeat`, `dispatch.ack.stall` |
| `scheduler.*` | M2/M4 | `scheduler.schedule_generator.heartbeat`, `scheduler.billing_job.heartbeat`, `scheduler.aggregator.lag` |
| `external.*` | M7 | `external.webhook.delivery_failed`, `external.weather.fetch_failed` |
| `m9.*` | M9 | `m9.runtime.self_emit_dropped` (spine self-protection fact) |

Domain alarm codes (`alarm.gateway.offline` etc.) are **not** in this registry. They belong to the alarm subsystem.

### 4.3 Fingerprint rule

```
fingerprint = hash(event_code ‖ source ‖ tenant_scope ‖ dedup_dimension?)
```

- `dedup_dimension` is declared per `event_code` in the registry. Default is empty.
- Example: `ingest.telemetry.stale` declares `dedup_dimension = gateway_id`, so two distinct gateways produce two distinct issues, but repeat observations on the same gateway collapse into one.
- Example: `db.pool.exhausted` declares no dimension, so all BFF pool-exhaustion hits in the same tenant fold into one active issue.

Fingerprint width is the primary knob for "event spam vs issue collapse". Starting narrow is deliberate; widening is a post-phase-1 review point (§13).

### 4.4 Emit contract

```ts
// shared/runtime/emit.ts
export function emit(evt: Omit<RuntimeEvent, 'event_id' | 'observed_at' | 'fingerprint'>): void;
```

- **Synchronous call site, asynchronous persistence.** `emit()` enqueues to an in-process ring buffer; a background flusher persists to DB via service pool. Caller is not blocked. Call-site failure must never propagate.
- **Bounded**. Ring buffer has a hard cap (proposed default 1024). Overflow drops oldest with an internal `m9.runtime.self_emit_dropped` counter-event.
- **Feature-flag gated.** When `RUNTIME_GOVERNANCE_ENABLED=false`, `emit()` is a no-op.
- **No throw**. Any persistence error is caught, logged to stdout (fallback), and counted. The spine cannot fail a business request.

## 5. Severity Taxonomy and Lifecycle State Model

### 5.1 Severity taxonomy

| Severity | Platform meaning | Example |
|----------|------------------|---------|
| `info` | Normal-operation fact. Retained only if needed for lifecycle recovery evidence. | `bff.boot.ok`, `scheduler.billing_job.heartbeat` (on success) |
| `notice` | Worth retaining for triage; not a degradation. | `bff.auth.jwt_invalid_burst` (elevated but below abuse threshold) |
| `warning` | Subsystem imperfect but functional; operator should notice, not respond. | `ingest.fragment.backlog` (non-zero but below threshold) |
| `degraded` | Subsystem impaired; partial-service operation. | `db.pool.exhausted` (queries succeeding but queued) |
| `critical` | Subsystem failing or failed; operator must respond. | `db.connection.failed`, `ingest.telemetry.stale` past threshold |

Severity is declared by the emitter but validated against the event-code registry. If an emitter declares a severity outside the registry-allowed set for that code, the spine coerces to the registry default and emits an `m9.runtime.self_*` notice. This is the enforcement mechanism preventing "module-by-module severity semantics with no central taxonomy" (REQ §10.3 anti-pattern).

Severity is **not** additive across events in the same issue. The issue's current severity is the max severity observed in its current open cycle.

### 5.2 Lifecycle state model

```
                ┌─────────────┐   recovery_hint / self-check passed
                │             │──────────────────────┐
   emit(open)   ▼             │                      ▼
              ┌─────────┐    ┌─────────┐    ┌───────────┐
              │detected │──► │ ongoing │──► │ recovered │
              └────┬────┘    └─────────┘    └─────┬─────┘
                   │                               │ TTL_recovered OR operator_close
                   │ operator_mute                 │
                   ▼                               ▼
              ┌────────────┐                 ┌─────────┐
              │ suppressed │────────────────►│ closed  │
              └─────┬──────┘  close / TTL    └─────────┘
                    │
                    └──── new open event ───► detected (same row, next cycle)
```

Transition rules (all owned by `runtime/projection.ts`, not by emitters):

| From → To | Trigger |
|-----------|---------|
| (none) → `detected` | First event with `lifecycle_hint='open'` for a new fingerprint |
| `detected` → `ongoing` | Any subsequent `open` event before recovery |
| `ongoing` / `detected` / `suppressed` → `recovered` | Event with `lifecycle_hint='recovery'` OR self-check for the same subsystem passes after failure |
| `recovered` → `closed` | Retention TTL expiry (default 24h) OR operator close |
| `ongoing` / `detected` → `closed` | Operator close (explicit) |
| `ongoing` / `detected` / `recovered` → `suppressed` | Operator mute; row remains queryable but is removed from active-summary posture while suppression is in force |
| `suppressed` → `closed` | Operator close OR retention expiry after recovery/stale closure logic |
| `closed` / `recovered` / `suppressed` → `detected` | New `open` event for the same fingerprint starts a new cycle on the same projection row: clear `recovered_at` / `closed_at` / suppression fields, increment `cycle_count`, reset current-cycle counters |

Notes:

- `heartbeat` lifecycle_hint does **not** open or transition issues; it is consumed only by self-check liveness logic (§6.2).
- No auto-suppression. Silencing is always operator-explicit, to avoid the junk-drawer failure mode.
- `suppressed` is a visibility/action posture, not a second kind of closure. `closed` means the cycle is finished; `suppressed` means the cycle still exists but is intentionally muted from active summary output until recovery, closure, or suppression expiry.
- The spine never reasons about "same root cause across event codes". Correlation across codes is an out-of-scope inference problem.

## 6. Health Summary and Self-Check Model

### 6.1 Platform health summary

The health summary is a **derived read**, not a persisted truth. It is computed on demand by `runtime/health.ts` from:

1. Active `runtime_issues` (not `closed`, not `suppressed`) grouped by `source` and max-severity.
2. Recent `runtime_self_checks` results (see §6.2).
3. Last `runtime_health_snapshots` row for the trailing posture (last-known-good).

Shape (API response, see §9):

```jsonc
{
  "overall": "ok" | "warning" | "degraded" | "critical" | "disabled",
  "generated_at": "ISO-8601",
  "components": [
    { "source": "bff",          "state": "ok",       "open_count": 0, "max_severity": null,       "last_check_at": "..." },
    { "source": "db",           "state": "degraded", "open_count": 1, "max_severity": "degraded", "last_check_at": "..." },
    { "source": "m1.ingest",    "state": "warning",  "open_count": 2, "max_severity": "warning",  "last_check_at": "..." },
    { "source": "m3.dispatch",  "state": "ok",       "open_count": 0, "max_severity": null,       "last_check_at": "..." },
    { "source": "scheduler",    "state": "ok",       "open_count": 0, "max_severity": null,       "last_check_at": "..." }
  ],
  "critical_open": [ /* issue summaries */ ],
  "recovered_recent": [ /* issue summaries, last 24h */ ],
  "self_check": {
    "last_run_at": "...",
    "last_full_pass_at": "...",
    "stale": false
  }
}
```

`overall: "disabled"` is an **API-only top-level posture** returned only while `runtime_governance_enabled=false`. It is not a component state, not a severity, and not a persisted value in `runtime_health_snapshots`.

Derivation rules:

- `overall` is the highest component `state` across components, bounded by severity.
- `components` set is **fixed** in phase-1 to the onboarded sources (§8). Unknown sources are ignored, preventing one rogue emitter from polluting the summary.
- `last_full_pass_at` is the most recent snapshot where all self-checks passed and no component was `degraded` or `critical`.

### 6.2 Self-check registry

Self-checks are the spine's answer to "don't let the absence of bad events be confused with health". A quiet subsystem is not a healthy subsystem unless a self-check confirms it.

Each self-check has:

| Field | Meaning |
|-------|---------|
| `check_id` | Stable identifier |
| `source` | Component identifier the check answers for |
| `cadence` | Proposed interval (e.g. `60s`, `5m`, `1h`) |
| `run_host` | `bff` (in-process) or `cron` (scheduled worker) |
| `emits_on_fail` | The `event_code` it emits on failure |
| `emits_on_recover` | The `event_code` it emits on recovery |

Read-model decision for phase-1: self-check latest state is **stored explicitly** in `runtime_self_checks`, not reconstructed ad hoc from `runtime_events`. The events stream still records failures/recoveries for historical context, but `/api/runtime/self-checks` and health-summary freshness read from the table.

Phase-1 mandatory self-check set:

| check_id | source | cadence | run_host | Purpose |
|----------|--------|---------|----------|---------|
| `db.app_pool.reachable` | `db` | 60s | `bff` | `SELECT 1` on app pool |
| `db.service_pool.reachable` | `db` | 60s | `bff` | `SELECT 1` on service pool |
| `db.critical_query` | `db` | 5m | `cron` | Representative RLS-scoped read against a known org |
| `ingest.freshness` | `m1.ingest` | 5m | `cron` | Max(`recorded_at`) across active gateways vs wall-clock |
| `dispatch.loop.alive` | `m3.dispatch` | 2m | `cron` | Last `dispatch.dispatcher.heartbeat` within 2× cadence |
| `scheduler.jobs.alive` | `scheduler` | 15m | `cron` | Each cron emitter produced a heartbeat within its expected window |
| `bff.listen` | `bff` | 30s | `bff` | Self-reflective: emit heartbeat while serving |

Non-goals for self-check in v6.10: synthetic user flows, chaos probes, external-dependency (weather/CCEE) liveness, load-side-effect checks.

`runtime_self_checks` semantics:

- one mutable row per registered `check_id`
- every run updates `last_run_at`, `last_status`, `last_duration_ms`, and `latest_detail`
- a passing run updates `last_pass_at`
- a failing run increments `consecutive_failures` and may emit the configured failure event
- the first passing run after failures resets `consecutive_failures`, updates `last_pass_at`, and may emit the configured recovery event

This keeps the self-check read path precise without forcing event replay for simple "what is the latest status of each check?" questions.

### 6.3 Health snapshot persistence

`runtime_health_snapshots` is written on a 5-minute cadence (cron-driven) and captures the derived summary. This enables:

- "Last time the platform was fully green" — directly answered from `last_full_pass_at`.
- Simple historical posture review without needing a time-series database.
- Trend behavior **deferred** to a later release; v6.10 writes the data, does not chart it.

## 7. Data Model — Active Issue vs Retained History

Three runtime tables for three distinct query patterns, plus one snapshot table for derived history.

### 7.1 `runtime_events` — retained raw history

Append-only. Partitioned by `observed_at` monthly RANGE, consistent with `telemetry_history` partitioning pattern.

| Column | Type | Notes |
|--------|------|-------|
| `event_id` | VARCHAR(32) | ULID, **PK(event_id, observed_at)** for partition compat |
| `event_code` | VARCHAR(80) | Registry-validated |
| `source` | VARCHAR(40) | |
| `severity` | VARCHAR(16) | `info` / `notice` / `warning` / `degraded` / `critical` |
| `lifecycle_hint` | VARCHAR(16) | `open` / `recovery` / `heartbeat` |
| `occurred_at` | TIMESTAMPTZ | Emitter clock |
| `observed_at` | TIMESTAMPTZ | Spine clock, partition key |
| `fingerprint` | VARCHAR(64) | Hash |
| `correlation_id` | VARCHAR(64) NULL | |
| `tenant_scope` | VARCHAR(50) NULL | org_id or NULL (platform-wide) |
| `summary` | TEXT | |
| `detail` | JSONB | |

Indexes: `(observed_at DESC)`, `(fingerprint, observed_at DESC)`, `(source, observed_at DESC)`, `(severity, observed_at DESC)`.

Retention: monthly partitions; phase-1 retains 90 days. Older partitions detach + drop via the same pattern as telemetry.

### 7.2 `runtime_issues` — active projection

`runtime_issues` follows **Model A** for phase-1: one mutable projection row per `fingerprint`. Re-open does **not** allocate a second issue identity. Instead, the same row is updated to begin a new cycle, while prior cycle history remains in `runtime_events`. This keeps `/api/runtime/issues/:fingerprint` stable and preserves operator annotations on the same issue identity.

| Column | Type | Notes |
|--------|------|-------|
| `fingerprint` | VARCHAR(64) | **PK** |
| `event_code` | VARCHAR(80) | Current-cycle primary code for this fingerprint |
| `source` | VARCHAR(40) | |
| `tenant_scope` | VARCHAR(50) NULL | |
| `cycle_count` | INTEGER | Starts at 1; increments when a closed/recovered/suppressed issue is re-opened |
| `current_cycle_started_at` | TIMESTAMPTZ | Start timestamp of current cycle |
| `first_detected_at` | TIMESTAMPTZ | First-ever detection for this fingerprint |
| `last_observed_at` | TIMESTAMPTZ | Last event seen for this fingerprint |
| `recovered_at` | TIMESTAMPTZ NULL | Current-cycle recovery timestamp |
| `closed_at` | TIMESTAMPTZ NULL | Current-cycle closure timestamp |
| `suppressed_until` | TIMESTAMPTZ NULL | Optional operator mute expiry |
| `state` | VARCHAR(16) | `detected` / `ongoing` / `recovered` / `closed` / `suppressed` |
| `current_severity` | VARCHAR(16) | Max severity observed in current cycle |
| `observation_count` | INTEGER | Events that hit this fingerprint in current cycle |
| `summary` | TEXT | Latest summary text |
| `latest_detail` | JSONB | Latest detail payload (not merged; replaced) |
| `operator_note` | TEXT NULL | Most recent operator annotation |
| `operator_actor` | VARCHAR(100) NULL | Last operator who acted |
| `updated_at` | TIMESTAMPTZ | |

Indexes: `(state, last_observed_at DESC)`, `(source, state)`, `(tenant_scope, state)` with a partial index on `state IN ('detected','ongoing','recovered')` for the active-query hot path.

Projection semantics:

- first `open` event for a new fingerprint inserts the row with `cycle_count=1`, `current_cycle_started_at=observed_at`, `state='detected'`
- subsequent `open` events within the same cycle update `last_observed_at`, `observation_count`, `summary`, `latest_detail`, and severity maxing
- a recovery signal updates the same row to `state='recovered'` and sets `recovered_at`
- operator close or TTL close updates the same row to `state='closed'` and sets `closed_at`
- a later `open` event for that same fingerprint clears `recovered_at`, `closed_at`, and `suppressed_until`, resets `observation_count`, sets `current_cycle_started_at` to the new observation, increments `cycle_count`, and moves the row back to `state='detected'`

**Why a projection and not a view**: the lifecycle transition is stateful even under the single-row model. A view cannot cleanly express per-fingerprint mutable state such as `cycle_count`, suppression expiry, operator notes, and TTL-driven closure without replaying event history on every read. A projection table with explicit upsert gives O(log n) reads and stable row identity for operator actions.

### 7.3 Retention, dedup, TTL

| Rule | Value (phase-1 default) |
|------|-------------------------|
| Raw event retention | 90 days (monthly partitions) |
| Recovered issue auto-close TTL | 24 hours after `recovered_at` |
| Stale-detected auto-close (no observations, not recovered) | 72 hours — emits `m9.runtime.auto_closed_stale` info |
| Max open issues per source per tenant | 200; spine refuses new fingerprint allocation above this and emits `m9.runtime.issue_quota_exceeded` warning |
| Fingerprint dedup window | Infinite across the fingerprint identity; repeated observations update the same row, with historical cycles distinguished by `cycle_count` and event history |

Retention defaults are tunable via `feature_flags` rows (`runtime.retention.*`). DESIGN does not commit the numbers to DDL.

### 7.4 `runtime_self_checks` — latest self-check read model

| Column | Type | Notes |
|--------|------|-------|
| `check_id` | VARCHAR(80) | **PK**; registry identifier |
| `source` | VARCHAR(40) | Component summarized by the check |
| `run_host` | VARCHAR(16) | `bff` / `cron` |
| `cadence_seconds` | INTEGER | Expected cadence for staleness math |
| `last_status` | VARCHAR(16) | `pass` / `fail` / `unknown` |
| `last_run_at` | TIMESTAMPTZ NULL | Most recent attempted execution |
| `last_pass_at` | TIMESTAMPTZ NULL | Most recent passing execution |
| `last_duration_ms` | INTEGER NULL | Useful for triage, not SLO math |
| `consecutive_failures` | INTEGER | Resets on pass |
| `latest_detail` | JSONB | Last structured result payload |
| `updated_at` | TIMESTAMPTZ | |

Indexes: `(source, last_status)`, `(last_run_at DESC)`.

This table is intentionally latest-state only. Historical run-by-run evidence belongs in `runtime_events` and in periodic snapshots, not in an ever-growing per-check history table for phase-1.

### 7.5 `runtime_health_snapshots` — periodic derived history

| Column | Type | Notes |
|--------|------|-------|
| `id` | BIGSERIAL | **PK** |
| `captured_at` | TIMESTAMPTZ | Unique |
| `overall` | VARCHAR(16) | `ok` / `warning` / `degraded` / `critical` |
| `component_states` | JSONB | Array of `{ source, state, open_count, max_severity }` |
| `critical_open_count` | INTEGER | |
| `self_check_all_pass` | BOOLEAN | |
| `snapshot_source` | VARCHAR(40) | `cron` / `boot` / `manual` |

Retention: 30 days.

### 7.6 Indexes, RLS, and pool usage

- All four runtime tables live under the service pool (BYPASSRLS). Platform runtime is **not** tenant-scoped by default; `tenant_scope` is an informational field, not a security boundary. This diverges from the tenant-RLS pattern used by domain tables, and it is deliberate: a platform-wide DB pool exhaustion issue must be visible across the SOLFACIL admin view without RLS gating.
- Operator read endpoints (§9) execute as `SOLFACIL_ADMIN` and use the service pool. Non-admin roles have no access to `/api/runtime/*` in phase-1.
- No RLS policies are created for `runtime_*` tables. This mirrors `algorithm_metrics`-class choices and is justified by the access model.

## 8. Phase-1 Onboarding Set and Justification

Phase-1 commits to **narrow but deep** onboarding. The five sources below are selected because they cover the full runtime critical path — from request entry, through data access, through ingest, through dispatch, through cron-driven correctness — with minimal overlap and maximum truth density.

| # | Source | What gets emitted | Why it is phase-1 |
|---|--------|-------------------|-------------------|
| 1 | **BFF request lifecycle** (`bff.*`) | Unhandled exception boundary, auth middleware invalid-JWT burst, `wrapHandler` catch-all failures, boot-time self-emit | Top of every user-facing path. If the BFF is silent while handlers are erroring, every downstream signal is suspect. Covers ~all synchronous user-facing runtime. |
| 2 | **Shared DB pool** (`db.*`) | App-pool & service-pool reachability, query timeouts, connection failures, self-check results | Every request and every background job depends on DB. Failures here masquerade as everything-is-broken. Single instrumentation point (in `shared/db.ts`) covers M1/M2/M3/M4/M5/M7/M8 simultaneously. |
| 3 | **M1 ingest pipeline** (`ingest.*`) | Telemetry-stream stale-detection, fragment backlog, parser failure bursts, heartbeat-missed as platform fact | Ingest is the data backbone. Without it, the platform has no fresh truth to govern. Also proves the separation from `gateway_alarm_events` — the same incident produces both kinds of events without confusion. |
| 4 | **M3 dispatch loops** (`dispatch.*`) | `CommandDispatcher` and `TimeoutChecker` heartbeats, ack-stall detection | Dispatch loops are the clearest case where "system is running but no work is happening" masks failure. Heartbeat semantics make silence actionable. |
| 5 | **Scheduler / cron** (`scheduler.*`) | ScheduleGenerator, Telemetry5MinAggregator, TelemetryAggregator, DailyBillingJob heartbeats and error facts | Time-bounded correctness. Billing missing a daily run is a platform-runtime defect, not a business defect. Cron silence is the classic "operational semantics are inconsistent" failure (REQ §4 item 2). |

Explicitly **not in phase-1** (rationale):

| Deferred source | Why deferred |
|-----------------|--------------|
| External webhook delivery (M7 `external.*`) | One of-the-cuff emitter is fine (listed in §2.2), but building meaningful delivery-health semantics requires retry/backoff context better defined in a later phase. |
| Redis runtime path (`redis.*`) | Phase-1 reserves the namespace only. Redis is **not** part of the fixed phase-1 health-summary component set and no redis self-check row is expected unless a later release confirms redis as an actual governed runtime dependency. |
| M1 MQTT broker liveness | Broker health is an external-dependency question; v6.10 treats ingest freshness as the proxy. |
| M8 Admin operations | Low runtime-incident rate; not worth phase-1 cost. |
| P5 / P6 handler-specific emitters | Domain-adjacent; risk of confusing platform vs domain. Deliberately skipped. |

The onboarding set is closed at design time. Post-release review (§13) is the forum for expansion.

## 9. Backend Contract — Operator API Surface

All endpoints are registered in `backend/scripts/local-server.ts` and the BFF CDK stack, consistent with the normalized v6.9 BFF lineage (`05_BFF_MODULE_v6.9.md`). All are **SOLFACIL_ADMIN only** in phase-1 — this is an internal-ops-first release.

### 9.1 Route table

| # | Method | Route | Handler file | Description |
|---|--------|-------|--------------|-------------|
| R1 | GET | `/api/runtime/health` | `get-runtime-health.ts` | Derived platform health summary (§6.1 shape) |
| R2 | GET | `/api/runtime/issues` | `get-runtime-issues.ts` | Active issues with filters: `state` (default `detected,ongoing,recovered`), `source`, `severity`, `since`, `limit` (default 100, max 500) |
| R3 | GET | `/api/runtime/issues/:fingerprint` | `get-runtime-issue-detail.ts` | One issue by fingerprint + last N events (default 50) |
| R4 | GET | `/api/runtime/events` | `get-runtime-events.ts` | Raw event stream tail: `source`, `event_code`, `severity`, `since`, `limit` (default 200, max 1000) |
| R5 | GET | `/api/runtime/self-checks` | `get-runtime-self-checks.ts` | Latest state of every registered self-check, read from `runtime_self_checks` |
| R6 | POST | `/api/runtime/issues/:fingerprint/close` | `post-runtime-issue-close.ts` | Operator-close an issue (`SOLFACIL_ADMIN`); body: `{ note?: string }` |
| R7 | POST | `/api/runtime/issues/:fingerprint/suppress` | `post-runtime-issue-suppress.ts` | Operator-suppress; body: `{ reason: string, until?: ISO-8601 }` |
| R8 | POST | `/api/runtime/issues/:fingerprint/note` | `post-runtime-issue-note.ts` | Annotate an issue; body: `{ note: string }` |

Response envelope is the existing `ok()` / `fail()` from `shared/types/api`. Handlers use the service pool (not `queryWithOrg`) because the spine is platform-scoped.

### 9.2 What the surface intentionally does not offer in phase-1

| Deferred | Why |
|----------|-----|
| SSE push channel for runtime issues | Valuable, but requires new pg_notify wiring; defer to avoid conflating with REQ §13.3 "minimal UI" and phase-1 scope boundary. |
| Alert routing (email/Slack/webhook) | A downstream product decision; skeleton-first release should not prejudge it. |
| Per-tenant operator-facing subset | Phase-1 is internal-engineering/ops audience (REQ §13.1). Per-tenant read model is phase-2+. |
| Time-range aggregations / charts | Handled by snapshots table; surface is deferred. |
| Event ingest from external sources | No third-party emitter in phase-1. |

### 9.3 Webhook / internal ingestion

No webhook. The spine is an **in-process** sink: every emitter runs in a Solfacil process (BFF / M1 / cron). External-process ingestion is a phase-2 consideration.

## 10. Minimal UI Stance and Explicit Deferrals

Phase-1 ships **one** frontend module, `p7-runtime.js` at hash `#runtime`, registered in `PAGES` with `admin`-only visibility. This is deliberate minimalism consistent with REQ §13.3 ("backend contract first with minimal operator UI").

### 10.1 What the page is

A single vertical layout consuming the backend contract directly:

1. **Top band**: `overall` state chip + `last_full_pass_at` + self-check freshness indicator.
2. **Components row**: one compact state pill per source in the fixed phase-1 component list (§6.1 — BFF, DB, M1 ingest, M3 dispatch, Scheduler).
3. **Active issues table**: fingerprint, source, severity, state, first_detected_at, last_observed_at, observation_count, summary. Filter bar for `state` and `source`.
4. **Issue detail drawer**: recent events, detail payload (rendered as JSON), operator actions (close / suppress / note).
5. **Self-check panel**: each check with last-run timestamp and pass/fail.

Everything above is rendered from `/api/runtime/*`. No charts, no trends, no live push. Refresh is manual + a soft 30-second poll.

### 10.2 What the page is explicitly **not**

| Deferred UI concern | Owner of the deferral |
|---------------------|------------------------|
| Dashboard polish, visual hierarchy passes | Phase-2 UX |
| Trend charts / time-series graphs | Phase-2 (data is already captured by snapshots) |
| Real-time push via SSE | Phase-2 (contract surface not committed) |
| i18n full coverage beyond pt-BR + en | Phase-2 |
| Merging with P5 Strategy or P6 Alerts | Never — the separation is the point |
| Responsive mobile layout | Phase-2 |
| Customer-facing (non-admin) view | Phase-2+ (REQ §13.1 audience cut) |
| Accessibility pass beyond baseline keyboard nav | Phase-2 |

The minimal UI serves one function: prove that the backend contract is operationally usable. It is not the release center and must not grow into one.

## 11. Separation from P6 Alerts and Strategy Intents — enforced

Because this is the primary anti-pattern risk (REQ §10.3, §5.3), the separation is restated as a contract:

| Property | `runtime_*` (v6.10) | `gateway_alarm_events` (P6) | `strategy_intents` + `posture_overrides` (P5) |
|----------|--------------------|-----------------------------|-----------------------------------------------|
| Writer | `shared/runtime/emit.ts` exclusively | M1 alarm handler (MSG#5) | M2/M5 governance logic |
| Reader | `/api/runtime/*` exclusively | `/api/alerts*`, P6 UI | `/api/p5/*`, P5 UI |
| Semantic | "Is the platform's own plumbing healthy?" | "Does this gateway's device/site have a physical or protocol alarm?" | "Is a strategy-level business intent pending operator decision?" |
| Severity vocabulary | `info`/`notice`/`warning`/`degraded`/`critical` | `alarm_level` string from device | `urgency` + `governance_mode` |
| Lifecycle | Spine-owned (§5.2) | Device-driven `triggered_at` → `resolved_at` | Intent-status machine (see `types/p5.ts`) |
| Operator action surface | close/suppress/note | (P6 read-only in v6.8) | approve/defer/suppress/escalate |
| Cross-link | Detail payload MAY reference gateway_id / alarm id for operator drill-down | Does not reference runtime issues | Does not reference runtime issues |

No data migration moves rows between these systems. No background job cross-writes. If phase-2 introduces correlation, it is a read-side join in a new endpoint, not a storage merge.

## 12. Rollout, Mixed-Version, and Rollback Boundary (design-level)

v6.10 is a structural release; safe rollout is architected, not merely planned.

### 12.1 Feature flag

```
flag_name            = 'runtime_governance_enabled'
scope                = platform (org_id NULL)
default              = false  (phase-1 ship-disabled; enable post-migration)
```

All emit call-sites check the flag once at process boot + on SIGHUP or 60-second refresh. When flagged off, `emit()` short-circuits; read endpoints return an `overall: "disabled"` health response and empty collections.

### 12.2 DB migration

Single idempotent file `backend/src/shared/migrations/002_runtime_governance.sql`, following the pattern established in `001_p5_strategy_triggers.sql`:

- `CREATE TABLE IF NOT EXISTS` for all four runtime tables.
- Partition declarations for `runtime_events`: current + next 2 months, plus `default`.
- `CREATE INDEX IF NOT EXISTS`.
- No RLS policies (§7.6).
- Pre-seed `feature_flags` with `runtime_governance_enabled = false`.

The migration is safe to apply against v6.9 production without enabling the spine.

### 12.3 Mixed-version posture

During rollout, older containers (`solfacil-m1` pre-v6.10, pre-emit-aware) simply do not emit. The spine tolerates partial emitters — health summary's `components` array is fixed, but components with no events and no self-check show `state: "unknown"` with a freshness warning rather than erroring.

There is no wire-protocol change. No existing MQTT/SSE/HTTP endpoint contract moves.

### 12.4 Rollback

Because the spine is additive, rollback reduces to:

1. Set `runtime_governance_enabled = false`.
2. Revert BFF image; `/api/runtime/*` 404s disappear.
3. Retain `runtime_*` tables (empty or sparse). Drop is optional and deferred — data is cheap and forensic.

No business-flow code path depends on the spine succeeding. A `runtime_*` write failure cannot fail a user request, cannot fail MQTT ingest, cannot fail a cron job.

### 12.5 Release boundary

- Phase-1 goes live **disabled** in production. Enablement is a flag flip after migration confirmation.
- Enablement order: local → dev → production, separated by at least one full business day of observation.
- First 7 days post-enablement: phase-1 sources only. No new event-codes added. This is the governed observation window.

## 13. Risks, Tradeoffs, and Open Review Points

Only items where a meaningful design question is still open. Items closed by REQ decisions are not re-litigated.

### 13.1 Fingerprint width

**Risk**: too-narrow fingerprint produces event spam (one issue per request); too-wide fingerprint collapses distinct problems into a single confusing issue.

**Current design**: start narrow. `dedup_dimension` in the event-code registry defaults to empty; codes that need per-tenant or per-gateway dimension declare it explicitly.

**Open review point**: after 2 weeks of phase-1 observation, audit the top 20 fingerprints by observation_count and review whether dimensions should be widened or narrowed. Commit this as a post-release calibration pass, not a pre-release guess.

### 13.2 Lifecycle recovery detection

**Risk**: many failure modes have no natural recovery signal (a transient query timeout "recovers" by not happening again). Without a recovery event, issues stay `ongoing` indefinitely.

**Current design**: combine three recovery paths — (a) explicit `lifecycle_hint: 'recovery'` from emitters that can produce it, (b) self-check pass for the corresponding subsystem, (c) stale-detection TTL (72h) that auto-closes with an info-level annotation.

**Open review point**: whether (c)'s 72h is too long for operator utility or too short for noisy transient issues. Tunable via `feature_flags`; design does not hard-commit.

### 13.3 Projection: app-level vs DB-level

**Risk**: app-level projection logic and DB truth can drift if `emit()` is replayed, or if two processes race on the same fingerprint.

**Current design**: app-level projection using `INSERT ... ON CONFLICT (fingerprint) DO UPDATE` with deterministic transition rules; every update carries `updated_at` and is idempotent. Re-entry from replay is benign because transitions are monotone within a cycle and cycle resets are explicit (`cycle_count` increment + field clears).

**Open review point**: whether a DB trigger on `runtime_events` could take over projection. Downside: couples two concerns at storage layer and complicates testing. Deferred unless projection contention proves real.

### 13.4 Self-check host division (BFF in-process vs cron)

**Risk**: BFF-in-process self-checks produce spurious failures under BFF restart; cron-host self-checks add coupling to scheduler liveness (which itself is a check target).

**Current design**: reachability checks (fast, cheap) run in-process in BFF on a 60s tick; heavier freshness/representative checks run under the existing cron infrastructure (Telemetry5MinAggregator pattern). BFF-liveness self-check is a separate reflective check that the BFF emits a heartbeat about itself.

**Open review point**: whether to consolidate into a dedicated worker process. Rejected for phase-1 on complexity grounds.

### 13.5 Platform-vs-tenant surfacing

**Risk**: a platform-wide issue (e.g. `db.pool.exhausted`) affects every tenant, but phase-1 exposes it only to `SOLFACIL_ADMIN`. Tenants see degraded service without any explanation.

**Current design**: accepted. REQ §13.1 fixes the audience at internal ops for phase-1. Phase-2 can introduce a tenant-filtered read model.

**Open review point**: none for v6.10. Flagged for phase-2 design input.

### 13.6 SSE vs poll for the operator UI

**Risk**: 30s poll can mask fast-moving incidents.

**Current design**: poll. SSE push deferred because (a) the minimal-UI stance in REQ §13.3 discourages surface-side sophistication, and (b) the existing SSE handler already multiplexes 3 domain channels; adding runtime traffic complicates that stream.

**Open review point**: whether a separate SSE endpoint `/api/runtime/events-stream` is a phase-2 addition; current answer is yes, but not committed.

### 13.7 Cost of partitioned `runtime_events`

**Risk**: high-traffic environments could produce millions of events; monthly partitions may be too coarse for efficient retention.

**Current design**: monthly RANGE partitions + 90-day retention mirrors the proven `telemetry_history` pattern. Event volume is expected to be orders of magnitude lower than telemetry (not per-device, per-incident).

**Open review point**: monitor post-enablement. If monthly partitions prove too coarse, move to weekly. Partition strategy change is an internal-only refactor.

### 13.8 What is **not** on this list

The following are not open review points because REQ closed them:

- Audience (internal ops). §13.1 closed.
- Depth vs breadth (narrow-but-deep). §13.2 closed.
- UI centrality (not central). §13.3 closed.
- Whether the spine replaces business modules (no). §2 / REQ §2, §10 closed.
- Whether platform health equals domain alarm health (no). §11 / REQ §5.3 closed.

## 14. Summary of Architectural Commitments

| Commitment | Concrete expression |
|------------|---------------------|
| Spine as shared-layer citizen | `backend/src/shared/runtime/*` + `types/runtime.ts` + `migrations/002_runtime_governance.sql` |
| One emit API for all modules | `emit(event): void`, fire-and-forget, flag-gated, ring-buffered |
| One event-code registry | `runtime/contract.ts`; phase-1 namespace set closed at release |
| One severity taxonomy | 5-tier, registry-enforced |
| One lifecycle state machine | Projection-owned, not emitter-owned |
| One active-issue projection | `runtime_issues` single-row-per-fingerprint upsert with `cycle_count` |
| One health summary shape | Derived read, phase-1 component set fixed |
| One self-check registry | 7 mandatory checks, BFF + cron hosted, latest state persisted in `runtime_self_checks` |
| One operator backend surface | `/api/runtime/*`, SOLFACIL_ADMIN only |
| One minimal operator UI | `#runtime` hash, admin-only, no polish budget |
| Strict separation from P5/P6 | Independent tables, readers, writers, namespaces |
| Additive, flag-gated rollout | Emit is best-effort; no business flow depends on spine success |

**v6.10 design one-liner**: Install one shared-layer runtime-governance spine into the normalized v6.9 matrix lineage, with a closed event contract, a governed lifecycle, a narrow phase-1 onboarding set, an admin-only backend surface, and a deliberately minimal UI — such that the platform can finally express its own operational condition through standardized facts instead of scattered logs and module-local judgment.
