## 1. Overall verdict

The PLAN set is structurally good and implementation-oriented enough to be useful.
It already has:
- proper phase ordering,
- file-level grounding,
- migration / bridge / rollback awareness,
- backend-focused test commands,
- release-level acceptance criteria.

However, the testing model is still uneven.

Current weakness:
- backend segmented tests are much clearer than frontend segmented tests;
- final integrated testing exists, but it is spread across several sections instead of being expressed as one explicit test matrix;
- the plan does not yet make it impossible for implementers to skip frontend/browser verification or to confuse slice tests with release acceptance.

This revision pass should strengthen the PLAN, not redesign it.

## 2. Keep unchanged

Do not change these plan-level choices unless necessary:
1. the current phase ordering
2. REQ/DESIGN decisions already settled
3. matrix workstream structure FE / BE / DEP / DOC
4. bridge/removal model
5. rollout/rollback framing

## 3. Mandatory PLAN revision items

### 3.1 Add an explicit test strategy split

The revised PLAN must clearly separate three layers of testing:

1. segmented / slice-level testing
   - tests run immediately after each slice or workstream step
   - backend unit/integration tests, local curl checks, focused browser checks

2. cross-workstream integration testing
   - tests after FE + BE connect together
   - tests after DEP hardening is applied
   - staging tests with compatibility bridge ON and OFF

3. final release acceptance testing
   - one explicit integrated checklist/table for the full v6.9 contract
   - not scattered across several sections only

Do not leave the reader to infer this structure.

### 3.2 Add a single explicit test matrix to the master PLAN

The master PLAN must contain one dedicated section that lists the minimum required end-to-end / integrated scenarios.

At minimum, include these scenarios explicitly as named rows/cases:
1. Unauthenticated login page and SPA entry behavior
2. Successful login -> session bootstrap -> protected API -> SSE
3. Logout -> cookie cleared -> SSE closed -> subsequent API 401
4. Expired/invalid session -> redirect/401 handling
5. Stale localStorage token does not restore access
6. Repeated invalid login triggers throttle/backoff
7. Public-boundary headers and CSP are correct on login + SPA entry
8. Cookie attributes are correct on real login response
9. Compatibility bridge ON behavior (if used during migration)
10. Compatibility bridge OFF steady-state behavior

For each row, include:
- purpose
- where it runs (local / staging / public boundary)
- method (automated test / curl / browser / mixed)
- pass condition

### 3.3 Strengthen frontend segmented testing

Current frontend plan has good verification bullets, but the test contract is still softer than backend.

Required revision:
- for each major FE slice, explicitly state whether verification is:
  - automated test,
  - local browser/manual smoke,
  - network/devtools inspection,
  - staging browser verification.
- if there is no automated suite yet, say so clearly instead of implying one exists.
- make the browser verification mandatory, not optional, for auth/storage/CSP behavior.

### 3.4 Make the final full-test gate impossible to misunderstand

The revised PLAN must make clear that final full testing is not just:
- `npm test`
- `npm run build`
- a couple of curl checks

It must explicitly state that release approval requires:
- segmented backend tests passing,
- segmented frontend/browser verification passing,
- staging integrated flow verification passing,
- public boundary header/CSP verification passing,
- bridge OFF steady-state verification passing.

### 3.5 Tie tests to phases more tightly

The plan should make phase-gate testing explicit:
- what must pass before leaving Phase 1
- what must pass before leaving Phase 2
- what must pass before leaving Phase 3
- what must pass before production rollout

Right now some gates exist, but testing expectations per phase should be more explicit.

### 3.6 Clarify what is automated vs what is manual

The revised PLAN should not blur these.

For each important verification path, indicate one of:
- automated repo test
- command-line probe
- browser/manual verification
- operator/infrastructure verification

This is especially important for:
- frontend storage/auth behavior
- CSP behavior
- effective public-boundary headers
- SSE close/reconnect behavior
