## 1. Scope

This document covers the deployment and infrastructure boundary responsibilities for v6.9:

- Public boundary header ownership (app vs. ingress)
- Trusted-proxy and secure-cookie correctness
- App-owned vs. ingress-owned responsibilities
- CSP posture aligned with frontend self-hosting

## 3. Boundary Ownership — Fixed Design

### 3.1 Responsibility Matrix

| Responsibility | Owner | Rationale |
|----------------|-------|-----------|
| **Security response headers** (CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy) | **App (Express middleware)** | Headers must be effective in all topologies including local dev. App-layer ownership ensures they are never silently missing due to proxy misconfiguration |
| **HSTS** (`Strict-Transport-Security`) | **Ingress (Nginx)** | HSTS requires HTTPS guarantee. Only the TLS-terminating layer can make this promise |
| **TLS termination** | **Ingress (Nginx / ALB)** | App runs HTTP behind proxy |
| **`__Host-solfacil_session` cookie** | **App (Express)** | Cookie attributes are set in login handler. The `Secure` flag works because the browser sees HTTPS (terminated at ingress). Express `trust proxy` ensures `req.secure` is true behind proxy |
| **`X-Forwarded-For` / `X-Real-IP`** | **Ingress sets, App trusts** | Nginx sets `X-Real-IP` from downstream connection. Express `trust proxy` reads it for abuse-control IP extraction |
| **Login abuse control** | **App (Express middleware)** | Redis-backed; app-owned logic. Not an ingress responsibility |
| **Static asset caching** | **Ingress or App** | Cache-busting via `?v=` query parameter already in v6.8. No change |
| **Request size limits** | **Ingress (Nginx)** | `client_max_body_size` — existing configuration, no change |

### 3.2 Why App-Layer for Security Headers

Alternative considered: set security headers in Nginx. Rejected because:
1. Local dev (`http://127.0.0.1:3100`) has no Nginx — headers would be missing
2. Docker Compose dev has Nginx but configuration drift is common
3. App-layer headers are testable in integration tests without requiring a proxy
4. Defense-in-depth: Nginx MAY also set headers, but the app layer is the authoritative source

## 4. Trusted-Proxy Configuration

### 4.1 Express `trust proxy`

Required for two purposes:
1. `req.secure` returns `true` when behind HTTPS-terminating proxy (needed for `__Host-` cookie `Secure` flag)
2. `req.ip` returns the real client IP (needed for abuse-control keying)

**Fixed design**:

```typescript
// In local-server.ts
app.set('trust proxy', 1); // Trust first proxy hop
```

`trust proxy = 1` means Express trusts the first `X-Forwarded-For` value. This matches the single-proxy topology (Nginx → Express).

### 4.2 Nginx Configuration Requirements

Nginx must set these headers for the proxy pass to Express:

```nginx
proxy_set_header X-Forwarded-For   $remote_addr;
proxy_set_header X-Forwarded-Proto $scheme;
proxy_set_header X-Real-IP         $remote_addr;
proxy_set_header Host              $host;
```

**Critical**: `X-Forwarded-Proto` must be set correctly for `req.secure` to work. Without it, Express does not know the original request was HTTPS, and the `__Host-` cookie's `Secure` flag behavior depends on this.

### 4.3 Cookie Correctness Chain

```
Browser (HTTPS) ──► Nginx (TLS termination, sets X-Forwarded-Proto: https)
                     ──► Express (trust proxy=1, req.secure=true)
                          ──► res.cookie('__Host-solfacil_session', jwt, { secure: true })
                               ──► Set-Cookie header sent to browser
                                    ──► Browser accepts Secure cookie (connection is HTTPS)
```

If any link in this chain breaks:
- Missing `X-Forwarded-Proto` → Express thinks request is HTTP → `req.secure = false` → cookie may not be sent with `Secure` flag (depending on Express version behavior)
- Nginx not terminating TLS → browser connection is HTTP → browser rejects `__Host-` prefixed cookie

### 4.4 Local Development Exception

Local dev runs plain HTTP. The dev-mode cookie (`solfacil_session` without `__Host-` prefix, `secure: false`) works without any proxy. `trust proxy` is still set to 1 for consistency but has no effect when running directly.

## 5. Security Headers — App-Layer Implementation

### 5.1 Express Middleware

A new `securityHeaders` middleware is added early in the Express middleware chain (before route handlers):

```typescript
function securityHeaders(req, res, next) {
  res.setHeader('Content-Security-Policy', CSP_POLICY);
  res.setHeader('X-Frame-Options', 'DENY');
  res.setHeader('X-Content-Type-Options', 'nosniff');
  res.setHeader('Referrer-Policy', 'strict-origin-when-cross-origin');
  res.setHeader('Permissions-Policy', 'camera=(), microphone=(), geolocation=(), payment=()');
  next();
}
```

This middleware applies to **all responses** (HTML, API JSON, static assets).

### 5.2 CSP Policy Value

After frontend self-hosting of ECharts and Google Fonts (see frontend design):

```
default-src 'none';
script-src 'self';
style-src 'self';
img-src 'self' data:;
font-src 'self';
connect-src 'self';
frame-ancestors 'none';
base-uri 'self';
form-action 'self'
```

### 5.3 CSP for API Responses

The CSP header is set on all responses, including JSON API responses. This is intentional — it provides defense-in-depth against content-type confusion attacks and costs nothing.

### 5.4 CSP Report-Only Phase

**Bounded open point**: Whether to deploy CSP in `Content-Security-Policy-Report-Only` mode first (for 24-48h) before switching to enforcement. This is a deployment-time decision, not a design decision. The policy itself is fixed.

## 6. HSTS — Ingress-Layer

### 6.1 Nginx Configuration

```nginx
# In the HTTPS server block for solfacil.alwayscontrol.net
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

### 6.2 Why Ingress-Only

HSTS is set at the ingress layer because:
1. It promises "this domain is always HTTPS" — only the TLS-terminating layer can make that promise
2. Setting HSTS in Express behind a proxy is confusing: the Express app runs HTTP, but the header claims HTTPS
3. If Express sets HSTS and the proxy is misconfigured (no TLS), the header creates a broken promise

### 6.3 Dev Environment

Dev (`http://188.166.184.87/solfacil/`) does **not** set HSTS. HSTS on an HTTP-only endpoint would lock browsers into expecting HTTPS that doesn't exist.

## 7. Environment Variables — v6.9 Requirements

### 7.1 New or Changed Environment Variables

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `JWT_SECRET` | **Required** in non-dev | None (fatal if missing) | JWT signing/verification secret |
| `NODE_ENV` | Recommended | `undefined` (treated as production) | Controls dev-mode exceptions (cookie name, local-only in-memory rate limiting). When unset or non-`development`, all production fail-fast validations apply |
| `RATE_LIMIT_REDIS_URL` | **Required** in production | None (**fatal if missing in non-dev**) | Redis connection for login abuse control. Process must not start without a working connection in production |
| `LEGACY_BROWSER_BEARER` | Optional | `false` | Emergency: enable legacy browser bearer auth |

### 7.2 Existing Variables — Unchanged

| Variable | Purpose |
|----------|---------|
| `APP_DATABASE_URL` | App pool DSN |
| `SERVICE_DATABASE_URL` | Service pool DSN |
| `DATABASE_URL` | Legacy fallback DSN |
| `PORT` | Express listen port (default 3000) |

### 7.3 Startup Validation Order

At process startup, before any route is registered:

```
1. Validate JWT_SECRET (fail-fast if non-dev and missing/weak)
2. Initialize rate-limit store:
   - Non-dev: connect to Redis via RATE_LIMIT_REDIS_URL (fail-fast if missing or connection fails)
   - Dev without RATE_LIMIT_REDIS_URL: use in-memory store (local dev only)
3. Register security headers middleware
4. Register auth middleware
5. Register routes
6. Start listening
```

## 8. Nginx Configuration Changes — v6.9

### 8.1 Required Changes

| Change | Current | v6.9 |
|--------|---------|------|
| HSTS header | Not set | `add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;` |
| `X-Forwarded-Proto` | May or may not be set | **Must be set**: `proxy_set_header X-Forwarded-Proto $scheme;` |
| `X-Forwarded-For` | May or may not be set | **Must be set**: `proxy_set_header X-Forwarded-For $remote_addr;` |

### 8.2 No Nginx Security Header Duplication

Security headers (CSP, X-Frame-Options, nosniff, Referrer-Policy, Permissions-Policy) are **not** set in Nginx. The app layer is the single authoritative source. This avoids:
- Duplicate headers (some browsers handle duplicates inconsistently)
- Configuration drift between Nginx config and app code
- Testing gaps (local dev tests would miss Nginx-only headers)

### 8.3 Nginx Proxy Pass — Reference Config

```nginx
server {
    listen 443 ssl;
    server_name solfacil.alwayscontrol.net;

    # TLS config (existing — not shown)

    # HSTS (NEW in v6.9)
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $remote_addr;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE support (existing)
        proxy_set_header Connection '';
        proxy_http_version 1.1;
        chunked_transfer_encoding off;
        proxy_buffering off;
        proxy_cache off;
    }
}
```

## 9. Docker Compose Impact

### 9.1 `solfacil-bff` Service

| Change | Detail |
|--------|--------|
| New env var | `JWT_SECRET` — must be set in `docker-compose.yml` or `.env` file |
| New env var | `RATE_LIMIT_REDIS_URL` — points to Redis service |
| New service | **Redis** container for rate limiting |

### 9.2 Redis Service Addition

```yaml
solfacil-redis:
  image: redis:7-alpine
  ports:
    - "127.0.0.1:6379:6379"
  restart: unless-stopped
```

**Bounded open point**: Whether Redis is a new Docker Compose service or an external managed service in production. For local dev, a Docker Compose Redis service is sufficient. Production may use managed Redis (e.g., ElastiCache, DigitalOcean Managed Redis).

### 9.3 Static Asset Serving

Self-hosted ECharts and fonts are served by Express's static file middleware (already configured for `frontend-v2/`). No Nginx or Docker change needed for static assets.

## 10. Verification Expectations — Deployment Layer

These are observable properties that deployment verification should confirm after v6.9 rollout. They are not test scripts — they are acceptance criteria for the deployment boundary.

| # | Verification | Method |
|---|-------------|--------|
| V1 | `__Host-solfacil_session` cookie is set on login with correct attributes | Inspect `Set-Cookie` header in browser dev tools or `curl` |
| V2 | CSP header present on HTML page response | `curl -I https://solfacil.alwayscontrol.net/` |
| V3 | `X-Frame-Options: DENY` present | Same as V2 |
| V4 | `X-Content-Type-Options: nosniff` present | Same as V2 |
| V5 | `Referrer-Policy` present | Same as V2 |
| V6 | `Permissions-Policy` present | Same as V2 |
| V7 | HSTS header present (production only) | Same as V2 |
| V8 | No `token` field in browser login response body | `curl -X POST .../api/auth/login` (without `X-Auth-Contract: machine` header) |
| V9 | `GET /api/auth/session` returns 200 with valid cookie | `curl --cookie "..." .../api/auth/session` |
| V10 | `GET /api/auth/session` returns 401 without cookie | `curl .../api/auth/session` |
| V11 | Login rate limiting returns 429 after threshold | Repeated `curl -X POST .../api/auth/login` with wrong password |
| V12 | Process exits if `JWT_SECRET` is missing in non-dev mode | Start BFF without `JWT_SECRET` env var with `NODE_ENV=production` |
| V13 | ECharts and fonts served from self-hosted paths | Check network tab — no requests to `cdn.jsdelivr.net` or `fonts.googleapis.com` |
| V14 | No inline scripts in HTML pages | View source of `login.html` and `index.html` |
| V15 | Process exits if `RATE_LIMIT_REDIS_URL` is missing or unreachable in non-dev mode | Start BFF without `RATE_LIMIT_REDIS_URL` env var with `NODE_ENV=production` — must fail to start |

## 11. Affected vs. Unaffected Deployment Artifacts

| Artifact | Changed | Reason |
|----------|---------|--------|
| Nginx site config (production) | **Yes** | HSTS header; ensure `X-Forwarded-Proto` and `X-Forwarded-For` are set |
| Nginx site config (dev) | **Minimal** | Ensure `X-Forwarded-Proto` and `X-Forwarded-For` are set. No HSTS |
| `docker-compose.yml` | **Yes** | New Redis service; new env vars (`JWT_SECRET`, `RATE_LIMIT_REDIS_URL`) |
| `.env` or secret management | **Yes** | `JWT_SECRET` must be provisioned with a strong random value |
| Express `local-server.ts` | **Yes** | `trust proxy = 1`; startup validation; security headers middleware; rate-limit init |
| `frontend-v2/` static files | **Yes** | New files: `js/vendor/echarts.min.js`, `js/login.js`, `fonts/*.woff2` |
| PostgreSQL schema | **Unchanged** | No schema changes for v6.9 |
| M1 MQTT pipeline | **Unchanged** | No deployment changes |
| CDK / Lambda configuration | **Unchanged** (v6.9 targets local/Docker/Nginx topology) | If CDK deployment is used, equivalent changes needed there — out of v6.9 scope |

## 12. Bounded Open Points

| Open Point | Constraint |
|------------|------------|
| Whether to use CSP `Report-Only` mode for initial 24-48h | Deployment-time decision. Policy content is fixed |
| Managed Redis vs. Docker Compose Redis in production | Either is acceptable. Must support TTL key operations |
| Whether Nginx should also set security headers as defense-in-depth (duplicate with app layer) | Not recommended due to duplicate-header risks. If done, Nginx headers must be identical to app-layer headers |
| CDK/Lambda deployment alignment | Out of v6.9 scope. CDK deployment would need equivalent header and cookie changes if activated |
