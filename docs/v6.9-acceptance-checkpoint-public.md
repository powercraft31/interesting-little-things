# v6.9 Acceptance Checkpoint — Public Surface

Date: 2026-04-16
Target surface: `https://solfacil.alwayscontrol.net/`
Conclusion: `FAIL / BLOCKED FOR RELEASE`

## Executive Summary

The public production-like surface is not serving the v6.9 security-corrected app/runtime that was verified locally.

What is live on `solfacil.alwayscontrol.net` is still the older auth/frontend contract:
- browser login stores `solfacil_jwt` in `localStorage`
- login response body still returns `token`
- cookie is still `solfacil_jwt`, not `__Host-solfacil_session`
- cookie is `SameSite=Lax`, not `Strict`
- dedicated `GET /api/auth/session` contract is not working as required for the v6.9 browser flow
- shipped HTML still depends on external Google Fonts and jsDelivr ECharts
- shipped HTML still contains inline `onclick`
- top-level HTML responses do not carry the expected hardening headers (`Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`)

So the local v6.9 implementation work is real, but the public surface has not been rolled forward to that truth.

This is not a DB problem and not a reason to reopen implementation scope. This is deployment/release state mismatch.

## Evidence

### A1. Top-level security headers on public HTML

Command:
```bash
curl -sSI https://solfacil.alwayscontrol.net/
```

Observed root headers:
- present:
  - `Strict-Transport-Security: max-age=31536000`
- missing on HTML response:
  - `Content-Security-Policy`
  - `X-Frame-Options`
  - `X-Content-Type-Options`
  - `Referrer-Policy`
  - `Permissions-Policy`

Observed response excerpt:
```http
HTTP/1.1 200 OK
Server: nginx/1.26.3
Content-Type: text/html; charset=utf-8
X-Powered-By: Express
Strict-Transport-Security: max-age=31536000
```

Assessment:
- HSTS exists.
- The expected v6.9 browser-hardening header set is not active on the public HTML surface.

### A2. Public shipped assets still use external origins

Command pattern:
```bash
curl -sS https://solfacil.alwayscontrol.net/index.html | grep -nE 'fonts.googleapis|fonts.gstatic|cdn.jsdelivr|onclick='
```

Observed in public `index.html`:
- Google Fonts preconnect and stylesheet links still present
- jsDelivr ECharts still present
- inline `onclick` still present on sidebar hamburger

Observed lines:
```html
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=Inter..." rel="stylesheet">
<button class="hamburger-btn" onclick="document.querySelector('.sidebar').classList.toggle('sidebar-open')">☰</button>
<script src="https://cdn.jsdelivr.net/npm/echarts@5/dist/echarts.min.js"></script>
```

Assessment:
- This directly contradicts the v6.9 CSP/self-hosting closure work.
- Public frontend is still the old surface.

### A3. Browser login contract is still old

Browser login on the public site succeeded with:
- email: `admin@solfacil.com.br`
- password: `solfacil2026`

Observed post-login runtime state in browser:
- URL: `https://solfacil.alwayscontrol.net/index.html#fleet`
- `localStorage.solfacil_jwt` exists
- no evidence of cookie-based browser session contract replacing bearer storage

Observed browser state excerpt:
```json
{
  "url": "https://solfacil.alwayscontrol.net/index.html#fleet",
  "token": {
    "solfacil_jwt": "<jwt-present>"
  }
}
```

Assessment:
- Browser auth is still localStorage bearer model.
- That is the exact old contract v6.9 was supposed to retire for browser flows.

### A4. Login API response still returns token and old cookie

Command:
```bash
curl -sS -D headers.txt -o body.json \
  -H 'Content-Type: application/json' \
  --data '{"email":"admin@solfacil.com.br","password":"solfacil2026"}' \
  https://solfacil.alwayscontrol.net/api/auth/login
```

Observed response headers:
```http
HTTP/1.1 200 OK
Set-Cookie: solfacil_jwt=<jwt>; Max-Age=86400; Path=/; Expires=...; HttpOnly; SameSite=Lax
Strict-Transport-Security: max-age=31536000
```

Observed response body excerpt:
```json
{
  "success": true,
  "data": {
    "token": "eyJhbG...",
    "user": {
      "userId": "USER_ADMIN_001",
      "email": "admin@solfacil.com.br",
      "orgId": "ORG_ENERGIA_001",
      "role": "SOLFACIL_ADMIN"
    }
  }
}
```

Assessment:
- Wrong cookie name: `solfacil_jwt` instead of `__Host-solfacil_session`
- Wrong browser response shape: still returns `token`
- Wrong cookie semantics: `SameSite=Lax` instead of `Strict`
- Public surface has not picked up the v6.9 auth contract

### A5. `/api/auth/session` contract mismatch on public surface

Without cookie:
```bash
curl -sS -i https://solfacil.alwayscontrol.net/api/auth/session
```
Observed:
- `401 Unauthorized`
- body says: `Authorization header or auth cookie required`

With the login cookie jar captured from the public login response:
```bash
curl -sv -b /tmp/solcookie.txt https://solfacil.alwayscontrol.net/api/auth/session
```
Observed:
- request sends `Cookie: solfacil_jwt=...`
- response becomes `404 Not Found`
- body: `Cannot GET /api/auth/session`

Assessment:
- The dedicated browser session contract required by v6.9 is not consistently present on the public surface.
- At minimum, the public surface is not serving the verified local `/api/auth/session` behavior.

### A6. Browser resource origins on live page

Observed via browser runtime:
- page loads external origins:
  - `https://fonts.googleapis.com`
  - `https://fonts.gstatic.com`
  - `https://cdn.jsdelivr.net`

Assessment:
- Self-hosted asset closure is not deployed on the public surface.
