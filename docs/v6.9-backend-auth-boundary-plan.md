```
1. securityHeaders()                       ← B1 (all responses)
2. express.json()                          ← existing
3. cors()                                  ← existing
4. authMiddleware                          ← B3 (cookie-first + bearer)
5. [rate-limit on /api/auth/login only]    ← B6
6. Route handlers                          ← existing + B4 (session) + B5 (login changes)
```

## 4. Secret Validation — `tenant-context.ts` Change Detail

### Current code (v6.8) — being retired:

The v6.8 `tenant-context.ts` contains a hardcoded fallback to a well-known dev secret. v6.9 removes this entirely — no code path evaluates or references the legacy fallback value.

### Target code (v6.9):
```typescript
// Option A: Module-level init (preferred — see §8 Bounded Open)
let jwtSecret: string;

export function initJwtSecret(secret: string): void {
  jwtSecret = secret;
}

// In verifyTenantToken — no fallback, no hardcoded default
jwt.verify(token, jwtSecret);  // jwtSecret is guaranteed non-null by startup validation
```

Or:

```typescript
// Option B: Parameter passing
export function verifyTenantToken(token: string, secret: string): TenantClaims {
  return jwt.verify(token, secret) as TenantClaims;
}
```

**Constraint**: Option A or B are both acceptable for secret *passing mechanism* (bounded open decision, §8). The non-negotiable requirement is that no code path in v6.9 contains a hardcoded secret fallback — not at startup, not at runtime.

## 5. Machine Contract Route Scoping — Explicit Status

The design declares machine bearer auth as a bounded open point for route scoping. This plan makes it explicit:

| Route Category | Machine Bearer Accepted? | Rationale |
|----------------|-------------------------|-----------|
| `GET /api/auth/session` | **No** — cookie only by design | Session endpoint is browser-only |
| `POST /api/auth/login` | **Yes** — with `X-Auth-Contract: machine` header for machine response shape | Machine clients need a way to obtain tokens |
| `POST /api/auth/logout` | **Cookie contract only** | Machine clients don't have sessions to clear |
| All other `/api/*` data routes (45 handlers) | **Yes** — bearer accepted when no cookie present | Machine clients (CLI, scripts) use these routes |
| Non-`/api/*` paths (static files, HTML) | **No auth required** | Public static content |

**Implementation note**: The auth middleware handles this uniformly. `GET /api/auth/session` handler additionally checks that auth came from cookie, not bearer, and returns 401 if bearer-only. All other routes accept either contract via the middleware.

## 6. Slice Dependencies and Parallelism

```
B1 (Security headers)    ──── no dependency ──────────┐
B2 (Secret fail-fast)    ──── no dependency ──────────┤
B3 (Auth middleware)      ──── no dependency ──────────┤
B4 (Session endpoint)     ──── no dependency ──────────┤── G1
B5 (Login handler split)  ──── depends on B3 (cookie) ┤
B6 (Abuse control)        ──── depends on B2 (startup)┤
B7 (SSE cleanup)          ──── depends on B3 ─────────┘
```

**Parallelism**: B1, B2, B3, B4 can all execute concurrently. B5 and B7 require B3 (for `SESSION_COOKIE_NAME` constant). B6 requires B2 (for startup validation pattern). B5, B6, B7 can execute concurrently once their dependencies are met.
