# Solfacil v6.10 Runtime Governance — Implementation Plan

Date: 2026-04-18
Status: PLAN generated from accepted REQ + finalized canonical design set
Baseline: v6.9 normalized matrix
Target release: v6.10

---

## 1. Goal / architecture baseline

v6.10 is an **internal-ops-first, narrow-but-deep, backend-contract-first** release that inserts a **runtime-governance spine into M9 shared layer** and onboards a bounded phase-1 emitter set around it.

This PLAN is implementation-oriented. It does **not** reopen the accepted design.

Fixed architectural baseline carried into implementation:
- v6.10 is an **M9 shared-layer extension**, not a new standalone module.
- runtime governance remains distinct from **P5 / P6 / domain alarm surfaces**.
- rollout is **additive** and **feature-flag-gated**.
- business flows must continue to function when the runtime spine is disabled or degraded.
- Redis namespace may be reserved, but Redis is **not mandatory phase-1 scope** unless later explicitly confirmed.
- canonical phase-1 deltas are in **M1 / M2 / M3 / M4 / M5 / M9 / DB / frontend**.

Phase-1 implementation outcome:
1. additive `runtime_*` schema exists and is bootstrapped first
2. M9 exposes canonical event, issue, self-check, projection, persistence, and health logic
3. M5 exposes `/api/runtime/*` admin-only operator contract
4. selected emitters onboard BFF + DB substrate + M1 + M2 + M3 + M4
5. frontend ships one minimal `#runtime` page proving operator usability of the backend contract
6. release can be enabled, observed, and rolled back via feature flags without breaking normal product flows

## 3. Phase ordering and gates

### Gate A — bootstrap ready
Required before any runtime fact is emitted outside isolated tests.

Exit criteria:
- feature flags defined with defaults = disabled
- rollout toggle model is explicit in code/config: one global gate plus per-slice enable flags for `bff_db`, `m1_ingest`, `m2_scheduler`, `m3_dispatch`, `m4_billing`, and `frontend_runtime`
- migration execution path decided and wired into local/runtime bootstrap
- disabled-mode contract documented in code for API and emitter no-op behavior
- explicit separation preserved between runtime governance and existing business flows

Required pass evidence before Gate A is considered passed:
- WS0 slice tests for flag parsing/defaults and disabled-mode helpers pass
- one local smoke check proves all flags default to off and runtime API still returns disabled posture when only schema exists

### Gate B — storage foundation ready
Required before M9 persistence or operator API work.

Exit criteria:
- `002_runtime_governance.sql` exists and is idempotent
- four `runtime_*` tables exist with required indexes
- current + near-future `runtime_events` partitions bootstrap successfully
- local DB verification script extended to assert new tables/indexes/partition bootstrap
- migration can be applied while feature flag remains off

Required pass evidence before Gate B is considered passed:
- WS1 DB contract script passes on first apply
- WS1 DB contract script passes on second apply (idempotence proof)
- no existing business table diff/regression is reported by local DB rebuild/contract verification

### Gate C — M9 spine ready
Required before module onboarding.

Exit criteria:
- canonical runtime types and event-code registry compiled
- `emit(...)` is feature-flag-gated, best-effort, and non-throwing
- persistence writes events/issues/self-checks/snapshots through service pool path
- projection rules for detected/ongoing/recovered/closed/suppressed are implemented
- health derivation returns `disabled` only as API output when feature flag is off

Required pass evidence before Gate C is considered passed:
- WS2 slice suite passes for contract registry, projection semantics, self-check latest-state updates, and health derivation
- explicit tests prove reopen increments `cycle_count`, suppress/close remain distinct, and persistence failure does not throw to caller

### Gate D — operator contract ready
Required before frontend work and before operator action testing.

Exit criteria:
- `/api/runtime/health`, `/issues`, `/issues/:fingerprint`, `/events`, `/self-checks`, and action POSTs exist
- all `/api/runtime/*` routes require `SOLFACIL_ADMIN`
- operator actions mutate through shared-layer interfaces, not ad hoc handler SQL
- local-server and deployed BFF route maps both expose the same runtime endpoints

Required pass evidence before Gate D is considered passed:
- WS3 route/auth/envelope/wiring test suite passes
- one local scripted smoke proves admin access succeeds and non-admin access is rejected on runtime routes

### Gate E — emitter onboarding ready
Required before final acceptance.

Exit criteria:
- BFF + DB substrate facts emitting under flag
- M1, M2, M3, M4 bounded emitter slices merged
- self-check cadence writes observable latest-state rows
- no emitter failure can fail a business request or background job’s core path

Required pass evidence before Gate E is considered passed:
- WS4 slice suite passes, plus isolated integration proof for BFF/DB substrate path
- WS5 isolated integration proof passes before M1 enablement is considered complete
- WS7 isolated integration proof passes before M3 enablement is considered complete
- WS6 isolated integration proof passes before M2 enablement is considered complete
- WS8 isolated integration proof passes before M4 enablement is considered complete

### Gate F — operator usability ready
Required before release signoff.

Exit criteria:
- `#runtime` page renders health, component states, active issues, issue detail, self-checks
- admin-only visibility works
- manual refresh / soft poll works without SSE
- page remains clearly separate from P5 and P6

Required pass evidence before Gate F is considered passed:
- WS9 named frontend smoke/contract suite passes
- manual browser/operator checklist completes successfully on a live local runtime with at least one induced issue and one operator action

### Gate G — release-ready
Final signoff gate.

Exit criteria:
- slice-level, integration, and final acceptance matrices all pass
- rollout/rollback rehearsal completed
- feature-flag-off regression verified against normal business flows
- residual risk list accepted as bounded follow-up, not release blocker

Required pass evidence before Gate G is considered passed:
- every acceptance row in §5.3 has an attached evidence type and recorded pass result
- rollout rehearsal with incremental flags succeeds in intended enablement order
- rollback rehearsal with global off + one per-slice flag rollback path succeeds without business-flow regression

## 5. Testing strategy

Testing is split into three layers and must be executed in order.

### 5.1 Slice-level / module-level verification

This layer proves each workstream in isolation before broader integration.

#### WS1 DB schema slice
- apply `002_runtime_governance.sql` twice against local DB and confirm idempotence
- extend `verify-local-db-contract` to assert:
  - all four `runtime_*` tables exist
  - canonical indexes exist
  - `runtime_events` partition bootstrap exists for current + near-future periods
- confirm migration succeeds while feature flag remains off
- confirm no existing business table diff is introduced

#### WS2 M9 slice
- unit tests for runtime type guards / contract registry / invalid event-code rejection
- projection tests covering:
  - first detect creates issue row
  - repeat observation updates same fingerprint row
  - recovery changes issue state and timestamps correctly
  - close and suppress semantics remain distinct
  - reopen increments `cycle_count` on same row
- emitter tests proving:
  - feature-off = no-op behavior
  - persistence failure does not throw to caller
  - bounded queue behavior under burst
- self-check tests proving latest-state row update semantics per `check_id`
- health derivation tests proving `disabled` is API-only and never persisted to snapshots

#### WS3 M5 route slice
- handler tests for each GET route and each POST action route
- auth tests proving non-admin receives denial and admin succeeds
- envelope tests proving `ok()` / `fail()` conventions remain intact
- local-server route smoke test proving `/api/runtime/*` endpoints map to correct handlers
- stack test proving `backend/lib/bff-stack.ts` includes runtime route lambdas/bindings

#### WS4 BFF + DB substrate slice
- boot-path emitter test
- top-level error-boundary emitter test
- auth anomaly emission test with bounded/noise-disciplined thresholds
- DB self-check tests for app pool, service pool, and critical query failure/success transitions
- regression tests confirming normal login/session/business requests still succeed when runtime emission fails internally

#### WS5 M1 slice
- stale ingest produces runtime fact and updates latest self-check state
- parser failure emits structured detail payload but does not rewrite domain alarms
- backlog threshold emits once per fingerprint cycle rather than spamming duplicates
- recovery path transitions issue to recovered when ingest freshness resumes

#### WS6 M2 slice
- optimization scheduler heartbeat success path
- scheduler error path emission
- missed-run / lag path emission without generic all-cron scope bleed

#### WS7 M3 slice
- dispatcher heartbeat success path
- timeout checker heartbeat success path
- ack-stall detection path
- recovery path after dispatch resumes

#### WS8 M4 slice
- billing job heartbeat success path
- billing error path emission
- missed-run path emission and recovery semantics

#### WS9 frontend slice
- router tests or smoke checks for `#runtime` navigation
- admin-only page visibility check
- issue list / detail / self-check rendering with mocked runtime API payloads
- operator action wiring check for close/suppress/note POSTs
- confirm no SSE/live-stream dependency exists

### 5.2 Cross-workstream integration verification

This layer proves the spine actually works as a system.

#### Integration set I1 — migration-to-API path
1. apply runtime migration with feature flag off
2. start local server
3. call `GET /api/runtime/health`
4. verify response returns `overall="disabled"` and empty runtime collections
5. enable feature flag
6. verify same endpoint now returns governed shape from runtime tables/self-checks

#### Integration set I2 — BFF/DB substrate path
1. start local server with runtime feature on
2. force one controlled DB critical-query failure or unavailable dependency simulation in test harness
3. verify runtime event row written
4. verify issue row projected under same fingerprint
5. verify self-check latest-state changes to failing
6. verify health summary degrades
7. restore dependency
8. verify recovery event, issue transition, and health summary recovery

#### Integration set I3 — M1 ingest path
1. simulate stale-ingest condition or halt ingest activity in local test harness
2. verify `ingest.freshness` self-check fails
3. verify runtime issue appears in `/api/runtime/issues`
4. resume ingest
5. verify issue moves to recovered and later auto-close path remains available
6. verify no P6/domain alert table mutation occurs

#### Integration set I4a — M3 dispatch path
1. enable only global runtime flag + `runtime_emit_bff_db` + `runtime_emit_m3_dispatch`
2. start dispatcher + timeout-checker path under runtime feature on
3. verify heartbeat/self-check path for dispatch only
4. inject one controlled ack-stall / dispatch-progress failure
5. verify one distinct dispatch fingerprint/source projection appears
6. restore dispatch progress
7. verify recovery path and health-summary correction

#### Integration set I4b — M2 scheduler path
1. enable only global runtime flag + `runtime_emit_bff_db` + `runtime_emit_m2_scheduler`
2. run optimization scheduler path
3. verify optimization scheduler heartbeat reaches runtime surfaces
4. inject one controlled scheduler failure or missed-run scenario
5. verify one distinct M2-owned scheduler fingerprint/source projection appears
6. restore scheduler run
7. verify recovery path and health-summary correction

#### Integration set I4c — M4 billing path
1. enable only global runtime flag + `runtime_emit_bff_db` + `runtime_emit_m4_billing`
2. run billing job path
3. verify billing heartbeat reaches runtime surfaces
4. inject one controlled billing-job failure or missed-run scenario
5. verify one distinct M4-owned billing fingerprint/source projection appears
6. restore billing path
7. verify recovery path and health-summary correction

#### Integration set I5 — operator action path
1. create live issue via controlled fault
2. fetch issue list and detail via runtime API
3. POST suppress
4. verify issue remains in projection with muted active-summary posture
5. POST note
6. verify note persistence visible on detail read
7. POST close after recovery or accepted manual closure rule
8. verify close semantics differ from suppress semantics

#### Integration set I6 — retention / auto-close executor path
1. run retention executor in local/test mode with injected clock or explicit cutoff parameters
2. verify a recovered issue older than 24h moves to `closed`
3. verify a stale/no-new-observation issue older than 72h auto-closes with the expected annotation path
4. verify old `runtime_events` partition/data selection logic marks >90 day data for cleanup without touching in-scope partitions
5. verify old `runtime_health_snapshots` cleanup selection logic marks >30 day snapshots for cleanup
6. confirm these behaviors are provable without waiting real wall-clock time

#### Integration set I7 — feature-flag isolation
1. with runtime tables present, disable feature flag
2. run normal auth, fleet, P5, and P6 endpoint smoke checks
3. verify business flows still work unchanged
4. verify emitters no-op and runtime API returns disabled posture
5. confirm disabling runtime governance is sufficient rollback for live release incident handling

## 6. Rollout / rollback

### 6.1 Rollout procedure

1. **Merge additive schema first**
   - ship `002_runtime_governance.sql`
   - apply migration in target environment(s)
   - keep runtime feature flag off

2. **Deploy M9 + M5 code with feature off**
   - deploy shared runtime spine and BFF route code
   - verify `GET /api/runtime/health` returns disabled posture
   - verify business endpoints remain healthy

3. **Enable self-checks and substrate emitters first**
   - turn on runtime feature for internal env only
   - validate DB + BFF runtime facts and snapshots
   - confirm no request-path breakage

4. **Enable module onboarding incrementally**
   - M1 ingest emitters
   - M3 dispatch emitters
   - M2 scheduler emitters
   - M4 billing emitters
   - after each enablement, complete the corresponding integration slice before proceeding

5. **Expose `#runtime` page to admin users only**
   - verify page against live runtime API
   - keep navigation/internal labeling explicit

6. **Enable slices incrementally using explicit flags**
   - enable order:
     - `runtime_emit_bff_db`
     - `runtime_emit_m1_ingest`
     - `runtime_emit_m3_dispatch`
     - `runtime_emit_m2_scheduler`
     - `runtime_emit_m4_billing`
     - `runtime_frontend_page_enabled`
   - after each enablement, complete the corresponding isolated integration set and record pass evidence before proceeding

7. **Promote to broader internal environments only after acceptance matrix passes**

### 6.2 Rollback procedure

Rollback priority is behavioral rollback, not schema deletion.

1. turn runtime-governance feature flag off
2. confirm `/api/runtime/health` returns disabled posture
3. confirm emitters no-op and core business flows continue
4. if a specific emitter slice is noisy, disable its corresponding slice flag (`runtime_emit_bff_db`, `runtime_emit_m1_ingest`, `runtime_emit_m2_scheduler`, `runtime_emit_m3_dispatch`, `runtime_emit_m4_billing`) before considering code revert
5. if flag-based rollback is insufficient, revert that module slice while keeping the global spine flag off
6. do **not** drop `runtime_*` tables during rollback
7. preserve collected runtime data for postmortem unless separate data-retention policy requires cleanup later

### 6.3 Release guardrails during rollout
- never make authentication, dispatch, ingest, or billing success depend on runtime spine success
- never gate P5/P6 surfaces on runtime-governance reads
- never promote Redis into mandatory health scope during v6.10 phase-1 unless a later explicit decision changes scope

## 7. Residual risks / bounded follow-ups

These are recognized but explicitly bounded so they do not reopen v6.10 design.

### 7.1 Noise-tuning risk
Initial fingerprints or thresholds may be too noisy, especially for ingest backlog and auth anomalies.
- bounded follow-up: tune event-code thresholds and dedup dimensions within the accepted registry model
- not a reason to widen scope into full observability stack work

### 7.2 Scheduler ownership wording drift
M2 and M4 ownership is clear in design, but implementation must preserve that split in code and tests.
- bounded follow-up: tighten naming or comments if `scheduler.jobs.alive` reads too generic in code

### 7.3 Retention executor shape
Canonical docs define retention defaults but not a single mandatory executor mechanism.
- bounded follow-up: if retention cleanup lands as a simple scheduled job in v6.10, deeper automation/polish can be postponed as long as defaults are enforceable and documented

### 7.4 Frontend polish pressure
The runtime page could attract dashboard-polish requests.
- bounded follow-up: charts, trend analysis, SSE, and richer UX stay deferred beyond v6.10
- phase-1 success is operational usability, not dashboard aesthetics

### 7.5 Broader emitter coverage pressure
Open API, Redis, and other modules may be logical next onboarding candidates.
- bounded follow-up: reserve namespaces and backlog items only
- do not expand v6.10 phase-1 beyond accepted M1/M2/M3/M4/M5/M9/DB/frontend scope
