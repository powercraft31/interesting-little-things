# Solfacil v6.9 Production Rollout Plan

> Read-only audit basis: production host `18.141.63.142` (`solfacil-prod-ec2`) on 2026-04-17.
> Goal is bounded promotion of v6.9 to production **without disturbing unrelated services on the shared host**.

## Executive judgment

Do **not** treat production rollout as a single `git pull && docker compose up` action.

Current production is a shared host with many unrelated services, and the local v6.9 `docker-compose.yml` is **not production-safe as-is** on that host because it introduces `solfacil-redis` bound to `127.0.0.1:6379`, while the host already has another listener on `0.0.0.0:6379`.

Therefore the correct rollout shape is:

1. freeze a **production-specific deployment contract**,
2. isolate Solfacil-only changes,
3. reset/rebuild Solfacil DB only after backup,
4. recreate only Solfacil containers,
5. verify public acceptance,
6. stop immediately if any step implies touching shared nginx/global redis/unrelated containers unexpectedly.

## Phase 1 — Create production-specific deployment contract

Do not deploy local compose to production unchanged.

Create a production-safe deployment shape that:
- does not touch unrelated host services,
- does not require shared nginx edits unless proven necessary,
- does not collide with existing host Redis port usage.

### Preferred production approach
Use a production-specific compose file or override with one of these explicit models:

#### Option A — Reuse existing production Redis service
- set `RATE_LIMIT_REDIS_URL` to an already-running Redis reachable on host
- do **not** start `solfacil-redis`
- requires explicit namespace / isolation judgment

#### Option B — Dedicated Solfacil Redis without host-port bind
- keep `solfacil-redis` internal to the Solfacil docker network
- remove `127.0.0.1:6379:6379` publish
- BFF talks to `redis://solfacil-redis:6379`
- avoids collision with shared host port 6379

**Default judgment:** Option B is cleaner because it reduces cross-service coupling on a shared host.

**Stop line:** do not run `docker compose up` on production until Redis strategy is frozen and port conflict removed.

## Phase 2 — Prepare rollback and backup

Before any destructive action:

1. capture production repo state
2. create DB dump backup
3. snapshot baseline identity rows if using seed/reseed path
4. keep a copy of current production compose/env inputs

Minimum artifacts:
- DB dump of `solfacil_vpp`
- current `/opt/solfacil/docker-compose.yml`
- current production commit SHA
- exported gateway/user baseline rows

**Stop line:** if backup is not verified, do not proceed.

## Phase 5 — Production execution (bounded window)

### 5.1 Pre-flight
- verify target commit `2afdf57` is present locally and ready
- verify production-specific compose/override is present
- verify env values including `JWT_SECRET` and `RATE_LIMIT_REDIS_URL`
- verify no nginx change is required for first rollout attempt

### 5.2 Code update
- update `/opt/solfacil` to target commit
- do not touch unrelated repos or containers

### 5.3 DB reset
- stop/recreate only Solfacil DB path as planned
- rebuild via canonical path
- apply production seed
- run DB verifier

### 5.4 Runtime recreation
- recreate only:
  - `solfacil-bff`
  - `solfacil-m1`
  - `solfacil-db`
  - optionally dedicated `solfacil-redis` if that strategy was chosen
- do **not** restart `nginx-web` unless a verified nginx config change is actually required

### 5.5 Acceptance
- browser login contract
- `GET /api/auth/session`
- no `localStorage` JWT for browser app flow
- security headers present
- self-hosted asset surface confirmed
- only intended production gateways visible

**Stop line:** if rollout requires touching shared nginx or host-wide Redis unexpectedly, pause and re-evaluate instead of improvising.

## Rollback model

### If DB reset completed but app verification fails
- restore DB dump
- return repo to old commit if necessary
- recreate only Solfacil containers

### If code/runtime fails before DB reset
- revert repo to previous commit
- restart only Solfacil containers

### If unrelated services show impact
- stop immediately
- revert Solfacil-only changes
- do not continue same window
