# v6.10 Runtime Governance DESIGN — Ryan Review Notes Round 1

Date: 2026-04-18 UTC
Target doc: `docs/v6.10-runtime-governance-design.md`
Status: revision required before acceptance

## Overall judgment

The draft is broadly strong.

What is correct:
- It preserves the REQ center of gravity: runtime-governance spine, not dashboard theater.
- It anchors the spine in M9 shared layer, which is the right architectural placement.
- It keeps phase-1 narrow, internal-ops-first, and backend-contract-first.
- It separates platform runtime from P5/P6 domain surfaces clearly.
- It gives a concrete backend contract and explicit rollout / rollback boundary.

But two design defects remain and they are material enough that this draft should not be accepted as-is.

## Issue 1 — `runtime_issues` row model is internally inconsistent

### Where the inconsistency is
The draft says all of the following:

1. `runtime_issues` is "one row per fingerprint" and uses `fingerprint` as the PK.
2. lifecycle can move `closed` / `recovered` → `detected` with a **new issue row**, not resurrection of the old one.
3. the reason for a projection table instead of a view is that "re-open produces a new row, not an update to the old closed one".

Those three statements cannot all be true simultaneously.

If `fingerprint` is the PK and there is only one row per fingerprint, then a reopen must update that row, not create a second row with the same fingerprint.

### Why this matters
This is not wording trivia. It affects:
- storage schema
- operator note history semantics
- endpoint identity (`/api/runtime/issues/:fingerprint`)
- lifecycle transitions
- whether issue history is represented in `runtime_issues` or only in `runtime_events`

### Required fix
The design must choose one of these models explicitly:

#### Model A — single mutable projection row per fingerprint
- `runtime_issues` has one row per fingerprint
- reopen updates the same row and increments a cycle counter / clears recovered/closed fields
- historical cycles live in `runtime_events` only

#### Model B — multiple issue rows across cycles
- introduce stable `issue_id` PK
- `fingerprint` becomes non-unique
- reopen creates a new `issue_id` row for the same fingerprint
- operator endpoints must key by `issue_id`, not raw fingerprint alone

My recommendation: **Model A for phase-1**.
Reason: phase-1 is backend-governance-first and narrow-but-deep. Model A is simpler, keeps operator contract compact, and avoids over-designing issue genealogy before real usage evidence exists.

## Issue 2 — self-check persistence/read model is under-specified

### Where the gap is
Section 6.1 says health summary is derived from:
- active `runtime_issues`
- recent `runtime_self_checks` results
- last `runtime_health_snapshots` row

But the data model section defines only:
- `runtime_events`
- `runtime_issues`
- `runtime_health_snapshots`

There is no explicit design for where self-check last-run / pass-fail results live.

### Why this matters
Without a concrete self-check persistence/read model, these parts become fuzzy:
- `/api/runtime/self-checks`
- `last_run_at`
- `last_full_pass_at`
- freshness / staleness semantics
- whether self-checks are reconstructed from events or stored as first-class rows

### Required fix
The design must make one of these explicit:

#### Option A — add `runtime_self_checks` table
Store one row per check with latest status and update history via events/snapshots.

#### Option B — derive latest self-check state from `runtime_events`
Then the design must explicitly define:
- which event codes count as self-check pass/fail
- how last-run state is queried efficiently
- why a separate table is intentionally unnecessary

My recommendation: **Option A** for clarity, unless the author can make Option B precise without turning the read path into query soup.
