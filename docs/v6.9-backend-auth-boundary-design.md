## 2. Auth Middleware — Rewritten

### 2.1 v6.8 Auth Middleware (Being Replaced)

Located in `bff/middleware/auth.ts`. Current behavior:
1. Skip `/api/auth/login`, `/api/auth/logout`, non-`/api/*` paths
2. Extract `Authorization: Bearer <token>` header
3. `verifyTenantToken(token)` → JWT verify
4. Overwrite `req.headers.authorization` with raw JSON `{ userId, orgId, role }` for downstream handlers

**Problem**: Only accepts bearer tokens. Cookie auth is only used for SSE via a separate code path in `sse-events.ts`.

### 2.2 v6.9 Auth Middleware — Fixed Design

The rewritten middleware supports two contracts (browser cookie, machine bearer) and routes to the correct one.

**Public routes** (skip auth entirely):

| Route | Method |
|-------|--------|
| `/api/auth/login` | POST |
| `/api/auth/logout` | POST |
| Non-`/api/*` paths | Any (static files, frontend HTML) |

**Auth resolution order** for all other `/api/*` routes:

```
1. Check for __Host-solfacil_session cookie
   → If present: extract JWT from cookie value → verify → Browser Contract

2. Check for Authorization: Bearer <token> header
   → If present: extract token → verify → Machine Contract

3. Neither present → 401 Unauthorized
```

**Post-verification** (same for both contracts):
- Call `verifyTenantToken(token)` → get `{ userId, orgId, role }`
- Overwrite `req.headers.authorization` with raw JSON `{ userId, orgId, role }` (preserves backward compatibility with all 47 downstream handlers — zero handler changes)
- Call `next()`

### 2.3 Contract Discrimination Detail

| Signal | Contract | Token Source |
|--------|----------|-------------|
| Cookie present, no `Authorization` header | Browser | Cookie value |
| `Authorization` header present, no cookie | Machine | Header value (strip `Bearer ` prefix) |
| Both present | Browser (cookie wins) | Cookie value |
| Neither | Reject | 401 |

**Rationale for cookie-wins**: A browser request will always have the cookie attached (same-origin, auto-sent). If a browser-side bug also sets an `Authorization` header, the cookie is the authoritative credential. Machine clients will never have the cookie.

### 2.4 SSE Auth Path

The SSE handler (`sse-events.ts` at `GET /api/events`) already uses cookie auth in v6.8. In v6.9, the middleware handles cookie extraction uniformly for all routes, so the SSE handler no longer needs its own cookie-extraction logic.

| Aspect | v6.8 | v6.9 |
|--------|------|------|
| Cookie extraction | Custom code in `sse-events.ts` | Handled by auth middleware (unified) |
| Cookie name | `solfacil_jwt` _(historical, pre-v6.9)_ | `__Host-solfacil_session` (middleware reads this) |
| Downstream interface | Same `req.headers.authorization` rewrite | Same |

## 3. Session Endpoint — Fixed Contract

### 3.1 Route Definition

| Property | Value |
|----------|-------|
| **Method** | `GET` |
| **Path** | `/api/auth/session` |
| **Auth** | Cookie only |
| **Handler file** | `auth-session.ts` (new file) or added to `auth-login.ts` |
| **Response** | `ApiResponse<SessionData>` |

### 3.2 Behavior

```
1. Auth middleware extracts JWT from __Host-solfacil_session cookie
2. verifyTenantToken(jwt) → { userId, orgId, role }
3. Query users table for name, email (using userId from JWT)
4. Return 200: { success: true, data: { userId, orgId, role, name, email } }
```

On failure (missing cookie, expired JWT, invalid signature):
```
Return 401: { success: false, error: "Session expired or invalid" }
```

### 3.3 Bearer Rejection on This Endpoint

`GET /api/auth/session` is browser-only. If the request has an `Authorization: Bearer` header but no session cookie, it returns 401. This prevents machine clients from using the session endpoint (they have no session concept — they use stateless bearer tokens).

### 3.4 Database Query

The session endpoint queries `users JOIN user_org_roles` to return fresh user metadata:

```sql
SELECT u.user_id, u.email, u.name, uor.org_id, uor.role
FROM users u
JOIN user_org_roles uor ON u.user_id = uor.user_id
WHERE u.user_id = $1
LIMIT 1
```

Pool: **App Pool** with `queryWithOrg(sql, params, orgId)` — RLS-scoped. The `orgId` comes from the JWT claims.

**Rationale**: Querying the DB on each session check ensures the response reflects current user state (role changes, deactivation). The JWT claims are used for auth, but the DB is the source of truth for user metadata.

## 4. Login Handler — Browser/Machine Split

### 4.1 v6.8 Login (Historical — Being Replaced)

> **Historical context only** — the following describes the pre-v6.9 behavior that is being replaced. None of these attributes are part of the v6.9 design.

Pre-v6.9 `POST /api/auth/login` in `auth-login.ts`:
- Validated credentials
- Signed JWT
- Set `solfacil_jwt` cookie (HttpOnly, SameSite=Lax) — **not the v6.9 contract**
- Returned `{ token, user }` in body — **not the v6.9 contract**

**Problem being fixed**: The JWT was returned in the response body and stored in localStorage. The cookie used `SameSite=Lax` instead of `Strict` and a non-`__Host-` prefixed name.

### 4.2 v6.9 Login — Fixed Design

The login handler serves both browser and machine clients but with different response shapes.

**Contract discrimination**: The login endpoint uses a request header or parameter to distinguish browser from machine callers.

| Mechanism | Value | Contract |
|-----------|-------|----------|
| Default (no special header) | — | **Browser contract** |
| `X-Auth-Contract: machine` header | `machine` | **Machine contract** |

**Browser contract response** (default):

```
HTTP/1.1 200 OK
Set-Cookie: __Host-solfacil_session=<jwt>; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=86400

{
  "success": true,
  "data": {
    "user": { "userId": "...", "orgId": "...", "role": "...", "name": "...", "email": "..." }
  }
}
```

No `token` field in body.

**Machine contract response** (when `X-Auth-Contract: machine`):

```
HTTP/1.1 200 OK

{
  "success": true,
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIs...",
    "user": { "userId": "...", "orgId": "...", "role": "...", "name": "...", "email": "..." }
  }
}
```

No `Set-Cookie` header. Token in body for machine consumption.

**Machine contract boundary rules**:
- The machine contract exists only for non-browser callers (CLI tools, service-to-service, scripts). It must not be used by the SPA or any browser-originated code.
- The body-token response shape must not be returned to browser-contract callers. The absence of `X-Auth-Contract: machine` means browser contract; no fallback or convenience path may reintroduce token-in-body for browser clients.
- The exact set of routes eligible for machine bearer auth is a bounded open point for PLAN/implementation, but the default must be conservative: machine bearer is accepted on general `/api/*` data routes and `POST /api/auth/login` (with the machine header), **not** on `GET /api/auth/session` (which is browser-only by design).
- This separation must not be weakened by vague "backward compatibility" language during implementation.

### 4.3 Abuse-Control Integration

The abuse-control middleware runs **before** credential validation:

```
POST /api/auth/login
  │
  ├── abuse-control check (§6) → 429 if rate-limited
  │
  ├── validate email + password → 400/401 on failure
  │     └── on failure: increment abuse-control counters
  │
  ├── sign JWT
  │
  ├── contract discrimination → set cookie OR return token
  │
  └── on success: reset email counter
```

## 5. Cookie Lifecycle

### 5.1 Cookie Set (Login)

```typescript
res.cookie('__Host-solfacil_session', jwt, {
  httpOnly: true,
  secure: true,         // false in NODE_ENV=development
  sameSite: 'strict',
  path: '/',
  maxAge: 86400 * 1000, // 24h in milliseconds
  // no domain (required by __Host- prefix)
});
```

**Dev-mode exception**: When `NODE_ENV=development`, cookie name is `solfacil_session` (no `__Host-` prefix) and `secure: false`. This allows local HTTP development.

### 5.2 Cookie Clear (Logout)

```typescript
res.clearCookie('__Host-solfacil_session', {
  httpOnly: true,
  secure: true,
  sameSite: 'strict',
  path: '/',
});
```

The `POST /api/auth/logout` handler clears the cookie and returns `{ success: true }`.

### 5.3 Cookie Name Constant

The cookie name is defined as a constant shared between login, logout, auth middleware, and session endpoint:

```typescript
const SESSION_COOKIE_NAME = process.env.NODE_ENV === 'development'
  ? 'solfacil_session'
  : '__Host-solfacil_session';
```

## 6. Abuse-Control Middleware

### 6.1 Middleware Placement

Applied only to `POST /api/auth/login`. Runs before the login handler.

### 6.2 Interface to Backing Store

```typescript
interface RateLimitStore {
  /** Increment counter for key. Returns current count after increment. Sets TTL on first increment. */
  increment(key: string, windowSeconds: number): Promise<number>;
  /** Reset counter for key (on successful login). */
  reset(key: string): Promise<void>;
}
```

Two implementations:
- `RedisRateLimitStore` — production. Uses `RATE_LIMIT_REDIS_URL` env var
- `MemoryRateLimitStore` — development only. `Map<string, { count, expiresAt }>`

### 6.3 Middleware Logic

```
Extract IP from req (respecting trust proxy — see deployment design)
Extract email from req.body (normalized: trimmed, lowercased)

Check login_ip:<ip>:
  If count >= 10 → 429 with Retry-After: <remaining seconds in window>

Check login_email:<email>:
  If count >= 5 → 429 with Retry-After: <remaining seconds in window>

Allow request to proceed to login handler

After login handler:
  If login failed (401) → increment both counters (15-minute window)
  If login succeeded (200) → reset email counter; do NOT reset IP counter
```

### 6.4 429 Response

```json
{
  "success": false,
  "error": "Too many login attempts. Try again later."
}
```

Headers: `Retry-After: <seconds>` (seconds remaining in the 15-minute window).

### 6.5 Store Selection at Startup

```
IF NODE_ENV === 'development' AND RATE_LIMIT_REDIS_URL is not set:
  Use MemoryRateLimitStore (acceptable for local single-process dev only)

ELSE IF RATE_LIMIT_REDIS_URL is set:
  Create RedisRateLimitStore
  Test connection
  If connection fails:
    LOG FATAL: "Rate-limit Redis connection failed — cannot start with degraded abuse control"
    process.exit(1)

ELSE (non-dev, no RATE_LIMIT_REDIS_URL):
  LOG FATAL: "RATE_LIMIT_REDIS_URL is required in production"
  process.exit(1)
```

**Rationale**: Login abuse control is release-critical. Production must not start without the configured shared backing store. In-memory fallback is acceptable only for `NODE_ENV=development` single-process local runs.

## 7. Secret Validation — `tenant-context.ts` Changes

### 7.1 v6.8 State

```typescript
const jwtSecret = process.env.JWT_SECRET || 'solfacil-dev-secret';
```

Used in `verifyTenantToken()` for `jwt.verify(token, jwtSecret)`.

### 7.2 v6.9 Design

The hardcoded dev-secret fallback is removed from the production-capable code path. Secret loading is split into startup validation and runtime use.

**Startup** (in `local-server.ts`, before routes are registered):

```
const jwtSecret = loadAndValidateJwtSecret();

function loadAndValidateJwtSecret(): string {
  const secret = process.env.JWT_SECRET;
  if (process.env.NODE_ENV !== 'development') {
    if (!secret || secret === 'solfacil-dev-secret') {
      console.error('FATAL: JWT_SECRET must be set to a strong secret in production');
      process.exit(1);
    }
    return secret;
  }
  // Development mode — explicit local secret is the preferred path
  if (!secret) {
    // LEGACY TRANSITIONAL: this fallback exists only to avoid blocking
    // local onboarding. Developers should provision JWT_SECRET via .env
    // or shell export. This fallback is debt to eliminate in a future release.
    console.warn('WARNING: JWT_SECRET not set — using legacy dev fallback. ' +
                 'Provision an explicit JWT_SECRET in .env for local development.');
    return 'solfacil-dev-secret';
  }
  return secret;
}
```

**Design intent**: The `solfacil-dev-secret` literal is legacy debt, not an accepted design path. It remains only as a transitional convenience for local development and must not be referenced as a standard mechanism in PLAN or implementation.

**Runtime** (`verifyTenantToken`):

```typescript
// jwtSecret is now passed in or set at module load from validated startup
// NO fallback. If startup validation passed, jwtSecret is guaranteed to be set.
jwt.verify(token, jwtSecret);
```

### 7.3 Passing the Secret

The validated `jwtSecret` is passed to `verifyTenantToken` (or set as a module-level variable in `tenant-context.ts` via an initialization function). The exact wiring mechanism is flexible, but the requirement is fixed: **`verifyTenantToken` must never fall back to a hardcoded secret at runtime**.

### 8.2 Changed Routes

| Method | Route | Change |
|--------|-------|--------|

v6.8: 50 routes / 47 handler files
v6.9: 51 routes / 47–48 handler files (+1 route for `GET /api/auth/session`, +0–1 handler file depending on whether session is added to `auth-login.ts` or a new file)

## 11. Bounded Open Points

| Open Point | Constraint |
|------------|------------|
| Whether `GET /api/auth/session` is a new handler file or added to `auth-login.ts` | Either is acceptable. New file is cleaner but minor |
| Whether `jwtSecret` is passed to `verifyTenantToken` as a parameter or set via module-level init | Either is acceptable. Parameter passing is more explicit; module-level init is simpler. The requirement is that no fallback exists at runtime |
| Redis client library for rate limiting | Must support TTL key expiry. `ioredis` is expected |
| Whether `X-Auth-Contract: machine` header or a query parameter distinguishes machine logins | Header is preferred (cleaner). Query parameter is acceptable if header causes proxy issues |
| Whether to add a `POST /api/auth/machine-login` as a separate endpoint instead of header discrimination | Separate endpoint is an acceptable alternative. The requirement is that browser and machine responses are distinct |

## 12. Legacy Browser Bearer Bridge

If `LEGACY_BROWSER_BEARER=true` is set (default: `false`), the auth middleware additionally accepts `Authorization: Bearer` tokens even when no cookie is present, for browser-originated requests. This is an emergency rollback aid.

| Aspect | Value |
|--------|-------|
| Default | `false` (off) |
| When enabled | Auth middleware falls through to bearer check even for browser requests |
| Lifetime | Must be disabled within 48 hours |
| Logging | Every request authenticated via legacy bearer logs a warning: `"Legacy browser bearer auth used — migrate to cookie"` |

This bridge does not affect the machine contract, which always uses bearer tokens.
